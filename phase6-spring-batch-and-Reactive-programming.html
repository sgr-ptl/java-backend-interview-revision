Spring Batch
Batch Processing Essentials
Processing millions of records reliably — with restart, retry, skip, and audit built-in
🟡 Medium
📌 Concepts

🧠 Memory
🧠
In plain English: Spring Batch is for processing huge volumes of data reliably — importing 5 million CSV records, generating monthly invoices, migrating database records nightly. It adds restart-from-failure, retry, skip-bad-records, and audit trail — so a crash at record 4,823,001 doesn't mean starting from scratch.

🏗️
Job — The top-level batch unit composed of one or more ordered Steps. Launched via JobLauncher. Each run creates a unique JobExecution. The same job can run multiple times with different JobParameters — the parameters define uniqueness of each logical run.
🪜
Step — A single phase of a Job. Two types: Tasklet (executes once — file cleanup, table truncation, single notifications) and Chunk-oriented (read→process→write in batches — for bulk record processing). Steps can branch conditionally based on the exit status of the previous step.
💾
JobRepository — Persists all execution state (JobExecution, StepExecution, ExecutionContext) to a database. Enables: restart from failure, monitoring, preventing duplicate runs. Spring Batch provides DDL scripts for all major databases. Required for production use.
🔄
ExecutionContext — A key-value store persisted per job/step execution. Stores checkpoints: "last processed record ID = 4823000". On restart, batch resumes from there. Job-scoped context shared across steps; Step-scoped context is private to one step.
↩️
Skip & Retry — Skip: configure which exception types can be silently skipped up to a max count (bad records go to SkipListener). Retry: automatically retry on transient errors (DB deadlock, network timeout) with configurable attempts and backoff. Always set skipLimit — without a cap, all records could silently disappear.

Spring Batch — Job + Tasklet + Chunk + Skip/Retry
// ── Spring Batch Job Configuration ───────────────
@Configuration
public class InvoiceJobConfig {

    @Bean
    public Job invoiceJob(JobRepository repo,
                          Step cleanupStep, Step generateStep) {
        return new JobBuilder("invoiceJob", repo)
            .start(cleanupStep)
            .next(generateStep)
            .build();
    }

    // ── Tasklet Step (single operation) ──────────────
    @Bean
    public Step cleanupStep(JobRepository repo, PlatformTransactionManager tx) {
        return new StepBuilder("cleanup", repo)
            .tasklet((contribution, ctx) -> {
                deleteOldTempFiles();
                return RepeatStatus.FINISHED;
            }, tx)
            .build();
    }

    // ── Chunk Step with Skip & Retry ──────────────────
    @Bean
    public Step generateStep(JobRepository repo,
                              PlatformTransactionManager tx,
                              ItemReader<Order> reader,
                              ItemProcessor<Order, Invoice> processor,
                              ItemWriter<Invoice> writer) {
        return new StepBuilder("generate", repo)
            .<Order, Invoice>chunk(100, tx)   // 100 records per TX
            .reader(reader)
            .processor(processor)
            .writer(writer)
            .faultTolerant()
            .skip(DataIntegrityViolationException.class)
            .skipLimit(50)               // skip up to 50 bad records
            .retry(TransientDataAccessException.class)
            .retryLimit(3)              // retry transient errors 3x
            .listener(skipListener)
            .build();
    }
}

// ── Launch Job with parameters ────────────────────
@Service
public class BatchService {
    @Autowired private JobLauncher launcher;
    @Autowired private Job invoiceJob;

    public void run(YearMonth month) throws Exception {
        JobParameters params = new JobParametersBuilder()
            .addString("month", month.toString())         // identifying
            .addLong("runAt", System.currentTimeMillis(),
                      false)                             // non-identifying
            .toJobParameters();
        launcher.run(invoiceJob, params);
    }
}
◆ Interview Questions
Must Know
What is the difference between a Tasklet step and a Chunk-oriented step?
▾
Tasklet Step: runs a single operation once. The Tasklet interface has one method: execute() returning RepeatStatus.FINISHED or CONTINUABLE. Use for: file cleanup, table truncation, single API calls, sending one notification. Simple — no reader/writer needed.

Chunk-Oriented Step: reads N items → processes each → writes the batch → commits → repeats until reader returns null. The chunk size controls transaction boundaries. Use for: bulk record processing where read-process-write makes sense. Provides retry, skip, and restart semantics per item.
⚡ Rule: processing records one-by-one in bulk → Chunk. A single action (cleanup, notification) → Tasklet.
Must Know
How does Spring Batch support restart after failure?
▾
Spring Batch persists all execution state to the JobRepository database tables. When a job fails at step 3, restarting with the same JobParameters causes Spring Batch to skip steps 1 and 2 (status = COMPLETED in DB) and resume from step 3. Within a chunk step, the ExecutionContext stores the last committed chunk position — restart begins from the next unprocessed chunk, not record 1.
⚡ Add a timestamp to JobParameters to force a fresh run instead of restarting a failed one. Same params = same logical run = restart.
Tricky
What is the difference between skip and retry?
▾
Retry: for transient failures — network timeout, DB deadlock. Spring retries the same chunk N times with optional backoff. Assumes: it might succeed on the next attempt.

Skip: for permanent bad records — malformed data, constraint violations. The item is marked as skipped (logged via SkipListener) and processing continues with the next record. Item is never processed again.

Use both: retry first (transient), then skip if retry limit exceeded (permanent). Always set skipLimit — without a cap, all records could silently skip.
Tricky
What is the purpose of JobParameters and when would you use non-identifying parameters?
▾
Identifying parameters (default): included in the JobInstance key. Same identifying params = same logical run = Spring Batch tries to restart it. Use for: the business key of what's being processed (report month, file path, date range).

Non-identifying parameters (addLong("runAt", timestamp, false)): passed to the job but excluded from the identity calculation. Use for: run timestamp, user who triggered it, debug flags — metadata that shouldn't affect whether it's a new or restarted run.
⚡ To always create a fresh run (not restart): add a unique non-identifying timestamp as a non-identifying parameter or add a unique run ID as identifying.
Senior
How would you implement parallel processing in Spring Batch?
▾
Three approaches:
1. Multi-threaded Step: add a TaskExecutor to a chunk step — multiple threads process chunks from the same reader simultaneously. Simple but the reader must be thread-safe (use SynchronizedItemStreamReader wrapper).

2. Parallel Steps: use split() in the Job builder to run independent steps concurrently — e.g., process users and orders simultaneously if they don't depend on each other.

3. Partitioning: most powerful — a Partitioner splits work into N partitions (e.g., user IDs 1-10000, 10001-20000 etc.). A PartitionHandler runs each partition as a separate step execution, possibly on different threads or even different machines. Best for very large datasets.
◆ Memory Tip
💡
Spring Batch = assembly line with memory. If the line breaks at station 47, you don't restart from station 1 — you restart from station 48. That's JobRepository + ExecutionContext.

Step selector: Bulk records → Chunk. Single action → Tasklet.
Error selector: Transient (network, deadlock) → Retry. Bad data record → Skip.
Run uniqueness: Same JobParameters = restart. Different params = new run.

🧠 Memory
🧠
In plain English: Instead of reading 10 million records into memory at once, chunk processing reads 100 at a time, processes those 100, writes them, commits the transaction, then reads the next 100. Each chunk = one DB transaction. If a chunk write fails, only 100 records need reprocessing — not 10 million.

📖
ItemReader<T> — Reads one item at a time, returns null to signal end. Built-in: FlatFileItemReader (CSV, delimited), JdbcCursorItemReader (DB cursor — streams efficiently), JpaPagingItemReader (JPA pages — thread-safe), JsonItemReader, StaxEventItemReader (XML). Must be thread-safe for multi-threaded steps.
⚙️
ItemProcessor<I, O> — Transforms one input to one output. Return null to filter the item (silently dropped, not counted as skip). Can apply business rules, validation, enrichment. Chainable with CompositeItemProcessor. Optional — omit to pass input directly to writer.
✍️
ItemWriter<T> — Receives a Chunk<T> (list) and writes all at once. Built-in: JdbcBatchItemWriter (JDBC batch INSERT/UPDATE — much faster than individual inserts), JpaItemWriter, FlatFileItemWriter, CompositeItemWriter (write to multiple outputs).
📏
Chunk Size Trade-off — Small (10): frequent commits, more DB overhead, smaller failure scope. Large (10,000): fewer commits, higher throughput, more memory, larger failure scope. Rule of thumb: 50–500 for DB operations, 1000–5000 for file operations. Profile and tune.
🔧
@StepScope — Creates a new bean instance per step execution. Enables late binding: @Value("#{jobParameters['filePath']}") — evaluated at step runtime, not application startup. Critical for partitioned jobs where each worker gets different parameters. Without @StepScope, all workers share one bean instance (state conflicts).
⚠️
JdbcCursor vs JpaPaging Reader — JdbcCursorItemReader: streams via a single open DB cursor. Efficient for sequential single-threaded reads. Not thread-safe. JpaPagingItemReader: paginated JPA queries. Thread-safe. More memory per page. Better for multi-threaded steps but can be slow at high page offsets.

Chunk Processing — Reader + Processor + Writer + @StepScope
// ── FlatFileItemReader (CSV) ──────────────────────
@Bean
@StepScope
public FlatFileItemReader<OrderRow> csvReader(
        @Value("#{jobParameters['filePath']}") String path) {
    return new FlatFileItemReaderBuilder<OrderRow>()
        .name("orderReader")
        .resource(new FileSystemResource(path))
        .linesToSkip(1)                     // skip header
        .delimited().delimiter(",")
        .names("id", "customerId", "amount")
        .targetType(OrderRow.class)
        .build();
}

// ── JdbcCursorItemReader (streams from DB) ────────
@Bean
public JdbcCursorItemReader<Order> dbReader(DataSource ds) {
    return new JdbcCursorItemReaderBuilder<Order>()
        .name("orderDbReader")
        .dataSource(ds)
        .sql("SELECT * FROM orders WHERE status='PENDING' ORDER BY id")
        .rowMapper(new BeanPropertyRowMapper<>(Order.class))
        .fetchSize(1000)
        .build();
}

// ── ItemProcessor: transform + filter null ────────
@Component
public class InvoiceProcessor implements ItemProcessor<Order, Invoice> {
    @Override
    public Invoice process(Order order) {
        if (order.getAmount().signum() <= 0)
            return null;           // null = filter out (not a skip!)
        return Invoice.builder()
            .orderId(order.getId())
            .amount(order.getAmount().multiply(new BigDecimal("1.2")))
            .build();
    }
}

// ── JdbcBatchItemWriter ───────────────────────────
@Bean
public JdbcBatchItemWriter<Invoice> invoiceWriter(DataSource ds) {
    return new JdbcBatchItemWriterBuilder<Invoice>()
        .dataSource(ds)
        .sql("INSERT INTO invoices (order_id, amount) VALUES (:orderId, :amount)")
        .beanMapped()     // maps bean fields to :named params
        .build();          // uses JDBC batch — far faster than N inserts
}
◆ Interview Questions
Must Know
What does returning null from ItemProcessor.process() do? How is it different from a skip?
▾
Returning null from process() tells Spring Batch to filter out that item — it won't be passed to the writer. It's silently dropped, NOT counted as a skip, and no exception is required. Use for intentional business filtering: "only generate invoices for orders above $10." The StepExecution.filterCount tracks how many items were filtered.

Skip is for unexpected errors — an exception is thrown/caught, and the item is logged as a problem. Filter (null return) is for expected business logic exclusion.
⚡ Null return = intentional drop. Skip = error handling. Never throw an exception just to skip — return null instead for business filtering.
Tricky
Why is @StepScope important? What goes wrong without it?
▾
Without @StepScope, the reader bean is a singleton — created once at application startup. Problems:

1. Late binding failure: @Value("#{jobParameters['filePath']}") is evaluated at startup before any job runs — throws a SpEL error because no step execution context exists yet.

2. State conflicts in parallel jobs: Readers are stateful (track current position). Without @StepScope, all worker threads in a partitioned job share one reader instance — data duplication or skipped items.

With @StepScope, a fresh instance is created per step execution with the correct parameter values injected at runtime.
Tricky
When would you use JdbcCursorItemReader vs JpaPagingItemReader?
▾
JdbcCursorItemReader: opens a single DB cursor, streams results one-by-one. Single open connection held for the step duration. Very efficient for large sequential reads. Thread-UNSAFE — one thread only.

JpaPagingItemReader: executes paginated JPA queries (LIMIT/OFFSET). Opens/closes EntityManager per page. Thread-SAFE — can be used in multi-threaded steps. More overhead per page. Can be slow at very high page numbers (OFFSET 100000 is expensive on large tables).

Rule: Single-threaded large scans → JdbcCursor. Multi-threaded → JpaPaging. Need JPA entity features (lazy loading etc.) → JpaPaging.
Senior
What is the optimal chunk size strategy? How do you choose it?
▾
There is no single right answer — it depends on record size, processing complexity, and DB performance. Starting points:
• DB operations (inserts/updates): 50–500 records per chunk
• File operations: 1000–5000 records per chunk
• Very fast writes (simple inserts): up to 10,000

Trade-offs: Smaller chunk = more TX overhead but smaller failure blast radius. Larger chunk = higher throughput but more memory and more reprocessing on failure.

Method: start with 100, measure throughput (records/sec), double and measure again, stop when throughput plateaus or memory becomes a constraint. Monitor: DB commit rate, heap usage, step execution time in Batch metadata.
💡
Chunk = one database transaction.
Read N → Process N → Write N → COMMIT. Crash = reprocess only last uncommitted chunk.

The three operators:
• Reader: returns one item or null (done)
• Processor: returns transformed item or null (filter out)
• Writer: receives the whole chunk as a List — write in one batch

@StepScope = fresh instance per step run. Required for any bean using jobParameters or used in parallel steps.

Job Parameters & Scheduling
Controlling when and how batch jobs run — cron, parameters, and preventing duplicate runs
🟢 Essential
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Job Parameters are like function arguments for your batch job — they make each run configurable and uniquely identifiable. Scheduling is how you trigger jobs automatically (every night at 2am). @Scheduled handles simple cases; Quartz or Kubernetes CronJobs handle production clustered deployments.
🎛️
JobParameters — Key-value pairs passed when launching a job. Types: String, Long, Double, Date, LocalDate, LocalDateTime. Identifying (default): included in JobInstance key — same identifying params = same logical run = restart attempt. Non-identifying: passed to job but excluded from identity check (e.g., run timestamp, triggering user).
⏰
@Scheduled — Simple scheduling. cron = "0 0 2 * * ?" (2am daily). fixedRate = 5000 (every 5s from START of last run). fixedDelay = 5000 (5s after COMPLETION). initialDelay (wait before first run). Requires @EnableScheduling. Single-threaded by default — configure a TaskExecutor bean for concurrent scheduling.
📅
Cron Expression — Spring cron: 6 fields: second minute hour day-of-month month day-of-week. Examples: 0 0 2 * * ? (2am daily), 0 */15 * * * * (every 15 minutes), 0 0 9-17 * * MON-FRI (every hour 9-5 weekdays). ? = don't care. * = every value.
🏢
Quartz vs @Scheduled — @Scheduled: simple, in-process, no persistence, works on one node. Quartz: persistent (schedule survives restarts), clustered (one node fires at a time), rich API. Use Quartz when: multiple app instances, schedule must survive restarts, complex trigger logic.
🚫
Preventing Duplicate Cluster Runs — All app instances execute @Scheduled → job runs N times. Solutions: ShedLock (acquires DB/Redis distributed lock before running), Quartz clustered (built-in leader election), Kubernetes CronJob (one pod per run — cleanest for containers), Spring Integration LockRegistry.

◆ Interview Questions
Must Know
What is the difference between fixedRate and fixedDelay in @Scheduled?
▾
fixedRate: triggers every N ms measured from the start of the previous invocation. If task takes longer than N, executions can queue up (overlap if using a thread pool). Example: fixedRate=5000 fires at T+0, T+5s, T+10s regardless of execution time.

fixedDelay: waits N ms after completion of the previous invocation. Guarantees no overlap. Example: fixedDelay=5000, task runs 3s → next starts at T+8s (3s task + 5s delay).
⚡ Use fixedDelay when tasks must not overlap. Use fixedRate for periodic polling where small overlaps are acceptable.
Tricky
Why do you get duplicate batch runs in a clustered environment? How do you prevent it?
▾
When multiple app instances run, all instances execute @Scheduled simultaneously — the job runs N times in parallel. Spring Batch's JobRepository helps (same identifying JobParameters = already-running/completed detection) but doesn't prevent multiple launch attempts at the exact same moment.

Solutions: ShedLock (acquires a DB or Redis lock before executing — simplest), Quartz clustered mode (persistent scheduler with built-in leader election for trigger firing), Kubernetes CronJob (K8s runs exactly one pod per scheduled time — cleanest for containerized apps), Leader election (Spring Cloud Zookeeper/Consul).
Tricky
How do you trigger a batch job via REST API for on-demand execution?
▾
Inject JobLauncher and Job into a @RestController. Build JobParameters from request params. Call jobLauncher.run(job, params).

Async launcher is critical for HTTP: default JobLauncher is synchronous — it blocks the HTTP thread until the job completes (minutes or hours). Configure an async launcher: set setTaskExecutor(new SimpleAsyncTaskExecutor()) on the launcher bean — returns immediately with a JobExecution object (status: STARTING). The client receives the jobExecutionId and can poll for status later via JobExplorer.getJobExecution(id).
💡 Always use async JobLauncher for HTTP-triggered jobs. Sync launcher = blocked HTTP thread = timeouts.
💡
Cron mnemonic: "Seconds Minutes Hours Day Month Weekday"
0 0 2 * * ? = second 0, minute 0, hour 2, any day, any month = 2am daily

fixedRate vs fixedDelay:
Rate = "fire every 5s from START" → may overlap
Delay = "wait 5s after FINISH" → never overlaps

Cluster safety rule: @Scheduled in multi-instance = multiple runs. Use ShedLock (simplest) or K8s CronJob (cleanest) for guaranteed single execution.

Project Reactor
Reactive Programming Concepts
Non-blocking, backpressure-aware streams — the paradigm that enables massive concurrency
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Traditional code blocks a thread while waiting for DB/HTTP responses (sleeping, wasting CPU). Reactive code registers a callback and moves on — one thread handles thousands of concurrent requests by never sleeping. Like a waiter taking 50 orders vs standing at the kitchen waiting for each one.
🌊
Reactive Streams Spec — 4 interfaces: Publisher (produces data), Subscriber (consumes data), Subscription (connection between them, handles backpressure), Processor (both). Project Reactor implements this spec with Mono and Flux.
⚡
Non-Blocking I/O — Traditional: thread blocks while waiting for DB/HTTP response (sleeping, wasting CPU). Reactive: thread registers a callback, continues doing other work, processes result when ready. One event loop thread handles thousands of concurrent I/O operations. Requires non-blocking drivers: R2DBC (DB), WebClient (HTTP), reactive MongoDB.
🔄
Backpressure — Flow control mechanism. Consumer signals how much data it can handle via request(n). Publisher sends at most N items. Consumer requests more when ready. Prevents fast producer from overwhelming slow consumer (buffer overflow/OOM). Key advantage over simple callbacks (CompletableFuture has NO backpressure).
🧵
Schedulers — Control which thread executes pipeline stages. Schedulers.boundedElastic(): for blocking calls (DB, file I/O) — elastic thread pool. Schedulers.parallel(): for CPU work — fixed pool = core count. Schedulers.single(): single reusable thread. Switch with subscribeOn() (source thread) or publishOn() (downstream thread).
🚫
Never Block in Reactive — Calling block(), Thread.sleep(), or any blocking operation on the event loop thread starves all concurrent requests. Use Mono.fromCallable(() → blockingCall()).subscribeOn(Schedulers.boundedElastic()) to offload. Or use virtual threads (Java 21 + Tomcat) if your codebase is mostly blocking.
⚠️
Nothing Executes Until Subscribed — Reactive pipelines are lazy — they're just a description of work. A Mono/Flux runs only when subscribed to. In WebFlux, the framework subscribes automatically. In tests or manual code — don't forget to subscribe or call .block().
◆ Memorizable Code
Reactive Concepts — Schedulers + Safe Blocking + Error Handling
// ── subscribeOn: affects the SOURCE thread ────────
Mono<String> result = Mono.fromCallable(() -> readFileBlocking()) // blocking!
    .subscribeOn(Schedulers.boundedElastic()) // run on elastic pool
    .map(String::toUpperCase)                 // still on elastic
    .publishOn(Schedulers.parallel())          // switch to parallel pool
    .map(s -> s + " processed");              // runs on parallel thread

// ── Wrapping a blocking legacy call safely ────────
Mono<User> safeCall(Long id) {
    return Mono.fromCallable(() -> legacyRepo.findUser(id))  // blocking JPA
               .subscribeOn(Schedulers.boundedElastic());
}

// ── Error handling strategies ─────────────────────
Mono<User> user = fetchUser(id)
    .onErrorReturn(User.anonymous())           // default value on error
    .onErrorResume(e -> fetchFromCache(id))      // fallback Mono
    .onErrorMap(Exception.class,                 // wrap as different type
        e -> new ServiceException(id, e))
    .retryWhen(Retry.backoff(3, Duration.ofMillis(200))); // exp backoff

// ── Backpressure: subscriber controls rate ────────
Flux.range(1, 1_000_000)
    .onBackpressureBuffer(1000)   // buffer max 1000 items
    .subscribe(new BaseSubscriber<Integer>() {
        @Override
        protected void hookOnSubscribe(Subscription s) {
            request(10);               // request 10 items initially
        }
        @Override
        protected void hookOnNext(Integer v) {
            process(v);
            request(10);               // request 10 more after processing
        }
    });
◆ Interview Questions
Must Know
What is backpressure and why does it matter in reactive programming?
▾
Backpressure is a flow control mechanism where the consumer tells the producer how much data it can handle. Without it: a fast producer (reading file at 1M records/sec) overwhelms a slow consumer (writing to DB at 100/sec) — causing buffer overflow or OOM. Reactive Streams' Subscription.request(n) protocol: subscriber requests N items, publisher sends at most N, subscriber requests more when ready — demand-driven cooperative flow.

This is the key advantage of Reactive Streams over plain callbacks (CompletableFuture, RxJava1) — no flow control means no backpressure protection.
⚡ "Backpressure = the consumer holds the remote control." Publisher can only send as fast as the consumer says it can handle.
Tricky
What is the difference between subscribeOn() and publishOn()?
▾
subscribeOn(scheduler): affects the thread where the subscription happens and data is produced (the source). No matter where in the chain you put it, it affects the source. Only the first subscribeOn matters. Use for: blocking producers (file reads, legacy blocking code).

publishOn(scheduler): switches the thread for all downstream operations from that point. Multiple publishOn calls progressively switch threads. Use for: switching from a blocking pool back to a reactive pool after a blocking call, or moving CPU-intensive work to the parallel scheduler.
Tricky
When should you use WebFlux vs Tomcat? Is reactive always better?
▾
WebFlux is better when: high concurrency with I/O-heavy work (many downstream API calls), streaming data (SSE, WebSocket), all dependencies support reactive (R2DBC, reactive MongoDB, WebClient).

Stick with Tomcat when: team unfamiliar with reactive (steep learning curve, difficult debugging), any blocking dependency exists (JPA/JDBC in WebFlux blocks the event loop), Java 21 virtual threads available (Tomcat + virtual threads gives near-reactive throughput with synchronous code simplicity).
⚠️ Using JPA in WebFlux = blocking the event loop = worse than Tomcat. Either use R2DBC or switch to Tomcat + virtual threads.
Senior
What does "nothing executes until subscribed" mean in practice?
▾
A Mono/Flux is a lazy description of a computation — no code runs when you build the pipeline. The actual execution (HTTP call, DB query, file read) only happens when a subscriber subscribes. In Spring WebFlux, the framework subscribes when writing the HTTP response — so your controller code is never actually executed until WebFlux subscribes to it.

Practical implications: (1) Building a Mono in a test without subscribing = nothing happened. (2) A Mono returned from a method is not "in progress" — no execution until subscribed. (3) mono.cache() re-triggers execution on first subscriber then caches — without subscription, even cache doesn't help. (4) Side effects should be in doOnNext()/doOnSubscribe() — not outside the pipeline.
💡
Reactive = never block, never waste a thread. The event loop thread is precious — one blocked call stops all concurrent work on that thread.

subscribeOn = "where to START the pipeline" (affects source).
publishOn = "where to CONTINUE from here" (switches downstream).

Reactive decision tree:
• All deps are reactive + high concurrency → WebFlux
• Have JPA/JDBC + Java 21 → Tomcat + virtual threads
• Mixed blocking + reactive → wrap blocking in boundedElastic

In plain English: Mono<T> is like Optional or a single async result — 0 or 1 item. Flux<T> is like a List or stream — 0 to N items. Both are lazy, composable with operators, and propagate errors through the pipeline.

Key Mono Operators
map(fn): sync transform (T→R). flatMap(fn→Mono): async transform (chains another async call). filter(pred): empty if false. switchIfEmpty(Mono): fallback on empty. defaultIfEmpty(val): default value. zipWith(other): combine two Monos. cache(): cache result for multiple subscribers.
🌊
Key Flux Operators
map/filter/flatMap: same semantics as Mono but per item. take(n): first N. skip(n): skip first N. collectList(): Mono<List<T>>. groupBy(fn): group items. buffer(n): batch N items into List. merge/concat/zip: combine multiple Fluxes.
⚠️
flatMap vs concatMap — flatMap: subscribes to inner publishers eagerly (parallel) — results may arrive out of order, better throughput. concatMap: subscribes sequentially — preserves order, one at a time. flatMapSequential: subscribes eagerly but merges results in order. Use flatMap when order doesn't matter (parallel calls), concatMap when order matters.
🛠️
Creating Mono/Flux
Mono.just(val), Mono.empty(), Mono.error(ex), Mono.fromCallable(()→val) (blocking wrap). Flux.just(a,b,c), Flux.fromIterable(list), Flux.range(1,10), Flux.interval(Duration.ofSeconds(1)) (periodic), Flux.create(sink→...) (bridge from callbacks).

// ── Mono operators ────────────────────────────────
Mono<User> user = findUser(id)
    .map(u -> u.withName(u.getName().toUpperCase()))  // sync
    .flatMap(u -> enrichWithOrders(u))                  // async (→ Mono)
    .filter(User::isActive)                             // empty if false
    .switchIfEmpty(Mono.error(new NotFoundException(id)))
    .doOnNext(u -> log.info("User: {}", u))              // side effect only
    .cache();  // cache result — replay to multiple subscribers

// ── Combining two Monos (parallel) ───────────────
Mono<User> u = fetchUser(id);
Mono<List<Order>> o = fetchOrders(id);

Mono.zip(u, o)                          // runs both in parallel!
    .map(t -> new Dashboard(t.getT1(), t.getT2()));

// ── Flux operators ────────────────────────────────
Flux<String> words = Flux.just("hello", "reactive", "world");
words.map(String::toUpperCase)
     .filter(s -> s.length() > 5)
     .take(2)
     .collectList()           // → Mono<List<String>>
     .subscribe(list -> log.info("{}", list));

// ── flatMap vs concatMap ──────────────────────────
Flux.just(1, 2, 3)
    .flatMap(id -> fetchUser(id))    // parallel — order NOT guaranteed
    .subscribe(System.out::println);

Flux.just(1, 2, 3)
    .concatMap(id -> fetchUser(id))  // sequential — order preserved
    .subscribe(System.out::println);

// ── Combining multiple Fluxes ─────────────────────
Flux<Integer> a = Flux.just(1,2,3);
Flux<Integer> b = Flux.just(10,20,30);
Flux.merge(a, b);      // interleave — fast, unordered
Flux.concat(a, b);     // a completes first, then b — ordered
Flux.zip(a, b, (x,y) -> x+y);  // pair elements: 11, 22, 33

// ── Bridging from callback API ────────────────────
Flux<Event> events = Flux.create(sink -> {
    bus.onEvent(e -> sink.next(e));
    bus.onClose(() -> sink.complete());
    bus.onError(ex -> sink.error(ex));
});
◆ Interview Questions
Must Know
What is the difference between map() and flatMap() in Reactor?
▾
map(T → R): synchronous transformation. Returns a plain value which gets wrapped back in Mono/Flux. Use when the transform is a direct calculation (no async call). Example: mono.map(u → u.getName()) → Mono<String>.

flatMap(T → Mono<R>): asynchronous transformation. The function returns another Publisher. flatMap subscribes to the inner Publisher and flattens the result. Use when the transform needs another async call. Example: mono.flatMap(u → fetchOrders(u.getId())) → Mono<List<Order>>.
⚡ Rule: transform → map. Another async call → flatMap. Wrapping in Mono.just() inside flatMap is always a sign you should use map instead.
Tricky
What is the difference between Flux.merge(), concat(), and zip()?
▾
merge(a, b): subscribes to both simultaneously, emits items as they arrive — order not guaranteed. Fastest throughput. Use when order doesn't matter.

concat(a, b): subscribes to a first, waits for completion, then subscribes to b. Preserves source order. Use when results must be ordered.

zip(a, b): pairs elements by position (first from a + first from b → tuple). Stops when any source completes. Use for: combining related parallel data (price + quantity → total).
💡 merge = fast, unordered. concat = ordered, sequential. zip = paired, like a zipper.
Tricky
What does Mono.cache() do and when is it useful?
▾
By default, each subscriber to a Mono triggers a fresh execution — two subscribers = two DB queries, two HTTP calls. cache() executes the Mono on first subscription, stores the result, and replays it to subsequent subscribers immediately.

Use for: frequently accessed slow-to-compute data that doesn't change often (config lookups, reference data, feature flags). Optional TTL: cache(Duration.ofMinutes(5)). Without cache: a service that's called by 100 concurrent users would trigger 100 DB queries for the same config data.
Senior
How do you handle errors differently with onErrorReturn, onErrorResume, and onErrorMap?
▾
onErrorReturn(T): when any error occurs, emit a fallback value and complete normally. Terminal — stops error propagation. Use for: "if DB fails, return an empty/default response".

onErrorResume(fn → Publisher): when error occurs, switch to a different Publisher (Mono/Flux). More flexible — can return cached data, trigger a fallback service call. Use for: "if primary DB fails, try read replica".

onErrorMap(fn → Throwable): transform one error type into another. Doesn't recover — error still propagates but with a different type. Use for: wrapping low-level infrastructure exceptions in meaningful business exceptions before they reach the caller.
💡
Mono = one async result. Flux = many async results.

Operator cheat sheet:
• Sync transform → map()
• Async call → flatMap()
• On empty → switchIfEmpty() or defaultIfEmpty()
• Side effect → doOnNext()
• Combine two Monos parallel → Mono.zip()
• All items into list → flux.collectList()
• Parallel Flux merge → Flux.merge()
• Ordered Flux → Flux.concat()

🧠 Memory
🧠
In plain English: Spring WebFlux looks almost identical to Spring MVC but your controller returns Mono<T> instead of T. The server is Netty (event loop) instead of Tomcat (thread pool). WebClient replaces RestTemplate for making non-blocking HTTP calls to other services.
🎮
@RestController in WebFlux — Nearly identical to MVC! Return Mono<T> for single responses, Flux<T> for streaming. Same annotations: @GetMapping, @PostMapping, @RequestBody, etc. Key difference: the embedded server is Netty (non-blocking event loop) instead of Tomcat (thread pool). Add spring-boot-starter-webflux instead of starter-web.
🌐
WebClient — Replaces RestTemplate for non-blocking HTTP calls. Builder pattern: WebClient.builder().baseUrl("http://service").build(). Supports: timeouts, retry, filters (auth headers), streaming responses. RestTemplate is deprecated in Spring 5 — use WebClient for all new code (works in both reactive and non-reactive apps).
📡
Server-Sent Events (SSE) — Stream data to the browser over a single HTTP connection. Return Flux<ServerSentEvent<T>> with content type text/event-stream. Browser uses EventSource API. Perfect for: live dashboards, real-time notifications, progress updates — without WebSocket complexity.
🗄️
R2DBC — Reactive DB — JPA/Hibernate use blocking JDBC — incompatible with WebFlux event loop. Use R2DBC (Reactive Relational Database Connectivity) with Spring Data R2DBC repositories that return Mono/Flux. Supports: PostgreSQL, MySQL, H2, MSSQL. Simpler than JPA (no lazy loading, no EntityManager) but fully non-blocking.
🔀
Functional Routing — Alternative to annotations. RouterFunctions.route(GET("/users"), handler::list). More explicit, purely functional. Can coexist with annotation-based controllers. Preferred in some teams for its clarity and testability without mocking annotations.
◆ Memorizable Code
WebFlux Controller + WebClient + SSE Streaming
// ── WebFlux @RestController ───────────────────────
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping("/{id}")
    public Mono<ResponseEntity<UserDto>> get(@PathVariable Long id) {
        return userService.findById(id)
            .map(ResponseEntity::ok)
            .defaultIfEmpty(ResponseEntity.notFound().build());
    }

    @GetMapping
    public Flux<UserDto> list() {
        return userService.findAll();  // items stream as they arrive
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public Mono<UserDto> create(@Valid @RequestBody Mono<CreateReq> req) {
        return req.flatMap(userService::create);
    }

    // ── SSE: stream events to browser ────────────────
    @GetMapping(value = "/events", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> stream() {
        return Flux.interval(Duration.ofSeconds(1))
            .map(i -> ServerSentEvent.<String>builder()
                .id(String.valueOf(i))
                .event("tick")
                .data("ping at " + Instant.now())
                .build());
    }
}

// ── WebClient ─────────────────────────────────────
@Service
public class WeatherService {
    private final WebClient client = WebClient.builder()
        .baseUrl("https://api.weather.com")
        .defaultHeader("Accept", "application/json")
        .build();

    public Mono<WeatherDto> get(String city) {
        return client.get()
            .uri("/weather/{city}", city)
            .retrieve()
            .onStatus(HttpStatus::is4xxClientError,
                r -> Mono.error(new NotFoundException(city)))
            .bodyToMono(WeatherDto.class)
            .timeout(Duration.ofSeconds(5))
            .retryWhen(Retry.backoff(3, Duration.ofMillis(500)));
    }

    // Parallel calls with zip
    public Mono<Dashboard> getDashboard(String userId) {
        return Mono.zip(getProfile(userId), getOrders(userId))
                  .map(t -> new Dashboard(t.getT1(), t.getT2()));
    }
}
◆ Interview Questions
Must Know
Why is RestTemplate deprecated? What should you use instead?
▾
Spring deprecated RestTemplate in Spring 5 because it's blocking and not reactive-friendly. Spring recommends WebClient for all new code — it supports both sync and async modes, has a fluent API, better error handling, built-in retry/timeout/filter support, and works in both WebFlux and Servlet apps.

For synchronous calls in non-reactive code: call .block() on the Mono. In Spring Boot 3.2+, RestClient was introduced — a synchronous HTTP client with WebClient's fluent API (no reactive, no block()). Choose: RestClient (sync apps, no reactive), WebClient (reactive or async-in-sync).
⚡ If you're using RestTemplate in Spring Boot 3.x, migrate to RestClient (sync) or WebClient (async/reactive).
Tricky
Why can't you use JPA/Hibernate with Spring WebFlux directly?
▾
JPA/Hibernate use JDBC which is a blocking API. Calling entityManager.find() blocks the calling thread until the DB responds. In WebFlux, the server runs on Netty's event loop — blocking this thread stops all concurrent requests on that event loop thread, completely defeating the purpose of reactive.

Solutions: 1. R2DBC (Reactive Relational Database Connectivity) — non-blocking DB access, Spring Data R2DBC provides reactive repositories. 2. Offload to scheduler: Mono.fromCallable(() → jpaCall()).subscribeOn(Schedulers.boundedElastic()) — runs blocking code on a separate thread pool, doesn't block the event loop. Option 1 is preferred for new reactive projects.
Tricky
How do you test a WebFlux controller?
▾
WebTestClient: the reactive equivalent of MockMvc. Use with @WebFluxTest(UserController.class):
webTestClient.get().uri("/users/1").exchange().expectStatus().isOk().expectBody(UserDto.class).returnResult()

StepVerifier: test Mono/Flux pipelines directly without HTTP.
StepVerifier.create(userService.findById(1L)).expectNextMatches(u → u.getName().equals("Alice")).verifyComplete()

StepVerifier.withVirtualTime(): test time-based operators (Flux.interval, delay) without actually waiting — virtual time advances instantly.
💡 StepVerifier.withVirtualTime is essential for testing Flux.interval() or any timed operations without slow tests.
Senior
What is the difference between annotation-based controllers and functional routing in WebFlux?
▾
Annotation-based (@RestController, @GetMapping): familiar to MVC developers, Spring scans and registers handlers automatically, good IDE support, easier onboarding.

Functional routing (RouterFunction + HandlerFunction): routes and handlers are plain Java — no reflection, explicit wiring, easier to compose programmatically, simpler unit testing (no MockMvc/WebTestClient needed for handler logic), immutable and thread-safe by design.

Both can coexist in the same app. Most teams use annotation-based for familiarity. Functional routing is preferred when you want explicit, testable handler logic without Spring's annotation processing overhead.

WebFlux = Spring MVC but returns Mono/Flux instead of T.
The framework subscribes and writes the HTTP response automatically.

WebClient = non-blocking RestTemplate. Use in all new code — sync, async, and reactive apps.

JPA in WebFlux = blocks event loop = bad. Use R2DBC for reactive DB, or wrap JPA in Schedulers.boundedElastic().

SSE = return Flux with content-type text/event-stream. Browser gets a real-time stream without polling.
