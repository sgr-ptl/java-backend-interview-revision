Routing Pattern
API Gateway
The single front door for your entire microservices system — routes, secures, and transforms all incoming traffic
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Without an API Gateway, your mobile app would need to know the address of every single microservice. The API Gateway is like a hotel receptionist — the guest talks to one desk, the receptionist figures out which department to route to. It handles cross-cutting concerns (auth, logging, rate-limiting) in one place so your individual services don't have to repeat that logic.

🚦
Routing — Maps incoming URL paths to the right downstream service. /api/orders/** → Order Service, /api/users/** → User Service. The client speaks to one URL and never needs to know internal addresses. Routing rules can include path rewriting, query param manipulation, and load balancing across multiple instances of the same service.
🔐
Authentication & Authorization — Validate JWT tokens or API keys at the gateway level before the request ever reaches a service. This means your individual microservices can trust the caller's identity without doing their own auth. Only valid requests pass through. Invalid tokens get a 401 without touching any service logic.
🚧
Rate Limiting — Protect services from being overwhelmed by limiting how many requests a client can make per second/minute. Configured by IP address, API key, or user ID. When a client exceeds their limit, the gateway returns 429 Too Many Requests immediately. Services behind the gateway are shielded from burst traffic.
🔀
Request Aggregation (BFF) — A specialised form: the Backend for Frontend (BFF) pattern creates a gateway tailored per client type. The mobile BFF calls User, Order, and Product services in parallel and stitches results into a single response — saving the mobile client multiple round trips. One BFF per front-end channel (mobile, web, partner API).
📊
Cross-cutting Concerns — Because all traffic passes through one point, the gateway is the ideal place for: request/response logging, distributed tracing injection (add correlation ID to headers), CORS handling, SSL termination, and response caching. Do this once at the edge rather than in every service.
⚡
Spring Cloud Gateway — Spring's reactive API gateway built on WebFlux/Netty. Non-blocking. Configured via predicates (route matching rules) and filters (transforms applied before/after routing). Supports circuit breakers, retry, rate limiting with Redis, and JWT validation as built-in filter factories. Replaces the older Zuul gateway.

// application.yml ─ declarative route config
// spring:
//   cloud:
//     gateway:
//       routes:
//         - id: order-service
//           uri: lb://ORDER-SERVICE          # load-balanced via Eureka
//           predicates:
//             - Path=/api/orders/**
//             - Method=GET,POST
//           filters:
//             - StripPrefix=1               # strip /api before forwarding
//             - name: CircuitBreaker
//               args:
//                 name: ordersCB
//                 fallbackUri: forward:/fallback/orders
//             - name: RequestRateLimiter
//               args:
//                 redis-rate-limiter.replenishRate: 10
//                 redis-rate-limiter.burstCapacity: 20
//                 key-resolver: "#{@userKeyResolver}"

// Programmatic route + JWT filter
@Configuration
public class GatewayConfig {

    @Bean
    public RouteLocator routes(RouteLocatorBuilder builder) {
        return builder.routes()
            .route("users", r -> r
                .path("/api/users/**")
                .filters(f -> f
                    .stripPrefix(1)
                    .addRequestHeader("X-Source", "gateway")
                    .retry(c -> c.setRetries(3)
                        .setStatuses(HttpStatus.SERVICE_UNAVAILABLE)))
                .uri("lb://USER-SERVICE"))
            .build();
    }

    // Global filter: validate JWT on every request
    @Bean
    public GlobalFilter jwtValidationFilter(JwtUtil jwtUtil) {
        return (exchange, chain) -> {
            String auth = exchange.getRequest().getHeaders()
                .getFirst(HttpHeaders.AUTHORIZATION);
            if (auth == null || !auth.startsWith("Bearer ")) {
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            }
            String token = auth.substring(7);
            if (!jwtUtil.isValid(token)) {
                exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
                return exchange.getResponse().setComplete();
            }
            // forward userId downstream as trusted header
            ServerWebExchange mutated = exchange.mutate()
                .request(r -> r.header("X-User-Id", jwtUtil.getUserId(token)))
                .build();
            return chain.filter(mutated);
        };
    }

    // Rate-limit key: per user ID
    @Bean
    KeyResolver userKeyResolver = exchange ->
        Mono.just(exchange.getRequest().getHeaders()
            .getFirst("X-User-Id"));
}
◆ Interview Questions
Must Know
Why use an API Gateway instead of letting clients call services directly?
▾
Direct client-to-service calls create several problems: clients must know every service's address, every service must independently handle auth/rate-limiting, CORS, and logging — duplicating code everywhere. Adding a new service means updating every client. Renaming or moving a service breaks all clients.

The gateway solves all of this: one address for clients, one place for cross-cutting concerns, services are free to change their internal addresses without clients noticing. Also enables A/B testing (send 10% of traffic to v2), canary deployments, and protocol translation (REST outside, gRPC inside).
⚡ Key insight: The gateway is NOT about performance — it's about decoupling clients from the internal topology of your services. Services are free to evolve, split, merge, or move without breaking clients.
Tricky
What is the Backend for Frontend (BFF) pattern and when does it apply?
▾
BFF creates a dedicated gateway per client type. A mobile app has very different data needs than a web dashboard or a partner API. A single generic gateway forces all clients to make the same calls and receive the same response shape.

With BFF: the mobile gateway aggregates three service calls into one response perfectly shaped for mobile (smaller payload, device-optimised fields). The web BFF returns richer data. Each BFF is owned by the team building that client — no coordination needed for API shape changes.
💡 Real example: Netflix uses BFF extensively — the Smart TV app, mobile app, and browser each have dedicated API layers that aggregate and shape data specifically for that device's rendering requirements.
Senior
What are the risks of putting too much logic in the API Gateway?
▾
The gateway becomes a bottleneck and a single point of failure if overloaded with business logic. Symptoms: the gateway team becomes a blocker for all other teams, deploying new features requires gateway changes, the gateway codebase becomes a monolith.

Gateway should handle: routing, auth, rate-limiting, TLS, CORS, tracing injection, and simple request/response transforms. Gateway should NOT handle: business logic, complex data aggregation, service orchestration, domain validation. Those belong in dedicated services or a dedicated orchestration service. Keep the gateway thin and stateless.

💡

What stays OUT of the gateway: business logic, complex aggregation, domain validation.

BFF mnemonic: "One gateway per client type" — each client gets a gateway tailored to its exact shape of data.

Resilience Pattern
Circuit Breaker
Stop the cascade — when a downstream service is struggling, stop hammering it and fail fast instead
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Imagine your payment service starts responding slowly. Without a circuit breaker, every order request ties up a thread waiting 30 seconds for the payment call to time out. Soon all threads are stuck waiting, and your entire order service goes down too — a cascade failure. The circuit breaker is like an electrical fuse: it detects the problem, "trips open", and immediately returns a fallback response. The struggling service gets breathing room to recover.
◆ The Three States

🟢
CLOSED state — Normal traffic — All requests pass through to the real service. The circuit breaker silently monitors: counting failures, tracking slow responses, measuring error rate. When failures stay below the configured threshold, everything flows normally. Think of this as the fuse being intact.
🔴
OPEN state — Fast fail — Once failures exceed the threshold (e.g., 50% failure rate over 10 seconds), the circuit "trips". Now it immediately returns a fallback response without even trying to call the service. This is actually kinder to both caller and service — no wasted threads waiting for timeouts. The struggling service gets a chance to recover without being bombarded.
🟡
HALF-OPEN state — Testing recovery — After the configured wait duration (e.g., 60 seconds), the circuit breaker lets a small number of probe requests through. If those succeed, the circuit closes again (service recovered). If they fail, it trips back to OPEN and waits again. This prevents prematurely sending full traffic to a service that hasn't fully recovered.
🛟
Fallback Strategy — When OPEN, what do you return instead? Options: cached stale data ("showing results from 5 minutes ago"), a default value (empty list, zero), a response from a secondary/degraded service, or a meaningful error message telling the user to try again. The fallback should be defined per use case — not just return null.
🔧
Resilience4j in Spring Boot — The standard library for circuit breakers, retry, bulkhead, rate limiter, and time limiter in Spring. Add spring-cloud-starter-circuitbreaker-resilience4j. Configures via application.yml or annotations. Integrates with Micrometer for metrics (track open/close events, failure rates). Works with Spring Cloud Gateway and WebClient.

// resilience4j:
//   circuitbreaker:
//     instances:
//       paymentService:
//         registerHealthIndicator: true
//         slidingWindowType: COUNT_BASED
//         slidingWindowSize: 10          # last 10 calls
//         failureRateThreshold: 50       # >50% failures → OPEN
//         waitDurationInOpenState: 60s   # wait before HALF-OPEN
//         permittedCallsInHalfOpenState: 3
//         slowCallDurationThreshold: 2s  # 2s+ is a "failure"
//         slowCallRateThreshold: 80
//   retry:
//     instances:
//       paymentService:
//         maxAttempts: 3
//         waitDuration: 500ms
//         enableExponentialBackoff: true
//         retryExceptions:
//           - java.io.IOException
//           - org.springframework.web.client.ResourceAccessException

// Service layer — annotation based
@Service
public class PaymentClient {

    @CircuitBreaker(name = "paymentService", fallbackMethod = "fallback")
    @Retry(name = "paymentService")
    @TimeLimiter(name = "paymentService")           // enforce 2s timeout
    public CompletableFuture<PaymentResult> charge(PaymentRequest req) {
        return CompletableFuture.supplyAsync(() ->
            webClient.post().uri("/charge")
                .bodyValue(req).retrieve()
                .bodyToMono(PaymentResult.class)
                .block()
        );
    }

    // Fallback — called when OPEN or all retries exhausted
    public CompletableFuture<PaymentResult> fallback(
            PaymentRequest req, Exception ex) {
        log.warn("Payment service unavailable, returning pending: {}", ex.getMessage());
        // Queue for async retry instead of failing the order
        pendingQueue.enqueue(req);
        return CompletableFuture.completedFuture(PaymentResult.pending());
    }
}

// Programmatic access — query circuit state
@Autowired
CircuitBreakerRegistry registry;

public CircuitBreaker.State getPaymentCircuitState() {
    return registry.circuitBreaker("paymentService").getState();
}

◆ Interview Questions
Must Know
What problem does the Circuit Breaker pattern solve?
▾
It prevents cascade failures. In a microservices system, one slow or failing service can bring down the whole chain. Example: Service A calls Service B (slow). A's threads pile up waiting. A runs out of threads. Now A fails too. Service C calls A and fails. The whole system collapses — not because of C's problem, but because of B's.

Circuit breaker stops this chain: when B is struggling, A's circuit breaker opens and immediately returns a fallback without tying up threads waiting. A stays healthy. C stays healthy. B gets time to recover without being bombarded. The failure is contained at the source.
⚡ Thread pool exhaustion is the key mechanism of cascade failure. Circuit breakers prevent it by eliminating waiting. Fast fail is kinder than slow timeout.
Tricky
What is the difference between Circuit Breaker and Retry? When should you use each?
▾
Retry is for transient failures — the kind that resolve themselves quickly: a brief network glitch, a momentary database hiccup. Retry makes sense when the failure is short-lived and the operation is idempotent (safe to repeat). Use exponential backoff to avoid hammering the recovering service.

Circuit Breaker is for sustained failures — when the service is genuinely down or degraded for more than a few seconds. Retrying a broken service just wastes threads and delays the caller. The circuit breaker detects the pattern and stops trying.

In practice, combine both: Retry handles brief transient issues, Circuit Breaker handles longer outages. The retry runs first (inside the circuit breaker's context). If retries exhaust without success, those failures count toward the circuit breaker's threshold.
Senior
How do you monitor circuit breaker health in production?
▾
Resilience4j integrates with Micrometer out of the box. Expose via Spring Boot Actuator + Prometheus → Grafana. Key metrics to track: resilience4j.circuitbreaker.state (0=CLOSED, 1=OPEN, 2=HALF_OPEN — alert when not 0), resilience4j.circuitbreaker.failure.rate (alert when >10%), resilience4j.circuitbreaker.calls (failed, slow, not permitted).

Also expose the health endpoint: management.health.circuitbreakers.enabled=true → /actuator/health shows each circuit's state. Alert ops when any circuit is OPEN for >5 minutes — that means a service dependency is genuinely down and needs investigation, not just a transient blip.
💡
Circuit Breaker = electrical fuse for your service calls.

3 states to memorise:
• CLOSED = fuse intact, all requests pass, failures counted
• OPEN = fuse tripped, fail fast, return fallback immediately
• HALF-OPEN = send probe requests, if OK → CLOSE, if fail → OPEN again

Retry vs Circuit Breaker: Retry for brief glitches. Circuit Breaker for sustained outages. Layer them: Retry inside, Circuit Breaker outside.

Fallback principle: Define it per business use case. "Show stale cache" beats "return 500 error".

Data Consistency
Saga Pattern
Distributed transactions without a distributed lock — a sequence of local transactions connected by events or commands
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: You can't use a single database transaction across multiple microservices — each service has its own database. The Saga pattern solves this by breaking a "distributed transaction" into a chain of smaller local transactions, each in their own service's database. If step 3 fails, the saga runs compensating transactions to undo steps 1 and 2. Think of it like a hotel booking system: book flight → book hotel → book car. If the car is unavailable, cancel the hotel and cancel the flight.
◆ Choreography vs Orchestration

✅
Local Transaction = Saga Step — Each saga step is a completely normal local database transaction within one service. It either commits or rolls back locally. The saga coordinates these steps. There is no distributed lock, no two-phase commit, no distributed transaction coordinator. Each service's data stays consistent locally at all times.
↩️
Compensating Transactions — These are the "undo" steps. They don't literally reverse the DB transaction (which already committed), but they apply the business-level inverse. Example: step 2 was "debit account by $100" — the compensating transaction is "credit account by $100." Compensating transactions must always be defined for every step that could need rollback. They are idempotent (safe to run twice).
🔄
Choreography Saga Flow — Order Service saves order and publishes OrderCreated event. Payment Service listens, charges card, publishes PaymentCompleted or PaymentFailed. Inventory Service listens to PaymentCompleted, reserves stock. Each service reacts to events from the previous step. No central coordinator — the flow emerges from event subscriptions.
🎯
Orchestration Saga Flow — An Orchestrator service sends a command to Payment Service, waits for a reply, then sends a command to Inventory Service, and so on. The orchestrator holds all the flow logic and knows how to compensate if any step fails. Easier to reason about and debug. Axon Framework and Temporal are popular implementations.
⚠️
Eventual Consistency Trade-off — Sagas provide eventual consistency, not immediate consistency. Between steps, the system is temporarily inconsistent. Example: after step 1 the order exists but stock hasn't been reserved yet. Other queries during this window will see partially-applied state. Design your read models to handle this — show "pending" status rather than assuming completion.

Saga — Choreography via Kafka + Compensating Transaction
// ── STEP 1: Order Service creates order + publishes event ──
@Service
public class OrderService {
    @Transactional
    public Order createOrder(OrderRequest req) {
        Order order = orderRepo.save(Order.of(req, OrderStatus.PENDING));
        // Use Outbox pattern here in production
        publisher.publishEvent(new OrderCreatedEvent(order.getId(), req));
        return order;
    }

    // Compensating transaction — undo the order if payment fails
    @Transactional
    public void cancelOrder(Long orderId, String reason) {
        orderRepo.findById(orderId).ifPresent(order -> {
            order.setStatus(OrderStatus.CANCELLED);
            order.setCancelReason(reason);
            orderRepo.save(order);
        });
    }
}

// ── STEP 2: Payment Service listens + charges ──────────
@Component
public class PaymentSagaStep {

    @KafkaListener(topics = "order-created")
    public void onOrderCreated(OrderCreatedEvent event) {
        try {
            paymentService.charge(event.getCustomerId(), event.getAmount());
            kafka.send("payment-completed", new PaymentCompletedEvent(event.getOrderId()));
        } catch (PaymentException e) {
            // Publish failure → triggers compensating transactions upstream
            kafka.send("payment-failed",
                new PaymentFailedEvent(event.getOrderId(), e.getMessage()));
        }
    }
}

// ── COMPENSATION: Order Service listens to payment failure ─
@Component
public class OrderCompensationHandler {

    @KafkaListener(topics = "payment-failed")
    public void onPaymentFailed(PaymentFailedEvent event) {
        orderService.cancelOrder(event.getOrderId(), event.getReason());
        log.warn("Saga rolled back for order {}: {}",
            event.getOrderId(), event.getReason());
    }
}

// ── STEP 3: Inventory listens to payment success ────────
@Component
public class InventorySagaStep {

    @KafkaListener(topics = "payment-completed")
    public void onPaymentCompleted(PaymentCompletedEvent event) {
        inventoryService.reserveStock(event.getOrderId());
        kafka.send("order-fulfilled", new OrderFulfilledEvent(event.getOrderId()));
    }
}
◆ Interview Questions
Must Know
Why can't you use a regular database transaction across microservices?
▾
A database transaction requires all operations to share the same database connection — or at least the same database server for distributed transactions. In microservices, each service owns its own isolated database (the "database per service" pattern). There is no shared connection.

Even if you tried two-phase commit (2PC) across databases, it introduces: a distributed lock that blocks all participating services until the coordinator confirms, tight coupling between services, risk of indefinite blocking if the coordinator crashes, and massive latency overhead. Sagas trade immediate consistency for availability and independence — each service's data is always locally consistent, and the system moves toward eventual global consistency through events.
⚡ Key principle: Microservices trade ACID across services for BASE (Basically Available, Soft state, Eventually consistent). Sagas implement this trade-off gracefully.
Tricky
What makes a compensating transaction different from a rollback?
▾
A database rollback literally undoes uncommitted changes as if they never happened. A compensating transaction is a new, separate database operation that applies the business inverse of a previously committed action.

Example: Step 1 commits "debit $100 from account." The account balance is now $100 less. You can't roll that back — the transaction is committed. The compensating transaction is a separate "credit $100 to account" — it acknowledges the debit happened and reverses the business effect by applying the opposite.

This means compensating transactions must be idempotent: if they run twice (due to retries or duplicates), the end result must be the same. Use idempotency keys to detect and ignore duplicate compensation requests.
Compare
Choreography vs Orchestration — which would you choose for a 5-step order saga?
▾
For a 5-step saga with complex rollback logic, Orchestration is the better choice. Here's why:

Choreography with 5 steps creates an implicit flow spread across 5 services — understanding the full flow requires reading code in all 5 codebases simultaneously. When debugging a bug in step 4, you have to trace events through steps 1→2→3→4. Compensations become implicit chains of failure events that are hard to reason about.

Orchestration puts the flow in one place — a single Saga class (or Axon Aggregate/Temporal Workflow) that clearly shows the step sequence, success path, and all compensation paths. When step 4 fails, the orchestrator explicitly calls compensations for steps 3, 2, and 1 in reverse order. Any developer can read the saga class and understand the full business flow.
💡 Rule of thumb: 1–3 steps → Choreography (simple, low overhead). 4+ steps with complex rollback → Orchestration (maintainable, testable, debuggable).

Data Pattern
CQRS — Command Query Responsibility Segregation
Separate your write model from your read model — scale and optimise each independently
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Most applications read data far more than they write it. CQRS says: stop using the same data model for both. The write model is optimised for consistency and business rule enforcement. The read model is optimised for fast, denormalised queries that serve the UI exactly what it needs. They are separate, talk via events, and can scale independently — read replicas for the query side, smaller primary for writes.

📝
Commands — the Write Side — A Command represents an intent to change state: PlaceOrderCommand, CancelOrderCommand. Command handlers validate business rules and persist changes to the write database (fully normalised, relationship-correct). Commands do not return data — they return void or just an ID. The write model is the single source of truth for business rules.
🔍
Queries — the Read Side — Queries retrieve data from a pre-built read model (a denormalised view tailored exactly to what the UI needs). No joins — all required data is already merged into the read record. A query for "Order Summary Page" returns one flat document with order info, customer name, product names, and delivery status already combined. No runtime joins, no N+1 problems.
🔄
Projections — Syncing Read from Write — When the write side saves a change, it publishes an event (OrderPlaced, OrderShipped). A projector consumes these events and updates the read database accordingly — building or updating the denormalised read record. This is eventually consistent: there is a small delay between a write and the read model reflecting it.
⚡
Independent Scaling — The read side typically gets far more traffic than the write side. With CQRS, you can run 10 read replicas (Elasticsearch, Redis, materialized views) for every 1 write primary. The write DB can be a fully ACID relational database, while the read DB can be a document store or search engine optimised for the specific query patterns.
🔁
Read Model Rebuild — One powerful benefit: if you need a new read model (new report, new UI), replay all events from the event store and build a fresh projection. You don't need to migrate existing data — the events are the record of truth, and any projection can be rebuilt from scratch at any time.
CQRS — Command Handler + Query Handler + Projector
// ── Command (write intent) ─────────────────────────
public record PlaceOrderCommand(Long customerId, List<OrderLine> lines) {}

// ── Command Handler (write side) ──────────────────
@Service
public class OrderCommandHandler {
    private final OrderWriteRepository writeRepo;
    private final ApplicationEventPublisher events;

    @Transactional
    public Long handle(PlaceOrderCommand cmd) {
        // Business rule validation happens here, in the write model
        if (cmd.lines().isEmpty()) throw new BusinessRuleException("Empty order");

        Order order = Order.create(cmd.customerId(), cmd.lines());
        writeRepo.save(order);

        // Publish event → projector will update the read model
        events.publishEvent(new OrderPlacedEvent(order.getId(), cmd));
        return order.getId();
    }
}

// ── Read model (denormalised, query-optimised) ────
@Document("order_summaries")          // Stored in MongoDB/Elasticsearch
public class OrderSummary {
    private Long   orderId;
    private String customerName;     // pre-joined! no runtime join needed
    private String customerEmail;
    private String status;
    private BigDecimal totalAmount;
    private List<String> productNames; // pre-joined from products table
    private Instant placedAt;
}

// ── Projector: builds read model from events ──────
@Component
public class OrderSummaryProjector {
    private final OrderSummaryRepository readRepo;
    private final CustomerService customerSvc;  // fetch customer details

    @EventListener
    public void on(OrderPlacedEvent event) {
        Customer customer = customerSvc.findById(event.getCustomerId());
        OrderSummary summary = OrderSummary.builder()
            .orderId(event.getOrderId())
            .customerName(customer.getFullName())
            .customerEmail(customer.getEmail())
            .status("PENDING")
            .totalAmount(event.getTotalAmount())
            .productNames(event.getProductNames())
            .placedAt(Instant.now())
            .build();
        readRepo.save(summary);     // one fast lookup at query time
    }
}

// ── Query Handler (read side — zero joins!) ───────
@Service
public class OrderQueryHandler {
    private final OrderSummaryRepository readRepo;

    public OrderSummary getOrderSummary(Long orderId) {
        return readRepo.findById(orderId)     // single document lookup
            .orElseThrow(() -> new NotFoundException(orderId));
    }
}
Must Know
What is the main advantage of CQRS and when is it overkill?
▾
Main advantages: (1) Read performance — denormalised read models serve queries with zero joins. (2) Independent scaling — add read replicas without touching write side. (3) Multiple read models — build different projections optimised for different query patterns (list view, detail view, analytics view). (4) Rebuilding — if your read model is wrong, replay events and rebuild it.

When it's overkill: Simple CRUD applications where reads and writes have similar patterns. Small teams where maintaining two models is more burden than benefit. Systems with low read traffic or where join performance is acceptable. Start simple — apply CQRS only to bounded contexts with high read/write ratio or complex query requirements.
⚡ The extra complexity of CQRS pays off when reads outnumber writes by 10:1+ or when queries are significantly more complex than writes.
Tricky
How do you handle the eventual consistency lag in CQRS?
▾
After a command executes, the read model is not yet updated — there's a brief window (milliseconds to seconds) where a read will return stale data. Strategies:

1. UI optimistic update: The UI immediately shows what the user just did (without waiting for the read model). When the read model catches up, the UI reconciles. This is what Google Docs and most modern apps do.

2. Version-based reads: The command returns a version number. The client polls the query endpoint until it sees a result with that version, then displays it.

3. Inform the user: Show a "Processing..." message and let them know the change will be visible shortly. Not ideal for all cases but honest and simple.

4. Same-request shortcut: For read-your-own-writes, route the immediate post-write read to the write database directly before the projection updates.
Senior
How do you rebuild a read model from scratch?
▾
This is one of CQRS's most powerful capabilities. Steps: (1) Deploy the updated projector that builds the new read model schema. (2) Clear or create a new read model collection/table. (3) Replay all historical events from your event store (if using Event Sourcing) or from a replayable event log (Kafka with long retention). (4) The projector processes every event from the beginning, building the new read model from scratch. (5) Once caught up, switch traffic to the new read model.

This enables schema-free evolution: you never need to write migration scripts for your read data. Just rebuild the projection. It also means you can create entirely new read models for new features — build the projection now, and the historical data is already there.
💡
CQRS = split your data model in two. Write for correctness, read for speed.

Restaurant analogy: The kitchen (write side) cooks carefully from recipes (business rules). The waiter (projector) pre-plates everything (read model). When a customer asks for their order status, the waiter just looks at the pre-plated dish — no going back to the kitchen (no runtime joins).

Key rule: Commands change state, Queries don't.
A Command handler returns void or an ID. A Query handler never modifies state.

Read model rebuild: replay all events → zero data migrations needed.

Event Sourcing

🟡 Medium

❓ Interview Q&A
🧠 Memory
🧠
In plain English: Traditional databases store the current state of data. Event Sourcing stores every change that ever happened instead. Think of it like a bank statement vs just your balance. The bank stores every transaction (events); your balance is calculated by summing them up. If there's ever a discrepancy, you can trace every penny. Event Sourcing gives you a complete, immutable audit log plus the ability to "time travel" to any past state.
📜
The Event Log is the Database — Instead of storing order.status = "SHIPPED", you store the event OrderShippedEvent(orderId=42, timestamp=..., trackingNumber=...). The current state of any order is derived by replaying all its events in sequence. The event log is append-only and immutable — events are never deleted or updated, only appended.
⏳
Complete Audit Trail — You always know not just the current state, but the full history of how it got there: who changed it, when, and why. This is invaluable for debugging, compliance, fraud detection, and customer support ("why was my order cancelled?"). Traditional databases typically need separate audit tables; Event Sourcing makes this automatic.
⏪
Time Travel — Replay events up to any point in time to see what the state was then. Powerful for: debugging a production issue ("what did this order look like at 3:47pm?"), A/B testing new business logic against historical data, correcting past mistakes by replaying with fixed logic.
📸
Snapshots — Performance Optimisation — Replaying 10,000 events every time someone loads an entity is slow. Snapshots periodically capture the current state and save it. On load: get the latest snapshot (event 9,800) and replay only the 200 events since then. Snapshots are a performance cache — they can always be discarded and rebuilt from the event log.
🔗
Event Sourcing + CQRS — These patterns are natural partners but independent. Event Sourcing defines how state is stored (as events). CQRS defines how reads and writes are separated. Together: write side stores events → projectors consume events → build read models. The event store is the write database. CQRS uses those events as the sync mechanism for its read models.
⚠️
Schema Evolution Complexity — Events are immutable forever. If your event schema changes (rename a field, add required data), old events still have the old schema. You must handle this with: event upcasters (transform old events to new schema on read), versioned events (OrderPlacedV1, OrderPlacedV2), or schema registries. This is the main complexity of Event Sourcing — plan your event schemas carefully.
Event Sourcing — Aggregate + Event Store + Snapshot
// ── Events (immutable facts, stored forever) ──────
public record OrderPlacedEvent(Long orderId, Long customerId,
    List<OrderLine> lines, Instant at) {}
public record OrderShippedEvent(Long orderId, String trackingNo, Instant at) {}
public record OrderCancelledEvent(Long orderId, String reason, Instant at) {}

// ── Aggregate: state derived from events ──────────
public class OrderAggregate {
    private Long id;
    private OrderStatus status;
    private String trackingNumber;
    private final List<Object> uncommittedEvents = new ArrayList<>();

    // Rebuild state by applying events in sequence
    public static OrderAggregate rehydrate(List<Object> events) {
        OrderAggregate order = new OrderAggregate();
        events.forEach(order::apply);
        return order;
    }

    public void ship(String trackingNo) {
        if (status != OrderStatus.PAID)
            throw new BusinessRuleException("Can only ship paid orders");
        var event = new OrderShippedEvent(id, trackingNo, Instant.now());
        apply(event);                    // update in-memory state
        uncommittedEvents.add(event);   // queue for persistence
    }

    private void apply(Object event) {
        switch (event) {
            case OrderPlacedEvent e   -> { this.id = e.orderId(); status = OrderStatus.PENDING; }
            case OrderShippedEvent e  -> { status = OrderStatus.SHIPPED; trackingNumber = e.trackingNo(); }
            case OrderCancelledEvent e -> { status = OrderStatus.CANCELLED; }
            default -> throw new UnknownEventException(event);
        }
    }
}

@Repository
public interface EventStoreRepository extends JpaRepository<StoredEvent, Long> {
    List<StoredEvent> findByAggregateIdOrderBySequenceAsc(Long aggregateId);
    Optional<StoredEvent> findLatestSnapshotByAggregateId(Long id);
}

// ── Load aggregate (snapshot + events since) ─────
public OrderAggregate load(Long orderId) {
    Optional<Snapshot> snapshot = snapshotRepo.findLatest(orderId);
    long fromSeq = snapshot.map(Snapshot::getSequence).orElse(0L);
    List<Object> events = eventStore.findEventsAfter(orderId, fromSeq);

    OrderAggregate order = snapshot.map(Snapshot::toAggregate)
        .orElseGet(OrderAggregate::new);
    events.forEach(order::apply);   // only events since snapshot
    return order;
}

What is Event Sourcing and how is it different from traditional storage?
▾
Traditional storage: store the current state. UPDATE orders SET status='SHIPPED'. History is gone — you only know the current value, not how it got there.

Event Sourcing: store every change as an immutable event. OrderPlacedEvent, PaymentReceivedEvent, OrderShippedEvent. The current state is derived by replaying these events in order. The event log is the system of record — never updated, only appended.

Think of the difference like: a bank storing only your current balance (traditional) vs storing every debit and credit since the account opened (event sourcing). The latter lets you audit every transaction, detect fraud, trace disputes, and even correct errors by replaying with fixed logic.
💡 "The log is the database." Everything else (tables, views, caches) is a derived projection of the event log.
Tricky
What is an Event Sourcing snapshot and when do you need one?
▾
A snapshot is a periodic save of the aggregate's current state after processing N events. On load, instead of replaying from event 1, you load the snapshot (say, after event 9,800) and replay only the 200 events since then.

When do you need it: when aggregates accumulate hundreds or thousands of events and replay becomes noticeably slow (50ms+). Long-lived aggregates like bank accounts, user profiles, or heavily-updated orders eventually need snapshots.

Important: snapshots are a performance optimisation, not a requirement. They can always be discarded and rebuilt from the event log. Never delete events just because you have a snapshot — the events are the source of truth.
⚡ Add snapshots when load performance degrades, not upfront. Start simple with full replay, add snapshots when you actually hit performance issues.
Senior
How do you handle schema evolution in Event Sourcing?
▾
This is the hardest part of Event Sourcing. Events are stored forever and are immutable, so old events will always have the old schema. Three main strategies:

1. Upcasters: when loading old events, transform them to the new schema on the fly before applying. OrderPlacedV1Upcaster converts old events to the new format. Chain upcasters for multiple version jumps.

2. Versioned events: create OrderPlacedV2Event alongside the original. Event handlers handle both versions explicitly. Old events remain valid forever.

3. Schema compatibility rules: design events to be forward-compatible. Adding optional fields is safe. Renaming, removing, or making fields required is breaking. Use Avro or Protobuf with a Schema Registry to enforce compatibility.

Best practice: treat event schemas like public APIs — once published, they're effectively immutable. Plan carefully before writing new event types.

Reliability Pattern
Transactional Outbox Pattern
Guarantee exactly-once event publishing — if the DB write succeeded, the message will eventually be published
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: You place an order (saved to DB) and want to publish an "OrderPlaced" event to Kafka. If you save to DB and then publish to Kafka, what if the app crashes between those two steps? The order is saved but the event is never published — your downstream services never know the order exists. The Outbox Pattern solves this by saving the event in the same database transaction as the order. A separate process then reliably delivers it to Kafka. If the app crashes, the outbox record is still there — delivery will happen when the app restarts.

🔒
Why the Dual Write Problem Exists — Writing to a database and publishing to Kafka are two separate systems. There is no single transaction that spans both. If you write to DB first then crash → event never published. If you publish to Kafka first then crash → event published but DB write never happened (or rolls back). Either way, the systems get out of sync. The Outbox Pattern eliminates this dilemma entirely.
📬
The Outbox Table — A special table in your service's own database: outbox_events(id, aggregate_type, aggregate_id, event_type, payload, published, created_at). When you save business data, you also INSERT to this table in the same transaction. Either both succeed (DB transaction committed) or both fail (rolled back). The outbox record is the guaranteed proof that the event should be published.
⚡
Debezium — CDC-Based Relay (Best Practice) — Debezium is a Change Data Capture tool that streams database changes directly as Kafka events. Configure it to watch your outbox table — when a new row appears, Debezium automatically publishes it to Kafka. No polling delay, very low latency, and the database WAL (write-ahead log) ensures zero missed events even if Debezium is temporarily down. No code needed in your service.
🔄
Polling-Based Relay (Simpler Alternative) — A scheduled job (or a dedicated thread) periodically queries the outbox for unpublished events, publishes them to Kafka, and marks them as published. Simpler to set up than Debezium but slightly higher latency and the polling interval must be tuned. Good for lower-throughput systems or when CDC is not available.
🔁
At-Least-Once Delivery — The outbox guarantees that events are published at least once. If the relay crashes after publishing but before marking as published, the event gets published again on restart. Consumers of these events must be idempotent — they must safely handle receiving the same event twice. Use the event ID as an idempotency key.
Outbox Pattern — Same-TX Write + Polling Relay + Idempotency
// ── Outbox entity ─────────────────────────────────
@Entity @Table(name = "outbox_events")
public class OutboxEvent {
    @Id @GeneratedValue private Long id;
    private String aggregateType;   // "Order"
    private String aggregateId;     // "42"
    private String eventType;       // "OrderPlaced"
    @Lob private String payload;    // JSON of the event
    private boolean published = false;
    private Instant createdAt = Instant.now();
}

// ── Service: save business data + outbox in same TX ─
@Service
public class OrderService {
    private final OrderRepository orderRepo;
    private final OutboxRepository outboxRepo;
    private final ObjectMapper json;

    @Transactional          // ONE transaction for both writes
    public Order placeOrder(OrderRequest req) {
        Order order = orderRepo.save(Order.from(req));

        // Write event to outbox — same transaction, guaranteed
        OrderPlacedEvent event = new OrderPlacedEvent(order.getId(), req);
        outboxRepo.save(OutboxEvent.builder()
            .aggregateType("Order")
            .aggregateId(order.getId().toString())
            .eventType("OrderPlaced")
            .payload(json.writeValueAsString(event))
            .build());
        // If either save fails, BOTH roll back — no inconsistency
        return order;
    }
}

// ── Polling Relay: publishes outbox events to Kafka ─
@Component
public class OutboxRelay {
    @Scheduled(fixedDelay = 1000)          // poll every 1s
    @Transactional
    public void relay() {
        outboxRepo.findTop100ByPublishedFalseOrderByCreatedAtAsc()
            .forEach(event -> {
                kafka.send("orders",
                    event.getAggregateId(),  // message key
                    event.getPayload());      // JSON payload
                event.setPublished(true);
                outboxRepo.save(event);
            });
    }
}

        return;                         // idempotency check
    }
    inventoryService.reserve(event);
    processedEvents.add(event.getEventId()); // mark as processed
}
Must Know
Why does the Outbox Pattern exist? What problem does it solve?
▾
It solves the dual write problem: you can't atomically write to two different systems (database + message broker). Any approach that writes to both separately has a failure window: crash between the two writes = systems out of sync.

The Outbox Pattern eliminates the dual write by: (1) writing BOTH the business data AND the outbox record to your database in a single atomic transaction. (2) A separate process reads from the outbox and publishes to Kafka. The outbox record survives any crash, and the relay will eventually deliver it. The guarantee: if the DB transaction committed, the event WILL eventually be published.
⚠️ Without the Outbox Pattern: every "save to DB then publish to Kafka" has a crash window. Production services that don't handle this will eventually have silent data inconsistencies.
Tricky
What is Debezium and why is it preferred over a polling relay?
▾
Debezium is an open-source Change Data Capture (CDC) platform that streams database changes directly as events. Instead of your application polling the outbox table, Debezium tails the database's WAL (Write-Ahead Log — the same binary log used for replication). Every INSERT/UPDATE/DELETE appears as a Debezium event which it streams to Kafka.

Why preferred: (1) Near-zero latency (sub-millisecond from DB write to Kafka message, vs polling delay). (2) Zero missed events — WAL ensures every change is captured even if Debezium is temporarily down. (3) No extra load on your DB from polling queries. (4) Works for any table changes, not just outbox. The trade-off: Debezium requires operational overhead (it's an additional deployment component).
💡 Debezium + Outbox = the gold standard for guaranteed event delivery in production microservices. Many large-scale systems (LinkedIn, Airbnb) use this pattern.
Senior
How do you handle duplicate event delivery from the Outbox Pattern?
▾
The Outbox Pattern guarantees at-least-once delivery — the relay may publish an event more than once if it crashes after publishing but before marking as published. Consumers must be idempotent.

Implementation strategies: (1) Idempotency table: store processed event IDs in a DB table. On receive, check if already processed — if yes, skip. (2) Database UPSERT: use INSERT ... ON CONFLICT DO UPDATE — duplicate application of the same event produces the same end state. (3) Natural idempotency: design operations to be naturally idempotent (e.g., SET status='SHIPPED' is idempotent — applying it twice has no extra effect). Include a unique eventId in every event specifically for deduplication purposes.
💡
Outbox Pattern = write event to DB first, publish to Kafka later — never lose an event.

The two-step guarantee:
Step 1 — Save business data + outbox record in ONE database transaction.
Step 2 — A relay (Debezium or polling) reads outbox and publishes to Kafka.

If the app crashes between step 1 and step 2, the outbox record survives in the database. The relay retries until success.

At-least-once = consumers must be idempotent. Store processed event IDs and skip duplicates.

Debezium = no polling, near-zero latency, reads database WAL directly.

Service Discovery

🟡 Medium

🧠 Memory
🧠
In plain English: In a microservices system, services are deployed on dynamic infrastructure — instances start and stop, IPs change. You can't hardcode "Payment Service is at 10.0.0.5:8082" because that IP might change after the next deployment. Service Discovery is like a phone book that's always up to date — services register their current address when they start and deregister when they stop. Others look up the address when they need to call.
📋
Service Registry — A central database of service instances and their addresses. Services register on startup: "I am Payment Service, running at 10.0.0.5:8082." They send heartbeats to prove they're still alive. They deregister on shutdown. The registry removes instances that stop sending heartbeats (self-preservation mode in Eureka). Netflix Eureka and HashiCorp Consul are the main implementations.
🔎
Client-Side Discovery (Eureka + Spring Cloud LoadBalancer) — The client queries the registry for available instances, then picks one and calls it directly. The load-balancing logic runs in the client. Spring Cloud LoadBalancer replaces the old Ribbon library. Advantage: the client has full control over the selection algorithm. Disadvantage: every client must include discovery logic.
🔀
Server-Side Discovery (Kubernetes, AWS ALB) — The client calls a fixed address. The infrastructure (Kubernetes Service, Load Balancer) handles routing to healthy instances. The client doesn't know discovery is happening — it just calls the service name. This is the Kubernetes model: http://payment-service/charge — Kubernetes DNS resolves it and load-balances across pods. Simpler for clients, requires infrastructure support.
🛡️
Eureka Self-Preservation Mode — If Eureka loses contact with >85% of registered clients simultaneously, it assumes a network partition rather than mass failures, and stops deregistering instances. This prevents the registry from incorrectly evicting healthy services during a network blip. In production, set enable-self-preservation: false in development to avoid stale registrations confusing tests.
☸️
Kubernetes Service Discovery — In Kubernetes, Service objects provide stable DNS names and cluster-internal load balancing. Every pod can reach payment-service.default.svc.cluster.local automatically. No separate discovery server needed — Kubernetes DNS IS the discovery mechanism. Most modern microservices on Kubernetes skip Eureka entirely and use K8s Services + Spring Cloud Kubernetes.

Must Know
What is the difference between client-side and server-side service discovery?
▾
Client-side discovery: the calling service queries the registry (Eureka), receives a list of available instances, picks one using a load-balancing algorithm (round-robin, least-connections, zone-aware), and calls that instance directly. Spring Cloud LoadBalancer does this. Advantage: client has full control. Disadvantage: every service needs discovery library code.

Server-side discovery: the client calls a fixed virtual address. A load balancer or service proxy (Kubernetes Service, AWS ALB, Nginx) handles the routing to a healthy backend instance. The client is unaware of discovery. Advantage: clients are simpler. Disadvantage: depends on infrastructure.
💡 Modern practice: On Kubernetes, use server-side discovery via K8s Services — it's built into the platform. Use Eureka only when Kubernetes isn't available or for cross-cluster discovery.
Tricky
What happens if the Eureka server goes down?
▾
Eureka clients cache the registry locally. If the Eureka server becomes unavailable, clients continue using their last known registry snapshot to make calls. This is intentional — Eureka favours availability over consistency (AP in CAP theorem). Clients won't discover new instances until connectivity restores, but existing instances remain reachable.

Self-preservation: if the server loses heartbeats from most clients, it assumes a network issue rather than mass failure and stops evicting registrations. This prevents cascade: the server going down causes clients to miss heartbeats, causing mass deregistration, causing routing failures — Eureka prevents this spiral. Deploy Eureka as a cluster (2–3 nodes) for high availability in production.

Sidecar Pattern & Service Mesh
Move cross-cutting concerns out of your application code and into the infrastructure layer
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: A sidecar is a helper container that runs alongside your main service container in the same pod. It handles tasks your service shouldn't need to care about: mTLS encryption, retry logic, metrics collection, distributed tracing. A Service Mesh is what you get when you deploy sidecars to every service — they form a network of proxies that collectively handle all service-to-service communication infrastructure.
🏍️
Sidecar Pattern — Deploy a secondary container alongside your main application container in the same pod. They share the same network namespace (same localhost). The sidecar intercepts all inbound and outbound traffic. Your service code is completely unaware — it talks to localhost and the sidecar handles the rest. Envoy proxy (used by Istio) is the most common sidecar implementation.
🕸️
Service Mesh — What It Provides — When every service has a sidecar proxy, the sidecars collectively form a mesh. The mesh provides: mutual TLS (mTLS) between all services automatically, distributed tracing without code changes, retries and circuit breaking at the infrastructure level, traffic shaping (canary deployments, A/B testing by percentage), and policy-based access control. Istio and Linkerd are the leading service meshes.
🔑
Mutual TLS (mTLS) — Service-to-service calls are encrypted AND both sides authenticate each other. Each sidecar gets a certificate tied to its service identity. When Service A calls Service B, the sidecar handles certificate exchange transparently. Your Java code makes a plain HTTP call to localhost and gets mTLS for free. Even if someone intercepts the internal network traffic, they can't read or spoof it.
📊
Observability for Free — Because all traffic passes through sidecars, the mesh automatically generates metrics (request rate, error rate, latency per service pair), distributed traces (spans for every hop), and access logs. No instrumentation changes to your Java code. The "Golden Signals" (traffic, errors, latency, saturation) are automatically available for every service in the mesh via Prometheus + Grafana.
⚖️
Service Mesh Trade-offs — Complexity: Istio adds several control plane components (istiod, gateway controllers) and sidecars to every pod, doubling your container count. Latency: every request passes through an additional proxy hop (typically adds 1–5ms). Operational overhead: the mesh itself must be monitored and upgraded. Consider service mesh only when you have 10+ services and genuinely need the features — don't add it prematurely.

What does a Service Mesh give you that Spring Cloud doesn't?

• mTLS by default — encrypted & authenticated service-to-service calls, impossible to implement in all services consistently with Spring Cloud
• Zero-code observability — every request traced and metered automatically
• Language-agnostic — works the same for Java, Python, Go, Ruby services in the same cluster
• Traffic shaping — canary deployments, A/B testing, fault injection for chaos testing
⚡ The key insight: Service Mesh moves cross-cutting concerns from code (Spring Cloud) to infrastructure (Kubernetes + Istio). Code stays simple; infrastructure gets the complexity.
Senior
When is a Service Mesh overkill and when is it worth the complexity?
▾
Overkill when: fewer than 10 services (overhead not worth it), small team (no one has Istio expertise), all services are Java/Spring (Spring Cloud handles it without infrastructure complexity), or traffic patterns are simple (no need for canary deployments or fine-grained traffic control).

Worth it when: polyglot environment (multiple languages need the same resilience and mTLS without per-language libraries), strict security requirements (mTLS everywhere mandated), complex traffic routing needed (canary, A/B, shadowing), compliance requires full network audit logging, or platform team wants to standardise cross-cutting concerns across dozens of teams' services.

Consider Linkerd as a lighter alternative to Istio — simpler to operate with lower overhead, fewer features but covers 80% of use cases.
💡
Sidecar = a helper running next to your service that handles the boring bits.
Your service talks to localhost. The sidecar intercepts, encrypts (mTLS), retries, measures, and forwards — invisibly.

Service Mesh = sidecar on every service = a smart network that knows your services.

What you get for free: mTLS everywhere, distributed tracing without code changes, retries at the network level, canary deployments by splitting traffic.

When to add it: 10+ services, polyglot stack, or strict security/compliance requirements. Don't add it on day one.

Resilience Pattern
Bulkhead & Rate Limiting
Isolate failures and protect capacity — don't let one bad tenant or one slow service consume all your resources
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: A ship's hull is divided into watertight compartments — if one compartment floods, the others stay dry and the ship stays afloat. The Bulkhead pattern applies the same idea to thread pools: give each downstream dependency its own thread pool. If the payment service is slow and exhausts its pool of 10 threads, the remaining 190 threads still serve other requests. Rate Limiting protects the other direction — limiting how fast callers can consume your service.
🚢
Bulkhead Pattern — Allocate separate, limited resource pools (thread pools, connection pools, semaphores) per downstream dependency. Without bulkheads, your service has one shared thread pool. If payment service is slow, payment calls pile up and consume all threads — nothing else can be processed. With bulkheads, payment has its own pool of 10 threads. Slow payments exhaust those 10 threads, but 190 threads remain for order, user, and inventory calls.
🚦
Rate Limiting — Algorithms — Token Bucket: a bucket fills with tokens at a constant rate. Each request takes a token. When the bucket is empty, requests are rejected. Allows bursts up to bucket capacity. Leaky Bucket: requests queue up; the queue drains at a constant rate. Smooths out bursts. Fixed Window: count requests per time window. Simple but allows double the limit at window boundaries. Sliding Window: more accurate — counts requests over a rolling window. Redis is the standard backend for distributed rate limiting.
🔒
Bulkhead via Semaphore vs ThreadPool — Semaphore bulkhead: limits concurrent calls using a counting semaphore. Simpler, no extra threads, good for non-blocking (reactive) code. ThreadPool bulkhead: each dependency gets its own dedicated thread pool. Caller's thread returns immediately; work happens in the pool. Better isolation but more resource overhead. Resilience4j supports both via @Bulkhead(type = SEMAPHORE) or THREADPOOL.
📏
Sizing the Bulkhead — Thread pool size formula (Little's Law): pool_size = throughput × latency. If the payment service handles 50 requests/second and each takes 200ms, you need 50 × 0.2 = 10 threads to keep up. Add 20% headroom: 12 threads. Keep the pool small enough that exhaustion happens quickly and clearly (fast failure) rather than slowly stacking up.
⏱️
Time Limiter — A companion to bulkheads: cancel requests that take longer than a set timeout. Prevents slow calls from permanently holding pool resources. Combine: Bulkhead limits concurrent calls, TimeLimiter cancels calls that run too long. Together they ensure the bulkhead pool is never indefinitely blocked by a few stuck requests — making the fast-failure guarantee reliable.
Resilience4j Bulkhead + Rate Limiter
// application.yml
// resilience4j:
//   bulkhead:
//     instances:
//       paymentService:
//         maxConcurrentCalls: 10        # max 10 concurrent payment calls
//         maxWaitDuration: 100ms        # wait up to 100ms for a slot
//   thread-pool-bulkhead:
//     instances:
//       inventoryService:
//         maxThreadPoolSize: 8
//         coreThreadPoolSize: 4
//         queueCapacity: 20
//   ratelimiter:
//     instances:
//       api:
//         limitForPeriod: 100           # 100 calls per period
//         limitRefreshPeriod: 1s        # per second
//         timeoutDuration: 0           # fail immediately if limit reached

// ── Annotation-based ─────────────────────────────
@Service
public class PaymentClient {

    @Bulkhead(name = "paymentService", type = Bulkhead.Type.SEMAPHORE,
              fallbackMethod = "fallback")
    @CircuitBreaker(name = "paymentService")
    @TimeLimiter(name = "paymentService")
    public CompletableFuture<PaymentResult> charge(PaymentRequest req) {
        return CompletableFuture.supplyAsync(() -> paymentApi.charge(req));
    }

    // Bulkhead full → return queued/degraded response
    public CompletableFuture<PaymentResult> fallback(
            PaymentRequest req, BulkheadFullException ex) {
        log.warn("Payment bulkhead full — queuing for retry");
        return CompletableFuture.completedFuture(PaymentResult.queued());
    }
}

@GetMapping("/search")
@RateLimiter(name = "api", fallbackMethod = "searchRateLimited")
public List<Product> search(@RequestParam String q) {
    return productService.search(q);
}

public List<Product> searchRateLimited(String q, RequestNotPermitted ex) {
    throw new ResponseStatusException(HttpStatus.TOO_MANY_REQUESTS,
        "Rate limit exceeded. Retry after 1s.");
}

// Outermost → Innermost (execution order):
// @RateLimiter → @Bulkhead → @CircuitBreaker → @Retry → @TimeLimiter → actual call
// Rate limit first (cheapest). Call last. Fail fast at each stage.
Must Know
What is the Bulkhead pattern and how does it prevent cascade failures?
▾
The Bulkhead pattern isolates failures by giving each downstream dependency its own resource pool. A shared thread pool is vulnerable: if Payment service slows down, payment calls pile up consuming all threads. Other calls (user lookup, order saving) queue behind them and also slow down — the entire service is degraded by one slow dependency.

With bulkheads: payment has 10 threads, inventory has 8, user has 12. When payment is slow and its 10 threads fill up, new payment calls fail fast (BulkheadFullException) and the remaining 18 threads continue serving other requests. The failure is contained — it doesn't spread. Your service degrades gracefully: payment is slow but users can still browse, add to cart, and view orders.
⚡ Bulkhead = firewall for thread pools. Failures in one pool can't consume resources from other pools.
Tricky
What is the token bucket algorithm for rate limiting and why is it preferred?
▾
A token bucket works like a bucket that fills with tokens at a fixed rate (e.g., 10 tokens/second). Each request consumes one token. If the bucket has tokens, the request is allowed. If the bucket is empty, the request is rejected (or queued). The bucket has a maximum capacity — excess tokens overflow.

Why preferred: it allows short bursts up to the bucket capacity while enforcing a long-term average rate. If a service allows 10 req/s but a client has been quiet for 5 seconds, the bucket has accumulated 50 tokens and they can send a burst of 50 requests immediately — which is fair. Fixed window algorithms would deny those 50 because "50 > 10 per second" without considering the quiet period.
💡 Spring Cloud Gateway rate limiter uses the Token Bucket algorithm with Redis as the state backend — enabling distributed rate limiting across all gateway instances.
Senior
How do you apply all Resilience4j patterns together? What is the correct order?
▾
When stacking annotations, they execute from outermost to innermost: @RateLimiter → @Bulkhead → @CircuitBreaker → @Retry → @TimeLimiter → actual call.

Reasoning: Rate Limiter first — cheapest check, no dependencies. If over limit, fail immediately before consuming any resources. Bulkhead second — check if there's capacity. Circuit Breaker third — if the downstream is known-broken, fail immediately without taking a slot. Retry inside Circuit Breaker — retries count toward the circuit breaker's failure rate. TimeLimiter innermost — caps how long the actual call runs.

Wrong order: putting Retry outside Circuit Breaker would retry even when the circuit is open — wasting resources. Always think about what the outer wrapper sees from the inner wrapper's perspective.
💡
Bulkhead = watertight compartments on a ship. One compartment floods (payment is slow), the rest stay dry (other services keep working).

Without Bulkhead: One slow dependency → all threads stuck → entire service fails.
With Bulkhead: One slow dependency → its own thread pool exhausts → other dependencies unaffected.

Rate Limiting algorithms:
• Token Bucket — allows bursts, best for APIs
• Leaky Bucket — smooths bursts, best for constant-rate services
• Fixed Window — simple but vulnerable at window edges
• Sliding Window — most accurate, higher overhead

Stacking order: RateLimit → Bulkhead → CircuitBreaker → Retry → TimeLimiter.

🌱
Why Strangler Fig, Not Big Bang Rewrite — A full rewrite is high-risk: you spend months rebuilding, the business keeps adding features to the monolith, and when you finally deploy the rewrite, it's already behind. The strangler approach: extract one domain at a time, deploy it alongside the monolith, validate it works with real traffic, then move on to the next. The system is live and valuable at every stage — no big-bang deployment risk.
🚦
The Facade Layer — An API gateway or reverse proxy sits in front of both the monolith and the new services. It routes based on path, header, or feature flags. /api/orders/** → new Order Service. /api/* → monolith. The client never knows which backend serves the request. The facade is the key to safe, gradual migration without client-side changes.
🔀
Dark Launching / Shadow Mode — An advanced technique during migration: send real traffic to BOTH the monolith and the new service simultaneously. The monolith's response is returned to the client. The new service's response is compared against the monolith's response silently. Mismatches are logged and investigated. This validates the new service against production traffic before switching over — zero user impact during validation.
🗄️
Database Strangling — The Hard Part — The monolith and new service initially share the same database (easiest to start). As the service matures, you need to extract its data: (1) Synchronise — write to both schemas simultaneously. (2) Validate — verify new schema has correct data. (3) Migrate — switch reads to new DB. (4) Decouple — new service owns its data, monolith calls the service's API for that data. Data decoupling takes the longest and is the most complex phase.
📋
Extraction Prioritisation — Which parts to extract first: (1) Features with high change rate (painful to deploy because they require monolith deployment). (2) Features with clear boundaries and minimal shared state. (3) Features that need different scalability characteristics. (4) Features owned by a team that could deliver the service independently. Avoid extracting tightly coupled, cross-cutting logic early.

Must Know
What is the Strangler Fig pattern and why is it safer than a big-bang rewrite?
▾
Named after the strangler fig tree that grows around another tree and gradually replaces it. Applied to software: instead of rewriting the entire monolith at once (big bang), you extract one capability at a time into a microservice while keeping everything else running.

Why safer: (1) The system stays live and revenue-generating throughout — no extended code freeze. (2) You validate each extraction with real traffic before moving to the next. (3) If an extraction goes wrong, only that small piece is affected — the rest of the system still runs. (4) The team learns microservices patterns gradually rather than being overwhelmed. A big-bang rewrite often fails because the target is moving (business keeps adding features) and when finally deployed, it's already wrong and behind.
⚡ Martin Fowler popularised this pattern specifically because so many big-bang rewrites fail. "Never rewrite everything at once" is a core piece of advice in enterprise architecture.
Tricky
How do you handle the shared database during Strangler Fig migration?
▾
The shared database is the hardest part. Immediate separation is too risky. A safe incremental approach:

Phase 1 — Share the database: New service reads/writes the monolith's database tables directly. Simple but creates tight coupling.
Phase 2 — Add new tables: New service gets its own schema/tables alongside the monolith's. Dual-write to both.
Phase 3 — Validate: Run both schemas in parallel, compare data for correctness. Fix discrepancies.
Phase 4 — Switch reads: New service reads from its own tables, stops reading monolith tables.
Phase 5 — Stop dual write: New service only writes to its own tables. Monolith calls the service's API when it needs that data.
Phase 6 — Remove old tables: Delete monolith's now-unused tables for the extracted domain.
Senior
What is dark launching and how does it validate the new service against production traffic?
▾
Dark launching (shadow mode) sends every real request to BOTH the old system and the new service simultaneously. The client always receives the response from the old system (production). The new service processes the request but its response is compared internally to the old system's response, and any differences are logged.

This means: zero risk to users — they always get the proven old response. The new service processes real, representative production traffic rather than just test cases. Differences reveal bugs, missing data handling, edge cases, or performance issues before you switch over.

Implementation: the gateway duplicates the request, sends both calls, waits for the canonical response, returns it to client, and asynchronously compares the shadow response in the background. Mismatches are tracked as metrics and logged for investigation. Only when mismatch rate reaches 0% (or acceptable threshold) do you switch the facade to route clients to the new service.

When would you NOT use the Strangler Fig pattern?
▾
Strangler Fig requires a facade that can route traffic — which requires at least some infrastructure investment upfront. It's also inherently slower: you're maintaining two systems simultaneously.

Cases where it may not apply: (1) Greenfield project — no legacy to migrate from, just build microservices correctly from the start. (2) Legacy system is too tangled — if every function touches every other function with no clear boundaries, extracting one part is nearly impossible without the whole thing falling over. May need to modularise the monolith first (extract packages, clean up internal APIs) before you can do Strangler Fig. (3) Infrastructure doesn't support a facade — some very old systems have no way to put a proxy in front. (4) The monolith is working well enough — migration has a real cost; if the monolith isn't causing pain, don't fix what isn't broken. Evaluate ROI honestly.
💡
Strangler Fig = wrap the old system, extract piece by piece, retire when empty.

The three steps:
1. Put a facade (gateway/proxy) in front of the monolith
2. Extract one capability → new service → route a percentage of traffic there
3. Gradually increase traffic until monolith handles 0% of that capability
4. Repeat for the next capability until the monolith is empty

Dark launching: send real traffic to BOTH. Return old response. Compare new response silently. Switch only when differences = 0%.

Database last: share first, dual-write second, migrate reads, then writes, then delete old tables.
