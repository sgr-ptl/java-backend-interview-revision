◆ Core Concepts
📦
What is a Starter? — A Spring Boot starter is a convenience dependency descriptor — a single Maven/Gradle dependency that pulls in a curated set of transitive dependencies needed for a specific feature. No more hunting for compatible library versions. spring-boot-starter-web brings in Spring MVC, Tomcat, Jackson, validation — all at compatible versions.
🧩
Naming Convention — Official starters follow: spring-boot-starter-{feature}. Community/third-party starters: {name}-spring-boot-starter. Key ones: starter-web (MVC + Tomcat), starter-webflux (reactive), starter-data-jpa (JPA + Hibernate), starter-security, starter-test, starter-actuator, starter-cache, starter-validation.
🎯
spring-boot-starter-parent — The parent POM that provides: default dependency versions (via spring-boot-dependencies BOM), default plugin configurations, UTF-8 encoding, Java version defaults. Inherit it and you get consistent dependency management across the entire Spring ecosystem. Alternatively use spring-boot-dependencies BOM via dependencyManagement if you already have a parent POM.
🔧
Overriding Versions — Starter versions are managed by the parent BOM. Override a specific version via a property in pom.xml: <jackson.version>2.17.0</jackson.version>. Or exclude a transitive dependency and add your preferred version explicitly. Spring Boot's version management covers 300+ libraries.
🚀
spring-boot-maven-plugin / Gradle Plugin — Packages your app as an executable "fat JAR" (JAR with all dependencies embedded) or a WAR. The plugin also enables: mvn spring-boot:run for dev, spring-boot:build-image (Buildpacks for Docker), layered JAR for optimized Docker caching, properties encryption. Essential for deployment.

◆ Interview Questions
Must Know
What is a Spring Boot Starter and what problem does it solve?
▾
Before Spring Boot, adding web support meant manually finding and reconciling a dozen compatible libraries (Spring MVC, Servlet API, Jackson, logging, validation...) — version conflicts were common. A Starter is a single POM descriptor that bundles the right set of dependencies at tested, compatible versions. spring-boot-starter-web gives you everything for a web app in one line. The parent BOM manages all version numbers, so you never need to specify versions for Spring-managed libraries.
⚡ Starters contain zero actual code — they're just dependency descriptors (pom.xml only). All real code is in the libraries they pull in.
Tricky
How does a "fat JAR" work and how is it different from a regular JAR?
▾
A standard JAR contains only the compiled classes of your application. A fat (executable) JAR contains your classes PLUS all dependencies embedded inside. Spring Boot's plugin creates a nested JAR structure: your classes in BOOT-INF/classes, dependencies in BOOT-INF/lib, and a custom JarLauncher as the manifest's main class. The launcher sets up a classloader that reads from the nested JARs, then starts your application. This makes the JAR fully self-contained — run with just java -jar app.jar, no classpath needed.
⚡ Layered JARs (Spring Boot 2.3+) split the JAR into layers (dependencies, application, snapshot-dependencies) for Docker layer caching — unchanged layers are cached, only changed layers rebuild.
Tricky
What is the difference between spring-boot-starter-parent and spring-boot-dependencies?
▾
Both manage dependency versions via a BOM, but differ in usage context: spring-boot-starter-parent is a parent POM — inherits from it to get dependency management PLUS plugin configurations (compiler settings, encoding, resource filtering, Spring Boot plugin). spring-boot-dependencies is a BOM (Bill of Materials) only — used in <dependencyManagement> with scope=import when you can't use Spring Boot as parent (because you already have your own corporate parent POM). You get version management but not plugin configs.
💡 Use spring-boot-dependencies when corporate policy mandates a different parent POM.
◆ Memory Tip
💡
Starter = Meal Deal. Instead of ordering chips, sandwich, and drink separately (and hoping they're compatible), you pick the Meal Deal — everything packaged together at the right combination. spring-boot-starter-parent is like the restaurant menu that lists all Meal Deals and their prices (versions).

Naming rule: Official = spring-boot-starter-{thing}. Third-party = {thing}-spring-boot-starter.

⚙️
@EnableAutoConfiguration — Reads the list of auto-configuration classes from META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (Boot 2.7+; previously spring.factories). Each class in the list is a candidate @Configuration — Spring evaluates its @Conditional annotations to decide whether to apply it.
🔀
@Conditional Annotations — The engine behind auto-configuration:
• @ConditionalOnClass — apply if a class is on classpath
• @ConditionalOnMissingBean — apply only if you haven't defined your own bean
• @ConditionalOnProperty — apply if a property is set
• @ConditionalOnWebApplication — apply in web context
• @ConditionalOnMissingClass, @ConditionalOnResource, @ConditionalOnExpression
👤
User Beans Override Auto-config — @ConditionalOnMissingBean means: if you define your own DataSource bean, Spring Boot's auto-configured one won't be created. Your explicit configuration always wins over auto-configuration. This is how you customize — define your own bean and auto-config steps back.
🔍
Debug Auto-configuration — Add --debug flag or debug=true in application.properties to see a Conditions Evaluation Report — shows which auto-configurations were applied (Positive matches), which were skipped (Negative matches), and why. Invaluable for diagnosing unexpected behavior.
🚫
Excluding Auto-configuration — @SpringBootApplication(exclude = {DataSourceAutoConfiguration.class}) or via property: spring.autoconfigure.exclude=org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration. Use when you want full manual control or when auto-config conflicts with your setup.
◆ Memorizable Code
Auto-config — @Conditional + Custom AutoConfiguration
// ── What @SpringBootApplication actually is ───────
@Target(ElementType.TYPE)
@SpringBootConfiguration      // = @Configuration
@EnableAutoConfiguration      // ← triggers auto-config magic
@ComponentScan                // scans this package + sub-packages
public @interface SpringBootApplication {}

// ── How DataSourceAutoConfiguration looks internally ─
@AutoConfiguration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
@ConditionalOnMissingBean(type = "io.r2dbc.spi.ConnectionFactory")
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean   // ← backs off if YOU define DataSource
    public DataSource dataSource(DataSourceProperties props) {
        return props.initializeDataSourceBuilder().build();
    }
}

// ── Writing your own AutoConfiguration ───────────
@AutoConfiguration
@ConditionalOnClass(MyLibraryClient.class)       // only if lib present
@ConditionalOnProperty(prefix = "mylibrary",
    name = "enabled", havingValue = "true", matchIfMissing = true)
@EnableConfigurationProperties(MyLibraryProperties.class)
public class MyLibraryAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean   // user can override
    public MyLibraryClient myClient(MyLibraryProperties props) {
        return new MyLibraryClient(props.getUrl(), props.getTimeout());
    }
}
// Register in: META-INF/spring/
//   org.springframework.boot.autoconfigure.AutoConfiguration.imports
// com.example.MyLibraryAutoConfiguration

// ── Exclude specific auto-config ──────────────────
@SpringBootApplication(exclude = {
    DataSourceAutoConfiguration.class,   // no DB auto-config
    SecurityAutoConfiguration.class        // no auto security
})
public class App { public static void main(String[] a){SpringApplication.run(App.class,a);} }

// ── application.properties debug ─────────────────
// debug=true   → prints ConditionEvaluationReport
// or run with: java -jar app.jar --debug

// ── @ConfigurationProperties ──────────────────────
@ConfigurationProperties(prefix = "app.cache")
public class CacheProperties {
    private int ttlSeconds = 300;
    private int maxSize    = 1000;
    // getters + setters...
}

// app.cache.ttl-seconds: 600   (Boot converts kebab-case to camelCase)

◆ Interview Questions
Must Know
How does Spring Boot auto-configuration work? Walk through the mechanism.
▾
Step by step: 1. @SpringBootApplication includes @EnableAutoConfiguration. 2. Spring Boot reads META-INF/spring/AutoConfiguration.imports — a list of hundreds of auto-config class names. 3. Each listed class is evaluated — Spring checks its @ConditionalOnClass, @ConditionalOnMissingBean, @ConditionalOnProperty etc. 4. If all conditions pass, the @Configuration class is applied and its beans are registered. 5. @ConditionalOnMissingBean ensures user-defined beans always take priority — auto-config is a default, not a mandate.
⚡ Key insight: Auto-configuration is "opt-out" not "opt-in". It applies unless you override it. Your beans always win.
Tricky
What is the difference between @Configuration and @AutoConfiguration?
▾
@Configuration: a regular Spring configuration class, processed normally during component scanning. @AutoConfiguration (Spring Boot 2.7+): specifically marks a class as an auto-configuration — it's loaded from AutoConfiguration.imports (not component scan), processed after all user-defined @Configuration classes (ensuring user beans are available for @ConditionalOnMissingBean checks), and uses @AutoConfigureBefore/@AutoConfigureAfter for ordering. Don't scan @AutoConfiguration classes directly — they must be listed in the imports file.
⚠️ Never put @AutoConfiguration classes in packages that are component-scanned — they're designed to load via the imports mechanism only.
Tricky
How do you customize auto-configured beans? What approaches exist?
▾
3 approaches in order of preference:
1. Properties: Most auto-configs expose @ConfigurationProperties — set spring.datasource.hikari.maximum-pool-size=20. Simplest, most maintainable.
2. Define your own bean: @ConditionalOnMissingBean means your @Bean DataSource completely replaces the auto-configured one. Full control.
3. Customize via *Customizer interfaces: Spring Boot provides customizer interfaces (e.g., WebMvcConfigurer, HttpMessageConvertersCustomizer) — implement and return a bean to modify rather than replace the auto-configured object.
⚡ Always prefer properties first. Only define your own bean if properties don't give enough control.
Concept
What is @ConfigurationProperties and how does it differ from @Value?
▾
@Value("${property.key}"): injects one property at a time into a field. Simple but scattered across code, no type safety for complex types, doesn't support relaxed binding, no IDE completion.
@ConfigurationProperties(prefix="app.cache"): binds an entire prefix of properties to a Java class — type-safe, supports nested objects, collections, and maps, enables IDE autocompletion (with spring-boot-configuration-processor), supports relaxed binding (kebab-case, underscore, SCREAMING_SNAKE). Always prefer @ConfigurationProperties for groups of related config.
◆ Memory Tip
💡
Auto-configuration = polite assistant. It sets up everything for you, but as soon as you say "I'll handle that myself" (define your own bean), it steps aside. It never overrides your decisions — @ConditionalOnMissingBean is the "don't duplicate your boss" rule.

@Conditional cheat:
• OnClass = "is the library even there?"
• OnMissingBean = "did the user define their own?"
• OnProperty = "is this feature turned on?"
• OnWebApplication = "are we in a web context?"

◆ Core Concepts
🏗️
Embedded Server Concept — Traditionally, you deployed a WAR file to an external Tomcat/JBoss server. Spring Boot embeds the server inside the JAR — the servlet container is a dependency, not infrastructure. Your main() method starts everything. Run with java -jar app.jar. Simpler deployment, container configuration in code, reproducible environments.
🐱
Tomcat (Default) — Apache Tomcat is the default in spring-boot-starter-web. Servlet-based, blocking I/O model, mature and battle-tested. Thread-per-request model — each request gets a thread from the pool (default 200). Best for standard REST APIs and web apps. Configure with server.tomcat.* properties.
⛵
Jetty — Lightweight alternative to Tomcat. Smaller footprint, slightly lower memory usage. Good for microservices where resource efficiency matters. Swap by excluding Tomcat starter and adding Jetty starter. Also supports async and WebSocket.
🌊
Undertow — JBoss/Red Hat's server. Non-blocking core with excellent performance for high-concurrency scenarios. Smaller memory footprint than Tomcat. Used in WildFly/JBoss. Good choice for high-throughput APIs.
⚡
Netty (WebFlux) — When you use spring-boot-starter-webflux instead of starter-web, the default server becomes Netty (not a Servlet container). Fully non-blocking, event-loop based (like Node.js). Required for reactive programming with Project Reactor. Not compatible with Servlet API.
📋
WebApplicationContext Types — SERVLET (Tomcat/Jetty/Undertow — standard), REACTIVE (Netty — WebFlux), NONE (no web server — for CLI apps, batch jobs). Spring Boot auto-detects based on classpath: if both are present, SERVLET wins.

// pom.xml:
// <dependency>
//   <artifactId>spring-boot-starter-web</...>

        implements WebServerFactoryCustomizer<ConfigurableTomcatWebServerFactory> {

    @Override
    public void customize(ConfigurableTomcatWebServerFactory factory) {
        factory.setPort(9090);
        factory.addConnectorCustomizers(connector -> {
            connector.setMaxPostSize(10 * 1024 * 1024); // 10MB
        });
    }
}

◆ Interview Questions
Must Know
What is the advantage of an embedded server over a traditional external server deployment?
▾
5 key advantages: 1. Self-contained deployment — one JAR, no separate server installation. 2. Dev/Prod parity — same server config in all environments, no "works on my Tomcat" issues. 3. Simpler CI/CD — build the JAR, run the JAR, done. 4. Programmatic configuration — configure the server in code instead of XML config files. 5. Cloud-native — ideal for Docker containers and Kubernetes pods. Downside: slightly larger artifact, less suited for classic Java EE enterprise environments that expect WAR deployments.
💡 You can still deploy as WAR to external Tomcat when corporate requirements demand it — just extend SpringBootServletInitializer.
Tricky
What is Tomcat's max-threads setting? What happens when all threads are busy?
▾
Tomcat uses a thread-per-request model. max-threads (default 200) is the maximum number of concurrent requests that can be processed simultaneously. When all threads are busy: new connections queue in the accept-count queue (default 100). If the queue is also full, new connections are refused with a connection refused error. Tuning: for I/O-bound workloads (DB queries, HTTP calls), increase threads (threads spend most time waiting). For CPU-bound workloads, keep threads ≈ CPU cores. Too many threads = excessive context switching. Too few = requests queuing. Monitor with /actuator/metrics/tomcat.threads.busy.
⚡ With virtual threads (Java 21 + Spring Boot 3.2), set spring.threads.virtual.enabled=true — Tomcat uses virtual threads, effectively unlimited concurrency without tuning max-threads.
Tricky
When would you use WebFlux (Netty) over the standard servlet stack (Tomcat)?
▾
Use WebFlux (Netty/reactive) when: 1. High concurrency with I/O-heavy operations — reactive handles thousands of concurrent connections with fewer threads. 2. Streaming data — Server-Sent Events, streaming responses. 3. Calling many downstream services concurrently — compose async calls naturally with Mono/Flux. 4. All dependencies support reactive (R2DBC, reactive MongoDB). Use Servlet (Tomcat) when: 1. Team is unfamiliar with reactive programming. 2. Dependencies don't support reactive. 3. Standard CRUD app — blocking Tomcat + virtual threads (Java 21) gives comparable throughput with simpler code.
⚡ Don't mix blocking code in a reactive stack — it blocks the event loop and defeats the purpose. If you must use blocking dependencies, use Tomcat.
◆ Memory Tip
💡
Embedded server = restaurant with a built-in kitchen. You don't need to find a commercial kitchen (external Tomcat) and get approval to use it. Your restaurant (app) comes with its own kitchen (server) — opens anywhere, runs anywhere.

Server selection: Most apps → Tomcat (default). High-throughput I/O → Undertow. Reactive streaming → Netty (WebFlux). Footprint matters → Jetty. Java 21 virtual threads → Tomcat + spring.threads.virtual.enabled=true (best of both worlds).

// ── Custom Health Indicator ───────────────────────
@Component
public class ThirdPartyApiHealthIndicator implements HealthIndicator {

    private final ThirdPartyClient client;

    @Override
    public Health health() {
        try {
            String status = client.ping();
            return Health.up()
                .withDetail("status", status)
                .withDetail("latencyMs", client.getLastLatency())
                .build();
        } catch (Exception e) {
            return Health.down()
                .withDetail("error", e.getMessage())
                .build();
        }
    }
}
// Shows at: /actuator/health/thirdPartyApi

// ── Custom Metric with Micrometer ────────────────
@Service
public class OrderService {
    private final Counter orderCounter;
    private final Timer orderTimer;

    public OrderService(MeterRegistry registry) {
        orderCounter = Counter.builder("orders.placed.total")
            .tag("app", "order-service")
            .description("Total orders placed")
            .register(registry);

        orderTimer = Timer.builder("orders.processing.duration")
            .description("Order processing time")
            .register(registry);
    }

    public Order placeOrder(OrderRequest req) {
        return orderTimer.record(() -> {
            Order order = processOrder(req);
            orderCounter.increment();
            return order;
        });
    }
}

◆ Interview Questions
Must Know
What is Spring Boot Actuator and what does it provide out of the box?
▾
Actuator adds production-ready features to Spring Boot apps via the spring-boot-starter-actuator dependency. Out of the box: Health checks (aggregated from DB, disk, cache, custom indicators), Metrics (JVM memory/GC/threads, HTTP request counts/durations, DB connection pool, custom via Micrometer), Environment inspection (all properties, profiles active), Bean listing, Request mapping exploration, Runtime log level changes, Thread dumps, and Heap dumps. All accessible via HTTP endpoints (or JMX). Zero lines of code needed.
Tricky
How do Kubernetes liveness and readiness probes use Actuator health endpoints?
▾
Enable K8s probes with management.endpoint.health.probes.enabled=true (auto-enabled when K8s is detected). Liveness probe → /actuator/health/liveness: answers "is this process healthy enough to continue running?" If FAIL → K8s kills and restarts the pod. Use for: app deadlocked, OOM, non-recoverable states. Readiness probe → /actuator/health/readiness: answers "is this pod ready to receive traffic?" If FAIL → K8s removes it from the load balancer (no kill). Use for: startup not complete, DB disconnected, briefly overloaded. Never fail liveness on temporary issues (DB restart) — that kills your pod unnecessarily.
⚡ Liveness = restart. Readiness = remove from rotation. Use them for different purposes!
Tricky
What is Micrometer and how does it relate to Actuator metrics?
▾
Micrometer is the metrics instrumentation facade for JVM applications — like SLF4J for metrics. Spring Boot Actuator uses Micrometer internally. You instrument your code with Micrometer's API (Counter, Timer, Gauge, DistributionSummary) and Micrometer adapts the output to your chosen backend via a registry implementation. Add micrometer-registry-prometheus → metrics export to Prometheus format at /actuator/prometheus. Swap to micrometer-registry-datadog → metrics push to Datadog. Your instrumentation code stays the same.
Concept
How do you secure Actuator endpoints in production?
▾
3-layer approach: 1. Expose only what you need: management.endpoints.web.exposure.include=health,info,prometheus. Never include env, heapdump, shutdown in production unless absolutely necessary. 2. Separate management port: management.server.port=9090 — reachable only from internal network/monitoring tools, never from the internet. 3. Spring Security: add requestMatchers("/actuator/**").hasRole("ADMIN") or hasIpAddress("10.0.0.0/8") for management port. For /health details: show-details: when-authorized — anonymous gets UP/DOWN, authenticated gets full detail.
◆ Memory Tip
💡
Actuator = your app's dashboard and diagnostic tools, all at HTTP endpoints.

K8s probe memory trick:
• Liveness = "Am I alive?" → FAIL = restart (use for dead states)
• Readiness = "Am I ready?" → FAIL = pause traffic (use for temporary issues)

Production exposure rule: if in doubt, leave it out. Default expose only health and info. Add others on internal port only.

🎯
DispatcherServlet — The Front Controller — single entry point for all HTTP requests. It delegates to specialized components instead of handling everything itself. Auto-configured by Spring Boot as a bean mapped to /. There can be multiple DispatcherServlets (each with its own context) but Spring Boot creates one by default.
🗺️
HandlerMapping — Determines which handler (controller method) should process the request based on URL, HTTP method, headers, and params. Multiple implementations: RequestMappingHandlerMapping (annotation-based, most common), BeanNameUrlHandlerMapping, RouterFunctionMapping (functional). Returns a HandlerExecutionChain (handler + interceptors).
🔌
HandlerAdapter — Invokes the handler found by HandlerMapping. Handles argument resolution (@PathVariable, @RequestBody, @RequestParam), method return value handling (@ResponseBody, ModelAndView), and validation. RequestMappingHandlerAdapter handles @RequestMapping methods.
🔄
HandlerInterceptor vs Filter — Both intercept requests but at different layers. Filter (Java Servlet): runs before DispatcherServlet, operates on raw HttpServletRequest — used for auth, CORS, rate limiting. HandlerInterceptor (Spring MVC): runs inside DispatcherServlet after handler is resolved — has access to Spring context, handler method, ModelAndView — used for auditing, locale, security at MVC level.
📤
HttpMessageConverter — Converts between Java objects and HTTP request/response body. MappingJackson2HttpMessageConverter handles JSON, StringHttpMessageConverter handles plain text, ByteArrayHttpMessageConverter handles binary. Content negotiation: client sends Accept: application/json, Spring picks the right converter. Customizable via WebMvcConfigurer.

// ── Servlet Filter (before DispatcherServlet) ─────
@Component
@Order(1)
public class RequestLoggingFilter implements Filter {

    @Override
    public void doFilter(ServletRequest req, ServletResponse res,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpReq = (HttpServletRequest) req;
        log.info("→ {} {}", httpReq.getMethod(), httpReq.getRequestURI());
        long start = System.currentTimeMillis();
        chain.doFilter(req, res);             // proceed
        log.info("← {}ms", System.currentTimeMillis() - start);
    }
}

// ── HandlerInterceptor (inside DispatcherServlet) ─
@Component
public class AuditInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest req,
                             HttpServletResponse res,
                             Object handler) {
        if (handler instanceof HandlerMethod hm) {
            log.info("Calling: {}.{}",
                hm.getBeanType().getSimpleName(),
                hm.getMethod().getName());
        }
        return true; // return false to abort request
    }

    @Override
    public void postHandle(HttpServletRequest req, HttpServletResponse res,
                           Object handler, ModelAndView mv) {
        // called after handler, before view render
    }

    @Override
    public void afterCompletion(HttpServletRequest req, HttpServletResponse res,
                                Object handler, Exception ex) {
        // always called after request complete (like finally)
    }
}

Must Know
Explain the DispatcherServlet request processing flow step by step.
▾
1. Request arrives at Servlet container (Tomcat) → passes through Filter chain (CORS, auth, logging). 2. DispatcherServlet receives the request. 3. Consults HandlerMapping — finds the matching @Controller method + any registered interceptors (returns HandlerExecutionChain). 4. Runs interceptor.preHandle() — if returns false, request aborted. 5. HandlerAdapter invokes the controller method — resolves @RequestBody, @PathVariable, @RequestParam, runs @Valid. 6. Controller returns response object or ResponseEntity. 7. Runs interceptor.postHandle(). 8. HttpMessageConverter serializes return value to JSON/XML. 9. Runs interceptor.afterCompletion(). 10. Response written back through Filter chain.
Tricky
What is the difference between a Servlet Filter and a Spring HandlerInterceptor?
▾
Filter: Servlet API level — runs before DispatcherServlet is involved, sees raw HTTP streams, can intercept ANY request (static resources too), can short-circuit the entire chain. No access to Spring context or handler info. Use for: auth, SSL redirect, request wrapping, rate limiting, CORS.
HandlerInterceptor: Spring MVC level — runs inside DispatcherServlet after handler is resolved. Has access to the handler method (HandlerMethod), can inspect method annotations, has access to Spring beans. Three hooks: preHandle (before), postHandle (after handler, before view), afterCompletion (always). Use for: auditing (know which method was called), locale setting, transaction markers.
⚡ Cannot access Spring beans from a Filter easily (runs before Spring context on some servers). HandlerInterceptor can @Autowire Spring beans freely.
Tricky
What happens when DispatcherServlet cannot find a handler for a request?
▾
If no handler is found: DispatcherServlet throws NoHandlerFoundException (if throwExceptionIfNoHandlerFound=true) or forwards to the default servlet (if configured). By default in Spring Boot: returns a 404 response via the BasicErrorController at /error. The NoResourceFoundException is handled by ResponseEntityExceptionHandler. Tip: Enable spring.web.resources.add-mappings=false to stop Spring from mapping static resources and let NoHandlerFoundException propagate — useful for building pure REST APIs where you want explicit 404 handling.

💡
DispatcherServlet = Airport customs officer. Every passenger (request) must pass through customs (DS), which checks documents (HandlerMapping), assigns a processing desk (HandlerAdapter), and sends you to the right agent (Controller). After processing, response goes back the same way.

Filter vs Interceptor: Filter = security check before entering the terminal. Interceptor = check at the gate — knows your flight details (handler method).

◆ REST Principles
📐
REST Constraints — Representational State Transfer: Client-Server (separation of concerns), Stateless (no session state on server — each request self-contained), Cacheable (responses labeled cacheable/non-cacheable), Uniform Interface (resource identification, manipulation through representations), Layered System (client can't tell if directly connected or via intermediaries), Code on Demand (optional — server can send executable code).
🏷️
HTTP Methods & Semantics — GET: read only, safe, idempotent. POST: create, not idempotent (calling twice creates two resources). PUT: full replace, idempotent (calling twice has same result). PATCH: partial update, not necessarily idempotent. DELETE: remove, idempotent (deleting twice = same state). HEAD: like GET but no body. OPTIONS: allowed methods.
🔢
HTTP Status Codes — 2xx Success: 200 OK, 201 Created (POST success), 204 No Content (DELETE success). 3xx Redirect: 301 Moved, 304 Not Modified. 4xx Client Error: 400 Bad Request, 401 Unauthorized (not authenticated), 403 Forbidden (authenticated but no permission), 404 Not Found, 405 Method Not Allowed, 409 Conflict, 422 Unprocessable Entity (validation). 5xx Server Error: 500 Internal Server Error, 503 Service Unavailable.
📍
Resource URL Design — Use nouns (resources), not verbs: /users (not /getUsers). Plural for collections: /users, /orders. Nested for relationships: /users/{id}/orders. Query params for filtering/sorting: /users?role=admin&sort=name. API versioning: /v1/users or Accept: application/vnd.api.v1+json.
📦
ResponseEntity<T> — Full control over HTTP response: set status code, headers, and body. Use when: you need to return a specific status code, add response headers (Location, custom headers), or conditionally return different status codes. If you always return 200 OK, just return the object directly from the method.

@RestController                        // = @Controller + @ResponseBody
@RequestMapping("/api/v1/users")
public class UserController {
    private final UserService userService;

    // GET /api/v1/users?page=0&size=20&sort=name
    @GetMapping
    public Page<UserDto> list(Pageable pageable) {
        return userService.findAll(pageable);
    }

    // GET /api/v1/users/42
    @GetMapping("/{id}")
    public ResponseEntity<UserDto> getById(@PathVariable Long id) {
        return userService.findById(id)
            .map(ResponseEntity::ok)                   // 200 if found
            .orElse(ResponseEntity.notFound().build()); // 404 if not
    }

    // POST /api/v1/users → 201 Created + Location header
    @PostMapping
    public ResponseEntity<UserDto> create(
            @Valid @RequestBody CreateUserRequest req) {
        UserDto created = userService.create(req);
        URI location = ServletUriComponentsBuilder
            .fromCurrentRequest()
            .path("/{id}")
            .buildAndExpand(created.getId())
            .toUri();
        return ResponseEntity.created(location).body(created);
    }

    // PUT /api/v1/users/42 — full replace
    @PutMapping("/{id}")
    public UserDto replace(@PathVariable Long id,
                           @Valid @RequestBody UpdateUserRequest req) {
        return userService.replace(id, req);
    }

    // PATCH /api/v1/users/42 — partial update
    @PatchMapping("/{id}")
    public UserDto update(@PathVariable Long id,
                          @RequestBody Map<String, Object> updates) {
        return userService.patch(id, updates);
    }

    // DELETE /api/v1/users/42 → 204 No Content
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        userService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
◆ Interview Questions
Must Know
What is idempotency in REST and why does it matter?
▾
An operation is idempotent if calling it multiple times produces the same result as calling it once. GET, PUT, DELETE are idempotent. POST and PATCH (usually) are not. Why it matters: in distributed systems, network failures mean clients retry requests. If an operation is idempotent, retries are safe — no double-charging, no duplicate records. Design: use idempotency keys for non-idempotent operations (include a unique key in POST request; server ignores duplicates with same key). Stripe's payment API uses this pattern.
⚡ DELETE is idempotent: deleting a non-existent resource returns 404 but has same state (resource not there). Some debate exists, but HTTP spec says DELETE should be idempotent.
Tricky
What is the difference between PUT and PATCH? When would you use each?
▾
PUT: full replacement — send the entire resource representation. Fields not included in the request body are set to null/default. Idempotent. Use when you want to replace the entire resource.
PATCH: partial update — send only the fields to change. Fields not included are unchanged. Use when you only want to update specific fields (e.g., just update email, leave everything else). PATCH is generally preferred for updates as it's more bandwidth-efficient and avoids accidentally nullifying fields. Implementation: accept Map<String, Object> and apply only provided fields, or use JSON Patch (RFC 6902) for structured partial updates.
Tricky
What is the difference between 401 Unauthorized and 403 Forbidden?
▾
Despite the name, 401 Unauthorized actually means unauthenticated — "I don't know who you are, please authenticate." The response should include WWW-Authenticate header. 403 Forbidden means unauthorized — "I know who you are, but you don't have permission to do this." Key distinction: 401 = no credentials or invalid credentials. 403 = valid credentials but insufficient permissions. Mixing them up is a common interview mistake and a real-world API design error.
⚡ Spring Security returns 401 when not logged in, 403 when logged in but missing role.
Concept
How do you version a REST API? What are the tradeoffs of each approach?
▾
1. URL versioning (/v1/users): Simple, cacheable, explicit. Breaks REST principle (URL = resource, not version). Most widely used.
2. Query parameter (/users?version=1): Flexible, optional. Cache-unfriendly. Messy.
3. Header versioning (Accept: application/vnd.api.v1+json): Clean URLs, proper REST. Harder to test (need curl/Postman). Not browser-friendly.
4. Subdomain (v1.api.example.com): Full isolation, routing at DNS. Infrastructure complexity.
Recommendation: URL versioning for public APIs (easiest for consumers), header versioning for mature APIs with developer-friendly tooling. Never break existing versions — deprecate with sunset headers.
◆ Memory Tip
💡
REST HTTP method cheat — "GPPPD":
• GET = Get (read, safe, idempotent)
• POST = Post new (create, NOT idempotent)
• PUT = Put back (full replace, idempotent)
• PATCH = Patch up (partial update)
• DELETE = Delete it (idempotent)

Status code families: 2xx = ✅ success · 3xx = ↗️ redirect · 4xx = ❌ client fault · 5xx = 💥 server fault
401 ≠ unauthorized (it means unauthenticated!). 403 = truly unauthorized.

📍
@PathVariable — Binds a URI template variable. /users/{id} → @PathVariable Long id. Name must match template variable name, or use @PathVariable("userId") for different names. Can be Optional.
❓
@RequestParam — Binds query string parameters. ?name=Alice&age=30. required=false for optional params. defaultValue="0" for defaults. Can bind to a List<String> for multi-value params (?tag=java&tag=spring).
📦
@RequestBody — Deserializes HTTP request body into a Java object (via Jackson). Use with @Valid for Bean Validation. Only one @RequestBody per method — the request body is a stream, can only be read once.
🏷️
@RequestHeader — Binds a specific HTTP header to a method parameter. @RequestHeader("Authorization") String auth. Optional: required=false. Bind all headers: @RequestHeader HttpHeaders headers.
🍪
@CookieValue — Binds a cookie value. @CookieValue("sessionId") String sessionId. Use required=false if cookie might not exist.
📋
@ModelAttribute — Binds form data or query params to a Java object (for non-JSON, form-based requests). Also used at class level to add data to all model views. In REST APIs, @RequestBody is typically preferred.
🔒
consumes / produces — Restrict which content types a method accepts/returns. @PostMapping(consumes = "application/json", produces = "application/json"). Used for content negotiation — different methods can handle different media types.

RequestMapping — All Parameter Binding Patterns
@RestController
@RequestMapping("/api/users")
public class UserController {

    // @PathVariable — from URL template
    // GET /api/users/42/orders/99
    @GetMapping("/{userId}/orders/{orderId}")
    public Order getOrder(
            @PathVariable Long userId,
            @PathVariable("orderId") Long oid) {
        return orderService.find(userId, oid);
    }

    // @RequestParam — from query string
    // GET /api/users?name=Alice&role=ADMIN&page=0
    @GetMapping
    public Page<User> search(
            @RequestParam(required = false) String name,
            @RequestParam(defaultValue = "USER") String role,
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size) {
        return userService.search(name, role, page, size);
    }

    // @RequestParam multi-value
    // GET /api/users?tag=java&tag=spring
    @GetMapping("/tagged")
    public List<User> byTags(@RequestParam List<String> tag) {
        return userService.findByTags(tag);
    }

    // @RequestBody + @Valid
    @PostMapping
    public ResponseEntity<User> create(
            @Valid @RequestBody CreateUserRequest req) {
        return ResponseEntity.status(HttpStatus.CREATED)
                            .body(userService.create(req));
    }

    // @RequestHeader
    @GetMapping("/profile")
    public User profile(
            @RequestHeader("X-User-Id") Long userId,
            @RequestHeader(value = "X-Request-Id", required = false) String reqId) {
        return userService.findById(userId);
    }

    // Content negotiation: different method for XML vs JSON
    @GetMapping(value = "/{id}", produces = "application/json")
    public UserDto getUserJson(@PathVariable Long id) {
        return userService.findById(id);
    }

    @GetMapping(value = "/{id}", produces = "application/xml")
    public UserXml getUserXml(@PathVariable Long id) {
        return userService.findByIdAsXml(id);
    }

    // Matrix variables: /users/42;role=admin;dept=eng
    @GetMapping("/{id}")
    public User getWithMatrix(
            @PathVariable Long id,
            @MatrixVariable String role) { // requires enableMatrixVariables
        return userService.findById(id);
    }
}
◆ Interview Questions
Must Know
What is the difference between @PathVariable, @RequestParam, and @RequestBody?
▾
@PathVariable: from URL path template. /users/{id} — value is part of the URL structure. Required by default (404 if missing).
@RequestParam: from query string. ?name=Alice — optional, has defaults, can be a List for multi-value. Good for filtering, sorting, pagination.
@RequestBody: from HTTP request body — deserializes the full body JSON/XML into a Java object. Only one per method. Use @Valid for validation. Not used with GET (no body by convention).
Use them together: @GetMapping("/users/{id}/orders") with @PathVariable Long id and @RequestParam String status.
Tricky
What happens if two @RequestMapping methods map to the same URL and method? How does Spring resolve conflicts?
▾
Spring's RequestMappingHandlerMapping tries to find the most specific match. If two mappings have identical URL and method, Spring throws IllegalStateException: Ambiguous mapping at startup — it cannot resolve the conflict. Resolution via specificity: more specific patterns win (/users/admin over /users/{id} for GET /users/admin). Tiebreakers: explicit method over wildcard, with consumes/produces over without, with params over without. If truly identical, it's a startup error that you must fix by differentiating the paths or conditions.
⚡ Add produces/consumes to differentiate handlers for the same URL.
Tricky
How does Spring handle argument binding when @RequestParam doesn't exactly match the parameter name?
▾
By default, @RequestParam uses the parameter name as the query string key. If the names don't match (e.g., query param is user_id but Java param is userId), use @RequestParam("user_id") Long userId — explicitly specify the key. For method parameters without any binding annotation, Spring tries: @RequestParam for simple types (String, int, etc.) in non-@RequestBody context, or model attribute binding for complex objects. Spring Boot enables parameter name retention via compiler flags — needed for implicit name resolution.

💡
WHERE is the data coming from?
• In the URL path? → @PathVariable (/users/{id})
• After the ? in URL? → @RequestParam (?name=Alice)
• In the body? → @RequestBody (JSON payload)
• In a header? → @RequestHeader (Authorization: Bearer...)

◆ Exception Handling Strategy
⚠️
Problem without @ControllerAdvice: Every controller method needs try-catch blocks. Exception handling is duplicated, inconsistent error response formats, mixing business logic with error handling.
✅
Solution: @ControllerAdvice = a global exception handler applied to ALL controllers. Define @ExceptionHandler methods once, used everywhere. Consistent error responses. Clean controllers.
🎯
@ControllerAdvice — A specialization of @Component that is shared across all @Controller classes. Works with @ExceptionHandler (handle exceptions), @ModelAttribute (add model data to all controllers), and @InitBinder (customize data binding). Use @RestControllerAdvice = @ControllerAdvice + @ResponseBody — all handler methods return JSON automatically.
🔍
@ExceptionHandler — Maps a specific exception type to a handler method. Spring selects the most specific matching handler. Can handle multiple exceptions: @ExceptionHandler({EntityNotFoundException.class, ResourceNotFoundException.class}). Method can return ResponseEntity, a response body directly, or a model for view resolution.
📊
ResponseEntityExceptionHandler — A base class provided by Spring that already handles all Spring MVC exceptions (MethodArgumentNotValidException, HttpMessageNotReadableException, NoHandlerFoundException, etc.) and returns RFC 7807 Problem Detail responses. Extend it and @Override specific handlers to customize. In Boot 3.x: ProblemDetail support is built-in.
📝
RFC 7807 Problem Detail — Standard JSON format for HTTP error responses. Spring Boot 3.x supports it natively. Fields: type (URI for error docs), title (human-readable), status (HTTP code), detail (specific error detail), instance (URI of specific occurrence). Enable with spring.mvc.problemdetails.enabled=true.
🏷️
Scope Limiting — @ControllerAdvice applies to ALL controllers by default. Limit scope: @ControllerAdvice(assignableTypes = {UserController.class, OrderController.class}) or @ControllerAdvice(basePackages = "com.example.user") or @ControllerAdvice(annotations = RestController.class).
◆ Memorizable Code
@RestControllerAdvice — Global Exception Handler + Problem Detail
// ── Standard error response ────────────────────────
public record ErrorResponse(
    int status, String error, String message,
    String path, Instant timestamp
) {}

// ── Global Exception Handler ───────────────────────
@RestControllerAdvice               // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler
        extends ResponseEntityExceptionHandler { // handle Spring MVC exceptions

    // 404 — custom domain exception
    @ExceptionHandler(EntityNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            EntityNotFoundException ex, HttpServletRequest req) {
        return ResponseEntity
            .status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse(
                404, "Not Found", ex.getMessage(),
                req.getRequestURI(), Instant.now()
            ));
    }

    // 409 — business rule violation
    @ExceptionHandler(DuplicateResourceException.class)
    public ResponseEntity<ErrorResponse> handleConflict(
            DuplicateResourceException ex, HttpServletRequest req) {
        return ResponseEntity
            .status(HttpStatus.CONFLICT)
            .body(new ErrorResponse(409, "Conflict", ex.getMessage(), req.getRequestURI(), Instant.now()));
    }

    // 400 — @Valid validation failures (override Spring's handler)
    @Override
    protected ResponseEntity<Object> handleMethodArgumentNotValid(
            MethodArgumentNotValidException ex,
            HttpHeaders headers, HttpStatusCode status,
            WebRequest request) {
        Map<String, String> errors = new LinkedHashMap<>();
        ex.getBindingResult().getFieldErrors().forEach(err ->
            errors.put(err.getField(), err.getDefaultMessage())
        );
        return ResponseEntity.badRequest().body(
            Map.of("status", 400, "errors", errors)
        );
    }

    // 500 — catch-all (never expose stack traces in production!)
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneric(
            Exception ex, HttpServletRequest req) {
        log.error("Unhandled exception", ex);  // log full stack trace
        return ResponseEntity
            .internalServerError()
            .body(new ErrorResponse(500, "Internal Server Error",
                "An unexpected error occurred",  // never expose ex.getMessage()!
                req.getRequestURI(), Instant.now()));
    }
}

// ── RFC 7807 Problem Detail (Spring Boot 3.x) ─────
// application.yml: spring.mvc.problemdetails.enabled: true
@ExceptionHandler(EntityNotFoundException.class)
public ProblemDetail handleNotFound(EntityNotFoundException ex) {
    ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
    pd.setTitle("Resource Not Found");
    pd.setType(URI.create("https://api.example.com/errors/not-found"));
    pd.setProperty("timestamp", Instant.now());

}
◆ Interview Questions
Must Know
What is @ControllerAdvice and why is it preferred over try-catch in controllers?
▾
@ControllerAdvice is a global exception handler that intercepts exceptions from all (or specific) controllers. Benefits over try-catch: 1. DRY — define error handling once, apply everywhere. 2. Separation of concerns — business logic in controller, error handling in advice. 3. Consistency — uniform error response format across all endpoints. 4. Type safety — strongly typed exception handlers vs generic catch blocks. 5. Testability — test error handling separately from business logic. The alternative (try-catch in every method) leads to code duplication and inconsistent error formats.
Tricky
How does Spring choose which @ExceptionHandler to invoke when multiple could match?
▾
Spring selects the most specific handler — the handler whose exception class is closest to the thrown exception in the class hierarchy. If you throw UserNotFoundException extends EntityNotFoundException: Spring prefers @ExceptionHandler(UserNotFoundException.class) over @ExceptionHandler(EntityNotFoundException.class) if both exist. If a local @ExceptionHandler in the controller and a global one in @ControllerAdvice match: the local controller handler wins. Catch-all @ExceptionHandler(Exception.class) is the last resort — always put it in @ControllerAdvice, never in a specific controller.
⚡ Order of resolution: most-specific local handler → less-specific local → most-specific global → less-specific global.
Tricky
What is ResponseEntityExceptionHandler and when should you extend it?
▾
ResponseEntityExceptionHandler is a base class that handles all standard Spring MVC exceptions by default — MethodArgumentNotValidException (validation failures), HttpMessageNotReadableException (malformed JSON), HttpRequestMethodNotSupportedException (wrong HTTP method), NoHandlerFoundException (404), and ~15 more. Without extending it, these exceptions bubble up to Spring Boot's default error controller and return a generic error. Extend it when: you want to customize the response format for Spring's built-in exceptions. Override specific methods to change format. Don't override all — just the ones you need to customize.
💡 In Spring Boot 3.x, extending ResponseEntityExceptionHandler and enabling spring.mvc.problemdetails.enabled=true gives you RFC 7807 Problem Detail responses out of the box.
Concept
Should you expose exception messages to API clients? What are the security implications?
▾
Never expose internal exception messages for 5xx errors in production. Reasons: stack traces reveal internal architecture (class names, packages, DB schema), error messages may include sensitive data (SQL queries with user data, file paths), and they help attackers understand your internals. Best practice: for 4xx errors (validation, not found, business rule) — safe to return the error message (you know what those contain). For 5xx errors — always return a generic "An unexpected error occurred" message and log the full exception server-side with a correlation/trace ID. Return the trace ID to the client so you can cross-reference in logs.
⚠️ Log the full exception, expose only the safe message. Include a trace ID in the response for debugging without exposing internals.
Must Know
How do you handle @Valid validation failures and return field-level error messages?
▾
When @Valid @RequestBody fails, Spring throws MethodArgumentNotValidException. Handle it in @ControllerAdvice by overriding handleMethodArgumentNotValid() from ResponseEntityExceptionHandler. The exception contains a BindingResult with a list of FieldError objects. Extract: ex.getBindingResult().getFieldErrors() → build a map of fieldName: errorMessage. Return 400 Bad Request with this map. Example response: {"errors": {"email": "must be a valid email", "age": "must be between 0 and 120"}}. This gives clients actionable field-level validation errors.

💡

Exception handling priority:
Local controller @ExceptionHandler → Global @ControllerAdvice (specific) → Global @ControllerAdvice (generic)

Golden rule for 5xx: Log everything, expose nothing. Return trace ID so you can look it up.
