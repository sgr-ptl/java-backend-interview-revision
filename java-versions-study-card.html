Java 8
Topic 41 — Lambdas, Streams & Optional
// The release that transformed Java into a functional-capable language
▾
🔑 KEY CONCEPTS
λ
Lambda — Anonymous function. Syntax: (params) -> expression. Implements a @FunctionalInterface (interface with exactly 1 abstract method). Enables passing behavior as data.
🌊
Streams — A pipeline to process collections declaratively. Two types of operations: Intermediate (lazy, return Stream: filter, map, flatMap, sorted, distinct, limit) and Terminal (eager, produce result: collect, forEach, reduce, count, findFirst). Streams are not reusable after terminal op.
📦
Optional<T> — A container that may or may not hold a value. Eliminates NullPointerException. Key methods: of(), ofNullable(), empty(), isPresent(), ifPresent(), orElse(), orElseGet(), orElseThrow(), map(), flatMap(), filter().
🔗
Method References — Shorthand for lambdas that call a single method. 4 types: ClassName::staticMethod, instance::method, ClassName::instanceMethod, ClassName::new (constructor).
⚙️
Functional Interfaces — Built-in: Predicate<T> (test→boolean), Function<T,R> (apply→R), Consumer<T> (accept→void), Supplier<T> (get→T), BiFunction<T,U,R>. All in java.util.function.
🗺️
Default & Static Interface Methods — Interfaces can now have implemented methods. default keyword allows backward-compatible interface evolution without breaking existing implementations.

Runnable r = () -> System.out.println("Hello Lambda");

// ── Stream Pipeline ──────────────────────────────
List<String> names = List.of("Alice", "Bob", "Anna", "Charlie");

List<String> result = names.stream()
    .filter(n -> n.startsWith("A"))         // intermediate
    .map(String::toUpperCase)               // intermediate
    .sorted()                                // intermediate
    .collect(Collectors.toList());           // terminal
// → ["ALICE", "ANNA"]

// flatMap: List of Lists → flat List

    .stream().flatMap(Collection::stream)
    .collect(Collectors.toList());  // [1,2,3,4]

// reduce: sum
int sum = Stream.of(1,2,3,4).reduce(0, Integer::sum); // 10

// ── Optional ─────────────────────────────────────
Optional<String> opt = Optional.ofNullable(getUserName());

String name = opt
    .filter(s -> s.length() > 3)
    .map(String::toUpperCase)
    .orElse("GUEST");                       // default value

opt.orElseThrow(() -> new RuntimeException("Not found"));

// ── Default Interface Method ──────────────────────
interface Greeter {
    void greet(String name);
    default void greetLoud(String name) {
        greet(name.toUpperCase());  // reuses abstract method
    }
}

🧠 MEMORY TIP
💡
Stream = Builder pattern for data. Think of it like a production line: raw material (source) → conveyor belts (intermediate ops, lazy) → packaging machine (terminal op, triggers execution). A stream is lazy — nothing runs until a terminal op fires. map transforms shape, flatMap flattens + transforms.
❓ INTERVIEW QUESTIONS
What is the difference between map() and flatMap() in Streams?
map() transforms each element into one other element (1-to-1 mapping). flatMap() transforms each element into zero or more elements, then flattens the result into a single stream (1-to-many). Example: a List<List<String>> — using map gives you a Stream<List<String>>, while flatMap(Collection::stream) gives you a flat Stream<String>.
What is a Functional Interface? Can we have multiple abstract methods?
A Functional Interface has exactly one abstract method, which is the target for lambda expressions. You can have multiple default/static methods. The @FunctionalInterface annotation is optional but causes a compile error if the contract is violated. Key built-ins: Predicate, Function, Consumer, Supplier, BiFunction.
Why should you avoid Optional.get() without checking first? What's the better way?
Optional.get() throws NoSuchElementException if empty — it's no safer than a null check. Prefer: orElse("default") for a constant fallback, orElseGet(() -> compute()) for lazy evaluation, orElseThrow() for explicit failure, or ifPresent(val -> use(val)) for side effects. Never use Optional as a field type or method parameter — it's designed for return values.
What is the difference between intermediate and terminal stream operations?
Intermediate operations (like filter, map, sorted) are lazy — they return a new Stream and don't execute until a terminal operation is called. Terminal operations (like collect, forEach, reduce, count) are eager — they trigger the entire pipeline and produce a result or side-effect. After a terminal op, the stream is consumed and cannot be reused.
What are the 4 types of Method References in Java 8?
1. ClassName::staticMethod — e.g. Integer::parseInt
2. instance::instanceMethod — e.g. System.out::println
3. ClassName::instanceMethod — e.g. String::toUpperCase (instance is the first param)
4. ClassName::new — Constructor reference, e.g. ArrayList::new

Java 11
Topic 42 — var keyword & New HttpClient API
// LTS release · Local type inference + modern HTTP without third-party libs
▾
🔑 KEY CONCEPTS
📝
var (Local Variable Type Inference) — Introduced in Java 10, widely adopted in 11. The compiler infers the type from the right-hand side at compile time. NOT dynamic typing — still statically typed. Only works for local variables (not fields, method params, or return types). Cannot be used with null alone.
🌐
HttpClient API — Replaces the old HttpURLConnection. Supports HTTP/1.1, HTTP/2, WebSocket. Supports synchronous (send()) and asynchronous (sendAsync() returns CompletableFuture). Builder pattern for both client and request. Immutable and reusable.
🧵
New String Methods — Must-know: isBlank() (true if empty or whitespace), strip() (Unicode-aware trim), stripLeading()/stripTrailing(), lines() (returns Stream of lines), repeat(n) ("ha".repeat(3) = "hahaha").
📁
Files Utility Improvements — Files.readString(path) and Files.writeString(path, content) — simple one-liner file I/O. Previously needed streams or Apache Commons.
🗑️
Removed/Deprecated — Java EE modules removed (JAX-WS, JAXB now need explicit dependencies). Thread.destroy() and Thread.stop() removed. Important for migration discussions.

🧠 MEMORY TIP
💡
var = lazy shorthand, not lazy typing. The compiler still knows the exact type — you just don't have to write it. Use it when the type is obvious from context (e.g., constructor on the right). Avoid var when it harms readability (e.g., var x = process() — what type is that?).

HttpClient = Builder pattern, two modes. Remember: newHttpClient() → newBuilder().uri().GET().build() → send(req, BodyHandlers.ofString()).
❓ INTERVIEW QUESTIONS
Is var dynamic typing? What are its limitations?
No — var is static typing with type inference. The type is resolved at compile time, not runtime. Limitations: only for local variables (not fields, method params, return types), cannot be used with null alone (type ambiguous), cannot be used in lambda parameter types (well, Java 11 allows it but it's unusual), and it can hurt readability when the right-hand side type is not obvious.
How does the Java 11 HttpClient differ from the old HttpURLConnection?
The new HttpClient supports HTTP/2 and WebSockets natively, provides both sync (send()) and async (sendAsync() → CompletableFuture) modes, uses an immutable builder pattern, handles redirects and authentication built-in, and is significantly less verbose. HttpURLConnection was HTTP/1.1 only, verbose, not thread-safe by design, and had no async support.
What's the difference between trim() and strip() in Java 11?
trim() removes only ASCII whitespace characters (characters with code ≤ U+0020). strip() is Unicode-aware and removes all Unicode whitespace as defined by Character.isWhitespace(). For modern applications, prefer strip(). Java 11 also added stripLeading() and stripTrailing() for one-sided trimming.

Java 17
Topic 43 — Sealed Classes & Pattern Matching instanceof
// LTS release · Sealed type hierarchies + smarter type checks
▾
🔑 KEY CONCEPTS
🔒
Sealed Classes — A class/interface that restricts which other classes can extend/implement it. Use sealed keyword + permits clause. Each permitted subclass must be final, sealed, or non-sealed. Enables exhaustive pattern matching — the compiler knows all possible subtypes.
🎯
Pattern Matching for instanceof — Eliminates the manual cast after an instanceof check. Old: if (obj instanceof String) { String s = (String) obj; }. New: if (obj instanceof String s) — variable s is automatically bound and scoped. Reduces boilerplate and avoids cast errors.
📝
Text Blocks (finalized in 15, widely used in 17) — Multi-line string literals using triple quotes """. Handles indentation automatically. Perfect for JSON, SQL, HTML in code. Supports \ line continuation and \s for trailing space.
📦
Records (finalized in 16, standard in 17) — Immutable data classes with auto-generated constructor, getters, equals, hashCode, toString. Syntax: record Point(int x, int y) {}. Cannot extend other classes (implicitly extends Record). Can implement interfaces.
🧱
Strong Encapsulation of JDK Internals — Access to internal APIs (sun.*) is now strongly encapsulated. Important for frameworks and libraries that relied on internal APIs — this is a common migration discussion point in senior interviews.

JAVA 17 · SEALED CLASSES + PATTERN MATCHING + RECORDS + TEXT BLOCKS
// ── Sealed Class Hierarchy ───────────────────────
sealed interface Shape permits Circle, Rectangle, Triangle {}

record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}
final class Triangle implements Shape { double base, height; }

// compiler knows ALL possible subtypes of Shape

// ── Pattern Matching instanceof ──────────────────
void describe(Object obj) {
    // OLD WAY:
    if (obj instanceof String) {
        String s = (String) obj; // redundant cast
        System.out.println(s.length());
    }

    // NEW WAY (Java 17):
    if (obj instanceof String s) {          // s is bound here
        System.out.println(s.length());      // no cast needed
    }

    // Can combine with conditions:
    if (obj instanceof String s && s.length() > 5) {
        System.out.println("Long string: " + s);
    }
}

// ── Records ──────────────────────────────────────
record Point(int x, int y) {
    // Compact canonical constructor (validation)
    Point {
        if (x < 0 || y < 0) throw new IllegalArgumentException("Negative!");
    }
    double distanceFromOrigin() {
        return Math.sqrt(x*x + y*y); // custom method OK
    }
}

var p = new Point(3, 4);
p.x();                    // getter (no "get" prefix)
p.distanceFromOrigin();   // 5.0

// ── Text Blocks ───────────────────────────────────
String json = """
        {
            "name": "Alice",
            "age": 30
        }
        """;

String sql = """
        SELECT u.name, o.total
        FROM users u
        JOIN orders o ON u.id = o.user_id
        WHERE u.active = true
        """;
🧠 MEMORY TIP
💡
Sealed = "I control my family tree." Like a parent who lists every allowed child. Useful when you want exhaustive switch/pattern matching — the compiler guarantees you haven't missed a case.

Records = "POJO on steroids, immutable by default." Think DTOs, value objects, or event payloads. If you need mutation → use a regular class. If you need inheritance → records can't extend classes (but can implement interfaces).
❓ INTERVIEW QUESTIONS
What are Sealed Classes and why would you use them over a regular abstract class?
Sealed classes give you controlled, closed type hierarchies. With a regular abstract class, anyone can subclass it anywhere. With sealed, only the permitted subclasses can extend it. This enables the compiler to verify exhaustiveness in pattern matching (like switch expressions). Use case: modeling a finite set of states (e.g., Success | Failure | Pending) where you want the compiler to catch missed cases.
What's generated automatically in a Record? Can you override anything?
A record auto-generates: canonical constructor, accessor methods (named after components, no "get" prefix), equals(), hashCode(), toString(). You can override any of these. You can also add a compact canonical constructor for validation (without listing parameters — they're implicit). You can add custom methods and static methods. What you can't do: add mutable instance fields, extend other classes (records implicitly extend java.lang.Record).
How does pattern matching for instanceof improve code safety?
The old pattern of instanceof check followed by cast was error-prone — you could cast to the wrong type or the variable scope was unclear. With pattern matching, the binding variable s in if (obj instanceof String s) is only in scope when the check passes, eliminating ClassCastException risk. It also reduces boilerplate significantly, especially in visitor-like code or when processing heterogeneous collections.

Java 21
Topic 44 — Pattern Matching for switch & Record Patterns
// LTS release · Expressive switch + structural deconstruction of records
▾
🔑 KEY CONCEPTS
🔀
Pattern Matching for switch — switch can now match on types, not just values. Combines type check + binding in one step. Works with sealed hierarchies for exhaustive matching (compiler enforces all cases). Supports guarded patterns with when clause.
📦
Record Patterns — Deconstruct a record directly in a pattern. Instead of if (obj instanceof Point p) { p.x(); p.y(); }, you write if (obj instanceof Point(int x, int y)) — the components are extracted inline. Works in both instanceof and switch patterns.
🧵
Virtual Threads (Project Loom) — Lightweight threads managed by the JVM, not the OS. Create millions of virtual threads without overhead. Use Thread.ofVirtual().start(runnable) or Executors.newVirtualThreadPerTaskExecutor(). Key for high-throughput I/O bound applications.
📡
Sequenced Collections — New interfaces: SequencedCollection, SequencedSet, SequencedMap — provide uniform access to first/last elements via getFirst(), getLast(), addFirst(), addLast(), reversed(). Works on List, LinkedHashSet, LinkedHashMap etc.
⚠️
null Handling in switch — Previously, switching on null threw NPE. Java 21 allows case null explicitly in switch expressions. Combine with case null, default for clean null-safe switching.
💻 MEMORIZABLE CODE
JAVA 21 · PATTERN SWITCH + RECORD PATTERNS + VIRTUAL THREADS
// ── Pattern Matching for switch ──────────────────
sealed interface Shape permits Circle, Rectangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double w, double h) implements Shape {}

double area(Shape shape) {
    return switch (shape) {
        case Circle c    -> Math.PI * c.radius() * c.radius();
        case Rectangle r -> r.w() * r.h();
        // No default needed — sealed hierarchy is exhaustive!
    };
}

// ── Guarded patterns with 'when' clause ─────────
String classify(Object obj) {
    return switch (obj) {
        case Integer i when i > 0  -> "positive int";
        case Integer i when i < 0  -> "negative int";
        case Integer i              -> "zero";
        case String s               -> "string: " + s;
        case null                    -> "null value";
        default                      -> "other";
    };
}

// ── Record Patterns (deconstruction) ────────────
record Point(int x, int y) {}
record Line(Point start, Point end) {}

void printLine(Object obj) {
    // Nested record deconstruction!
    if (obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2))) {
        System.out.printf("(%d,%d) → (%d,%d)%n", x1, y1, x2, y2);
    }
}

// ── Virtual Threads ──────────────────────────────
// Create one virtual thread
Thread.ofVirtual().start(() -> System.out.println("Virtual!"));

// ExecutorService with virtual threads (for HTTP servers, etc.)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        final int id = i;
        executor.submit(() -> processRequest(id));
    }
} // auto-close waits for all tasks

// ── Sequenced Collections ────────────────────────
List<String> list = new ArrayList<>(List.of("a","b","c"));
list.getFirst();     // "a"
list.getLast();      // "c"
list.reversed();    // reversed view: ["c","b","a"]
🧠 MEMORY TIP
💡
Pattern switch = switch evolved into a type-safe visitor. When combined with sealed classes, the compiler is your safety net — miss a case, get a compile error. This replaces many uses of the Visitor design pattern with cleaner, less verbose code.

Virtual Threads = threads as cheap as objects. 1 platform thread = ~1MB stack. 1 virtual thread = ~1KB. You can run 1M virtual threads on a typical JVM. Ideal for blocking I/O (REST calls, DB queries). Not for CPU-bound tasks — use platform threads there.
❓ INTERVIEW QUESTIONS
How do Virtual Threads differ from Platform Threads? When should you use them?
Platform threads are OS-managed (1:1 mapping to OS thread). Virtual threads are JVM-managed (M:N mapping — many virtual threads multiplexed onto fewer platform threads). Virtual threads have much lower memory (~1KB vs ~1MB) and creation overhead. Use virtual threads for I/O-bound, blocking tasks (HTTP calls, DB queries) where threads spend most time waiting. Avoid for CPU-bound computation — platform threads or ForkJoinPool are better there.
What is a Record Pattern and how does it differ from Pattern Matching instanceof from Java 17?
Java 17's pattern matching binds the whole object: obj instanceof Point p — you get p and then call p.x() separately. Java 21's record patterns deconstruct the record inline: obj instanceof Point(int x, int y) — components are extracted directly. You can even nest: obj instanceof Line(Point(var x1, var y1), Point(var x2, var y2)). This is structural pattern matching similar to Scala/Kotlin.
Why doesn't pattern matching switch need a default when used with sealed classes?
Because sealed classes have a finite, compiler-known set of subtypes. If the switch covers all permitted subclasses, the compiler can prove exhaustiveness — similar to how an enum switch doesn't need default if all values are covered. Adding default is still allowed but triggers a warning that it's redundant. This exhaustiveness checking is the main reason to combine sealed classes with switch pattern matching.

Java 25
Topic 45 — Scalability Focus
// LTS · Loom, Valhalla, Panama — the next platform generation
▾
🔑 KEY CONCEPTS
🧵
Project Loom (Matured) — Virtual threads are mainstream. Structured Concurrency (StructuredTaskScope) — treats a group of concurrent tasks as a single unit of work. Fail-fast with ShutdownOnFailure, or wait for first success with ShutdownOnSuccess. Cleaner than CompletableFuture chains.
💎
Project Valhalla (Value Types) — value class — Objects with value semantics (like primitives). No identity (no == reference comparison), no null, stored inline in memory (not on heap as separate objects). Eliminates "wrapper penalty" — List<int> without boxing overhead. Major performance improvement for data-heavy code.
🌉
Project Panama (Foreign APIs) — Foreign Function & Memory API — Safe, high-performance way to call native libraries and access off-heap memory. Replaces JNI (Java Native Interface) which was verbose and unsafe. Key for performance-critical native interop (e.g., ML libraries, OS APIs).
🔗
Scoped Values — Like ThreadLocal but designed for virtual threads. Immutable, inherit into child threads, automatically cleaned up when scope ends. Use for passing context (user ID, trace ID) through a call stack without passing as parameters.
🎭
Stable Primitives in Generics — Part of Valhalla: universal generics that work with both reference types AND primitives. List<int>, Map<long, double> — no boxing, no wrapper classes. Massive memory and CPU cache efficiency gain.

JAVA 25 · STRUCTURED CONCURRENCY + SCOPED VALUES (PREVIEW/CURRENT)
// ── Structured Concurrency ────────────────────────
// All subtasks treated as one unit — fail together, clean up together
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

    var userTask  = scope.fork(() -> fetchUser(userId));
    var orderTask = scope.fork(() -> fetchOrders(userId));

    scope.join();           // wait for both
    scope.throwIfFailed(); // propagate any failure

    var user   = userTask.get();   // guaranteed non-null here
    var orders = orderTask.get();
    return new Dashboard(user, orders);
} // scope closed → all threads cleaned up automatically

// ── ShutdownOnSuccess: first result wins ─────────
try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {
    scope.fork(() -> callPrimaryServer());
    scope.fork(() -> callFallbackServer());
    scope.join();
    return scope.result();  // whichever finished first
}

// ── Scoped Values (instead of ThreadLocal) ───────
static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

void handleRequest(User user) {
    ScopedValue.where(CURRENT_USER, user).run(() -> {
        processOrder();   // can access CURRENT_USER.get() here
        sendEmail();      // and in all called methods
    });
    // CURRENT_USER is unbound outside the run() block
}

void processOrder() {
    User u = CURRENT_USER.get();  // safely read, immutable
}

🧠 MEMORY TIP
💡
Structured Concurrency = "fork-join but smarter." Think of it like a try-with-resources for threads — tasks are scoped, cleanly cancelled on failure, no orphaned threads. Scoped Values = ThreadLocal done right — immutable, scope-bound, virtual-thread-friendly. For Java 25, the interview keyword is scalability without complexity: virtual threads + structured concurrency let you write sequential-looking code that scales like async/reactive.
❓ INTERVIEW QUESTIONS
What problem does Structured Concurrency solve over CompletableFuture?
CompletableFuture chains are hard to reason about — exceptions can be swallowed, threads can leak if one task fails but others keep running, and the relationship between parent and child tasks is implicit. Structured Concurrency makes the scope explicit — all forked tasks must complete before the scope exits. If any task fails with ShutdownOnFailure, all are cancelled. This prevents thread leaks, makes error handling straightforward, and makes the code read like sequential logic.
What are Scoped Values and why are they better than ThreadLocal for virtual threads?
ThreadLocal was designed for platform threads — it stores mutable, per-thread state that persists until explicitly removed, which can cause memory leaks. With millions of virtual threads, this doesn't scale. ScopedValue is immutable, automatically cleaned up when the scope ends, and naturally inherits into child virtual threads without copying. It's also faster because the JVM can optimize immutable scoped lookups better than mutable ThreadLocal.
What is Project Valhalla and why does it matter for performance?
Valhalla introduces value types — objects with value semantics (no identity, no null) that can be stored inline in memory rather than as heap references. Today, List<Integer> stores references to boxed Integer objects scattered on the heap (bad for CPU cache). With Valhalla, List<int> stores raw ints contiguously in memory — dramatically better cache performance. This is critical for data-intensive applications (financial systems, ML, game engines) where boxing overhead is a real bottleneck.
