CI/CD
What is CI/CD?
Continuous Integration, Delivery, and Deployment — automating the path from code to production
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Before CI/CD, developers would code for weeks, merge everything at the end ("integration hell"), manually test, and manually deploy — a process taking days and packed with surprise failures. CI/CD automates all of this: every code push triggers automated tests, every passing build creates a deployable artifact, and every merge to main can automatically go live in production.
◆ The Three Concepts
🔄
Continuous Integration (CI) — Developers push code frequently (multiple times per day). Each push automatically triggers: code checkout, compile/build, run unit tests, run integration tests, static analysis, code coverage. Goal: detect integration problems immediately. If it breaks, you know in minutes not weeks. The team keeps a single shared branch that always compiles and all tests pass.
📦
Continuous Delivery (CD) — After CI passes, the pipeline automatically packages the app (Docker image, JAR, etc.) and deploys it to a staging/QA environment. The app is always in a deployable state. Deployment to production still requires a human approval click — this gives teams confidence that production deployment is just "pressing a button" rather than a risky, manual process.
🚀
Continuous Deployment — The ultimate step: every commit that passes all tests is automatically deployed to production — no human approval. Only possible with comprehensive automated tests, feature flags, canary deployments, and rollback automation. Companies like Netflix, Amazon, and Etsy deploy hundreds of times per day using this model.
📊
Key Metrics (DORA) — DevOps Research and Assessment defines 4 metrics for high-performing teams: Deployment Frequency (how often you deploy), Lead Time for Changes (code commit → production), Change Failure Rate (% of deployments causing incidents), Time to Restore (how fast you recover from failures). Elite teams deploy multiple times/day with <1% failure rate.
🔀
Branch Strategy — Trunk-based development: small, frequent commits directly to main. Feature flags hide incomplete features. Simplest, fastest. Used by Google. GitFlow: feature/release/hotfix branches. More structure, slower. GitHub Flow: feature branches → PR → main. Middle ground. CI/CD works best with trunk-based or GitHub Flow — long-lived branches create "merge debt."

Must Know
What is the difference between Continuous Integration, Continuous Delivery, and Continuous Deployment?
▾
CI (Continuous Integration): developers integrate code into a shared repo frequently. Each push triggers automated build + test. Purpose: catch integration bugs early. Every commit should leave the codebase in a working state.

Continuous Delivery: extends CI — after tests pass, the pipeline automatically packages and deploys to a staging environment. Production deployment requires a human "approve" button. The promise: deployment to production is always a low-risk, one-button operation because it was already tested in staging.

Continuous Deployment: removes the human approval — every successful pipeline run deploys straight to production. Requires very high test coverage, feature flags, monitoring, and instant rollback capability.
⚡ Interview framing: CI = "tests run on every push." CD = "deployment is automated." The difference between Delivery and Deployment is: Delivery has a human gate before production; Deployment does not.
Tricky
Why is "integration hell" a problem and how does CI specifically solve it?
▾
Integration hell happens when developers work in isolation on feature branches for weeks. When they finally merge, the conflicting changes are extensive and painful to resolve — database schema changes conflict, API contracts diverged, tests fail in unexpected ways. Fixing takes longer than the feature took to build.

CI solves this with two principles: Integrate frequently (every day, every few hours) so conflicts are small and easy to fix. Automate the integration check — every merge triggers a full build + test suite, so the team gets immediate feedback rather than discovering problems days later.
💡 The fundamental insight: merging 10 small changes is far less painful than merging 1 huge change. CI enforces the habit of small, frequent integration.
Concept
What are the DORA metrics and why do they matter for CI/CD?
▾
DORA (DevOps Research and Assessment) metrics measure software delivery performance with data from thousands of organizations: Deployment Frequency (how often code goes to production), Lead Time for Changes (commit to deployed in production), Change Failure Rate (% of deployments causing production failures), Mean Time to Restore (how fast you recover from incidents).

They matter because they're predictive — elite performers (multiple deploys/day, <1hr lead time, <1% failure rate) also have better business outcomes (revenue growth, market share, customer satisfaction). CI/CD directly improves all four: automation increases deployment frequency, reduces lead time, and faster pipelines mean faster recovery.
💡
CI/CD = automate the boring, risky, manual parts of software delivery.

The three levels:
• CI = every push triggers automated build + tests
• Continuous Delivery = staging deploy is automatic, prod needs approval
• Continuous Deployment = prod deploy is automatic (no human gate)

Pipeline stages order: push → build → unit test → integration test → package → staging → (approve) → production

DORA goal: Deploy often, fail rarely, recover fast. CI/CD is the enabler.

GitHub Actions — Pipeline Configuration
Events, workflows, jobs, steps, secrets — the anatomy of a real CI/CD pipeline
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: GitHub Actions lets you define automated workflows in YAML files inside your repo. An event (push, pull request, schedule) triggers a workflow. Each workflow has jobs. Each job runs on a runner (VM) and has steps (commands to execute). Jobs can run in parallel or depend on each other. You get 2,000 minutes/month free on public repos.
📋
Workflow Structure — Lives in .github/workflows/*.yml. Top-level keys: name (workflow name), on (trigger events), env (shared environment variables), jobs (parallel/sequential units of work). Each job has: runs-on (runner OS), needs (job dependencies), steps (sequential commands).
⚡
Trigger Events — push: any push to specified branches. pull_request: PR opened/updated/merged. schedule: cron-based (e.g., nightly builds). workflow_dispatch: manual trigger with optional inputs. release: when a GitHub release is created. Multiple events can trigger the same workflow. Use branches filter to only trigger on specific branches.
🔐
Secrets & Environment Variables — Never hardcode credentials. Store as GitHub Secrets (Settings → Secrets → Actions). Reference with ${{ secrets.MY_SECRET }}. Environments (staging, prod) can have their own secrets and require approvals. Use env for non-sensitive config. GitHub automatically masks secret values in logs.
♻️
Caching Dependencies — actions/cache caches Maven/Gradle/npm dependencies between runs. Speeds up builds by 2–10x. Key includes a hash of the lock file (pom.xml, package-lock.json) — cache invalidates automatically when dependencies change. Without caching, each run re-downloads all dependencies from scratch.
🔗
Job Dependencies — Jobs run in parallel by default. Use needs: [build] to run a job only after another completes. Use if: ${{ github.ref == 'refs/heads/main' }} to conditionally run jobs. Chain: test → build → deploy-staging → (manual approval) → deploy-prod.
◆ Memorizable Code — Complete Spring Boot CI/CD Pipeline
GitHub Actions — Spring Boot CI/CD Pipeline
# .github/workflows/ci-cd.yml
name: Spring Boot CI/CD

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

env:
  JAVA_VERSION: '21'
  IMAGE_NAME: my-app

jobs:

  ## ── Job 1: Build & Test ──────────────────────────
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: ${{ env.JAVA_VERSION }}
          distribution: temurin

      - name: Cache Maven dependencies
        uses: actions/cache@v4
        with:
          path: ~/.m2/repository
          key: maven-${{ hashFiles('**/pom.xml') }}
          restore-keys: maven-

      - name: Run tests
        run: mvn verify --no-transfer-progress

      - name: Upload test report
        if: always()              # upload even on failure
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: target/surefire-reports/

  ## ── Job 2: Build & Push Docker Image ────────────
  docker:
    needs: test                   # only after tests pass
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            myorg/${{ env.IMAGE_NAME }}:latest
            myorg/${{ env.IMAGE_NAME }}:${{ github.sha }}

  ## ── Job 3: Deploy to Staging ─────────────────────
  deploy-staging:
    needs: docker
    environment: staging         # environment gates/secrets
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to K8s staging
        run: |
          kubectl set image deployment/app \
            app=myorg/${{ env.IMAGE_NAME }}:${{ github.sha }}
        env:
          KUBECONFIG_DATA: ${{ secrets.KUBECONFIG_STAGING }}

  ## ── Job 4: Deploy to Prod (requires approval) ────
  deploy-prod:
    needs: deploy-staging
    environment: production      # human approval required!
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to production
        run: ./scripts/deploy-prod.sh ${{ github.sha }}
Must Know
What is the difference between a job and a step in GitHub Actions?
▾
Job: a collection of steps that runs on a single runner (VM). Jobs run in parallel by default. Each job starts with a fresh VM — no files or state from other jobs unless you explicitly pass artifacts. Use needs: to create dependencies. Each job can have different environments, conditionals, and runner types.

Step: a single action within a job. Steps run sequentially on the same VM. They share the workspace (files from earlier steps are available). A step is either a shell command (run:) or a reusable action (uses:). Because steps share the workspace, step 2 can use files built by step 1.
⚡ Memory: Jobs = parallel VMs. Steps = sequential commands on one VM. If you need to share files between units → use steps. If you want parallelism → use jobs with artifacts.
Tricky
How do you securely pass secrets to a pipeline without hardcoding them?
▾
GitHub Secrets are the answer — encrypted at rest, masked in logs, never exposed in PR builds from forks. Store via GitHub UI (Settings → Secrets → Actions) or GitHub CLI. Reference in YAML: ${{ secrets.MY_SECRET }}.

For more structure: use Environments (staging, production) — each environment can have its own secrets and require mandatory reviewer approval before running. Production environment secrets are never exposed to staging jobs.

What NOT to do: hardcode in YAML (visible in git history forever), use env variables for sensitive values without secrets, print secrets to logs even "temporarily."
⚠️ Secrets from pull requests submitted from forks do NOT have access to repo secrets by default — GitHub protects against supply-chain attacks this way.
Tricky
Why is caching important in CI pipelines and how does it work?
▾
Every CI run starts on a fresh VM — no prior state. Without caching, Maven/Gradle downloads all dependencies (100s of MB) on every single run, adding 2–5 minutes per pipeline. Multiply by 50 developers × 10 pushes/day = significant waste.

The actions/cache action stores a named snapshot between runs. The cache key includes a hash of the dependency file (pom.xml, package-lock.json). If dependencies haven't changed, the cache hits and restores in seconds. When pom.xml changes, the hash changes, the old cache misses, and a fresh cache is built with the updated dependencies.
💡 Good caching practice: always include the lock file hash in the cache key. Use restore-keys for fallback to a partial match when the exact key misses.
💡
GitHub Actions anatomy: Event → Workflow → Jobs → Steps

Key YAML fields:
• on: push/pr/schedule/dispatch → WHEN to run
• runs-on: ubuntu-latest → WHERE to run
• needs: [job-name] → run AFTER another job
• if: condition → run ONLY WHEN condition is true
• uses: action@version → reuse community actions
• run: command → execute shell command

Secrets rule: never hardcode. Always ${{ secrets.NAME }}.
Caching rule: hash the lock file in the cache key.

Pipeline Patterns & Best Practices
Fast feedback, deployment strategies, rollbacks — patterns used by high-performing teams
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: A great CI/CD pipeline isn't just "run tests then deploy." It uses smart patterns: fail fast (run quick tests first), deploy gradually (canary before full rollout), test in production with flags, and automate rollbacks when metrics go wrong. These patterns reduce risk while maintaining deployment speed.
⚡
Fail Fast Principle — Order pipeline stages from cheapest to most expensive. Compilation (5s) → Unit tests (30s) → Integration tests (2min) → E2E tests (10min) → Deploy. If compilation fails, don't run tests. If unit tests fail, don't run integration tests. This minimises wasted compute and gives developers fast feedback on simple mistakes.
🐦
Canary Deployment — Deploy the new version to a small percentage of traffic (5–10%) first. Monitor error rates and latency. If metrics are healthy after 10 minutes, gradually roll out to 25%, 50%, 100%. If metrics degrade, automatically roll back the canary. Kubernetes + Argo Rollouts automates this. You get real production testing without full blast radius.
🔵🟢
Blue-Green Deployment — Run two identical production environments (Blue = current, Green = new). Deploy to Green while Blue serves traffic. Switch the load balancer from Blue to Green (instant, no downtime). If Green has problems, switch back to Blue in seconds. Requires double the infrastructure during deployment but gives near-zero-downtime deployments with instant rollback.
🏳️
Feature Flags — Deploy code without activating it. Use a feature flag service (LaunchDarkly, Unleash) to enable a feature for 1% of users, specific users, or specific environments. Decouple code deployment from feature release. Roll back by disabling the flag — no redeployment. Dark launching: deploy to all but show to 0% until testing complete.
↩️
Automated Rollback — Monitor deployment health metrics (error rate, p99 latency, HTTP 5xx rate). If metrics cross thresholds within N minutes of a deployment, automatically trigger a rollback to the previous known-good version. Kubernetes rolling deployments have built-in rollback: kubectl rollout undo deployment/app. Argo Rollouts automates analysis + rollback with Prometheus metrics.
Pipeline Patterns — Matrix Tests + Canary + Blue-Green Config
# ── Matrix testing: test on multiple JDK versions ────
jobs:
  test:
    strategy:
      matrix:
        java: [17, 21]
        os: [ubuntu-latest, windows-latest]
      fail-fast: false   # continue other matrix jobs on failure
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/setup-java@v4
        with:
          java-version: ${{ matrix.java }}
      - run: mvn test

# ── Semantic versioning from git tags ─────────────────
on:
  push:
    tags: [ 'v*.*.*' ]    # triggers on v1.2.3

steps:
  - name: Extract version
    run: echo "VERSION=${GITHUB_REF#refs/tags/v}" >> $GITHUB_ENV
  - name: Build with version
    run: mvn package -Dapp.version=${{ env.VERSION }}

# ── Blue-Green: switch K8s service selector ───────────
# Blue deployment (current live)
# apiVersion: apps/v1
# kind: Deployment
# metadata:
#   name: app-blue
# spec:
#   selector:
#     matchLabels: { app: myapp, version: blue }

# Green (new version)
# metadata:
#   name: app-green
# spec:
#   selector:
#     matchLabels: { app: myapp, version: green }

# Service — switch from blue to green by updating selector:
# kubectl patch svc app -p '{"spec":{"selector":{"version":"green"}}}'
# Instant traffic switch. Rollback: patch back to "blue"

# ── Automated rollback trigger in pipeline ────────────
steps:
  - name: Deploy
    run: kubectl rollout status deployment/app --timeout=5m
  - name: Smoke test post-deploy
    run: ./scripts/smoke-test.sh
  - name: Rollback on failure
    if: failure()
    run: |
      kubectl rollout undo deployment/app
      echo "Deployment failed! Rolled back."
Must Know
What is the difference between Blue-Green and Canary deployments?
▾
Blue-Green: two complete production environments. You deploy to the idle one (Green), test it, then switch ALL traffic at once. Instant switchover, instant rollback. Pro: zero-downtime. Con: requires 2x infrastructure cost during deployment. Best for: planned releases where you want tested, instant switchover.

Canary: gradually shift traffic from old to new version — 5% → 25% → 50% → 100%. The new version gets real production traffic at low blast radius. You observe real metrics before full rollout. Pro: catches issues with small user impact. Con: two versions running simultaneously for an extended period (complexity, state management). Best for: frequent releases where gradual exposure reduces risk.
⚡ Blue-Green = all-or-nothing switch. Canary = gradual percentage shift. Canary is safer for risky releases; Blue-Green is faster for high-confidence releases.
Senior
How do feature flags change the relationship between deployment and release?
▾
Traditionally, deployment = release. The moment code goes to production, users see it. Feature flags break this coupling: code can be deployed to production but hidden behind a flag that returns false for all users.

This enables: Continuous deployment without continuous release — you can merge and deploy every day even if features aren't ready for users. Controlled rollout — enable for 1% of users, monitor, gradually increase. Instant rollback — disable the flag without redeployment. A/B testing — show feature A to 50%, feature B to 50%, compare metrics. Kill switch — if a feature causes issues, disable it in seconds without a pipeline run.
💡 Feature flags are how large companies (Facebook, GitHub) deploy multiple times per day to hundreds of millions of users without risk. They're a key enabler of continuous deployment.
💡
Pipeline best practices:
• Fail fast: cheap tests first, expensive tests last
• Matrix: test across multiple Java/OS versions in parallel
• Canary: 5% → 25% → 100% with metric gates
• Blue-Green: instant traffic switch, instant rollback
• Feature flags: deploy code without releasing to users
• Automated rollback: if smoke tests fail → undo immediately

The goal: deploy often with confidence. Confidence comes from tests + gradual rollout + observability + fast rollback.

Docker Fundamentals
Images, containers, registries, and the key commands every developer needs daily
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Docker solves "it works on my machine." A container bundles your application AND all its dependencies (JDK, libraries, configs) into one portable unit. Run the same container image on your laptop, CI server, and production Kubernetes — identical behavior, no more environment drift. Unlike VMs, containers share the host OS kernel — they start in milliseconds and use megabytes of memory instead of gigabytes.
📸
Image — An immutable, read-only snapshot of your application and its environment. Built from a Dockerfile. Composed of layers — each instruction in a Dockerfile adds a layer. Images are stored in registries (Docker Hub, ECR, GCR). An image is the blueprint. You can't change an image — you rebuild it. Tag versions: myapp:1.0.0, myapp:latest.
📦
Container — A running instance of an image. Lightweight, isolated process. Has its own filesystem (from the image), network, and process space. Ephemeral by default — containers are designed to be disposable. Write a file in a container? It's lost when the container stops unless you use a volume. Multiple containers can run from the same image simultaneously.
📚
Layers & Caching — Each Dockerfile instruction creates a read-only layer. Docker caches layers — if a layer hasn't changed, Docker reuses it. Order matters: put infrequently-changing instructions first (base image, system deps), frequently-changing instructions last (application code). Copy dependencies BEFORE source code so the dependency layer is cached even when code changes.
🗂️
Registry — A storage and distribution service for Docker images. Docker Hub: default public registry, free tier available. AWS ECR: Amazon's private registry, integrates with ECS/EKS. GitHub Container Registry (ghcr.io): integrated with GitHub repos. Google Artifact Registry: GCP. Push an image: docker push myorg/myapp:1.0. Pull: docker pull myorg/myapp:1.0.
💾
Volumes — Persist data beyond container lifecycle. Managed by Docker, stored on the host. Mount a volume: -v myvolume:/data. Bind mount: -v /host/path:/container/path. Use volumes for: database data, config files, uploaded files. Never store stateful data in a container's filesystem — it disappears with the container.
🌐
Networking — Containers on the same Docker network can communicate by container name (DNS). Default bridge network: containers use IP addresses. User-defined networks: containers use names. docker network create mynet, then docker run --network mynet. Port mapping: -p 8080:8080 (host:container).
◆ Memorizable Code — Essential Docker Commands
Docker Commands — Build, Run, Inspect, Clean
# ── Build image ───────────────────────────────────────
docker build -t myapp:1.0 .           # build from ./Dockerfile
docker build -t myapp:1.0 -f Dockerfile.prod .  # custom Dockerfile
docker build --no-cache -t myapp:1.0 . # force rebuild all layers

# ── Run container ─────────────────────────────────────
docker run -p 8080:8080 myapp:1.0             # foreground
docker run -d -p 8080:8080 myapp:1.0          # detached (background)
docker run -d --name myapp -p 8080:8080 \
  -e SPRING_PROFILES_ACTIVE=prod \             # env variable
  -e DB_PASSWORD=secret \
  -v /data:/app/uploads \                      # mount volume
  --network mynet \                            # custom network
  --restart unless-stopped \                  # auto-restart policy
  myapp:1.0

# ── Container lifecycle ────────────────────────────────
docker ps                     # running containers
docker ps -a                  # all containers (incl. stopped)
docker stop myapp             # graceful stop (SIGTERM → 10s → SIGKILL)
docker start myapp            # start stopped container
docker rm myapp               # remove stopped container
docker rm -f myapp            # force remove (even if running)

# ── Inspect and debug ─────────────────────────────────
docker logs myapp             # view stdout/stderr
docker logs -f myapp          # follow logs (tail -f)
docker exec -it myapp bash    # open shell inside running container
docker inspect myapp          # full container metadata (JSON)
docker stats                  # live CPU/memory usage
docker top myapp              # processes running inside

# ── Images ────────────────────────────────────────────
docker images                 # list local images

docker push myorg/myapp:1.0   # upload to registry
docker tag myapp:1.0 myorg/myapp:1.0  # add registry tag

# ── Networking ────────────────────────────────────────

# ── Cleanup ────────────────────────────────────────────
docker system prune -a        # remove all unused (images, containers, etc.)

Must Know
What is the difference between a Docker image and a container?
▾
An image is a static, immutable snapshot — a read-only template containing the application and all its dependencies. Think of it as a class definition. You can't change an image; you rebuild it. Images are stored in registries and versioned with tags.

A container is a running instance of an image — it's the image brought to life as an isolated process. Think of it as an instance of a class. Multiple containers can run from the same image. Containers have writable layers on top of the image's read-only layers — any writes go there. When the container stops, its writable layer is lost (unless using volumes).
⚡ Analogy: Image = class definition (static blueprint). Container = object instance (running, has state). You can create many containers from one image.
Tricky
How does Docker layer caching work and why does layer order matter?
▾
Docker builds images layer by layer. Each Dockerfile instruction creates a new layer. Docker caches layers — if a layer is unchanged, Docker reuses the cached version. If a layer changes, all layers AFTER it must be rebuilt too (cache invalidated).

Order matters because of this chain invalidation. Wrong order: COPY . . (source code) then RUN mvn install (download deps) — every source change forces re-downloading all dependencies. Correct order: COPY pom.xml ., then RUN mvn dependency:resolve (downloads deps — cached until pom.xml changes), THEN COPY src ./src, then RUN mvn package. Now dependencies download once and are cached for subsequent code-only changes.
⚠️ The golden rule: put things that change INFREQUENTLY at the top. Things that change FREQUENTLY at the bottom. Dependencies before source code — always.
Concept
What is the difference between Docker volumes and bind mounts?
▾
Docker Volumes: managed by Docker, stored in a special Docker area on the host (/var/lib/docker/volumes/). Lifecycle independent of containers. Easy to backup, migrate. Access via -v volumename:/path. Best for: production database data, stateful app data. Don't depend on specific host path.

Bind Mounts: map a specific host directory into the container. -v /host/absolute/path:/container/path. Container sees host files directly. Best for: local development (mount source code for hot-reload), configuration files. Depends on specific host path — less portable across environments.
💡 Production rule: use named volumes for data that needs to persist. Use bind mounts only for local development hot-reload or config injection.
💡
Docker solves "works on my machine" by bundling everything into one portable image.

Core concepts:
• Image = immutable blueprint (class)
• Container = running instance (object)
• Registry = image storage (Docker Hub, ECR, GCR)
• Volume = persistent data storage
• Layer = each Dockerfile instruction, cached

Daily commands: build → run → logs → exec → stop → rm

Layer caching rule: infrequent changes first (deps), frequent changes last (code).

Dockerfile — Writing Production-Grade Images
Multi-stage builds, layer optimisation, security hardening, and Spring Boot best practices
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: A Dockerfile is a recipe for building a Docker image. Each instruction (FROM, RUN, COPY, etc.) adds a layer. The key technique is multi-stage builds: use a big "builder" image (JDK 21 + Maven) to compile your code, then copy only the compiled JAR into a small "runtime" image (JRE only). Your final image goes from 500MB to 80MB, has no build tools, and a smaller attack surface.
🏗️
Multi-Stage Builds — Use multiple FROM instructions. Each creates a new stage. You can selectively COPY --from=stage-name artefacts between stages. The final image only contains the last stage. Benefits: small production image (no JDK, Maven, build tools), faster pulls, smaller attack surface. The builder stage is discarded — it never appears in your registry or production environment.
📏
Key Dockerfile Instructions — FROM: base image. WORKDIR: set working directory (create if not exists). COPY: copy files from host. RUN: execute command, creates a layer. ENV: set environment variable. EXPOSE: document the port (doesn't publish it — just documentation). ENTRYPOINT: main command. CMD: default arguments. USER: switch to non-root user.
🔐
Security Best Practices — Run as non-root user (USER nonroot). Use minimal base images (distroless, alpine). Don't store secrets in the image (use env vars or mounted secrets). Scan images for vulnerabilities (docker scout, Trivy, Snyk). Pin base image versions — FROM eclipse-temurin:21-jre-alpine not FROM openjdk:latest. Running as root inside a container is a common security misconfiguration.
🍃
Spring Boot Layered JARs — Spring Boot 2.3+ supports layered JARs. Extract with java -Djarmode=layertools -jar app.jar extract. Creates separate directories for: dependencies (rarely change), spring-boot-loader, snapshot-dependencies, application code (changes most). Each directory becomes a Docker layer — Docker caches the unchanged dependency layers. Rebuilding after a code-only change only rebuilds the tiny application layer (seconds vs minutes).
📝
.dockerignore — Like .gitignore but for Docker build context. Exclude: target/, .git/, *.md, node_modules/, .env. Smaller build context = faster builds. Also prevents accidentally copying local environment files (credentials) into the image. Always create a .dockerignore alongside your Dockerfile.

What is a multi-stage Docker build and why is it important for Java apps?
▾
A multi-stage build uses multiple FROM instructions. Each stage can use a different base image. You copy artefacts between stages with COPY --from=stage-name.

For Java: Stage 1 (builder): full JDK 21 + Maven (~500MB) to compile the code. Stage 2 (runtime): just JRE (~80MB) to run the compiled JAR. The final image only contains Stage 2 — Stage 1 is discarded.

Why critical: a naive image with the full JDK is 500MB+. Multi-stage gives you ~80MB images. No build tools = smaller attack surface. No source code in production image. Faster pulls, faster startup. The build environment is completely separate from the runtime environment.
⚡ Without multi-stage: 500MB image with JDK, Maven, source code. With multi-stage: 80MB image with JRE + JAR only. Always use multi-stage for production Java images.
Tricky
Why run containers as a non-root user and how do you do it in Dockerfile?
▾
By default, Docker containers run as root. If an attacker exploits a vulnerability in your app, they get root access inside the container — and potentially the host if other vulnerabilities are combined (container escape).

Running as a non-root user is defense-in-depth: even if the app is exploited, the attacker has limited permissions. It's also a requirement for many Kubernetes environments (Pod Security Standards, most production clusters run with runAsNonRoot: true).

In Dockerfile: RUN addgroup -S appgroup && adduser -S appuser -G appgroup, then USER appuser:appgroup before ENTRYPOINT. The colon notation specifies both user and group.
⚠️ Many CI/CD security scanners (Trivy, Snyk) will fail images that run as root. Non-root is required for production Kubernetes deployments under Pod Security Standards.
Senior
How does Spring Boot's layered JAR improve Docker build performance?
▾
A plain Spring Boot fat JAR puts everything in one layer — changing one line of code invalidates the entire layer (100MB+ re-pushed to registry on every change). Spring Boot layered JARs split the fat JAR into ordered layers: dependencies (changes rarely — quarterly), spring-boot-loader (changes with Spring Boot upgrade), snapshot-dependencies (changes weekly), application code (changes multiple times per day).

In the Dockerfile, each layer is a separate COPY instruction — Docker caches each independently. A code-only change: only the application layer rebuilds (tiny, <1MB). Dependencies, loader, and snapshots are all served from cache. Result: rebuilds go from 2–5 minutes to seconds. Registry pushes go from 100MB to 1MB per deploy.
💡
Production Dockerfile checklist:
✓ Multi-stage: JDK to build → JRE to run
✓ Dependency layer before source code (cache!)
✓ Layered JAR for Spring Boot (COPY deps → loader → app)
✓ Non-root USER before ENTRYPOINT
✓ Pin base image version (not :latest)
✓ HEALTHCHECK for container orchestration
✓ .dockerignore (exclude target/, .git/, .env)
✓ JVM flags: -XX:+UseContainerSupport -XX:MaxRAMPercentage=75

Image size goal: <100MB for Java apps with JRE alpine base.

Must Know
Why does depends_on not ensure a service is ready, and how do you fix it?
▾
depends_on: db ensures Docker starts the db container before the app container. However, "started" just means the container process began — not that PostgreSQL inside has finished initialising and is accepting connections. PostgreSQL takes 2–5 seconds to start up after the container launches.

Without fix: your app starts, tries to connect to PostgreSQL, gets "connection refused", crashes, and Docker may or may not restart it depending on restart policy. Fix: add a healthcheck to the db service that uses pg_isready, then in app's depends_on use condition: service_healthy. Compose now waits until pg_isready returns success before starting the app.
⚡ Always add healthchecks to database services and use condition: service_healthy in depends_on. This is one of the most common Docker Compose mistakes.
Concept
When should you use Docker Compose vs Kubernetes?
▾
Docker Compose: single machine, simple services, local development, CI environment spin-up, small deployments. No auto-scaling, no self-healing across nodes, no load balancing across hosts. Great for: local dev stack, integration testing in CI, small internal tools.

Kubernetes: multi-node clusters, production workloads, auto-scaling, self-healing, rolling deployments, service discovery across nodes, resource management. Much more complex setup. Use when: high availability required, multiple replicas, horizontal scaling, zero-downtime deployments, managing dozens of microservices.
💡 Rule: development → Docker Compose. Testing → Docker Compose. Staging and production → Kubernetes (or managed equivalents like AWS ECS, Azure Container Apps).

What is a Pod and why is it the fundamental unit in Kubernetes rather than a container?
▾
A Pod abstracts a group of co-located containers that logically belong together. All containers in a pod share: the same network namespace (they communicate via localhost), the same storage volumes, and the same lifecycle. This enables the sidecar pattern: a main app container + helper containers (log forwarder, proxy, secret injector) that need to be tightly coupled.

K8s uses pods rather than containers as the unit because: (1) some workloads naturally need tightly-coupled helpers, (2) the pod provides a stable abstraction — the scheduler works with pods as atomic units, and (3) network and storage can be bound at the pod level (a single IP, shared volumes) rather than per-container, making the networking model simpler.
⚡ In practice: 99% of pods have exactly one app container. Use multiple containers per pod only for sidecars that must share localhost network or volumes (Envoy proxy, log agents).
Tricky
What happens when a Pod is deleted in Kubernetes?
▾
If the pod was created directly (bad practice): it's deleted permanently. Nothing recreates it.

If managed by a Deployment or ReplicaSet (normal practice): the controller immediately notices the pod count dropped below the desired replicas and creates a new pod. The new pod gets a new name and potentially runs on a different node — it's a fresh container. Any data in the old pod's local filesystem is gone.

This is why pods are called "ephemeral" and why stateful data must go in Volumes or external storage. The Deployment ensures the desired number of pods always run — you can delete pods freely during debugging and the system heals itself.
⚠️ Never create Pods directly in production — always use Deployments (or StatefulSets for stateful workloads). A lone Pod has no self-healing.
💡
Kubernetes architecture in one sentence: Control Plane (brain) schedules and manages Pods on Worker Nodes (muscle).

Key objects hierarchy: Cluster → Namespace → Node → Pod → Container

Pod = smallest deployable unit. Ephemeral. Gets new IP on restart. Never address by IP — use Service.

🧠 Memory
🧠
In plain English: Kubernetes has named building blocks. Deployment manages your app replicas and handles rolling updates. Service gives your pods a stable address and load balances traffic between them. ConfigMap stores non-sensitive config. Secret stores sensitive config (DB passwords, API keys). Together, these four objects cover 90% of what you deploy in K8s.
📦
Deployment — Manages a ReplicaSet which manages Pods. You declare: "I want 3 replicas of my-app:v2". K8s makes it happen and keeps it that way. Handles: rolling updates (gradually replace old pods), rollbacks (kubectl rollout undo), scaling (kubectl scale). Deployment → ReplicaSet → Pods. Use Deployment for stateless apps.
🌐
Service — A stable network endpoint for a group of Pods. Pods die and get new IPs — Services provide a consistent address. Types: ClusterIP (internal only — default), NodePort (expose on each node's port, dev use), LoadBalancer (creates cloud load balancer, production), ExternalName (DNS alias to external service). Selects pods by label selector.
⚙️
ConfigMap — Stores non-sensitive configuration as key-value pairs. Inject into pods as: environment variables, command-line arguments, or files mounted via volume. Separate config from image — same image runs in dev/staging/prod with different ConfigMaps. Update ConfigMap → pods pick up new config on restart (or rolling restart: kubectl rollout restart).
🔐
Secret — Like ConfigMap but for sensitive data (passwords, tokens, certificates). Base64 encoded (NOT encrypted by default — enable etcd encryption at rest for real security). Best practice: don't store credentials in Secrets; use external secret managers (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault) synced to K8s Secrets via operators (External Secrets Operator).
🚪
Ingress — HTTP/HTTPS routing for external traffic. One Ingress controller (NGINX, Traefik) handles all incoming HTTP traffic and routes based on host/path rules. api.example.com/orders → orders Service. api.example.com/users → users Service. Handles TLS termination. Replaces needing a LoadBalancer Service per microservice.
📊
Resource Requests & Limits — requests: minimum resources guaranteed (used by scheduler). limits: maximum allowed (container killed if exceeded). CPU: measured in millicores (250m = 0.25 cores). Memory: in bytes (256Mi = 256 megabytes). Always set both — without limits, one pod can starve others. Without requests, scheduler can't make good placement decisions.

Must Know
What is the difference between liveness and readiness probes?
▾
Liveness probe: answers "Is this container alive?" If it fails, Kubernetes kills the container and restarts it. Use for: detecting dead locks, infinite loops, unrecoverable application states. Map to Spring Boot Actuator's /actuator/health/liveness. Fail carefully — false positives cause restart loops. Only fail for genuinely unrecoverable states.

Readiness probe: answers "Is this container ready to receive traffic?" If it fails, Kubernetes removes the pod from the Service's endpoints (stops sending traffic). Container is NOT restarted. Use for: startup not complete yet, DB connection temporarily lost, temporarily overloaded. Map to /actuator/health/readiness.
⚡ Liveness = restart. Readiness = remove from rotation. Never fail liveness for temporary issues (DB restart) — that kills your pod unnecessarily. Fail readiness instead.
Tricky
What is the difference between requests and limits for CPU and memory?
▾
Requests: the minimum resources guaranteed to a pod. The scheduler uses this to place pods on nodes that have at least this much available. If requests = 250m CPU, the pod gets at least 250m CPU when available.

Limits: the hard maximum. CPU limit: if exceeded, CPU is throttled (not killed). Memory limit: if exceeded, the container is OOMKilled (killed and restarted). Always set both.

Practical advice: set memory requests = memory limits (avoid unexpected OOM). Set CPU requests = expected average usage, CPU limits = 2x requests (allows bursting). Without limits, one pod can consume entire node resources, starving others. Without requests, scheduler makes poor placement decisions.
⚠️ OOMKilled is one of the most common issues in K8s. Always set memory limits. Check with kubectl describe pod — if Reason: OOMKilled, increase memory limit.
Tricky
What is the difference between a ClusterIP, NodePort, and LoadBalancer Service?
▾
ClusterIP: creates a stable virtual IP only reachable within the cluster. Default type. Use for: service-to-service communication within the cluster. Other pods reach it by service name DNS. Not accessible from outside.

NodePort: exposes the service on a specific port (30000–32767) on EVERY node. External traffic hits any_node_ip:node_port → routed to the service. Useful for quick testing. Not for production (you need to know node IPs, ports are in ugly range).

LoadBalancer: provisions a cloud load balancer (AWS ALB, Azure LB, GCP LB) pointing to your service. Gets an external IP. Best for production external access. Expensive: each LoadBalancer costs money. Better: use one Ingress controller with one LoadBalancer in front of all services.
💡
Core K8s objects for every app:
• Deployment → manages replicas + rolling updates
• Service (ClusterIP) → stable DNS name for pods
• ConfigMap → non-sensitive config as env vars or files
• Secret → sensitive config (encrypted at rest ideally)
• Ingress → HTTP routing from outside world

Probe rule: Liveness = restart on unrecoverable. Readiness = remove from traffic on temporary issue. Both use /actuator/health/* in Spring Boot.

Always set requests AND limits. Memory: requests=limits. CPU: limits=2×requests.

In plain English: Basic K8s knowledge gets your app running. Horizontal Pod Autoscaler (HPA) — Automatically scales replica count based on metrics. CPU: scale up when average CPU across pods exceeds 70%. Custom metrics: scale based on Kafka lag, HTTP request queue depth, or any Prometheus metric. Define min and max replicas. KEDA extends HPA for event-driven scaling (scale to zero, scale on message queue depth). Essential for handling traffic spikes without over-provisioning 24/7.
🗃️
StatefulSet vs Deployment — Use StatefulSet for stateful apps (databases, Kafka, ZooKeeper). StatefulSet provides: stable, unique pod names (pod-0, pod-1), stable persistent volume per pod, ordered startup/shutdown (pod-0 starts before pod-1). Deployment pods are interchangeable and stateless. Use StatefulSet when each instance needs its own identity (Kafka partition leader, DB replica role).
🛡️
PodDisruptionBudget (PDB) — Limits the number of pods that can be down simultaneously during voluntary disruptions (node drain, rolling updates, cluster upgrades). minAvailable: 2: at least 2 pods must always be running. K8s won't drain a node if it would violate the PDB. Critical for high availability — prevents maintenance from taking down your entire service.
🏗️
Pod Affinity & Anti-Affinity — Control pod placement. Anti-affinity: spread pods across different nodes so a single node failure doesn't take out all replicas. podAntiAffinity: preferredDuringScheduling: soft preference. requiredDuringScheduling: hard rule (pod won't schedule if violated). Production rule: use anti-affinity for all multi-replica deployments to ensure nodes and availability zones are balanced.
💾
PersistentVolume & PersistentVolumeClaim — For stateful workloads needing persistent storage. PV: a piece of storage provisioned by cluster admin (or dynamic provisioner). PVC: a request for storage by a pod. StorageClass: defines provisioner and parameters (fast SSD, slow HDD, replicated). Pods request storage via PVC — abstract, doesn't need to know underlying storage. Dynamic provisioning: cloud creates the volume automatically when a PVC is created.

K8s Production — HPA + PodDisruptionBudget + Anti-Affinity
# ── HPA: auto-scale on CPU ────────────────────────────
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # scale up when >70% avg CPU
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # wait 5min before scale down
---
# ── PodDisruptionBudget: HA during maintenance ────────
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
spec:
  selector:
    matchLabels:
      app: myapp
  minAvailable: 2    # at least 2 pods always running
  # OR: maxUnavailable: 1
---
# ── Anti-Affinity: spread pods across nodes ───────────
# In Deployment spec.template.spec:
affinity:
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:  # soft rule
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchLabels:
              app: myapp
          topologyKey: kubernetes.io/hostname  # spread across nodes
  # For zone spread (production HA):
  # topologyKey: topology.kubernetes.io/zone
---
# ── Namespace ResourceQuota ────────────────────────────
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: myteam
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"
---

Must Know
What is a PodDisruptionBudget and why is it critical for production?
▾
A PodDisruptionBudget (PDB) defines the minimum number of pods that must remain available during voluntary disruptions — node drains, cluster upgrades, rolling deploys, or manual node maintenance.

Without PDB: when an admin drains a node (moving all pods to other nodes), Kubernetes could move all your pods simultaneously, causing a brief outage. Or during a rolling cluster upgrade, nodes could be drained faster than your pods can reschedule.

With PDB (minAvailable: 2): Kubernetes checks before draining a node — if draining would leave fewer than 2 pods available, it waits. This guarantees your service maintains minimum capacity even during maintenance operations.
⚡ Always set a PDB for any production service with multiple replicas. Without it, cluster maintenance can unexpectedly take down your service entirely.
Tricky
What is CrashLoopBackOff and how do you diagnose it?
▾
CrashLoopBackOff means your container starts, crashes immediately, Kubernetes tries to restart it, it crashes again, and K8s begins adding exponential backoff delays between restarts (10s → 20s → 40s → up to 5 minutes).

Diagnosis steps:
1. kubectl describe pod pod-name → look at Events and Last State (exit code, reason)
2. kubectl logs pod-name --previous → logs from the CRASHED container (not current)
3. Common causes: app throws exception on startup (check logs), misconfigured env var (DB URL wrong, missing secret), OOMKilled immediately (memory limit too low), liveness probe too aggressive (failing before app starts — increase initialDelaySeconds)
⚠️ Always use --previous flag with kubectl logs when troubleshooting CrashLoopBackOff — the current pod may be restarting and have no logs yet. You need the previous crashed pod's logs.
Senior
How does HPA decide when to scale and what are its limitations?
▾
HPA runs a control loop every 15 seconds. It calculates desired replicas: desiredReplicas = ceil(currentReplicas × (currentMetricValue / targetMetricValue)). If 3 pods averaging 90% CPU with target 70%: ceil(3 × 90/70) = ceil(3.86) = 4 replicas.

Scale-up is fast (within 3 minutes). Scale-down is intentionally slow (stabilization window, default 5 minutes) to prevent thrashing from brief traffic spikes.

Limitations: HPA can't scale beyond maxReplicas — if traffic exceeds what maxReplicas can handle, requests queue or fail. CPU metrics are delayed (~1 min) — by the time HPA scales up, a traffic spike may have already caused degradation. Custom metrics (Kafka lag, queue depth) from KEDA react faster to specific workload patterns. HPA doesn't help with stateful workloads (use StatefulSet with manual scaling).
💡
Production K8s checklist:
✓ HPA: minReplicas=2, maxReplicas=N, scale on CPU 70%
✓ PDB: minAvailable=2 (never go below 2 during maintenance)
✓ Anti-affinity: spread pods across nodes (topologyKey: hostname)
✓ Zone spread: for multi-AZ HA (topologyKey: zone)
✓ Resource limits: memory requests=limits, CPU limits=2×requests
✓ Liveness: restart on unrecoverable states only
✓ Readiness: remove from rotation on temporary issues

Troubleshooting order: describe pod → Events → logs --previous → exec bash
