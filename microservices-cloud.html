🔪
Decomposition Strategies — By Business Capability (DDD bounded contexts — User, Order, Payment, Inventory — recommended). By Subdomain (core, supporting, generic subdomains). Strangler Fig Pattern — gradually replace monolith by routing traffic incrementally to new microservices while keeping the monolith alive.
🗃️
Database Per Service — Each service owns its data store exclusively. No shared DB. Enables: independent scaling, polyglot persistence (each service picks the best DB for its needs). Challenge: distributed transactions — use SAGA pattern (choreography or orchestration) instead of 2PC.
📡
Inter-Service Communication — Synchronous: REST (HTTP) or gRPC (binary, faster, schema-driven). Asynchronous: Message brokers (Kafka, RabbitMQ) — loose coupling, better resilience. Use async when: result not immediately needed, fan-out to multiple consumers, eventual consistency is acceptable.
⚠️
Microservices Challenges — Distributed system problems: network latency, partial failures, data consistency, service discovery, distributed tracing, increased operational complexity. Not suitable for small teams or early-stage projects. Don't microservice-ize prematurely — start with a modular monolith.
📋
SAGA Pattern — Manages distributed transactions without 2PC. Choreography: services emit events, each service reacts and emits next event (event-driven, loose coupling, harder to debug). Orchestration: a central SAGA orchestrator tells each service what to do (centralized logic, easier to track, more coupling). Each step has a compensating transaction for rollback.
🏗️
12-Factor App Principles — The blueprint for cloud-native services: codebase in version control, dependencies declared explicitly, config in environment, backing services as attached resources, build/release/run stages separate, stateless processes, port binding, concurrency, disposability, dev/prod parity, logs as streams, admin processes.

Spring Boot Microservice — Setup + Inter-Service REST + Feign
// ── Spring Boot microservice entry point ──────────
@SpringBootApplication
@EnableDiscoveryClient        // register with Eureka
@EnableFeignClients            // enable Feign HTTP clients
public class OrderServiceApp {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApp.class, args);
    }
}

// ── Feign Client: calls UserService via service name ─
@FeignClient(name = "user-service")   // resolves via Eureka
public interface UserServiceClient {

    @GetMapping("/users/{id}")
    UserDto getUserById(@PathVariable Long id);

    @PostMapping("/users/validate")
    boolean validateUser(@RequestBody UserRequest req);
}

// ── Service using Feign ───────────────────────────
@Service
public class OrderService {
    private final UserServiceClient userClient;
    private final OrderRepository orderRepo;

    public OrderService(UserServiceClient uc, OrderRepository or) {
        this.userClient = uc;
        this.orderRepo  = or;
    }

    public Order placeOrder(Long userId, OrderRequest req) {
        UserDto user = userClient.getUserById(userId); // HTTP call via Feign
        if (user == null) throw new UserNotFoundException(userId);

        Order order = Order.builder()
            .userId(userId)
            .items(req.getItems())
            .status(OrderStatus.PENDING)
            .build();
        return orderRepo.save(order);
    }
}

◆ Interview Questions
Must Know
What is the difference between microservices and a monolith? When would you choose each?
▾
A monolith is a single deployable unit with all components tightly integrated — simple to develop, test, and deploy early on, but hard to scale parts independently. Microservices decompose the application into small, independently deployable services — each with its own data store, technology, and deployment pipeline. Choose monolith when: small team, early product, unclear domain boundaries. Choose microservices when: multiple teams working independently, need to scale specific components, need polyglot tech stacks. Modular monolith is often the right middle ground — clean internal boundaries without distribution complexity.
⚡ "Start with a monolith, extract microservices when you feel the pain" — Martin Fowler's recommendation.
Tricky
How do you handle distributed transactions across microservices without 2PC?
▾
Two-Phase Commit (2PC) is avoided in microservices because it creates tight coupling and availability issues. The alternative is the SAGA pattern:
Choreography SAGA: Each service publishes events after completing its local transaction; other services react. No central coordinator. Loose coupling but hard to track overall state.
Orchestration SAGA: A SAGA orchestrator service sends commands to each service and handles responses/compensations. Easier to manage flow but introduces coupling to the orchestrator.
Each step has a compensating transaction — if Payment fails, OrderService cancels the order, InventoryService restocks. Use event sourcing for audit trail.
⚡ In interviews: always mention compensating transactions for rollback — this shows you understand eventual consistency.
Tricky
What is Conway's Law and how does it relate to microservices?
▾
Conway's Law states: "Organizations design systems that mirror their own communication structure." Applied to microservices: your service boundaries should map to your team boundaries. A team that owns User and Orders will naturally create a monolith for both — because they communicate constantly. For microservices to succeed, you need teams that can work independently with minimal inter-team coordination. This is why the Inverse Conway Maneuver is used — restructure teams first (around business capabilities), then the architecture follows naturally.
Concept
Explain the Database Per Service pattern. What challenges does it create?
▾
Each microservice has its own private data store — no other service can access it directly (no shared DB). Benefits: services are truly independent (schema changes don't break others), can choose the best DB per service (polyglot persistence). Challenges: 1. No JOIN across services — must compose data in application code. 2. Distributed transactions — use SAGA. 3. Eventual consistency — data may be temporarily out of sync across services. 4. Reporting queries across services — use a separate read-model/CQRS pattern or a dedicated analytics DB fed by events.
Must Know
What is the Strangler Fig pattern and when is it used?
▾
The Strangler Fig pattern is a migration strategy for gradually replacing a monolith with microservices. Like a strangler fig vine that grows around a tree and eventually replaces it: 1) Deploy the new microservice alongside the monolith. 2) Route specific traffic to the new service (via API Gateway or routing layer). 3) Gradually migrate more functionality to microservices. 4) Retire the monolith when fully migrated. Benefits: no big-bang rewrite, validate each piece independently, rollback easily. Key tool: API Gateway as the routing layer controls traffic splitting.
◆ Memory Tip
💡
Microservices = team topology made tangible. Each service = one team = one business capability = one DB = one deployment pipeline. If two services always need to be deployed together, they should probably be one service.

Key patterns cheat sheet:
• Distributed transactions → SAGA (Choreography or Orchestration)
• Monolith migration → Strangler Fig
• Data consistency → Eventual Consistency + Event Sourcing
• Cross-service queries → CQRS + Read Models
• Service contract → API versioning + consumer-driven contracts

📖
Self-Registration — Services register themselves with the Eureka server on startup (sending service name, IP, port, health URL). They send heartbeats every 30s. If no heartbeat for 90s → Eureka marks the instance DOWN and removes it from the registry.
💾
Client-Side Discovery & Load Balancing — Clients (other services) pull the registry from Eureka and cache it locally. Spring Cloud LoadBalancer (replaces Ribbon) selects an instance from the cached list — Round Robin by default. Client-side = more flexible; server-side = simpler (e.g., AWS ALB).
🛡️
Self-Preservation Mode — If Eureka loses more than 15% of heartbeats in a short window, it enters self-preservation: stops evicting instances. This prevents Eureka from de-registering services due to network issues. Can cause stale entries — disable in dev with eureka.server.enable-self-preservation=false.
🔄
Eureka HA Setup — Run multiple Eureka server instances that replicate to each other (peer-to-peer). Each Eureka instance registers with others. Configure eureka.client.service-url.defaultZone to point to all peers. Clients can failover to another Eureka node if one goes down.
🔀
Alternatives — Consul (HashiCorp — health checks via TCP/HTTP/script, DNS-based discovery, KV store, multi-datacenter), Zookeeper (CP system — strong consistency, ZooKeeper-based coordination), Kubernetes Service Discovery (built-in DNS + Services — preferred in K8s deployments, no separate registry needed).

◆ Interview Questions
Must Know
What is service discovery and why is it needed in microservices?
▾
In microservices, instances are dynamic — they start/stop, scale up/down, and their IP:port changes constantly. Hardcoding service URLs is impossible. Service discovery solves this: services register their current address in a registry, and callers look up the address at runtime. Two patterns: Client-side discovery (client queries registry, chooses instance, calls directly — e.g., Eureka + Feign) and Server-side discovery (client calls a load balancer which queries registry — e.g., AWS ALB + ECS). Client-side gives more control; server-side is simpler for the client.
Tricky
What is Eureka's self-preservation mode and when can it cause problems?
▾
Self-preservation activates when Eureka receives fewer heartbeats than expected (default threshold: 85% of expected). Instead of de-registering instances, Eureka holds onto the stale registry — assuming it's a network partition, not actual failures. Problem: in a network partition scenario, clients may receive stale entries for services that are actually DOWN, causing failed calls. In development, this causes confusion — stopped services still appear as UP. Best practice: disable self-preservation in dev/test (enable-self-preservation: false), keep it on in production, and combine with client-side circuit breakers (Resilience4j) to handle stale entries gracefully.
⚡ Self-preservation = Eureka choosing AP over CP (CAP theorem). It prefers availability of registry data over consistency.
Tricky
How does Kubernetes service discovery differ from Eureka?
▾
Kubernetes has built-in service discovery via DNS and Services (kube-proxy). A K8s Service gets a stable DNS name and virtual IP — pods can call order-service:8080 without any registry client library. K8s handles load balancing across pod endpoints automatically. Differences: K8s discovery is infrastructure-level (no code changes needed), Eureka is application-level (requires SDK). For K8s deployments, you typically don't need Eureka — replace it with K8s Services and optionally spring-cloud-kubernetes for Spring Cloud integration.
Concept
What is the difference between Ribbon (deprecated) and Spring Cloud LoadBalancer?
▾
Netflix Ribbon was the original client-side load balancer in Spring Cloud — it integrated with Eureka and provided Round Robin balancing. It's now deprecated and in maintenance mode. Spring Cloud LoadBalancer (part of Spring Cloud Commons) is the replacement — lighter, reactive-friendly, supports Reactor-based WebClient, pluggable load balancing strategies (RoundRobin, Random, custom). Spring Cloud LoadBalancer is the default when you include spring-cloud-starter-loadbalancer. If you're still using Ribbon, migrate to Spring Cloud LoadBalancer.
◆ Memory Tip
💡
Eureka workflow in 4 steps:
1. Register — Service starts, posts its IP:port to Eureka
2. Heartbeat — Service pings Eureka every 30s to say "I'm alive"
3. Fetch — Clients pull registry and cache it locally (refreshed every 30s)
4. Resolve — http://service-name/path → Eureka resolves → Load Balancer picks instance

Self-preservation mnemonic: "Eureka is optimistic — it assumes YOU are wrong, not the services."

🏗️
Route = Predicate + Filters + URI — A route matches a request using Predicates (conditions: path, method, header, cookie, query param, host) and forwards it to a URI after applying Filters (pre and post processing). Spring Cloud Gateway is reactive (built on Project Reactor + Netty) — non-blocking, high throughput.
⚙️
Built-in Filters — AddRequestHeader, AddResponseHeader, RemoveRequestHeader, RewritePath (URL rewriting), StripPrefix (remove path prefix), CircuitBreaker (Resilience4j integration), RequestRateLimiter (Redis-based rate limiting), Retry, RequestSize.
🔑
JWT Authentication at Gateway — Gateway validates JWT tokens before forwarding to downstream services. Downstream services trust the gateway and don't need to re-validate (or receive validated user info in headers). Extract user claims, add as X-User-Id, X-User-Role headers to downstream request.
🚦
Rate Limiting — Built-in RequestRateLimiter filter uses Redis + Token Bucket algorithm. Configure: requests per second, burst capacity. Keys: by IP (#{@remoteAddrKeyResolver}) or by user ID (custom). Prevents API abuse and DDoS.
⚡
Gateway vs Zuul — Zuul 1.x (Netflix, blocking, Servlet-based) is deprecated. Zuul 2 was never officially integrated into Spring Cloud. Spring Cloud Gateway is the current standard: reactive, non-blocking, Netty-based, better performance, actively maintained. Use Spring Cloud Gateway for all new projects.

// ── Gateway Setup ─────────────────────────────────
@SpringBootApplication
@EnableDiscoveryClient
public class GatewayApp {
    public static void main(String[] args) {
        SpringApplication.run(GatewayApp.class, args);
    }
}

// ── application.yml — Route Configuration ─────────
// spring:
//   cloud:
//     gateway:
//       routes:
//         - id: user-service
//           uri: lb://user-service       # lb:// = load-balanced via Eureka
//           predicates:
//             - Path=/api/users/**
//           filters:
//             - StripPrefix=1            # strip /api prefix before forwarding
//             - name: CircuitBreaker
//               args:
//                 name: userCB
//                 fallbackUri: forward:/fallback/user
//             - name: RequestRateLimiter
//               args:
//                 redis-rate-limiter.replenishRate: 10
//                 redis-rate-limiter.burstCapacity: 20
//
//         - id: order-service
//           uri: lb://order-service
//           predicates:
//             - Path=/api/orders/**
//             - Header=X-Request-Id, \d+  # header must match regex
//           filters:
//             - AddRequestHeader=X-Gateway, spring-cloud-gateway
//             - RewritePath=/api/orders/(?<rest>.*), /orders/${rest}

// ── Custom JWT Global Filter ──────────────────────
@Component
public class JwtAuthFilter implements GlobalFilter, Ordered {

    private final JwtUtil jwtUtil;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();

        // Skip auth for public endpoints
        if (isPublicEndpoint(request.getPath())) return chain.filter(exchange);

        String token = extractToken(request);
        if (token == null || !jwtUtil.isValid(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // Add user info to downstream headers
        String userId = jwtUtil.getUserId(token);
        ServerHttpRequest mutatedReq = request.mutate()
            .header("X-User-Id", userId)
            .header("X-User-Role", jwtUtil.getRole(token))
            .build();

        return chain.filter(exchange.mutate().request(mutatedReq).build());
    }

    @Override
    public int getOrder() { return -1; } // run before other filters
}
◆ Interview Questions
Must Know
What is an API Gateway and what cross-cutting concerns does it handle?
▾
An API Gateway is a single entry point for all client requests in a microservices architecture. It handles: Routing (forward to correct service based on path/method/header), Authentication/Authorization (validate JWT, check roles before forwarding), Rate Limiting (throttle requests per IP/user), Load Balancing (across service instances), SSL Termination (HTTPS at gateway, HTTP internally), Request/Response Transformation (add/remove headers, rewrite paths), Circuit Breaking (fail fast when downstream is unhealthy), Observability (centralized logging, tracing, metrics).
⚡ Without a gateway, every microservice needs to implement auth, rate limiting, etc. — massive duplication.
Tricky
What is the Backend for Frontend (BFF) pattern and how does it relate to the API Gateway?
▾
The BFF pattern creates separate API Gateway instances tailored to specific client types (mobile app, web app, third-party). A mobile BFF might aggregate multiple service calls into one response (reducing chattiness), return compressed/minimal data, and handle mobile-specific auth flows. A web BFF might return richer data. This differs from a generic gateway: BFF is client-specific, may aggregate/transform data, and can evolve independently per client. Use BFF when different clients have very different data needs or access patterns.
Tricky
How does Spring Cloud Gateway differ from Zuul? Why was the switch made?
▾
Zuul 1.x (Netflix): Blocking I/O, Servlet-based, one thread per request. Simple but poor performance under high concurrency. Zuul 2 (non-blocking) was never officially integrated into Spring Cloud. Spring Cloud Gateway uses Project Reactor + Netty — fully non-blocking, reactive, single event loop handles thousands of concurrent connections. SCG supports WebSocket out of the box, integrates with Spring Security 's reactive stack, and has better performance benchmarks. The switch reflects the industry shift toward reactive microservices.
Concept
What is the difference between a Global Filter and a GatewayFilter in Spring Cloud Gateway?
▾
GlobalFilter: applies to ALL routes automatically. Implement GlobalFilter and register as a bean. Use for: authentication, request ID injection, logging. Order controlled via Ordered interface.
GatewayFilter: applies to specific routes only, configured in the route definition. Use for: route-specific path rewriting, adding service-specific headers, per-route rate limiting. Both can be pre-filters (modify request before forwarding) or post-filters (modify response after upstream returns).
Must Know
How does API Gateway handle JWT validation without hitting the auth service every time?
▾
The gateway validates the JWT locally using the public key (asymmetric RSA/ECDSA) or shared secret (HMAC). No call to the auth service for every request — JWT is self-contained with claims and signature. Steps: 1) Auth service issues JWT signed with private key. 2) Gateway has the public key (from Config Server or JWKs endpoint). 3) Gateway verifies signature + expiry locally in microseconds. 4) Extracts claims (userId, roles), adds as headers to downstream. Downstream services trust these headers (they trust the gateway). Only needs auth service contact on: token refresh, logout/blacklist checks, or key rotation.
⚡ This is why JWT is preferred over opaque tokens in microservices — stateless validation at every node.
◆ Memory Tip
💡
Gateway = Airport Security Check. Every passenger (request) goes through security (auth filter) before boarding their specific gate (route). Some gates have extra checks (rate limiting, role-based). The terminal (gateway) handles all the common rules — airlines (services) just handle their flight (business logic).

Route anatomy: ID + Predicate + Filter + URI
Predicate = does this request match? (path, method, header)
Filter = what to do before/after forwarding?
URI = where to forward? (lb://service-name for load-balanced)

⚡
Why Circuit Breaker? — In distributed systems, a slow/failed downstream service causes callers to pile up waiting (threads blocked on timeouts). This cascades — all threads exhausted, caller goes down, cascades upward. Circuit breaker fails fast when failure rate exceeds threshold — no waiting for timeout. Callers get an immediate error/fallback instead of hanging.
📊
Resilience4j Configuration — Key settings: slidingWindowType (COUNT_BASED or TIME_BASED), slidingWindowSize (number of calls or seconds), failureRateThreshold (% of failures to trip — default 50%), waitDurationInOpenState (time before HALF-OPEN), permittedNumberOfCallsInHalfOpenState, slowCallDurationThreshold + slowCallRateThreshold (also counts slow calls as failures).
🛡️
Fallback Strategy — When circuit is OPEN, return: cached data, default value, partial response, error message. Avoid: fallback that calls another slow service (nested calls defeat the purpose). A good fallback is pre-computed, in-memory, or from cache. Combine with @Retry for transient errors before circuit trips.
📦
Resilience4j Modules — CircuitBreaker (fail-fast on repeated failures), Retry (retry with backoff for transient errors), RateLimiter (limit calls per period), Bulkhead (limit concurrent calls — ThreadPool or Semaphore), TimeLimiter (timeout on async calls). Use as annotations: @CircuitBreaker, @Retry, @RateLimiter, @Bulkhead.
🗑️
Hystrix (Deprecated) — Netflix Hystrix entered maintenance mode in 2018. No longer actively developed. The Spring Cloud team recommends Resilience4j as the replacement. Key difference: Hystrix is thread-pool based (each command runs in a separate pool), Resilience4j is functional and decorator-based (works with existing threads), lighter weight, fully reactive-compatible.
◆ Memorizable Code
Resilience4j — CircuitBreaker + Retry + Bulkhead
// ── application.yml config ────────────────────────
// resilience4j:
//   circuitbreaker:
//     instances:
//       userService:
//         slidingWindowSize: 10         # evaluate last 10 calls
//         failureRateThreshold: 50      # >50% failures → OPEN
//         waitDurationInOpenState: 10s  # wait 10s before HALF-OPEN
//         permittedNumberOfCallsInHalfOpenState: 3
//         slowCallDurationThreshold: 2s # calls >2s = slow
//         slowCallRateThreshold: 80     # >80% slow calls → OPEN
//   retry:
//     instances:
//       userService:
//         maxAttempts: 3
//         waitDuration: 500ms
//         retryExceptions:
//           - java.net.ConnectException
//   bulkhead:
//     instances:
//       userService:
//         maxConcurrentCalls: 10        # max 10 concurrent calls

// ── Annotation-based usage ────────────────────────
@Service
public class UserApiService {

    @CircuitBreaker(name = "userService", fallbackMethod = "fallbackUser")
    @Retry(name = "userService")            // retry first, then CB
    @Bulkhead(name = "userService")
    public UserDto getUser(Long id) {
        return restTemplate.getForObject(
            "http://user-service/users/{id}", UserDto.class, id
        );
    }

    // Fallback: same signature + Throwable
    private UserDto fallbackUser(Long id, Throwable ex) {
        log.warn("Circuit open for user {}: {}", id, ex.getMessage());
        return UserDto.anonymous();      // return safe default
    }
}

// ── Programmatic usage (more control) ────────────
CircuitBreakerRegistry registry = CircuitBreakerRegistry.ofDefaults();
CircuitBreaker cb = registry.circuitBreaker("userService");

Supplier<UserDto> supplier = CircuitBreaker.decorateSupplier(cb,
    () -> userApiClient.getUser(id)
);

UserDto user = Try.ofSupplier(supplier)
    .recover(ex -> UserDto.anonymous())
    .get();

// ── Listen to CB state transitions ───────────────
cb.getEventPublisher()
    .onStateTransition(e ->
        log.warn("CB {} state: {} → {}",
            e.getCircuitBreakerName(),
            e.getStateTransition().getFromState(),
            e.getStateTransition().getToState()
        )
    );
◆ Interview Questions
Must Know
Explain the three states of a Circuit Breaker and what happens in each.
▾
CLOSED (normal): All calls pass through. CB tracks failure rate in a sliding window. When failure rate ≥ threshold, transitions to OPEN.
OPEN (failing fast): All calls are immediately rejected with CallNotPermittedException — no actual call made. Returns fallback or error. After waitDurationInOpenState, transitions to HALF-OPEN.
HALF-OPEN (testing recovery): Allows a limited number of test calls (configured). If those succeed → CLOSED. If they fail → OPEN again (with reset wait timer). This prevents immediately flooding a recovering service.
Tricky
What is the difference between Circuit Breaker, Retry, and Bulkhead? When do you use each?
▾
Retry: Handle transient failures — temporary network blips, occasional timeouts. Retry a few times with backoff before failing. Don't retry non-idempotent operations blindly (charging twice!).
Circuit Breaker: Handle sustained failures — when a downstream is consistently failing. Fails fast instead of waiting for timeouts; gives downstream time to recover.
Bulkhead: Limit resource consumption — prevents one service's slowness from consuming all threads/connections. Isolates failure impact. Semaphore bulkhead: limits concurrent calls. Thread pool bulkhead: separate thread pool per dependency.
⚡ Compose them: Retry → CircuitBreaker → Bulkhead → TimeLimiter. Retry first (transient), CB second (sustained), Bulkhead third (resource isolation), TimeLimiter fourth (async timeout).
Tricky
Why is Hystrix deprecated and what does Resilience4j do differently?
▾
Hystrix used a thread pool per dependency — each command ran in an isolated thread pool. Heavy, hard to configure correctly, and not reactive-friendly. Netflix stopped developing it in 2018. Resilience4j is lightweight (uses decorators/functional wrappers, no separate thread pools by default), fully reactive-compatible (works with Mono/Flux), modular (use only what you need), configurable per-instance, and has better observability via Micrometer/Actuator. Spring Cloud officially recommends Resilience4j as the Hystrix replacement.
Concept
What is a good fallback strategy? What are the anti-patterns?
▾
Good fallback strategies: Return cached data (stale but better than nothing), return a safe default (anonymous user, empty list), return a degraded response (partial data), queue the request for async retry, fail gracefully with a user-friendly error message.
Fallback anti-patterns: 1. Calling another slow service in the fallback — you're just delaying the failure. 2. Returning null — causes NPE downstream. 3. Hiding errors silently — always log fallback invocations with the cause. 4. Complex fallback logic — fallbacks should be fast and simple. 5. Not alerting on high fallback rates — a healthy system should rarely use fallbacks.
◆ Memory Tip
💡
Circuit Breaker = Electrical circuit breaker analogy.
When too many failures (overload), the breaker OPENS (trips) — no more current flows. After cooling down, it tests with HALF-OPEN (try manually). If safe → CLOSED (normal). If still dangerous → OPEN again.

Resilience4j annotation stacking order (outer to inner):
@Bulkhead → @TimeLimiter → @CircuitBreaker → @Retry → @RateLimiter

CB vs Retry trigger: Retry = "maybe it'll work this time". CB = "stop trying, it's definitely broken".

🔐
Secret Encryption — Config Server supports encrypting sensitive values. Use {cipher}... prefix for encrypted values in config files. Server decrypts before serving to clients. Key sources: symmetric (text key), asymmetric (Java keystore). Alternatively use HashiCorp Vault as backend — Spring Cloud Vault provides native integration for secrets management.
🔄
Dynamic Refresh with @RefreshScope — Annotate beans with @RefreshScope — when a POST /actuator/refresh is triggered, the bean is recreated with fresh config. Beans without @RefreshScope keep old values. For broadcast refresh across all services: use Spring Cloud Bus (RabbitMQ or Kafka) to publish RefreshRemoteApplicationEvent to all instances at once.
📂

Multiple backends can be used simultaneously with composite environment.
⚡
Bootstrap vs Application Config — In Spring Boot 2.4+, bootstrap.yml is no longer needed by default. Use spring.config.import=configserver: in application.yml to pull config from Config Server. Pre-2.4: bootstrap.yml was loaded first (before application context) to fetch config server location — critical for Config Server address to be known before loading the rest of config.

◆ Interview Questions
Must Know
Why is externalized configuration important in microservices?
▾
The 12-Factor App principle #3: config should be in the environment, not the code. In microservices: you have dozens of services, each with multiple instances across environments (dev/staging/prod). Embedding config in code means repackaging/redeploying for every config change. Externalized config enables: 1. Same artifact deployed across environments (different configs applied at runtime). 2. Config changes without redeployment. 3. Centralized audit trail (Git history of config changes). 4. Secrets managed separately from code (no API keys in source control). 5. Feature flags toggled at runtime.
Tricky
How does @RefreshScope work? What happens when refresh is triggered?
▾
@RefreshScope creates a scoped proxy around the bean. When a refresh is triggered (via POST /actuator/refresh or Spring Cloud Bus event): 1) The bean's current instance is destroyed (scope cache cleared). 2) On next access, the proxy creates a new instance, pulling fresh config values. 3) New instance has updated @Value fields. Beans without @RefreshScope retain old values until restart. Important: @ConfigurationProperties beans are automatically refreshed without @RefreshScope when using @RefreshScope on a config class or by default in Boot 2.4+.
⚡ Don't put @RefreshScope on @Configuration classes — it can cause unexpected bean recreation cycles.
Tricky
What is Spring Cloud Bus and when do you need it?
▾
Spring Cloud Bus links all service instances via a message broker (RabbitMQ or Kafka). When a config change is pushed: instead of hitting /actuator/refresh on every instance individually, you hit /actuator/busrefresh on any instance (or the config server). The Bus broadcasts a RefreshRemoteApplicationEvent to all connected instances — they all refresh simultaneously. Needed when: you have multiple instances of services (horizontal scaling) and need all to pick up config changes atomically. Without Bus, you'd need to call refresh on each instance separately — error-prone and slow at scale.
Concept
How would you handle secrets (DB passwords, API keys) in a Config Server setup?
▾
Multiple options, in order of security: 1. Config Server encryption: Store {cipher}encrypted-value in Git. Config Server decrypts using a key. Key must be secured (env var, not in code). 2. HashiCorp Vault: Config Server uses Vault as a backend for secrets only. Vault provides fine-grained access control, secret rotation, audit logs. Best for production. 3. Kubernetes Secrets: Mount secrets as environment variables or files. Managed by K8s RBAC. 4. Environment Variables: For sensitive values in containerized environments — injected at runtime, never in code or Git.
⚡ Never store plain-text passwords in Git config files — even private repos can be compromised.
Must Know
What is the Config Server config file resolution order?
▾
For a service named order-service with profile prod, Config Server fetches in order (most specific wins): 1. order-service-prod.yml (application + profile — highest priority). 2. order-service.yml (application only). 3. application-prod.yml (shared + profile). 4. application.yml (shared defaults — lowest priority). All matched files are loaded and merged — properties from higher-priority files override lower. Local application.yml in the service itself is lowest priority and overridden by Config Server values.
◆ Memory Tip
💡
Config Server = centralized source of truth for all configuration.
Think of it like a library — all services check out their config at startup. When books (configs) are updated, use /actuator/refresh to return the old book and get a new one.

Refresh flow cheat sheet:
• Single instance: POST /actuator/refresh on that instance
• All instances: POST /actuator/busrefresh → Spring Cloud Bus broadcasts
• Without restart: bean must be @RefreshScope
• Git webhook: GitHub/GitLab webhook → Config Server → Bus refresh → All services update

Config resolution: SPECIFIC beats GENERIC. Profile-specific > App > Shared.
