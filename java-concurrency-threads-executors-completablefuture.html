01
Core Concept
Thread Basics — Creating & Running
What is a thread, how to create one, and three ways to do it in Java
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: A thread is the smallest unit of execution — an independent path through your program that runs concurrently with other threads. The JVM starts with one thread (the main thread). You create more threads to do work in parallel: download a file while updating the UI, process multiple HTTP requests simultaneously, or run CPU-intensive calculations on multiple cores at once.
◆ Three Ways to Create a Thread
1️⃣
Extend Thread class — Override run() and call start(). Simple but couples your class to Thread — can't extend any other class. Rarely used in modern Java. Call start() not run() — calling run() directly executes on the current thread, no new thread is created.
2️⃣
Implement Runnable — Pass a Runnable to the Thread constructor. Better separation of task from thread machinery. The class can still extend something else. Use a lambda: new Thread(() -> doWork()).start(). This is the preferred way over extending Thread.
3️⃣
Submit to ExecutorService — Don't manage thread creation at all. Submit a Runnable or Callable to an executor and let the thread pool handle it. This is the modern, correct approach for production code. Avoids the overhead of creating new threads per task. You almost never need raw thread creation in production Java.
⚡
Runnable vs Callable — Runnable.run(): returns void, cannot throw checked exceptions. Callable<T>.call(): returns a value of type T, can throw checked exceptions. Submit Callable to ExecutorService → get a Future<T> back. Use Callable whenever the task produces a result or might throw a checked exception.
⚠️
start() vs run() — This is a classic interview trap. thread.start() creates a new OS thread and then calls run() on it. thread.run() just calls the run() method synchronously on the calling thread — no new thread is created. Calling start() twice throws IllegalThreadStateException.
🔢
Thread Priority & Daemon Threads — Threads have priority 1–10 (default 5). Higher priority may get more CPU time — but JVM makes no guarantee. Daemon threads (thread.setDaemon(true)) are background threads that JVM kills automatically when all non-daemon threads finish. Timer threads, GC threads are daemon. Always set daemon BEFORE calling start().
◆ Memorizable Code
Thread Creation — 3 Approaches + Daemon + Priority
// ── Way 1: Extend Thread (avoid in production) ───────
class WorkerThread extends Thread {
    @Override
    public void run() {
        System.out.println("Running in: " + Thread.currentThread().getName());
    }
}
new WorkerThread().start();

// ── Way 2: Implement Runnable (preferred) ─────────────
Runnable task = () -> System.out.println("Task running");
Thread t = new Thread(task, "my-worker-thread");
t.start();                 // creates new OS thread — call start(), NOT run()!

// ── Way 3: ExecutorService (modern, production-grade) ─
ExecutorService exec = Executors.newFixedThreadPool(4);
exec.submit(() -> System.out.println("Pool thread"));
exec.shutdown();           // always shutdown when done

// ── Runnable vs Callable ──────────────────────────────
Runnable runnable = () -> System.out.println("No result");

Callable<Integer> callable = () -> {
    // Can throw checked exceptions, returns a value
    return computeExpensiveResult();
};
Future<Integer> future = exec.submit(callable);
Integer result = future.get();   // blocks until result is ready

// ── Daemon Thread ─────────────────────────────────────
Thread daemon = new Thread(() -> {
    while (true) monitorHealth();  // runs in background
});
daemon.setDaemon(true);       // MUST be before start()
daemon.start();
// When all user threads finish → JVM kills this daemon automatically

// ── Thread info ───────────────────────────────────────
Thread curr = Thread.currentThread();
System.out.println(curr.getName());    // thread name
System.out.println(curr.getId());      // unique ID
System.out.println(curr.isDaemon());   // is daemon?
System.out.println(curr.getState());   // NEW/RUNNABLE/BLOCKED/etc.
System.out.println(curr.getPriority()); // 1-10, default 5
◆ Interview Questions
Must Know
What is the difference between extending Thread and implementing Runnable?
▾
Both create tasks that can run in a thread, but they differ in design:

Extend Thread: your class becomes a thread — it can only be submitted to exactly one thread (itself). Since Java has single inheritance, you can't extend anything else. Tightly couples your task logic to thread machinery.

Implement Runnable: separates the task (what to do) from the thread (execution mechanism). Your class can still extend another class. The same Runnable instance can be submitted to any Executor, scheduled repeatedly, or wrapped in multiple threads. Always prefer Runnable (or Callable) over extending Thread.
⚡ Production rule: Never extend Thread in real code. Use Runnable/Callable with an ExecutorService. Raw thread creation is an anti-pattern in modern Java.
Tricky
What happens if you call thread.run() instead of thread.start()?
▾
Calling run() directly is a very common mistake. It simply calls the run() method like a regular method — no new thread is created, and the code executes synchronously on the calling thread. The Thread object exists but its thread-creation machinery is bypassed entirely.

start() is the only way to actually create a new OS thread and begin concurrent execution. After calling start(), the JVM creates the native thread, schedules it, and eventually calls run() on the new thread.

Calling start() twice on the same Thread throws IllegalThreadStateException — a Thread can only be started once.
⚠️ Classic interview trick: If asked "will this run in parallel?" — check whether start() or run() is called. run() = always sequential. start() = parallel.
Tricky
What is a daemon thread and what happens when all non-daemon threads finish?
▾
A daemon thread is a background thread that the JVM considers "non-essential." When all user (non-daemon) threads finish, the JVM exits — it doesn't wait for daemon threads to complete, killing them abruptly.

Use daemon threads for: background monitoring, GC helpers, cache cleanup, internal timer tasks. Never use daemon threads for: database writes, file operations, or any work that must complete for correctness — it may be killed mid-operation.

Must call setDaemon(true) BEFORE start(). After start(), calling setDaemon() throws IllegalThreadStateException. The main thread and threads spawned from non-daemon threads are non-daemon by default.
Must Know
What is the difference between Runnable and Callable?
▾
Runnable.run(): executes a task, returns void, cannot throw checked exceptions. Use for fire-and-forget tasks where you don't need the result.

Callable<T>.call(): executes a task, returns a value of type T, can throw checked exceptions. Use when you need a result or may throw a checked exception. Submit a Callable to ExecutorService → get back a Future<T>. Call future.get() to retrieve the result (blocking) or future.get(timeout, unit) with a timeout.
⚡ Rule: if you need the result → Callable. If fire-and-forget → Runnable. In modern code, CompletableFuture often replaces both for async work.
💡
Thread creation in 3 levels of abstraction:
Level 1 (old): extends Thread → rarely use
Level 2 (ok): implements Runnable + new Thread(r).start()
Level 3 (modern): Submit to ExecutorService → always prefer

Critical distinction: start() creates a new thread. run() doesn't.

Runnable = void + no checked exceptions. Callable = returns T + throws.

Daemon thread = background service. JVM won't wait for it. Never do critical I/O in daemon threads.

Thread Lifecycle — States & Transitions
The six states a thread moves through from creation to termination — and the key methods that trigger transitions
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: A thread doesn't just "run" — it moves through distinct states during its lifetime. Understanding these states is critical for debugging concurrency issues: a deadlocked thread will be BLOCKED forever, a thread waiting for notify() is WAITING, a thread sleeping is TIMED_WAITING. Knowing which state tells you exactly why the thread isn't running.

🟦
NEW — Thread object created with new Thread(...) but start() not yet called. The thread has no associated OS thread yet. No system resources consumed. Transitions to RUNNABLE when start() is called.
🟩
RUNNABLE — Thread is either actively executing on a CPU core, or ready to execute and waiting for CPU time from the OS scheduler. Java doesn't distinguish between "executing" and "ready-to-execute" — both are RUNNABLE. Most healthy active threads are in this state.
🟥
BLOCKED — Thread is waiting to acquire a synchronized monitor lock that another thread currently holds. Example: two threads trying to enter the same synchronized block. The first enters and holds the lock; the second becomes BLOCKED until the first releases it. BLOCKED threads are the signature of lock contention — too many BLOCKED threads = synchronization bottleneck.
🟨
WAITING — Thread is indefinitely waiting for another thread to take a specific action. Triggered by: Object.wait() (waiting for notify), Thread.join() (waiting for another thread to finish), LockSupport.park(). Must be woken explicitly — will stay WAITING forever if notify/unpark never comes.
🟪
TIMED_WAITING — Like WAITING but with a maximum time limit. Triggered by: Thread.sleep(n), Object.wait(n), Thread.join(n), LockSupport.parkNanos(n). Automatically returns to RUNNABLE when the timer expires — no explicit wake-up needed. Distinguishes "intentionally paused" from "potentially deadlocked."
⬛
TERMINATED — Thread's run() method has completed (either normally or by throwing an uncaught exception). The Thread object still exists in memory but can't be restarted — calling start() again throws IllegalThreadStateException. Thread pools avoid this by recycling threads.
Thread States + sleep/join/wait/interrupt
// ── sleep(): RUNNABLE → TIMED_WAITING → RUNNABLE ─────
Thread.sleep(2000);      // pause 2 seconds — throws InterruptedException
// sleep() holds locks — does NOT release synchronized monitors

// ── join(): wait for another thread to finish ─────────
Thread worker = new Thread(() -> heavyWork());
worker.start();
worker.join();          // current thread → WAITING until worker TERMINATES
worker.join(5000);      // or TIMED_WAITING with 5s timeout
System.out.println("Worker done!");

// ── wait/notify: classic producer-consumer ────────────
Object lock = new Object();

// Consumer thread
synchronized (lock) {
    while (!dataReady) {        // ALWAYS check in a while loop!
        lock.wait();           // releases lock → WAITING
    }
    consume(data);
}

// Producer thread
synchronized (lock) {
    data = produce();
    dataReady = true;
    lock.notifyAll();          // wake all waiting threads
}

// ── Thread interruption ───────────────────────────────
Thread t = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            // sleep was interrupted — re-interrupt and exit
            Thread.currentThread().interrupt();  // restore flag!
            break;
        }
    }
});
t.start();
t.interrupt();  // sets interrupted flag / throws InterruptedException in sleep

// ── Check thread state programmatically ──────────────
System.out.println(t.getState());    // Thread.State enum value
System.out.println(t.isAlive());     // true if RUNNABLE/BLOCKED/WAITING
System.out.println(t.isInterrupted()); // check flag without clearing it
Thread.interrupted();              // check AND clear interrupt flag (static)
Must Know
What is the difference between BLOCKED and WAITING states?
▾
Both mean a thread is paused, but the cause and wake-up mechanism differ fundamentally:

BLOCKED: the thread is trying to enter a synchronized block/method but another thread holds the monitor lock. It's waiting for the lock to be released. This is the hallmark of lock contention. The JVM automatically moves the thread back to RUNNABLE when the lock becomes available — no explicit action from the lock-holder is needed.

WAITING: the thread explicitly called wait(), join(), or park() and is waiting for a specific external signal. It won't wake up automatically — another thread must call notify()/notifyAll() or unpark(), or the joined thread must complete.
💡 Thread dump diagnosis: Many threads in BLOCKED state → lock contention (bottleneck). Threads in WAITING → possibly deadlock or correct waiting. TIMED_WAITING → sleeping or waiting with timeout (usually fine).
Tricky
Why should wait() always be called in a while loop, not an if statement?
▾
This is the "spurious wakeup" problem. A thread calling wait() can wake up without any thread calling notify() — this is called a spurious wakeup and is allowed by the JVM specification (the JVM makes no guarantee against it for performance reasons on some OS implementations).

If you use if (condition) wait(): the thread wakes up spuriously, condition is not checked again, the thread proceeds as if the condition is true — incorrect behavior and potential race condition.

Correct pattern: while (!condition) { wait(); } — after waking (for any reason, spurious or real), the condition is re-evaluated. Only proceeds when the condition is genuinely true.
⚠️ This is a well-known bug pattern. Always: while(!condition) { wait(); }
Tricky
How does thread interruption work? What is the interrupted flag?
▾
Thread interruption is a cooperative mechanism — it doesn't forcefully stop the thread. It sets the thread's interrupted flag to true and, if the thread is in SLEEPING/WAITING/BLOCKED state, throws InterruptedException.

The thread must cooperate by periodically checking Thread.currentThread().isInterrupted() and exiting gracefully. When InterruptedException is caught, the interrupted flag is automatically cleared. If you catch it, always restore the flag: Thread.currentThread().interrupt().

Key methods: t.interrupt() — set flag/throw exception. t.isInterrupted() — check flag without clearing. Thread.interrupted() — static, checks and clears the flag.
⚡ Golden rule: never swallow InterruptedException with an empty catch block. Either propagate it or re-interrupt the thread.
💡
6 Thread States to memorise (Thread.State enum):
NEW → RUNNABLE → [BLOCKED / WAITING / TIMED_WAITING] → TERMINATED

State diagnosis cheatsheet:
• BLOCKED = waiting for a lock → lock contention
• WAITING = waiting for notify()/join() → possibly deadlock
• TIMED_WAITING = sleeping/waiting with timeout → usually fine
• Many BLOCKED threads = synchronization bottleneck

sleep() holds locks. wait() releases them.
Always: while(!condition) { wait(); } ← never if(!condition)
03
Safety Critical
Synchronization, Visibility & Locks
volatile, synchronized, Lock, atomic classes — controlling access to shared state
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: When multiple threads share data, two problems arise: (1) Visibility — one thread's changes may not be visible to another thread (CPU caches). (2) Atomicity — operations like i++ are NOT atomic (read-modify-write), so two threads can corrupt the value. Synchronization mechanisms solve both: volatile fixes visibility, synchronized/Lock fix atomicity, atomic classes fix both without locking.
👁️
volatile — Visibility Only — Declaring a field volatile guarantees that reads and writes go directly to main memory, not a CPU cache. This fixes the visibility problem: a write in Thread A is immediately visible to Thread B. However, volatile does NOT make compound operations atomic. volatile int counter; counter++ is still a race condition — read-increment-write is three separate operations.
🔒
synchronized — Mutual Exclusion + Visibility — Guarantees only one thread executes the synchronized block at a time (using the object's intrinsic monitor lock). Also provides visibility — all writes before releasing the lock are visible to the next thread that acquires it. Can synchronize on any object. Method-level synchronized uses this as the lock object. Best for protecting a block of statements as a single atomic operation.
🔑
ReentrantLock — Flexible Locking — From java.util.concurrent.locks. Explicit lock/unlock. Must always unlock in a finally block. Advantages over synchronized: tryLock() (try to acquire without blocking), lockInterruptibly() (interruptible while waiting), fairness option (longest-waiting thread gets the lock first), multiple condition objects per lock.
⚛️
Atomic Classes — Lock-Free Thread Safety — AtomicInteger, AtomicLong, AtomicBoolean, AtomicReference. Use CPU CAS (Compare-And-Swap) instructions for lock-free, thread-safe operations. counter.incrementAndGet() is genuinely atomic. Much faster than synchronized for simple counter/flag operations. AtomicReference.compareAndSet(expected, update) is the key method for lock-free algorithms.
🚦
ReadWriteLock — Concurrent Reads — When reads are far more frequent than writes, ReentrantReadWriteLock allows multiple simultaneous readers but exclusive access for writers. Readers don't block other readers. A writer blocks until all readers release, then holds exclusive access. Perfect for: caches, configuration data, reference data. Can improve throughput dramatically for read-heavy workloads.
🚫
Deadlock — The Four Conditions — Four conditions must ALL be true simultaneously: (1) Mutual exclusion: resources can't be shared. (2) Hold and wait: threads hold locks while waiting for others. (3) No preemption: locks can't be forcibly taken. (4) Circular wait: T1 waits for T2's lock, T2 waits for T1's lock. Prevention: always acquire locks in the same order across all threads, or use tryLock() with timeout.
volatile + synchronized + ReentrantLock + AtomicInteger + Deadlock Prevention
// ── volatile: visibility only (NOT atomicity) ─────────
volatile boolean running = true;     // write visible to all threads
volatile int counter = 0;
counter++;  // ⚠ NOT atomic! Read-increment-write = race condition

// ── synchronized method ───────────────────────────────
public class SafeCounter {
    private int count = 0;

    public synchronized void increment() { count++; }   // lock = this
    public synchronized int  get()       { return count; }

    // synchronized block — finer granularity
    private final Object lock = new Object();
    public void increment2() {
        synchronized (lock) { count++; }  // explicit lock object
    }
}

// ── ReentrantLock: explicit, flexible ─────────────────
ReentrantLock lock = new ReentrantLock(true);  // fair=true
lock.lock();
try {
    criticalSection();
} finally {
    lock.unlock();  // ALWAYS in finally!
}

// tryLock: attempt without blocking
if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
    try { criticalSection(); } finally { lock.unlock(); }
} else {
    System.out.println("Could not acquire lock, doing something else");
}

// ── AtomicInteger: lock-free, fast ────────────────────
AtomicInteger atomicCount = new AtomicInteger(0);
atomicCount.incrementAndGet();      // atomic ++ (returns new value)
atomicCount.getAndIncrement();      // atomic ++ (returns old value)
atomicCount.addAndGet(5);          // atomic += 5
atomicCount.compareAndSet(10, 20); // if value==10, set to 20 (CAS)

// ── ReadWriteLock: concurrent reads ───────────────────
ReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock  = rwLock.readLock();
Lock writeLock = rwLock.writeLock();

public String getData() {
    readLock.lock();
    try { return data; } finally { readLock.unlock(); }
    // Multiple threads can hold readLock simultaneously
}
public void setData(String d) {
    writeLock.lock();
    try { data = d; } finally { writeLock.unlock(); }
    // Exclusive — blocks all readers and other writers
}

// ── Deadlock prevention: consistent lock ordering ─────
// ALWAYS acquire lockA before lockB across ALL methods
void transfer(Account from, Account to, double amount) {
    Account first  = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to   : from;
    synchronized (first) { synchronized (second) { doTransfer(); } }
}
Must Know
What is the difference between volatile and synchronized?
▾
volatile guarantees only visibility — writes to a volatile variable are immediately visible to other threads (bypasses CPU cache). It does NOT guarantee atomicity. volatile int i; i++ is NOT thread-safe — it's three operations: read i, increment, write i. Another thread can interleave.

synchronized guarantees both visibility and atomicity — only one thread at a time executes the synchronized block, and all changes made inside are visible to the next thread that enters. More expensive (mutex overhead) but provides full thread safety.

Use volatile for: simple boolean flags, single-write-multiple-read patterns where the write is a single operation (not compound). Use synchronized or atomic classes for: compound operations (read-modify-write), protecting invariants across multiple fields.
⚠️ volatile int counter; counter++ — this is NOT thread-safe. Use AtomicInteger instead.
Tricky
When would you choose ReentrantLock over synchronized?
▾
synchronized is simpler and should be the default. Choose ReentrantLock when you need capabilities not available with synchronized:

1. tryLock(): attempt to acquire the lock without blocking — useful for avoiding deadlocks or implementing "try and give up" patterns.
2. tryLock(timeout, unit): wait up to N milliseconds for the lock — bounded waiting prevents indefinite blocking.
3. lockInterruptibly(): allow the thread to be interrupted while waiting for the lock — not possible with synchronized.
4. Fairness: new ReentrantLock(true) guarantees longest-waiting thread gets the lock next — prevents starvation.
5. Multiple Conditions: one lock can have multiple Condition objects (like multiple wait queues), vs. synchronized which only has one.
⚡ Rule: start with synchronized. Switch to ReentrantLock only when you specifically need tryLock, fairness, interruptible waiting, or multiple conditions.
Tricky
What is a deadlock and how do you prevent it?
▾
A deadlock is when two or more threads are each waiting for a lock held by the other — neither can proceed.

Classic example: Thread 1 holds Lock A, waits for Lock B. Thread 2 holds Lock B, waits for Lock A. Both wait forever.

Prevention strategies:
1. Consistent lock ordering: always acquire locks in the same global order (e.g., sort by account ID before locking). If everyone follows the same order, circular wait can't occur.
2. tryLock with timeout: if you can't acquire all needed locks within a timeout, release what you have and retry. Eliminates indefinite wait.
3. Lock striping: instead of one big lock, use many smaller locks keyed by data (e.g., ConcurrentHashMap's segment locking).
4. Avoid nested locks: redesign to not hold one lock while acquiring another.
💡 Diagnosis: take a thread dump (jstack PID). Look for threads in BLOCKED state forming a cycle.
Concept
What are the happens-before guarantees in Java Memory Model?
▾
The Java Memory Model (JMM) defines when one thread's writes are guaranteed to be visible to another thread's reads. Key "happens-before" relationships:

1. Monitor lock: a thread releasing a synchronized lock "happens-before" any thread subsequently acquiring the same lock.
2. volatile write: a write to a volatile field "happens-before" subsequent reads of that field.
3. Thread start: actions before thread.start() "happen-before" actions in the started thread.
4. Thread join: all actions in a thread "happen-before" any thread that successfully calls join() on it.

Without a happens-before relationship, the JVM can reorder operations and cache writes — two threads can see different values for the same variable.
💡
Two problems with shared data:
1. Visibility — changes not seen by other threads → fix with volatile
2. Atomicity — compound ops like i++ are not atomic → fix with synchronized / AtomicInteger

Tool selector:
• Simple flag/stop signal → volatile boolean
• Counter increments → AtomicInteger
• Protect a block of code → synchronized
• Need tryLock/fairness → ReentrantLock
• Many reads, few writes → ReadWriteLock

Deadlock prevention: always acquire locks in the same order. Or use tryLock with timeout.

Thread Pools
ExecutorService — Thread Pool Management
The right way to manage threads in production — submit, shutdown, Future, and exception handling
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Creating a new thread per task is expensive (OS thread creation takes time) and uncontrolled (10,000 requests = 10,000 threads = JVM crash). An ExecutorService is a thread pool — a fixed set of reusable threads that pick up tasks from a queue. You submit tasks, the pool assigns them to available threads, and you get the results via Future. The pool manages thread lifecycle so you don't have to.
◆ ExecutorService Key Operations
MethodReturnsUse When
execute(Runnable)voidFire-and-forget, no result needed
submit(Runnable)Future<?>Need to know when task finishes
submit(Callable<T>)Future<T>Need the result of the task
invokeAll(Collection<Callable>)List<Future<T>>Run many tasks, wait for all
invokeAny(Collection<Callable>)TFirst successful result wins
shutdown()voidGraceful: accept no new, finish existing
shutdownNow()List<Runnable>Interrupt all, return pending tasks
awaitTermination(n, unit)booleanWait up to N for shutdown to complete
📬
Future<T> — Represents the result of an async computation. get() blocks until the result is ready. get(timeout, unit) blocks for at most N time. isDone() checks without blocking. cancel(true) attempts to interrupt the running task. Limitation: Future is awkward to chain — you must block to get the result, making async composition difficult. CompletableFuture fixes this.
🛑
Shutdown — Two-Phase Shutdown Pattern — shutdown() stops accepting new tasks but lets queued/running tasks complete. Then call awaitTermination(60, TimeUnit.SECONDS) to wait. If it returns false (timeout), call shutdownNow() to interrupt. Always shut down executors — unreleased thread pools keep the JVM alive and leak resources. Use try-with-resources or Spring's @PreDestroy lifecycle hook.
⚠️
Exception Handling in Executors — If a submitted Runnable throws an uncaught exception, it's silently swallowed — you won't see it. The Future from submit() wraps it: future.get() throws ExecutionException containing the original. For tasks submitted via execute(): set an UncaughtExceptionHandler on the ThreadFactory. Never lose exceptions silently — they indicate bugs.
🏭
ThreadFactory — Custom Thread Names — Instead of generic "pool-1-thread-1" names, provide a custom ThreadFactory that names threads meaningfully (e.g., "order-processor-1"). Invaluable during debugging — thread dumps with meaningful names immediately tell you which pool is struggling. Also use for: setting daemon status, priority, UncaughtExceptionHandler, and logging.
ExecutorService — Submit, Future, shutdown, invokeAll, exception handling
// ── Basic usage ───────────────────────────────────────
ExecutorService executor = Executors.newFixedThreadPool(4);

// submit Callable → get Future back
Future<String> future = executor.submit(() -> {
    Thread.sleep(1000);
    return "result";
});

// future.get() blocks until done
try {
    String result = future.get(2, TimeUnit.SECONDS);  // 2s timeout
} catch (ExecutionException e) {
    log.error("Task threw: ", e.getCause());  // unwrap actual exception
} catch (TimeoutException e) {
    future.cancel(true);   // cancel the task
}

// ── invokeAll: run many, wait for all ─────────────────
List<Callable<Integer>> tasks = List.of(
    () -> processPartition(0),
    () -> processPartition(1),
    () -> processPartition(2)
);
List<Future<Integer>> results = executor.invokeAll(tasks); // blocks until all done
int total = results.stream()
    .mapToInt(f -> { try { return f.get(); } catch (Exception e) { return 0; } })
    .sum();

// ── invokeAny: first success wins ────────────────────
String fastest = executor.invokeAny(List.of(
    () -> queryPrimary(), () -> queryReplica(), () -> queryCache()
)); // returns first successful result, cancels others

// ── Two-phase shutdown (production pattern) ───────────
void shutdownAndAwait(ExecutorService executor) {
    executor.shutdown();                         // no new tasks
    try {
        if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
            executor.shutdownNow();               // interrupt running tasks
            executor.awaitTermination(30, TimeUnit.SECONDS);
        }
    } catch (InterruptedException e) {
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
}

// ── Custom ThreadFactory: named threads ───────────────
ThreadFactory factory = new ThreadFactory() {
    private final AtomicInteger count = new AtomicInteger();
    @Override
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r, "order-processor-" + count.incrementAndGet());
        t.setDaemon(false);
        t.setUncaughtExceptionHandler((th, ex) ->
            log.error("Uncaught exception in {}: ", th.getName(), ex));
        return t;
    }
};
ExecutorService namedPool = Executors.newFixedThreadPool(8, factory);
Must Know
What is the difference between execute() and submit() on an ExecutorService?
▾
execute(Runnable): defined by Executor interface, takes a Runnable, returns void. Any exception thrown by the Runnable is handled by the thread's UncaughtExceptionHandler — if none is set, the exception is silently lost or printed to stderr. You have no way to know if the task failed.

submit(Runnable) or submit(Callable): returns a Future. Exceptions are captured and wrapped in the Future. Calling future.get() re-throws them as ExecutionException. You must unwrap: executionException.getCause() to get the actual exception.
⚠️ Using execute() and ignoring exceptions is a common bug. If you use execute(), always set an UncaughtExceptionHandler. If you use submit(), always check future.get() to surface failures.
Tricky
What happens if you don't shut down an ExecutorService?
▾
An ExecutorService with a thread pool that's never shut down keeps its threads alive indefinitely. Three problems:

1. JVM won't exit: unless the threads are daemon threads, the JVM stays alive even after your main logic finishes — waiting for threads that will never receive shutdown signal.
2. Thread leak: the thread pool objects can't be garbage collected while their threads are alive. In a web app with multiple contexts/reloads, this creates a new pool on each reload without cleaning up old ones.
3. Resource waste: idle threads still consume OS resources and JVM memory.

Always shut down executors in: Spring's @PreDestroy, shutdown hooks, finally blocks, or try-with-resources if the executor is AutoCloseable.
Senior
How do you handle exceptions thrown by tasks submitted to an ExecutorService?
▾
Three layers of exception handling:

1. Via Future (submit): catch ExecutionException from future.get(). Get the actual cause via e.getCause(). This is the primary way — always check futures from submit().

2. UncaughtExceptionHandler on ThreadFactory: catch exceptions from tasks submitted via execute() or submit() where the future isn't checked. Set during thread creation: thread.setUncaughtExceptionHandler((t, e) -> log.error(...)).

3. Override afterExecute in ThreadPoolExecutor: subclass ThreadPoolExecutor and override afterExecute(Runnable, Throwable) — called after each task, even those submitted via execute(). The Throwable parameter is null for tasks submitted via submit() (exception is in the Future).
⚡ In production: always log all task failures. Silent exceptions = bugs you'll never know about until data is corrupted.
💡
ExecutorService = reusable thread pool + task queue.
Submit tasks → pool assigns to free thread → get result via Future.

Key operations:
• execute() → void, fire-and-forget (exceptions lost!)
• submit(Callable) → Future<T>, result + exceptions captured
• invokeAll() → wait for ALL tasks to complete
• invokeAny() → first success wins, others cancelled

Always shut down: shutdown() → awaitTermination() → shutdownNow()

Always name threads with ThreadFactory. "order-processor-3" beats "pool-1-thread-3" in every thread dump.

// ── Production-grade custom pool (ALWAYS prefer this) ─
ExecutorService pool = new ThreadPoolExecutor(
    4,                              // corePoolSize: keep 4 alive always
    8,                              // maximumPoolSize: burst up to 8
    60, TimeUnit.SECONDS,           // keepAliveTime: idle burst threads die after 60s
    new ArrayBlockingQueue<>(100), // bounded queue (prevent OOM!)
    new ThreadFactoryBuilder().setNameFormat("order-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy() // backpressure
);

// ── Thread pool sizing rules ──────────────────────────
// CPU-bound tasks (compute):  threads ≈ CPU cores (no waiting)
int cpuThreads = Runtime.getRuntime().availableProcessors();

// I/O-bound tasks (DB, HTTP):  threads >> CPU cores (threads wait a lot)
// Formula: threads = CPU × (1 + wait_time / cpu_time)
// If 90% of time waiting: threads = 4 × (1 + 0.9/0.1) = 40 threads

// ── Scheduled tasks ───────────────────────────────────
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);

// Run once after 5 seconds
scheduler.schedule(() -> cleanupExpiredSessions(), 5, TimeUnit.SECONDS);

// Run every 10 seconds, measured from START of previous (may overlap)
scheduler.scheduleAtFixedRate(() -> sendHeartbeat(), 0, 10, TimeUnit.SECONDS);

// Run 10 seconds after COMPLETION of previous (no overlap)
scheduler.scheduleWithFixedDelay(() -> pollMailbox(), 0, 10, TimeUnit.SECONDS);

// ── ForkJoinPool: divide and conquer ─────────────────
class SumTask extends RecursiveTask<Long> {
    private final long[] arr;
    private final int lo, hi;

    @Override
    protected Long compute() {
        if (hi - lo <= 1000) {
            long sum = 0;
            for (int i = lo; i < hi; i++) sum += arr[i];
            return sum;
        }
        int mid = (lo + hi) / 2;
        SumTask left  = new SumTask(arr, lo, mid);
        SumTask right = new SumTask(arr, mid, hi);
        left.fork();                    // submit left to pool
        return right.compute()         // compute right on this thread
             + left.join();            // wait for left
    }
}
long total = new ForkJoinPool().invoke(new SumTask(arr, 0, arr.length));
Must Know
Why are Executors.newFixedThreadPool and newCachedThreadPool dangerous in production?
▾
newFixedThreadPool(n): uses an unbounded LinkedBlockingQueue. If tasks are submitted faster than they're processed, the queue grows indefinitely → OutOfMemoryError when heap fills up. You have no protection against task overload.

newCachedThreadPool(): creates a new thread for every task if none are available. Under load, can create millions of threads → OutOfMemoryError or OS-level resource exhaustion. JVM typically crashes at 10,000–20,000 threads.

Solution: always use a custom ThreadPoolExecutor with a bounded ArrayBlockingQueue and a RejectedExecutionHandler. This way, overload causes controlled rejection (with logging) rather than a crash.
⚠️ Netflix famously outlines that unbounded queues are a top cause of production outages. Always bound your queues.
Tricky
How does ThreadPoolExecutor actually create threads? What is the exact lifecycle?
▾
ThreadPoolExecutor follows a specific flow when a new task is submitted:

1. If active threads < corePoolSize → create a new thread immediately (even if other threads are idle).
2. If active threads ≥ corePoolSize → try to add task to the queue.
3. If queue is full AND active threads < maximumPoolSize → create a burst thread.
4. If queue is full AND active threads = maximumPoolSize → invoke RejectedExecutionHandler.

Important: a FixedThreadPool with 10 threads and an unbounded queue never creates more than 10 threads — it just queues infinitely. CachedThreadPool creates a thread for every new task (queue is SynchronousQueue — no capacity). Burst threads (above core) die after keepAliveTime if idle.
⚡ Interview detail: ThreadPoolExecutor prefers queueing over creating burst threads. It only creates burst threads when the queue is full.
Compare
scheduleAtFixedRate vs scheduleWithFixedDelay — when does each matter?
▾
scheduleAtFixedRate(task, 0, 10, SECONDS): fires every 10 seconds measured from the START of the previous execution. If the task takes 12 seconds, the next execution starts immediately (no delay). If it takes 3 seconds, the next starts 7 seconds later. Tasks can overlap if the executor has enough threads, or queue behind if single-threaded.

scheduleWithFixedDelay(task, 0, 10, SECONDS): waits 10 seconds after the COMPLETION of the previous execution. If the task takes 3 seconds → next starts at T+13s. If it takes 30 seconds → next starts at T+40s. Guarantees no overlap — next task always waits for previous to finish.

Use fixedRate for: sending metrics every second regardless of how long sending takes. Use fixedDelay for: polling tasks where overlap is wrong (polling queue, checking DB for pending work).

Pool selector cheatsheet:
• CPU-intensive (N cores, no I/O wait): FixedThreadPool(N)
• I/O-intensive (lots of waiting): FixedThreadPool(N×10+) or cached
• Scheduled/periodic: ScheduledThreadPool
• Recursive divide-and-conquer: ForkJoinPool
• Production with SLA: Custom ThreadPoolExecutor with bounded queue

(core, max, keepAlive, boundedQueue, rejectionPolicy)

Queue full + maxThreads hit → CallerRunsPolicy = natural backpressure.

🧠 Memory
🧠
In plain English: A regular Future makes you block to get the result: future.get() — your thread sleeps waiting. CompletableFuture lets you say "when the result is ready, do THIS next" — no blocking. Chain operations like a pipeline: fetch user → then fetch orders → then calculate total → then send email. Each step starts as soon as the previous finishes, without any thread sleeping.
◆ Key Creation Methods
🚀
supplyAsync(Supplier) — Runs the supplier in the ForkJoin common pool (or your custom executor) and returns a CompletableFuture<T> that completes with the supplier's result. Non-blocking — the caller immediately gets the CF and can continue doing other work. CompletableFuture.supplyAsync(() -> fetchData(), myExecutor).
🔥
runAsync(Runnable) — Like supplyAsync but for tasks that return void. Returns CompletableFuture<Void>. Use for fire-and-forget background tasks: logging, sending notifications, cleanup operations.
🏭
completedFuture(value) — Creates an already-completed CF with the given value. Useful in tests (mock the result) and for returning cached values that don't need async computation. CompletableFuture.completedFuture("cached").
⛓️
thenApply vs thenCompose — thenApply(fn): synchronous transform — function returns a value (T→R). thenCompose(fn): async transform — function returns another CF (T→CF<R>). Like map vs flatMap in streams. If your callback needs to make another async call, use thenCompose. If it's just a data transformation, use thenApply. Using thenApply when you should use thenCompose creates nested CompletableFuture<CompletableFuture<T>>.
🤝
thenCombine — Runs two CFs in parallel and combines their results when BOTH complete. userCF.thenCombine(ordersCF, (user, orders) -> merge(user, orders)). The two futures run simultaneously — faster than sequential chaining.
👥
allOf vs anyOf — allOf(cf1, cf2, cf3): completes when ALL complete. anyOf(cf1, cf2, cf3): completes when ANY one completes (returns Object type). allOf returns CompletableFuture<Void> — collect individual results by calling .get() on each original CF after allOf completes.

    .supplyAsync(() -> userRepo.findById(42L),  myExecutor);
// caller continues immediately — no blocking

Must Know
What is the difference between thenApply() and thenCompose()?
▾
This is the most important distinction in CompletableFuture:

thenApply(T → R): the function returns a plain value. CF wraps it for you. Like map(). Use for: synchronous transformations — convert a User to a UserDTO, format a string, apply business logic.

thenCompose(T → CF<R>): the function returns another CompletableFuture. CF "flattens" the nested future. Like flatMap(). Use for: chaining async operations — after fetching the user, async-fetch the orders for that user.

Using thenApply when you should use thenCompose: you get CompletableFuture<CompletableFuture<Orders>> — a future of a future. You'd have to call .get() twice to unwrap, defeating the purpose.
💡 Memory trick: if your lambda returns T (plain value) → thenApply. If your lambda returns CF<T> (another future) → thenCompose.
Tricky
Which thread executes the callbacks in CompletableFuture?
▾
This is subtle and important for understanding performance:

If the CF has already completed by the time you register the callback, the callback runs on the thread that registered it (the current thread).

If the CF is still in progress when you register the callback, the callback runs on the thread that COMPLETES the CF — typically the pool thread that finished the async work.

To control which thread runs callbacks: use thenApplyAsync(fn, executor) instead of thenApply(fn). The Async variant always submits the callback to the specified executor (or ForkJoin common pool if omitted). This is critical when the callback does I/O — you don't want an I/O callback running on a ForkJoin compute thread.
⚠️ Never do blocking I/O in callbacks without specifying a dedicated I/O executor. Use thenApplyAsync(fn, ioExecutor) not thenApply(fn).
Tricky
What is the difference between get() and join() on CompletableFuture?
▾
Both block until the CompletableFuture completes, but they differ in exception handling:

get(): throws InterruptedException (if interrupted while waiting) and ExecutionException (if the computation threw an exception) — both are checked exceptions. Forces callers to handle them explicitly.

join(): throws CompletionException (unchecked) if the computation failed, and doesn't throw InterruptedException (though it respects interruption). More convenient in lambda callbacks (no try-catch needed).

Use get(timeout, unit) when you want bounded blocking with a deadline. Use join() inside lambda chains where checked exception syntax is awkward. In non-lambda code, prefer get(timeout, unit) for explicit timeout handling.
💡
CompletableFuture = Future with a callback pipeline. No blocking.

• Transform sync → thenApply (T → R)
• Chain another async → thenCompose (T → CF<R>)
• Consume result → thenAccept (T → void)
• Run after (no input) → thenRun

• Two CFs parallel → thenCombine

🧠
In plain English: Advanced CF patterns cover: how to timeout a future that's taking too long, how to run many async operations in parallel and collect all results, how errors propagate through the chain, and why you must use a dedicated I/O thread pool for database/HTTP calls instead of the default ForkJoin pool.
⏱️

Chain: if you don't handle an exception, it skips all transforms and surfaces at the final .get()/.join().

Must Know
Why should you never do blocking I/O inside a CompletableFuture using the default ForkJoin pool?
▾
The ForkJoin common pool (used by default in supplyAsync without an executor) is designed for CPU-bound, non-blocking work. It has a thread count equal to CPU cores minus 1 (e.g., 7 threads on an 8-core machine).

When you do a blocking DB query or HTTP call in a ForkJoin thread: that thread is sleeping (waiting for I/O). It's not available for other tasks. With 7 threads and 7 concurrent DB queries, all 7 threads are sleeping — no other CF callback can run, including unrelated ones. This starvation can cascade: parallel streams, other CFs, and CompletableFuture combinators all use the same pool.

Solution: always provide a dedicated I/O executor: supplyAsync(() -> dbCall(), ioExecutor). The I/O executor can have many threads (50+) since they spend most time waiting. ForkJoin stays clean for CPU work.
⚠️ This is one of the most common mistakes in CF usage. Using the default pool for DB/HTTP calls causes subtle, hard-to-debug performance degradation under load.
Tricky
How does exception handling work across a CompletableFuture chain?
▾
When one stage throws an exception, all subsequent thenApply/thenCompose/thenAccept stages are skipped — the exception propagates through the chain like a short-circuit. It travels through every stage until it hits an error handler.

exceptionally(fn): catches exception, provides fallback value, chain resumes normally with the fallback.
handle(biFunction): receives both result AND exception (one will be null). Can both recover and transform.
whenComplete(biConsumer): receives result AND exception, but can't change them — purely for side effects (logging, metrics). Like a finally block.

If no handler catches the exception and you call get(): throws ExecutionException wrapping the original. If you call join(): throws CompletionException.

How would you implement a fallback/retry pattern with CompletableFuture?
▾
Retry with CompletableFuture is implemented by creating a new CF on failure:
CompletableFuture<T> withRetry(Supplier<CF<T>> taskSupplier, int retries) {
    return taskSupplier.get().exceptionallyCompose(ex -> {
        if (retries > 0) {
            log.warn("Retry {} remaining", retries);
            return withRetry(taskSupplier, retries - 1);
        }
        return CF.failedFuture(ex);
    });
}
For fallback pattern: .exceptionally(ex → fallbackCF) or .exceptionallyCompose(ex → backupServiceCall()). exceptionallyCompose (Java 12+) is like exceptionally but the fallback function returns a CF, enabling async fallback (e.g., try primary DB → on failure, try replica DB).

ALWAYS provide ioExecutor for I/O tasks:
supplyAsync(() → dbCall(), ioPool) not supplyAsync(() → dbCall())

Timeout pattern: orTimeout(3, SECONDS) → exceptionally → fallback

whenComplete = always runs (success OR error) = CF's finally block

Virtual Threads — Project Loom
Millions of lightweight threads — blocking code that scales like async, without the complexity
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Traditional OS threads are expensive — each one consumes 1–2 MB of memory and OS resources. You can realistically run 1,000–10,000 threads before hitting limits. Virtual threads (Java 21) are JVM-managed, ultra-lightweight — they use only kilobytes and you can run millions simultaneously. When a virtual thread blocks on I/O, the JVM automatically parks it (frees the underlying OS thread) and parks another virtual thread instead. Your blocking code now magically scales like async code — without callbacks, without reactive programming.
◆ Platform Threads vs Virtual Threads

Platform (OS) Thread

  • ▸ 1 JVM thread = 1 OS thread
  • ▸ ~1–2 MB stack per thread
  • ▸ Max ~10,000 practical threads
  • ▸ Blocking wastes OS thread
  • ▸ Context switch expensive (microseconds)
  • ▸ Managed by OS scheduler

Virtual Thread (Java 21)

  • ▸ Many VTs share few OS threads (carrier threads)
  • ▸ ~few KB stack per VT (grows as needed)
  • ▸ Millions of concurrent VTs
  • ▸ Blocking → JVM parks VT, frees carrier
  • ▸ Context switch cheap (JVM managed)
  • ▸ Managed by JVM scheduler
🧵
Carrier Threads — The JVM manages a pool of platform (carrier) threads (default: CPU core count). Virtual threads are scheduled onto carrier threads by the JVM. When a virtual thread blocks on I/O, the JVM "unmounts" it from the carrier thread and parks it in a lightweight data structure. The carrier thread is immediately available to run another virtual thread. When the I/O completes, the VT is re-mounted on any available carrier.
🚀
Thread-per-Request with Virtual Threads — In Spring Boot 3.2+ with Java 21: enable virtual threads with spring.threads.virtual.enabled=true. Tomcat serves each HTTP request on a virtual thread. Your servlet code can block freely (DB query, HTTP call) — the carrier thread isn't blocked, it's running other requests. You get the scalability of reactive without the callback/Mono complexity.
⚠️
Pinning — The Main Limitation — Virtual threads can be "pinned" to their carrier thread (can't be unmounted) in two cases: (1) Inside a synchronized block while blocking on I/O. (2) Inside native code. Pinned VTs defeat the purpose — the carrier is blocked. Fix: replace synchronized with ReentrantLock for code paths that may do blocking I/O. Most modern Java libraries have been updated to avoid pinning.
🚫
Don't Pool Virtual Threads — Creating a pool of virtual threads defeats the point — VTs are cheap, just create them when needed and let them die. Executors.newVirtualThreadPerTaskExecutor() creates a new VT per task. Don't use setCorePoolSize or setMaxPoolSize for virtual thread executors — there's no value in limiting them.
🔗
Structured Concurrency (Java 21 Preview) — A new API that ensures: if you start multiple tasks as child VTs, they all complete (success or error) before the parent continues. Solves the "fire-and-forget child thread leaks" problem. try (StructuredTaskScope.ShutdownOnFailure scope = new ...()) — any child task failure cancels the others. Cleaner lifecycle management than CompletableFuture.allOf().

Must Know
What are virtual threads and what problem do they solve?
▾
Virtual threads (Project Loom, GA in Java 21) are lightweight, JVM-managed threads that solve the thread-per-request scaling problem. Traditional OS threads are expensive (~1MB each), limiting applications to ~10,000 concurrent threads. If each request blocks on DB I/O for 50ms, you need 10,000 threads to handle 200,000 req/s — impossible with OS threads.

Virtual threads solve this: each VT uses only ~few KB. You can run millions. When a VT blocks on I/O, the JVM "parks" it (unmounts from the underlying OS carrier thread) so the carrier can run another VT. 8 OS threads can handle millions of concurrent virtual threads by rapidly switching between those waiting for I/O.

Result: write normal blocking code (no async/reactive complexity) and get the throughput of reactive systems. The JVM handles the scheduling magic.
💡 Virtual threads eliminate the "concurrency vs simplicity" trade-off. You no longer have to choose between simple blocking code and scalable async code.
Tricky
What is "pinning" in virtual threads and when does it occur?
▾
Pinning happens when a virtual thread cannot be unmounted from its carrier thread, even when blocking on I/O. This defeats the purpose of virtual threads — the carrier is stuck, unable to run other VTs.

Pinning occurs in two situations:
1. Inside synchronized blocks while blocking: if a VT inside a synchronized block calls a blocking operation (DB query, sleep), it stays pinned to the carrier. The JVM cannot transfer the lock ownership to a different carrier.
2. Inside native (JNI) code: JVM can't unwind the native stack to unmount.

Fix for synchronized pinning: replace synchronized with ReentrantLock in hot paths that do I/O. Use -Djdk.tracePinnedThreads=full JVM flag to detect pinning in your code.

Good news: Java 23 introduced "Synchronized Pinning" improvements, and most Java ecosystem libraries have already been updated to avoid pinning.
⚠️ Only synchronized causes pinning. ReentrantLock, ReadWriteLock, StampedLock — all VT-friendly (no pinning).
Compare
Virtual threads vs reactive programming (WebFlux) — which should you use?
▾
Both solve the same problem (I/O-bound scalability) but with very different trade-offs:

Reactive (WebFlux): non-blocking from end to end. Maximum efficiency. But: requires reactive drivers (R2DBC instead of JPA, WebClient instead of RestTemplate), callback-based programming model (harder to reason about), difficult debugging (stack traces through operators), steep learning curve, no ThreadLocal support.

Virtual Threads: blocking code that scales. Write normal imperative code — JPA, RestTemplate, Thread.sleep — it all just works. Same debugging tools, normal stack traces, ThreadLocal works, no library changes needed. Trade-off: slightly less efficient than fully reactive for extreme I/O workloads, and doesn't support backpressure natively.

Recommendation (2024+): For most Spring Boot applications, virtual threads are the better choice. They provide 80–90% of reactive's throughput with 100% of the simplicity. Only choose reactive when you genuinely need backpressure control or are integrating with a reactive ecosystem.
⚡ Java 21 + Spring Boot 3.2 + spring.threads.virtual.enabled=true = the new default for scalable Spring apps. Reactive is now a specialist tool, not the default choice.
Senior
Do virtual threads eliminate the need for connection pools like HikariCP?
▾
No — connection pools are still essential. Here's why:

Virtual threads change how threads wait, not how databases handle connections. A database has a fixed connection limit (e.g., PostgreSQL defaults to 100). Without a connection pool: 10,000 concurrent VTs × each opens a DB connection = 10,000 connections → database refuses connections → application fails.

With HikariCP (say, pool size 20): 10,000 VTs compete for 20 connections. 9,980 VTs wait for a connection. With platform threads, those waiting VTs would block carrier threads. With virtual threads, waiting VTs are unmounted — their carrier threads run other VTs. When a connection is returned to the pool, a waiting VT is resumed on a carrier thread.

Virtual threads make connection pool waiting efficient — but the pool is still necessary to protect the database from connection exhaustion.
💡 Rule: VT + HikariCP(20 connections) efficiently handles 10,000 concurrent VTs — the wait is now "free" (no carrier thread wasted). Without HikariCP: 10,000 connections crash the DB.
💡
Virtual Threads = millions of lightweight threads that unblock like coroutines.
Platform thread blocks → OS thread sleeps (wasted). Virtual thread blocks → JVM parks VT, carrier thread runs another VT (efficient).

Enable in Spring Boot 3.2+: spring.threads.virtual.enabled=true — that's all.

Pinning = VT stuck to carrier (bad):
Cause 1: synchronized block + blocking I/O → fix: use ReentrantLock
Cause 2: native JNI code

Still need connection pools — VTs make waiting efficient, not unlimited DB connections.

Virtual threads vs Reactive: VTs win on simplicity. Use Reactive only when you need backpressure.
