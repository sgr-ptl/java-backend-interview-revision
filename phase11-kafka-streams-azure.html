🧠
In plain English: Kafka is a distributed, fault-tolerant, high-throughput message broker. Think of it as an append-only log database for events. Producers write to it, consumers read from it at their own pace, and Kafka keeps data for a configurable time. It can handle millions of events per second without losing anything.

📂
Topic — A named category/feed for messages. Like a database table or a folder. Messages are ordered within a partition, but not across partitions. Topics are split into partitions for parallelism and scalability. Retain messages for a configurable duration (default 7 days) — regardless of whether anyone consumed them.
🗂️
Partition — The unit of parallelism in Kafka. Each partition is an ordered, immutable, append-only log. Replicated across multiple brokers for fault tolerance. Partitions determine max parallelism: a topic with 12 partitions can have at most 12 consumers in a group actively consuming in parallel. More partitions = higher throughput but more overhead.
📏
Offset — A sequential ID assigned to each message within a partition. Consumers track which messages they've processed by committing offsets to Kafka (stored in the internal __consumer_offsets topic). This enables restart from failure: if a consumer crashes, it resumes from the last committed offset.
👥
Consumer Group — A group of consumers that together consume all partitions of a topic. Each partition is assigned to exactly ONE consumer in the group at a time. Multiple groups can independently consume the same topic — each gets its own copy. This is how pub-sub (each group = independent subscriber) and point-to-point (one group, one consumer) both work in Kafka.
🔑
Message Key & Partitioning — When a producer sets a key, the same key always goes to the same partition (hash of key % numPartitions). Use keys to guarantee ordering for related events: all events for user 42 go to partition 3 — they'll be consumed in order. Null key = round-robin distribution across partitions.
🛡️
Replication Factor & Leader — Each partition has N replicas across N brokers. One replica is the Leader (handles reads/writes), others are Followers (sync data). If the leader broker dies, ZooKeeper/KRaft elects a new leader from followers. min.insync.replicas=2: producer writes are acknowledged only when data is on 2+ replicas — prevents data loss.
⚙️
KRaft Mode (Kafka 3.3+ default) — Replaces ZooKeeper with Kafka's own built-in consensus protocol (Raft). Simpler deployment (no separate ZooKeeper cluster), faster metadata operations, easier scaling. As of Kafka 3.7+, ZooKeeper mode is deprecated. Spring Boot 3.x + Spring Kafka work seamlessly with KRaft.

◆ Interview Questions
Must Know
What is a Kafka partition and why does it matter for ordering and parallelism?
▾
A partition is an ordered, append-only log. Messages within a partition are strictly ordered by offset. Messages across different partitions have no ordering guarantee.

Parallelism: partitions are the unit of parallelism. A consumer group can have at most one consumer per partition actively consuming. 12 partitions = at most 12 consumers processing in parallel. If you add a 13th consumer, it sits idle until one of the 12 leaves.

Ordering use case: use message keys to route related messages to the same partition. orderId as key → all events for one order go to the same partition → consumed in order.
⚡ "How many partitions should I use?" Rule of thumb: target throughput / per-partition throughput. If you expect 120k msg/sec and each partition handles 10k/sec → 12 partitions minimum.
Must Know
What is a Consumer Group and how does Kafka balance partitions across consumers?
▾
A consumer group is a logical group of consumers that collaboratively consume all partitions of one or more topics. Kafka assigns each partition to exactly one consumer in the group at a time — this is called partition assignment.

Rebalancing: when consumers join/leave the group, Kafka triggers a rebalance — redistributes partitions. During rebalance, all consumers in the group pause briefly. Strategies: Range (contiguous), RoundRobin (even distribution), Sticky (minimizes movement on rebalance), CooperativeSticky (incremental — avoids stop-the-world pauses, default in modern Kafka).

Multiple groups: each consumer group maintains its own offset — they consume independently. This is Kafka's pub-sub model: N groups = N independent consumption pipelines from the same topic.
Tricky
What is the difference between acks=0, acks=1, and acks=all?
▾
acks=0: fire-and-forget. Producer doesn't wait for acknowledgment. Fastest, but messages can be silently lost (broker crash before write). Use for: non-critical metrics, logging where some loss is acceptable.

acks=1: leader writes to its log and acknowledges. Fast but risky: if leader crashes after ack but before followers sync, message is lost. Default in Kafka until 3.x.

acks=all (or acks=-1): producer waits until ALL in-sync replicas (ISR) have the message. Slowest but safest. Combine with min.insync.replicas=2: requires at least 2 replicas to be in-sync before ack. If fewer brokers are available, producer throws NotEnoughReplicasException — better than silent data loss.
⚡ Production recommendation: acks=all + min.insync.replicas=2 + enable.idempotence=true. This gives at-least-once delivery with deduplication.
Tricky
What is the difference between at-most-once, at-least-once, and exactly-once semantics?
▾
At-most-once: messages may be lost but never duplicated. Offset committed BEFORE processing. If consumer crashes after commit but before processing → message skipped. Use for: tolerable loss (metrics, click events).

At-least-once: no messages lost but duplicates possible. Offset committed AFTER processing. If consumer crashes after processing but before commit → message reprocessed. Consumer must be idempotent. Most common in practice.

Exactly-once: no loss, no duplicates. Achieved via Kafka Transactions (producer + consumer in same transaction) or Idempotent Producer (deduplication using producer ID + sequence). Kafka Streams uses this natively. Complex to implement with external systems (requires idempotent writes at the destination).
💡 Spring Kafka: set enable-idempotence=true + isolation-level=read_committed to enable exactly-once semantics.
Senior
How do you choose the right number of partitions for a Kafka topic?
▾
Key factors:
1. Throughput: each partition can sustain ~10–50 MB/s (depends on hardware). Target throughput / per-partition throughput = minimum partitions.
2. Consumer parallelism: partitions = max parallel consumers. Design for your expected peak consumer count.
3. Ordering requirements: fewer partitions = stronger ordering guarantees per key group.
4. Broker capacity: each partition = files on disk, memory for indexing. Too many partitions = broker overhead.

Rule of thumb: start with max(target_throughput/50MB, expected_consumers) × 2 for headroom. It's easy to increase partitions (keys may re-route to new partitions) but impossible to decrease without re-creating the topic.
⚡ Can't decrease partitions once created. Pick slightly higher than needed upfront. Common values: 6, 12, 24 (multiples of common broker counts).
◆ Memory Tip
💡
Kafka = distributed append-only log with subscriber groups.

Key mental model:
• Topic = table. Partition = table shard. Offset = row number.
• Consumer Group = team of workers sharing partitions.
• Same key → same partition → ordered events for that key.
• Multiple groups → each group gets ALL messages independently.

acks cheatsheet: 0 = fastest/unsafe · 1 = default/moderate · all = safe/slowest

Partition parallelism rule: max concurrent consumers = number of partitions. Extra consumers in a group sit idle.

🧠 Memory
🧠
In plain English: Producers push messages into topics; consumers pull messages from topics at their own pace. The critical decisions are: how many retries on failure? when to commit the offset (before or after processing)? what happens when processing fails — dead letter or retry?
📤
Producer Internals — Producer batches messages (batch.size: max bytes per batch) and waits for more (linger.ms: how long to wait before sending). Larger batches = better throughput, slightly higher latency. Compression (compression.type=snappy/lz4/zstd) reduces network and disk usage — especially effective for repetitive JSON. max.block.ms: how long to block if the buffer is full.
📥
Consumer Polling Loop — Consumer calls poll(timeout) in a loop to fetch records. max.poll.records: max records per poll (default 500). max.poll.interval.ms: if the consumer doesn't call poll() within this window, Kafka considers it dead and triggers rebalance. Long processing tasks must either: increase max.poll.interval.ms, process records asynchronously, or use pause/resume.
💾
Offset Commit Strategies — enable.auto.commit=true (default): Kafka commits offsets automatically every auto.commit.interval.ms. Risk: at-most-once (commit may happen before processing completes). Manual commit: ackMode=MANUAL_IMMEDIATE in Spring Kafka — call acknowledgment.acknowledge() after successful processing. Risk: at-least-once (duplicate processing if crash between processing and commit).
🔁
Retry & Dead Letter Topic (DLT) — Spring Kafka's DefaultErrorHandler retries failed records with configurable backoff. After max retries exhausted, sends the failed record to a Dead Letter Topic (e.g., orders.DLT). Dead letter topics are critical for production: failed messages don't block the partition, are stored for investigation, and can be replayed. Configure via DeadLetterPublishingRecoverer.
⏸️
Consumer Lag — The difference between the latest offset in a partition and the consumer's committed offset. High lag = consumer is falling behind. Monitor with: kafka-consumer-groups.sh --describe --group my-group or Kafka's metrics. Alert when lag exceeds threshold. Increase consumers (up to partition count) or optimize processing to reduce lag.
🏊
Concurrency & Listener Containers — Spring Kafka's listener container can run N listener threads: @KafkaListener(concurrency = "3") — creates 3 consumer threads, each assigned different partitions. Effective when a single consumer can't keep up. Max useful concurrency = partition count. Set containerFactory for different concurrency configs per listener.

// ── KafkaTemplate: send with key ─────────────────
@Service
public class OrderEventProducer {
    private final KafkaTemplate<String, OrderEvent> kafka;

    public void send(OrderEvent event) {
        String key = "order-" + event.getOrderId();  // same order → same partition
        CompletableFuture<SendResult<String, OrderEvent>> future =
            kafka.send("orders", key, event);

        future.whenComplete((result, ex) -> {
            if (ex == null) {
                RecordMetadata meta = result.getRecordMetadata();
                log.info("Sent to partition={} offset={}",
                    meta.partition(), meta.offset());
            } else {
                log.error("Failed to send: {}", ex.getMessage());
            }
        });
    }

    // Send with headers (correlation ID, trace)
    public void sendWithHeaders(OrderEvent event, String traceId) {
        ProducerRecord<String, OrderEvent> record =
            new ProducerRecord<>("orders", null, event.getOrderId().toString(), event);
        record.headers().add("X-Trace-Id", traceId.getBytes());
        kafka.send(record);
    }
}

// ── @KafkaListener — manual acknowledgment ────────
@Component
public class OrderEventConsumer {

    @KafkaListener(
        topics = "orders",
        groupId = "order-processor",
        concurrency = "3",                           // 3 threads (need ≥3 partitions)
        containerFactory = "manualAckContainerFactory"
    )
    public void consume(
            ConsumerRecord<String, OrderEvent> record,
            Acknowledgment ack) {

        log.info("Processing partition={} offset={}",
            record.partition(), record.offset());
        try {
            orderService.process(record.value());
            ack.acknowledge();   // commit AFTER success (at-least-once)
        } catch (RetryableException e) {
            // don't ack — will be retried by error handler
            throw e;
        }
    }
}

// ── Error Handler + Dead Letter Topic ────────────
@Bean
public DefaultErrorHandler kafkaErrorHandler(
        KafkaTemplate<Object, Object> template) {

    DeadLetterPublishingRecoverer recoverer =
        new DeadLetterPublishingRecoverer(template,
            (record, ex) -> new TopicPartition(
                record.topic() + ".DLT",  // orders → orders.DLT
                record.partition()
            )
        );

    ExponentialBackOff backoff = new ExponentialBackOff(1000L, 2.0);
    backoff.setMaxAttempts(3);     // retry 3 times: 1s, 2s, 4s

    DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backoff);
    handler.addNotRetryableExceptions(ValidationException.class); // no retry
    return handler;
}

// ── ContainerFactory for manual ack ───────────────
@Bean
public ConcurrentKafkaListenerContainerFactory<String, OrderEvent>
        manualAckContainerFactory(ConsumerFactory<String, OrderEvent> cf) {
    var factory = new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(cf);
    factory.getContainerProperties().setAckMode(AckMode.MANUAL_IMMEDIATE);
    factory.setCommonErrorHandler(kafkaErrorHandler(kafkaTemplate));
    return factory;
}
◆ Interview Questions
Must Know
What is consumer lag and how do you handle it?
▾
Consumer lag = latest offset in partition MINUS consumer's committed offset. If a topic has offset 10,000 and the consumer has committed offset 9,500, lag = 500 messages.

Causes: slow processing, consumer crashes, insufficient consumers, GC pauses.

Solutions: (1) Add more consumers up to partition count. (2) Increase concurrency in @KafkaListener. (3) Optimize processing — batch writes, async calls. (4) Increase partition count (requires new consumers). (5) Scale out horizontally (more pods in K8s).

Monitoring: Kafka's built-in metrics (records-lag-max), Prometheus + Grafana via JMX exporter, Confluent Control Center, or Burrow (lag monitoring tool).
⚡ If lag keeps growing despite max consumers, the bottleneck is processing speed, not consumer count. Profile your processing logic.
Tricky
What causes a rebalance and how can you minimize its impact?
▾
Rebalance triggers: consumer joins group, consumer leaves/crashes, max.poll.interval.ms exceeded (consumer took too long to poll), topics subscribed to change, partition count changes.

Impact: during rebalance, all consumers in the group pause processing (stop-the-world for old protocol). Can cause duplicate processing (uncommitted offsets re-processed by new assignees).

Minimization strategies: (1) CooperativeSticky assignor — incremental rebalance, only affected partitions move, others keep processing. (2) Static group membership: group.instance.id — K8s pod restarts without triggering rebalance within session.timeout.ms. (3) Increase max.poll.interval.ms if processing takes long. (4) Reduce processing time per batch.
Tricky
What is a Dead Letter Topic (DLT) and why is it better than just logging the error?
▾
A Dead Letter Topic stores messages that failed processing after all retry attempts. Better than logging because: (1) Message is preserved intact (key, value, headers, original offset) — you can replay it after fixing the bug. (2) Doesn't block the main topic partition — processing continues for other messages. (3) Structured for tooling — you can monitor DLT size, alert on growth, write automated replay jobs. (4) Headers include failure reason, original topic/partition/offset, and stack trace — traceable.

Pattern: consumers of the DLT can: alert ops team, attempt different processing strategy, route to manual review, or replay after a fix is deployed.
💡 Spring Kafka automatically adds headers to DLT records: kafka_dlt-original-topic, kafka_dlt-exception-message, kafka_dlt-original-offset.
Senior
How do you implement exactly-once semantics end-to-end with Kafka?
▾
True exactly-once (EOS) requires coordination at both producer and consumer sides:

Producer side: enable idempotent producer (enable.idempotence=true) + transactions (transactional.id=unique-id). The broker deduplicates retried messages using producer ID + sequence number.

Consumer side: set isolation.level=read_committed — consumers only see messages from committed transactions, not in-progress ones.

Database sink: the hardest part. For a Kafka→DB pipeline: consume in a transaction, write to DB in the same transaction, then commit the Kafka offset. This works with databases that support transactions. If the DB write succeeds but Kafka commit fails → duplicate write on retry — make DB writes idempotent (UPSERT, unique constraints).

Kafka Streams does all of this natively with processing.guarantee=exactly_once_v2.
💡
Producer sends → Kafka stores → Consumer pulls at own pace.

Delivery semantics cheatsheet:
• Commit before processing → at-most-once (fast, may lose)
• Commit after processing → at-least-once (safe, may duplicate)
• Idempotent + transactions → exactly-once (complex, guaranteed)

DLT pattern: fail → retry N times with backoff → send to orders.DLT → keep processing rest → investigate DLT separately.

Consumer performance: lag growing? → add consumers (up to partition count) OR optimize processing OR both.

🧠
In plain English: Spring Kafka wraps the native Kafka client with Spring idioms — @KafkaListener for consuming, KafkaTemplate for producing, transaction support, retry/DLT wiring, and EmbeddedKafka for tests. You configure once in application.yml and use annotations everywhere.
📨
@KafkaListener Features — Listen to multiple topics: topics = {"orders", "payments"}. Listen to specific partitions: topicPartitions = @TopicPartition(topic="orders", partitions={"0","1"}). Filter records: filter attribute with a RecordFilterStrategy. Batch listening: receive List<ConsumerRecord> instead of one record at a time. Access message metadata via @Header(KafkaHeaders.RECEIVED_PARTITION).
🔗
Kafka Transactions in Spring — @Transactional on a method that both produces and consumes wraps both in a Kafka transaction. If the method throws, the message production is rolled back. Configure via spring.kafka.producer.transaction-id-prefix=tx-. Enables consume-process-produce atomicity: if you consume a message, transform it, and produce to another topic — all or nothing.
📊
Metrics & Monitoring — Spring Kafka exposes Micrometer metrics: kafka.consumer.records-lag, kafka.producer.record-send-rate, etc. Add spring-boot-starter-actuator + Prometheus dependency → metrics auto-exported to /actuator/prometheus. Pair with Grafana dashboards for real-time visibility.
🧪
@EmbeddedKafka for Testing — @EmbeddedKafka(partitions = 1, topics = {"orders"}) starts an in-process Kafka broker. Use @SpringBootTest or @KafkaTest. Override bootstrap-servers with ${spring.embedded.kafka.brokers}. Can send test messages with KafkaTemplate and verify received records with ConsumerRecord capture using a BlockingQueue.
🎯
Idempotent Consumer Pattern — Kafka guarantees at-least-once delivery — consumers must handle duplicates. Strategies: (1) Check if message ID already processed in DB before processing. (2) Use UPSERT operations (INSERT ... ON CONFLICT DO UPDATE). (3) Use distributed locks. Store messageId or kafkaOffset in a processed-events table to detect replays.

◆ Interview Questions
Must Know
How do you handle serialization/deserialization of complex Java objects in Kafka?
▾
Options in order of recommendation:

1. JSON with JsonSerializer/JsonDeserializer (Spring Kafka): human-readable, easy debugging, schema-free. Configure spring.json.trusted.packages to prevent deserialization of untrusted classes. Works immediately with POJOs.

2. Apache Avro + Schema Registry: binary format (smaller, faster), schema evolution support, schema validation. Producer registers schema; consumer validates. Confluent Schema Registry or AWS Glue Schema Registry. io.confluent:kafka-avro-serializer. Best for production at scale.

3. Protobuf: Google's binary format — extremely compact, strongly typed. Good for polyglot environments (Java, Python, Go all consume the same messages).
⚡ JSON for dev/small scale. Avro + Schema Registry for production with high volume and schema evolution needs.
Tricky
What is the difference between @KafkaListener with auto-commit vs manual acknowledgment?
▾
Auto-commit (enable.auto.commit=true): Kafka commits offsets automatically every auto.commit.interval.ms (default 5s). Risk: consumer processes record, auto-commit fires, then consumer crashes before completing — offset committed, record treated as consumed even if processing failed (at-most-once for failed records).

Manual acknowledgment (AckMode.MANUAL_IMMEDIATE): you call acknowledgment.acknowledge() only after successful processing. If crash before ack → record reprocessed on restart (at-least-once). This is the safer production choice — combined with idempotent processing, it's reliable.

AckMode options: RECORD (ack per record), BATCH (ack after processing all polled records), MANUAL, MANUAL_IMMEDIATE (ack immediately when you call it).
Tricky
How do you test Kafka consumers and producers without a real Kafka cluster?
▾
@EmbeddedKafka (Spring Test): starts an in-memory Kafka broker for your test. Real Kafka behavior — topics, partitions, offsets. Override spring.kafka.bootstrap-servers with ${spring.embedded.kafka.brokers}. Works great for integration tests.

Testcontainers: start a real Kafka Docker container per test class. More realistic (real Kafka version), slightly slower. @Container KafkaContainer kafka = new KafkaContainer(DockerImageName.parse("confluentinc/cp-kafka:7.4")). Inject URL with @DynamicPropertySource.

Mock KafkaTemplate: for unit tests, mock KafkaTemplate with @MockBean and verify .send(topic, key, value) was called.
💡 For consumer tests: publish to EmbeddedKafka via KafkaTemplate, listen with a test @KafkaListener that puts records into a BlockingQueue, assert with poll(timeout).
💡
Spring Kafka = Spring idioms over native Kafka client.

@KafkaListener is to Kafka what @RestController is to HTTP. Define a method, add the annotation — Spring handles threading, deserialization, offset management, and error recovery.

Testing strategy:
• Unit test service logic with mocked KafkaTemplate
• Integration test with @EmbeddedKafka (fast, in-process)
• System test with Testcontainers (real Kafka, Docker)

Serialization rule: JSON for simplicity, Avro+Schema Registry for production scale.

🧠 Memory
🧠
In plain English: Kafka Streams is a library (not a separate cluster) that lets you process data IN Kafka using Java code. Read from a topic, transform/filter/join/aggregate, write back to a topic — all in a single Java application. No Spark, no Flink, no separate infrastructure needed.
◆ Core Abstractions

🌊
KStream — Represents an unbounded sequence of records. Every record is an independent event. Operations: filter, map, flatMap, foreach, to (write to topic), branch (split into multiple streams). Like Java Streams but infinite and backed by Kafka topics. Processing is stateless by default.
📋
KTable — A table of latest values keyed by key. Each new record for a key updates/replaces the previous value (like a database). Backed by a compacted topic and a local RocksDB state store. When joined with a KStream: each stream event is enriched with the latest table value for the matching key. Think: event-driven database lookup.
🌍
GlobalKTable — Like KTable but replicated across ALL application instances. Partitioned KTable: each instance only has a subset of data. GlobalKTable: every instance has ALL data. Enables lookups using non-partition-key fields. Uses more memory. Good for: small reference data (country codes, product catalog) used in joins.
⏱️
Windowing — Group stream events into time windows for aggregation. Types: Tumbling (fixed, non-overlapping: 5-min windows), Hopping (overlapping: 5-min window, advance every 1 min), Session (activity-based: inactivity gap defines window boundary). Use with windowedBy() in an aggregation chain.
🗃️
State Stores & RocksDB — Kafka Streams uses local RocksDB instances for stateful operations (aggregations, joins). State is backed by changelog topics in Kafka — if an instance crashes, another instance restores state from the changelog. State stores are partitioned — each instance only manages state for its assigned partitions. Interactive Queries API allows querying local state.
💡
Exactly-Once v2 (EOS) — Set processing.guarantee=exactly_once_v2. Kafka Streams wraps read-process-write in a transaction. If processing fails mid-way, the output is rolled back. The most efficient exactly-once implementation available. Works only with Kafka as source and sink (no external systems in the exactly-once boundary).

// ── Topology: filter + transform + to ────────────
@Component
public class OrderStreamTopology {

    @Autowired
    void buildTopology(StreamsBuilder builder) {

        KStream<String, Order> orders = builder.stream("orders");

        // ── Filter + Map ───────────────────────────────
        orders
            .filter((key, order) -> order.getAmount().compareTo(BigDecimal.TEN) > 0)
            .mapValues(order -> new LargeOrderEvent(order.getId(), order.getAmount()))
            .to("large-orders");

        // ── Branch: split stream ───────────────────────
        Map<String, KStream<String, Order>> branches = orders.split(Named.as("split-"))
            .branch((k, v) -> v.getStatus().equals("PLACED"),   Branched.as("placed"))
            .branch((k, v) -> v.getStatus().equals("SHIPPED"), Branched.as("shipped"))
            .defaultBranch(Branched.as("other"));
        branches.get("split-placed").to("placed-orders");

        // ── Aggregation: count orders per customer ─────
        orders
            .groupBy((key, order) -> order.getCustomerId()) // re-key by customer
            .count(Materialized.as("order-counts"))       // state store name
            .toStream()
            .to("customer-order-counts");

        // ── Windowed aggregation: orders per 5 min ────
        orders
            .groupByKey()
            .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
            .count()
            .toStream()
            .map((window, count) ->
                KeyValue.pair(window.key(),
                    new WindowCount(window.window().startTime(), count)))
            .to("order-window-counts");

        // ── KStream-KTable Join ───────────────────────
        KTable<String, Customer> customers = builder.table("customers");
        orders
            .join(customers,
                (order, customer) -> new EnrichedOrder(order, customer),
                Joined.with(Serdes.String(), orderSerde, customerSerde))
            .to("enriched-orders");
    }
}
◆ Interview Questions
Must Know
What is the difference between KStream, KTable, and GlobalKTable?
▾
KStream: an infinite sequence of individual events. Each record is independent — doesn't replace previous records with the same key. Like a database's INSERT-only log. Use for: event streams, activity logs, transactions.

KTable: represents the latest state per key — a materialized view. A new record with the same key updates/replaces the previous. Backed by a local RocksDB store. Partitioned like the source topic. Use for: user profiles, account balances, order statuses (current state).

GlobalKTable: a KTable fully replicated to all application instances. Every instance has ALL the data. Enables non-key-based joins (lookup by a field that's not the partition key). More memory, but no network hop on join. Use for: small, stable reference data (product catalog, country codes).
⚡ KStream = facts (immutable events). KTable = state (current snapshot). GlobalKTable = broadcast state (available everywhere).
Tricky
How does state store persistence work in Kafka Streams?
▾
Kafka Streams uses local RocksDB (embedded key-value store) for stateful operations. Every state store is backed by a Kafka changelog topic — every change to the store is written to the changelog.

On application failure/restart: the new instance reads the changelog topic from the beginning and reconstructs the state store. This can take time for large stores (offset by using standby replicas).

Standby replicas: num.standby.replicas=1 — Kafka Streams maintains a copy of each state store on another instance. If primary fails, standby takes over immediately (near-instant failover instead of rebuilding from scratch).

Interactive Queries: query a state store from outside: streams.store(StoreQueryParameters.fromNameAndType("order-counts", QueryableStoreTypes.keyValueStore())). Enables a REST API backed by the stream state.
Tricky
What are the different window types in Kafka Streams and when do you use each?
▾
Tumbling Window: fixed-size, non-overlapping. Events 0–5min in window 1, 5–10min in window 2. Each event belongs to exactly one window. Use for: per-5-minute aggregations, hourly reports.

Hopping Window: fixed-size, overlapping. 5-min window advancing every 1 min. Events may belong to multiple windows. Use for: rolling averages, sliding metrics.

Session Window: dynamic size based on activity. Groups events within an inactivity gap (e.g., all user events within 30 minutes of each other). Use for: user sessions, activity tracking, burst detection.

All windows have grace period (withGracePeriod()) — late-arriving events within the grace period are included in the window. Events after the grace period are dropped.
⚡ Use Session Windows for behavioral analytics (user sessions, clickstreams). Use Tumbling for fixed-period reporting. Use Hopping for smooth rolling metrics.
Senior
Kafka Streams vs Apache Flink vs Spark Streaming — when to use which?
▾
Kafka Streams: embedded library, no separate cluster, runs as part of your app. Simple deployment. Kafka-only (source and sink). Best for: teams already on Kafka wanting stream processing without new infrastructure, microservices pattern, moderate complexity processing.

Apache Flink: dedicated cluster, more powerful (event time semantics, complex stateful processing, exactly-once with external systems). Better for: complex CEP (Complex Event Processing), ML pipelines, joins between non-Kafka sources, when you need fine-grained resource management.

Spark Streaming / Structured Streaming: micro-batch processing, tight Spark ecosystem integration, great for batch+streaming hybrid. Best for: teams already using Spark, ETL pipelines, batch ML + streaming scoring, HDFS/data lake integration.
💡 Start with Kafka Streams if Kafka is your source/sink. Only add Flink/Spark when you outgrow Kafka Streams' capabilities.

Join Types in Kafka Streams
KStream-KStream: windowed join (events within a time window).
KStream-KTable: lookup join (enrich each event with latest table value).
KStream-GlobalKTable: non-key join (join on non-partition key field).
KTable-KTable: equi-join (update the join result whenever either table updates).
All joins require co-partitioned sources (same partition count and key type) except GlobalKTable joins.
🏗️
CQRS with Kafka Streams — Pattern: Commands → Kafka topic → Kafka Streams processor → update state store → Query side reads from state store. Event Sourcing + CQRS: every state change is an event; the current state is derived by replaying events (changelog). State store = read model. Interactive Queries = query API for the read model.
🔍
Interactive Queries — Query local state stores from outside the Streams topology. Example: expose order count via REST endpoint. ReadOnlyKeyValueStore<String, Long> store = streams.store(...). In a distributed deployment (multiple app instances), some queries may require forwarding to the instance that owns that key's partition. Use Kafka's Streams Metadata API to find the right instance.
📊
Aggregation Operations — count(): count records per key. reduce(): combine values (sum, min, max). aggregate(): custom aggregation with initializer and adder. All produce a KTable. Chain after groupByKey() or groupBy(). All backed by local state stores with changelog topics.
⚠️
Repartitioning & Co-partitioning — When you groupBy() on a non-partition-key field, Kafka Streams creates an internal repartition topic and reroutes records. This is transparent but has overhead (extra topic, extra read/write). For joins: all joined topics must have the same number of partitions and the same key type. Verify before deploying or you get a TopologyException.

@RestController
public class OrderCountController {
    private final KafkaStreams streams;

    @GetMapping("/counts/{customerId}")
    public Long getCount(@PathVariable String customerId) {
        ReadOnlyKeyValueStore<String, Long> store = streams.store(
            StoreQueryParameters.fromNameAndType(
                "order-counts",
                QueryableStoreTypes.keyValueStore()
            )
        );
        Long count = store.get(customerId);
        return count != null ? count : 0L;
    }

    // Get all entries (for small stores)
    @GetMapping("/counts")
    public Map<String, Long> getAllCounts() {
        ReadOnlyKeyValueStore<String, Long> store = getStore();
        Map<String, Long> result = new HashMap<>();
        store.all().forEachRemaining(kv -> result.put(kv.key, kv.value));
        return result;
    }
}

◆ Interview Questions
Must Know
What is co-partitioning and why is it required for Kafka Streams joins?
▾
Co-partitioning means two topics have the same number of partitions and use the same partitioning strategy (same key → same partition number in both topics). Kafka Streams joins require co-partitioning because each stream instance processes specific partitions — for a join, the matching keys from both topics must be on the same instance.

If topics have different partition counts, the join key might be on different instances for the two topics — they'd never meet. Kafka Streams throws a TopologyException at startup if it detects non-co-partitioned topics in a join.

Fix: create both topics with the same partition count. If they differ: use repartition() to create a new repartitioned topic before the join. GlobalKTable is the exception — it's fully replicated so co-partitioning is not required.
⚡ Rule: always create related Kafka topics (that you'll join) with the same partition count. Plan this upfront — changing partition counts is disruptive.
Tricky
How do you handle late-arriving events in Kafka Streams windows?
▾
By default, Kafka Streams windows close after their end time and discard late events. Configure a grace period: TimeWindows.ofSizeAndGrace(Duration.ofMinutes(5), Duration.ofMinutes(1)) — events up to 1 minute late are still accepted into the window.

After the grace period, late records are dropped (no error). If you need to handle late records differently: use a KStream with punctuators to manage time-based processing manually, or use the Processor API for fine-grained control.

Event time vs processing time: Kafka Streams can use the record's timestamp (set by the producer at event time) rather than wall clock. Configure TimestampExtractor to extract event time from record headers or payload for more accurate windowing.
💡 Grace period is the trade-off between result completeness and latency. Larger grace = more complete but slower results.
Senior
How would you implement a CQRS/Event Sourcing pattern with Kafka Streams?
▾
Pattern: Commands → Kafka topic → Streams processor → State store (Read Model) → REST API.

1. Write side: commands produce events to Kafka (OrderPlaced, OrderShipped, OrderCancelled). Each event is immutable and represents what happened.

2. Streams processor: aggregate events by order ID using groupByKey().aggregate() — maintains current order state (read model) in a local state store backed by a changelog topic.

3. Read side: REST API queries the local state store via Interactive Queries. For queries hitting a different partition's data: use Kafka Streams' metadata API to locate the right instance and forward the HTTP request there.

4. Replay: to rebuild read models, reset the consumer group and replay from offset 0. The state stores get rebuilt automatically from the changelog.
💡
Kafka Streams joins cheatsheet:
• Stream + Stream = windowed (events within time window)
• Stream + KTable = enrich each event with latest state
• Stream + GlobalKTable = enrich with any key (no co-partition needed)
• KTable + KTable = delta join (updates trigger new join result)

Co-partitioning rule: same topic partition count + same key for Stream-KTable and Stream-Stream joins.

Interactive Queries = turn your stream state into a queryable REST API without an extra database.

What is KEDA and how does it enable Kafka-driven autoscaling on Kubernetes?
▾
KEDA (Kubernetes Event-Driven Autoscaling) is a Kubernetes operator that scales deployments based on external event sources — including Kafka consumer lag. Normal HPA only scales on CPU/memory. KEDA watches the Kafka consumer group lag:

1. KEDA polls Kafka broker for consumer group lag on configured topic.
2. When lag exceeds threshold (e.g., 100 messages per pod), KEDA tells HPA to scale up.
3. New pods join the consumer group → Kafka rebalances → more partitions processed → lag reduces.
4. When lag drops below threshold, KEDA scales down (including scale-to-zero when no messages).

Configuration: ScaledObject CRD pointing to the Kafka topic and consumer group, min/max replica bounds. Pairs perfectly with Azure Event Hubs (same Kafka protocol) or self-managed Kafka on AKS.
⚡ KEDA + Kafka = autoscaling perfectly aligned to processing demand. No wasted pods when Kafka is quiet. Instantly scales when messages arrive.
