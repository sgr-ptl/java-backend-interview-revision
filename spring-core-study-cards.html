Dependency Injection & Inversion of Control
"Don't call us, we'll call you" — the Hollywood Principle
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory Tip
◆ Core Concepts
🎯
IoC (Inversion of Control) — A design principle where the control of object creation and wiring is inverted from the application code to a container/framework. You don't new dependencies yourself — Spring does it for you.
💉
DI (Dependency Injection) — The implementation technique of IoC. Spring injects dependencies into your class instead of you creating them. 3 types: Constructor Injection (preferred), Setter Injection, and Field Injection (@Autowired on field — avoid in production).
🏗️
Constructor Injection — Preferred because: dependencies are immutable (final fields), object is always in valid state, makes dependencies explicit, enables easier unit testing without Spring container. Spring 4.3+ auto-injects single-constructor beans.
⚙️
Setter Injection — Used for optional dependencies. Allows reconfiguration after construction. Beware: object can be in partially initialized state if setter not called.
⚠️
Field Injection — @Autowired directly on field. Avoid in production: hides dependencies, can't be used with final, cannot test without Spring context, violates encapsulation. Fine only for test classes.
🔄
Circular Dependency — When A depends on B and B depends on A — Spring throws BeanCurrentlyInCreationException with constructor injection. Fix options: use setter injection for one side, use @Lazy, or redesign to break the cycle.
◆ Memorizable Code
DI — CONSTRUCTOR vs SETTER vs FIELD
// ── ✅ Constructor Injection (PREFERRED) ─────────
@Service
public class OrderService {
    private final PaymentService paymentService;
    private final InventoryService inventoryService;

    // @Autowired optional with single constructor (Spring 4.3+)
    public OrderService(PaymentService ps, InventoryService is) {
        this.paymentService   = ps;  // final = immutable ✅
        this.inventoryService = is;
    }
}

// ── ⚠️ Setter Injection (optional deps only) ──────
@Service
public class ReportService {
    private NotificationService notifier; // optional

    @Autowired(required = false)
    public void setNotifier(NotificationService n) {
        this.notifier = n;
    }
}

// ── ❌ Field Injection (avoid in prod) ────────────
@Service
public class BadService {
    @Autowired
    private SomeService someService; // hidden dep, not testable
}

// ── Circular Dependency Fix with @Lazy ───────────
@Service
public class A {
    private final B b;
    public A(@Lazy B b) { this.b = b; } // B created on first use
}
@Service
public class B {
    private final A a;
    public B(A a) { this.a = a; }
}
◆ Interview Questions
Must Know
What is the difference between IoC and DI?
▾
IoC is a design principle — it means you give up control of object lifecycle to a container. DI is a concrete implementation of IoC — the container injects required dependencies into your object rather than you creating them inside the class. Think of IoC as the "what" and DI as the "how". Spring IoC container manages bean creation, wiring, and lifecycle; DI is the mechanism it uses to wire beans together.
Must Know
Why is Constructor Injection preferred over Field Injection?
▾
4 key reasons:
1. Immutability — fields can be final, preventing accidental reassignment
2. Testability — you can instantiate the class directly in unit tests without Spring context
3. Explicit contract — dependencies are visible in the constructor signature, not hidden inside the class
4. Fail-fast — if a required bean is missing, Spring fails at startup, not at runtime
⚡ Tricky add-on: With field injection you can have a bean that compiles and starts but NPEs at runtime because Spring didn't inject it (e.g., if the class is new'd manually). Constructor injection makes this impossible.
Tricky
What happens when there's a circular dependency in Spring? How do you fix it?
▾
With constructor injection, Spring throws BeanCurrentlyInCreationException at startup because it can't instantiate A without B and can't instantiate B without A. Fixes:
1. @Lazy on one constructor param — defers creation to first use
2. Use setter injection for one side — Spring can create both objects first, then inject
3. Best fix: redesign — circular deps usually indicate poor separation of concerns (extract a third class)
⚡ Note: Spring Boot 2.6+ disabled circular dependency by default. You'll get a startup error with a hint to refactor, or use spring.main.allow-circular-references=true (not recommended).
Tricky
Can you inject a prototype-scoped bean into a singleton bean? What goes wrong?
▾
Yes, but it doesn't work as expected. When a singleton bean has a prototype dependency injected via constructor/field, the prototype is injected only once at startup — you always get the same instance, defeating the purpose of prototype scope. Fixes:
1. Inject ApplicationContext and call ctx.getBean() each time
2. Use Method Injection — @Lookup annotation on an abstract method
3. Use ObjectFactory<YourBean> or Provider<YourBean> for lazy retrieval
⚡ Classic interview trap! Many devs assume the prototype scope is respected automatically — it's not.
Concept
What is the Hollywood Principle and how does it relate to IoC?
▾
The Hollywood Principle states: "Don't call us, we'll call you." In traditional code, your classes control the flow — they create dependencies via new and call framework code. With IoC, the framework (Spring) controls the flow — it creates your objects and calls your code. You register components; Spring decides when and how to instantiate and wire them. This inverts the dependency direction — high-level modules no longer depend on low-level modules directly.
Compare
When would you use Setter Injection over Constructor Injection?
▾
Use setter injection for optional dependencies — ones the bean can function without. Annotate with @Autowired(required = false). Also used when dealing with circular dependencies that can't be refactored (Spring handles them by first creating both objects, then calling the setter). However, always document which dependencies are optional to maintain clear contracts. Default to constructor injection for everything required.
◆ Memory Tip
💡
IoC = Vending Machine. DI = The Slot.
You don't go into the machine and manually connect components. You insert a coin (configure) and the machine (IoC container) delivers the product (injected dependency) to you. Constructor Injection = the machine delivers everything upfront. Setter Injection = add-ons delivered separately. Field Injection = sneaking parts directly into the machine's internals — messy and untestable.

🏗️
Lifecycle Hooks — Preferred: @PostConstruct (runs after DI, before use) and @PreDestroy (runs before bean removed from context). Alternatives: implement InitializingBean.afterPropertiesSet() / DisposableBean.destroy(), or use @Bean(initMethod="start", destroyMethod="stop").
📦
Singleton Scope — Default. One instance per ApplicationContext. Shared across all requests. Spring manages the full lifecycle. Not thread-safe by design — you must handle thread safety in your code.
🔄
Prototype Scope — New instance every time getBean() is called. Spring creates it but does NOT manage destruction — @PreDestroy is never called. Your responsibility to clean up.
🌐
Web Scopes — request (new per HTTP request), session (new per HTTP session), application (one per ServletContext, like singleton for web), websocket. Only available in web-aware ApplicationContext.
🔧
BeanPostProcessor — Intercepts ALL beans during initialization. postProcessBeforeInitialization() runs before @PostConstruct; postProcessAfterInitialization() runs after. This is how Spring creates AOP proxies — replaces bean with proxy at this stage.

BEAN LIFECYCLE + SCOPES
// ── Lifecycle hooks ───────────────────────────────
@Component
public class CacheService {

    @PostConstruct                // runs after injection, before use
    public void init() {
        // warm up cache, open connections
        System.out.println("Cache warming up...");
    }

    @PreDestroy                   // runs before context closes
    public void shutdown() {
        // close connections, flush to disk
        System.out.println("Cache shutting down...");
    }
}

// ── Prototype scope ───────────────────────────────
@Component
@Scope("prototype")           // new instance every getBean()
public class ReportBuilder { }

// ── Inject prototype into singleton via @Lookup ───
@Component
public abstract class ReportService {

    @Lookup                      // Spring overrides this method
    public abstract ReportBuilder getReportBuilder();

    public void generate() {
        ReportBuilder builder = getReportBuilder(); // fresh instance
    }
}

// ── Web scopes ────────────────────────────────────
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class RequestContext { }
// proxyMode needed when injecting short-lived scope into singleton

// ── @Bean with explicit lifecycle ─────────────────
@Bean(initMethod = "start", destroyMethod = "stop")
public DataSource dataSource() {
    return new HikariDataSource(config);
}
◆ Interview Questions
Must Know
What is the default bean scope in Spring and is it thread-safe?
▾
The default scope is singleton — one instance per ApplicationContext, shared across all threads. It is NOT inherently thread-safe. If your singleton bean holds mutable state, you must synchronize access manually or use ThreadLocal. This is why Spring services and repositories should be stateless by design. Thread-safety is your responsibility, not Spring's.
Tricky
Does Spring call @PreDestroy on prototype-scoped beans?
▾
No. Spring creates prototype beans but doesn't track them after handing them over. Because Spring doesn't hold a reference to prototype beans post-creation, it can't call @PreDestroy or destroy() on them. This is by design — prototype beans can have long or short lives, and Spring doesn't know when you're done with them. You are responsible for cleanup. If you need lifecycle management, use singleton scope with careful state management, or implement a custom factory.
⚡ Classic trick: Interviewer asks "Name a case where @PreDestroy is NOT called." Answer: prototype-scoped beans.
Tricky
What is BeanPostProcessor and how does it relate to AOP?
▾
BeanPostProcessor intercepts every bean before and after initialization. Spring's AOP auto-proxy creator (AnnotationAwareAspectJAutoProxyCreator) is itself a BeanPostProcessor. It detects beans with matching pointcuts and replaces them with CGLIB/JDK proxy objects in postProcessAfterInitialization(). This is why AOP doesn't work if you call methods on this inside a bean — you bypass the proxy.
Concept
What is the order of lifecycle callbacks: @PostConstruct, afterPropertiesSet(), and initMethod?
▾
Order: 1. @PostConstruct → 2. afterPropertiesSet() (InitializingBean) → 3. custom initMethod (from @Bean or XML). All run after properties are set by Spring. For destruction: 1. @PreDestroy → 2. destroy() (DisposableBean) → 3. custom destroyMethod. Recommendation: use @PostConstruct/@PreDestroy — they're JSR-250 standard annotations and don't couple your code to Spring interfaces.
Compare
When would you use request scope vs session scope?
▾
Request scope: New bean per HTTP request. Use for data specific to a single request — form data, request-level caching, per-request audit context. Destroyed after response is sent.

Session scope: New bean per HTTP session. Use for user-specific data that should persist across multiple requests — shopping cart, user preferences, authentication state. Destroyed when session expires.
⚡ Important: When injecting a request/session-scoped bean into a singleton, you MUST use proxyMode = ScopedProxyMode.TARGET_CLASS. Without this, Spring injects the proxy at startup (when no request exists) and the singleton holds a stale reference forever.
Tricky
What is the difference between ApplicationContext and a singleton? Can you have two instances of a singleton bean?
▾
Spring's singleton scope means one instance per ApplicationContext, NOT one instance per JVM. If you create two ApplicationContexts (e.g., in tests, or parent/child contexts in Spring MVC), you'll have two instances of the same singleton bean. This is different from the Gang-of-Four Singleton pattern. You can also call context.getBean() multiple times — you'll get the same instance. But in a multi-context scenario, they're different objects.
⚡ Senior-level nuance: Spring MVC has two contexts — root context (Services, Repos) and servlet context (Controllers). A singleton bean in the root context is not the same instance as a bean with the same name in the servlet context.
◆ Memory Tip
💡
Lifecycle = Birth → Setup → Work → Cleanup → Death.
@PostConstruct = "I'm ready for work" (connect to DB, warm cache).
@PreDestroy = "I'm clocking out" (close connections, flush state).

Scopes mnemonic — SPRAWS: Singleton, Prototype, Request, Application, Websocket, Session. Singleton = one for all. Prototype = one for each ask. The rest = web-only.

ApplicationContext & BeanFactory
The IoC container — BeanFactory is the engine, ApplicationContext is the full car
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory Tip
◆ Core Concepts
🏭
BeanFactory — The root interface (org.springframework.beans.factory.BeanFactory). Basic DI container. Uses lazy initialization — beans created only when first requested. Lightweight — suitable for resource-constrained environments. Does NOT support AOP, events, or i18n out of the box.
🏙️
ApplicationContext — Extends BeanFactory. Adds: eager initialization (all singleton beans on startup), event publishing (ApplicationEvent), i18n (MessageSource), AOP integration, environment abstraction. Use this in all real applications.
📁
Common Implementations — ClassPathXmlApplicationContext (XML from classpath), FileSystemXmlApplicationContext (XML from filesystem), AnnotationConfigApplicationContext (Java config, most common today), WebApplicationContext (Spring MVC web apps).
👨‍👩‍👧
Parent-Child Context — Spring MVC creates two contexts: Root context (loaded by ContextLoaderListener — Services, Repos, Config) and Servlet context (loaded by DispatcherServlet — Controllers, MVC beans). Child context can see parent beans; parent CANNOT see child beans.
🌱
Spring Boot Auto-context — SpringApplication.run() creates a AnnotationConfigServletWebServerApplicationContext automatically. You rarely create contexts manually in Spring Boot.

ApplicationContext ctx = new
    AnnotationConfigApplicationContext(AppConfig.class);

UserService svc = ctx.getBean(UserService.class);

// ── Multiple config classes ───────────────────────
ApplicationContext ctx = new
    AnnotationConfigApplicationContext(
        AppConfig.class, SecurityConfig.class, DbConfig.class

// ── Implement ApplicationContextAware ─────────────
@Component
public class AppContextHolder implements ApplicationContextAware {
    private static ApplicationContext context;

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        context = ctx;
    }

    public static <T> T getBean(Class<T> type) {
        return context.getBean(type); // utility for static access
    }
}

// ── Check bean existence ──────────────────────────
boolean exists = ctx.containsBean("myBean");
String[] names = ctx.getBeanNamesForType(UserService.class);

// ── Spring Boot — context is auto-created ─────────
@SpringBootApplication
public class MyApp {
    public static void main(String[] args) {
        ConfigurableApplicationContext ctx =
            SpringApplication.run(MyApp.class, args);
        // ctx is AnnotationConfigServletWebServerApplicationContext
    }
}
◆ Interview Questions
Must Know
What is the difference between BeanFactory and ApplicationContext?
▾
Both are IoC containers but ApplicationContext extends BeanFactory with additional enterprise features. BeanFactory: lazy bean initialization, no AOP, no events, no i18n, lightweight. ApplicationContext: eager initialization of singletons at startup (catches config errors early), automatic BeanPostProcessor detection, event publishing, i18n via MessageSource, environment abstraction. In practice, always use ApplicationContext. BeanFactory is mainly for embedded or resource-constrained systems.
Tricky
What is the advantage of eager initialization in ApplicationContext?
▾
Fail-fast at startup. Since all singleton beans are initialized when the ApplicationContext starts, configuration errors (missing beans, wrong types, failed @PostConstruct) are detected immediately — not when a user first hits that code path in production. With BeanFactory's lazy initialization, a misconfigured bean might go unnoticed until it's first requested at runtime, potentially hours or days after deployment.
⚡ Tip: You can selectively opt into lazy loading even with ApplicationContext using @Lazy on a bean or spring.main.lazy-initialization=true in Spring Boot (useful to speed up startup in dev).
Concept
In Spring MVC, why are there two ApplicationContexts (parent and child)?
▾
Spring MVC historically had two contexts for separation of concerns: the root context (created by ContextLoaderListener) holds shared infrastructure beans — services, repositories, datasources. The servlet context (created by DispatcherServlet) holds web-tier beans — controllers, view resolvers, handler mappings. This separation allows multiple DispatcherServlets to share the root context. The child (servlet) context can access parent beans; the parent cannot see child beans — this is why you sometimes can't inject a Controller into a Service (the parent can't see the child). Note: In Spring Boot, this distinction is largely invisible as Boot creates a single context.
Tricky
How would you get access to ApplicationContext inside a bean?
▾
Three ways:
1. @Autowired — inject ApplicationContext directly (Spring can inject itself)
2. Implement ApplicationContextAware — Spring calls setApplicationContext() after construction
3. @Autowired ApplicationContext — simplest in modern Spring

Use case: accessing beans dynamically at runtime (e.g., plugin system, strategy pattern where the bean name is determined at runtime).
⚡ Anti-pattern warning: Overusing ApplicationContext.getBean() is the "Service Locator" pattern — it hides dependencies and makes testing harder. Prefer DI.
◆ Memory Tip
💡
BeanFactory = Engine. ApplicationContext = Full Car.
You can drive with just the engine theoretically, but you need the full car (AC, GPS, radio = events, i18n, AOP) for real-world use. ApplicationContext IS-A BeanFactory — every ApplicationContext implements BeanFactory. Never use raw BeanFactory in production Spring apps.

Spring Config
XML vs. Java-based Configuration
Evolution of Spring configuration — from verbose XML to concise @Configuration
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory Tip
◆ Core Concepts
📄
XML Configuration — Original Spring approach (Spring 1.x). Beans defined in applicationContext.xml. Verbose but: externalized from code (can change without recompile), supports all Spring features, useful for legacy systems. Rarely used in new projects.
☕
Java-based Config (@Configuration) — Modern approach (Spring 3.x+). A class annotated @Configuration acts as the bean definition source. Methods annotated @Bean define beans. Benefits: type-safe, refactorable, IDE-friendly, can use Java logic in bean creation.
🔍
Component Scanning (@ComponentScan) — Auto-detects classes with @Component, @Service, @Repository, @Controller in specified packages. Eliminates manual bean registration. Enabled by @SpringBootApplication in Boot.
🔀
Mixing Both — You can import XML into Java config with @ImportResource("classpath:beans.xml") or import Java config classes with @Import(OtherConfig.class). Common during legacy migration.
🔧
@Configuration vs @Component — Both register the class as a bean. Key difference: @Configuration classes are CGLIB-proxied — calls between @Bean methods are intercepted to return the same bean instance. @Component (with @Bean methods) is NOT proxied — each call creates a new object (called "lite mode").
◆ Memorizable Code
XML vs JAVA CONFIG
// ══ XML (legacy) ══════════════════════════════════
// applicationContext.xml
// <bean id="userService" class="com.app.UserService">
//   <constructor-arg ref="userRepository"/>
// </bean>
// <bean id="userRepository" class="com.app.UserRepository"/>

// ══ Java Config (modern) ══════════════════════════
@Configuration
@ComponentScan("com.app")              // auto-detect @Component etc.
@PropertySource("classpath:app.properties")
public class AppConfig {

    @Bean                               // manually register bean
    public DataSource dataSource() {
        HikariDataSource ds = new HikariDataSource();
        ds.setJdbcUrl("jdbc:mysql://localhost/mydb");
        return ds;
    }

    @Bean
    public JdbcTemplate jdbcTemplate(DataSource ds) {
        return new JdbcTemplate(ds);    // Spring injects dataSource()
    }
}

// ── @Configuration CGLIB proxy magic ─────────────
@Configuration
public class Config {
    @Bean
    public A a() { return new A(); }

    @Bean
    public B b() {
        return new B(a()); // calls a() — Spring intercepts, returns
    }                        // the SAME singleton 'a' bean, not new A()
}

// ── Import between configs ────────────────────────
@Configuration
@Import({SecurityConfig.class, DbConfig.class})
@ImportResource("classpath:legacy-beans.xml") // import XML
public class RootConfig { }
◆ Interview Questions
Tricky
What's the difference between @Configuration and @Component when used with @Bean methods?
▾
@Configuration classes are enhanced by CGLIB — Spring subclasses them to intercept @Bean method calls. If b() calls a(), Spring returns the existing singleton A bean (not a new instance). This is called full mode. @Component with @Bean methods is lite mode — no CGLIB proxying. Each call to a() creates a brand new A instance. This is a subtle but important distinction — use @Configuration when beans depend on each other.
⚡ Watch out: If you annotate a config class with @Configuration(proxyBeanMethods = false) (default in Spring Boot auto-configs for performance), you get lite mode behavior.
Must Know
What does @ComponentScan do and what happens without it?
▾
@ComponentScan tells Spring which packages to scan for stereotype annotations (@Component, @Service, @Repository, @Controller, @RestController). Without it, Spring only registers beans explicitly defined via @Bean in @Configuration classes. In Spring Boot, @SpringBootApplication includes @ComponentScan that scans the package of the main class and all sub-packages. Beans in packages outside this tree won't be auto-detected.
Concept
Why would you still use XML configuration in a modern Spring project?
▾
Rare but valid cases: 1. Legacy codebases — gradual migration using @ImportResource. 2. External configuration — XML is externalized from compiled code; you can change bean wiring without recompiling (though @PropertySource + externalized properties handles most of this). 3. Third-party integration — some older frameworks or tools still generate or expect Spring XML config. In greenfield projects, always prefer Java config.
◆ Memory Tip
💡
XML = Blueprint on paper. Java Config = Blueprint in code.
Both describe the same building, but Java config is type-safe (no XML typos), refactorable, and IDE-autocompleted. The key trick: @Configuration CGLIB-proxies your class so that @Bean method calls are intercepted — you always get the singleton, not a new object. Think of it as Spring saying "I'll take over that method call."

Spring Core
Core Annotations: @Component, @Autowired, @Qualifier, @Value
The annotation vocabulary every Spring developer must know cold
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory Tip
◆ Core Concepts
🏷️
@Component — Generic stereotype. Marks a class as a Spring-managed bean. Specialized aliases: @Service (business logic layer — semantic only), @Repository (data layer — adds exception translation), @Controller (MVC controller), @RestController = @Controller + @ResponseBody.
💉
@Autowired — Triggers Spring DI. Resolution order: 1) by type → 2) by name (if multiple matches). Can be on constructor (preferred), setter, or field. required=false makes it optional (inject null if not found). In Spring 4.3+, a single-constructor class is auto-wired without the annotation.
🎯
@Qualifier — Disambiguates when multiple beans of the same type exist. Used alongside @Autowired. Specify the bean name: @Qualifier("mysqlDataSource"). You can also create custom qualifier annotations for cleaner code.
📋
@Value — Injects values from properties files, environment variables, or Spring Expression Language (SpEL). Syntax: @Value("${property.key}") for property placeholder, @Value("#{beanName.method()}") for SpEL expression. Default value: @Value("${key:defaultValue}").
🥇
@Primary — Marks one bean as the default when multiple beans of the same type exist. Lower precedence than @Qualifier — if both are present, @Qualifier wins.
StereotypeLayerExtra Behavior
@ComponentAnyGeneric — just registers bean
@ServiceBusinessSemantic only (same as @Component)
@RepositoryDataTranslates persistence exceptions to Spring's DataAccessException
@ControllerWeb MVCReturns view name, used with @RequestMapping
@RestControllerWeb REST= @Controller + @ResponseBody (returns data, not view)
◆ Memorizable Code
@COMPONENT · @AUTOWIRED · @QUALIFIER · @VALUE
// ── Stereotype annotations ────────────────────────
@Repository
public class UserRepository { }  // exception translation

@Service
public class UserService { }

@RestController                   // = @Controller + @ResponseBody
public class UserController { }

// ── @Autowired resolution + @Qualifier ───────────
public interface NotificationService { void send(String msg); }

@Service("emailNotifier")           // explicit bean name
public class EmailNotificationService implements NotificationService { }

@Service("smsNotifier")
public class SmsNotificationService implements NotificationService { }

@Service
public class AlertService {
    private final NotificationService notifier;

    public AlertService(@Qualifier("emailNotifier") NotificationService n) {
        this.notifier = n;  // Spring picks emailNotifier
    }
}

// ── @Primary — default when no qualifier ─────────
@Service
@Primary                           // default for NotificationService
public class EmailNotificationService implements NotificationService { }

// ── @Value ────────────────────────────────────────
@Service
public class AppSettings {
    @Value("${app.name}")                  // from application.properties
    private String appName;

    @Value("${app.max-retries:3}")         // with default value
    private int maxRetries;

    @Value("${app.feature.enabled:false}") // boolean
    private boolean featureEnabled;

    @Value("#{systemProperties['os.name']}") // SpEL expression
    private String osName;

    @Value("#{T(java.lang.Math).PI}")       // SpEL: static method
    private double pi;
}
◆ Interview Questions
Must Know
What is the difference between @Component, @Service, and @Repository?
▾
All three are specializations of @Component — they register the class as a Spring bean and are detected by component scanning. The differences are: @Service is purely semantic — communicates intent (business logic layer) but has no additional behavior. @Repository adds one concrete feature: it enables Spring's persistence exception translation — vendor-specific exceptions (like Hibernate's HibernateException) are translated to Spring's DataAccessException hierarchy, making your code database-agnostic. @Controller marks MVC controllers and enables handler mapping detection.
Tricky
What's the difference between @Autowired resolution by type vs by name?
▾
Spring first tries to match by type. If exactly one bean of that type exists, it's injected. If multiple beans exist, Spring tries to match by field/param name as the bean name. If that also fails, it throws NoUniqueBeanDefinitionException. Use @Qualifier to be explicit. Example: if you have EmailService emailService and two beans of type EmailService — one named "emailService" and one named "smsEmailService" — Spring picks "emailService" by name match on the field.
⚡ Tricky: Changing a field name from emailNotifier to notifier can break autowiring in code that relied on name-matching as the tiebreaker!
Tricky
What is the difference between @Value("${...}") and @Value("#{...}")?
▾
${...} is a property placeholder — resolves a key from a properties file, environment variable, or system property. Simple string substitution.
#{...} is Spring Expression Language (SpEL) — a full expression language that can reference other beans, call methods, evaluate math, access system properties, etc. Example: #{userService.getDefaultTimeout()} calls a method on a bean. You can also combine them: #{'${app.name}'.toUpperCase()}.
Compare
@Primary vs @Qualifier — which takes precedence?
▾
@Qualifier always wins over @Primary. @Primary is a default preference — used when no qualifier is specified and multiple beans of the same type exist. @Qualifier is an explicit selection — it overrides any primary designation. Think of it as: @Primary = "pick me if you don't have an opinion"; @Qualifier = "I specifically want this one."
Must Know
What happens if @Autowired finds no matching bean? Or too many?
▾
No matching bean: throws NoSuchBeanDefinitionException at startup. Prevent with @Autowired(required=false) — Spring injects null instead. Or use Optional<T> as the parameter type — Spring injects an empty Optional if no bean found.
Multiple matching beans: throws NoUniqueBeanDefinitionException. Fix with @Qualifier, @Primary, or rename the field/param to match the desired bean name. You can also inject List<T> or Map<String,T> to get all matching beans.
◆ Memory Tip
💡
Autowiring resolution: Type → Name → Qualifier → Error.
Spring is like a smart recruiter: first looks for candidates with the right skills (type). If many qualify, picks by matching job title (field name). If still ambiguous, you need a specific reference letter (@Qualifier). @Primary = preferred candidate when no specific request.

Stereotype layers: Controller (door) → Service (worker) → Repository (vault). @Repository is the only one with extra power — it translates vendor exceptions.

◆ AOP Vocabulary — Must Know All 5
📍
Joinpoint — A specific point in program execution where AOP can plug in. In Spring AOP, joinpoints are always method executions. Spring AOP does NOT support field-level or constructor-level joinpoints (AspectJ does).
🔍
Pointcut — An expression that selects which joinpoints to intercept. Uses AspectJ pointcut expression language. Example: execution(* com.app.service.*.*(..)) — all methods in all service classes. Pointcut = filter on joinpoints.
💊
Advice — The action to take at a matched joinpoint. Types: @Before (before method), @After (after method, always), @AfterReturning (after successful return), @AfterThrowing (after exception), @Around (wraps the method — most powerful, must call proceed()).
🏗️
Aspect — A class that contains Pointcuts + Advice. Annotated with @Aspect. It's the modularization of a cross-cutting concern (e.g., LoggingAspect, SecurityAspect).
🎭
Weaving — The process of linking aspects with the application code. Spring AOP uses runtime proxy weaving (at bean creation via BeanPostProcessor). AspectJ also supports compile-time and load-time weaving for finer control.
🛡️
Spring AOP vs AspectJ — Spring AOP: proxy-based, only method execution joinpoints, simpler, sufficient for 95% of use cases. AspectJ: bytecode manipulation, all joinpoints (fields, constructors, static methods), more powerful, more complex setup.
◆ Memorizable Code
AOP CONCEPTS — POINTCUT EXPRESSIONS
// ── Pointcut Expression Patterns ─────────────────
// execution([modifier] return-type [class].method(params) [throws])

"execution(* com.app.service.*.*(..))"
// Any method, any return type, any class in service pkg, any args

"execution(public String com.app.UserService.find*(String))"
// Public methods starting with "find", returning String, one String arg

"within(com.app.service..*)"
// All methods in service package and sub-packages

"@annotation(org.springframework.transaction.annotation.Transactional)"
// Methods annotated with @Transactional

"@within(org.springframework.stereotype.Service)"
// All methods in classes annotated with @Service

"bean(*Service)"                 // Spring AOP only: beans named *Service

// ── Combining pointcuts ───────────────────────────
"execution(* com.app..*(..)) && !execution(* com.app..*get*(..))'"
// All methods in com.app but NOT getters

// ── Advice types summary ──────────────────────────
//  @Before        → runs BEFORE method
//  @After         → runs AFTER method (like finally)
//  @AfterReturning→ runs only on SUCCESS (can access return value)
//  @AfterThrowing → runs only on EXCEPTION (can access exception)
//  @Around        → wraps method; MUST call joinPoint.proceed()
◆ Interview Questions
Must Know
Explain the difference between a Joinpoint and a Pointcut.
▾
A Joinpoint is any specific moment during execution where advice could be applied — in Spring AOP, that's every method call. A Pointcut is the filter/selector that says "out of all possible joinpoints, apply advice to these specific ones." Analogy: all intersections in a city are joinpoints; a Pointcut is the traffic camera placement rule (e.g., "install cameras at all intersections on Main Street").
Tricky
Why doesn't AOP work when you call a method on 'this' within the same bean?
▾
Spring AOP works by wrapping your bean in a proxy. When external code calls your bean, it goes through the proxy (which applies advice). But when your method calls another method on this, it's a direct Java call — it bypasses the proxy entirely. So @Transactional, @Cacheable, and custom aspects won't apply to self-invocations. Fixes: inject the bean into itself (@Lazy @Autowired MyBean self), use AopContext.currentProxy(), or use AspectJ load-time weaving.
⚡ This is the #1 Spring AOP gotcha in interviews and production bugs.
Compare
What is the difference between Spring AOP and AspectJ?
▾
Spring AOP: Runtime proxy-based, only intercepts method executions on Spring-managed beans, simpler setup (just add spring-aspects dependency), no compile step needed. Limited to public methods on proxy-eligible beans.
AspectJ: Full AOP framework, can intercept constructors, field access, static methods, private methods — any joinpoint. Requires compile-time weaving (ajc compiler) or load-time weaving (Java agent). Spring can use AspectJ annotations with its own proxy mechanism or full AspectJ weaving.
Tricky
What are the limitations of Spring AOP? When does it NOT work?
▾
Spring AOP does NOT work for: 1. Self-invocation (calling methods on this). 2. Private methods (proxies can't override private methods). 3. Final classes/methods (CGLIB proxying requires subclassing). 4. Non-Spring beans (objects created with new are not proxied). 5. Static methods (not intercepted by proxy). 6. Inner classes in some cases. If you need any of these, use AspectJ weaving.
◆ Memory Tip
💡
AOP in one sentence: "Intercept method calls to add behavior without changing the class."

Mnemonic — JWAPA: Joinpoint (where it could happen) → Pointcut (where it WILL happen) → Advice (what to do) → Aspect (the class holding it all) → Weaving (how it's applied).

Advice order to remember: Around wraps everything. Before runs first. After runs last (always). AfterReturning/Throwing run in the middle based on outcome.

Spring AOP
AOP Annotations: @Aspect, @Before, @After, @Around
Writing real aspects — logging, timing, and auditing in practice
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory Tip
◆ Advice Types Deep Dive
▶️
@Before — Runs before the method. Cannot prevent method execution (use @Around for that). Can access method args via JoinPoint. Common uses: input validation, logging, security checks.
⏹️
@After — Runs after method completes — always, whether it returned normally or threw an exception. Like a finally block. Cannot access return value.
✅
@AfterReturning — Runs only when method returns successfully. Can access and inspect the return value via returning attribute. Cannot modify the return value (use @Around for that).
💥
@AfterThrowing — Runs only when method throws an exception. Can access the exception via throwing attribute. Cannot suppress the exception (use @Around for that). Common use: error logging.
🔄
@Around — Most powerful. Wraps the entire method. Must call joinPoint.proceed() to execute the actual method (or not — to completely prevent execution). Can modify args, return value, and catch/suppress exceptions. Used for: caching, transactions, retries, timing.
◆ Memorizable Code
FULL ASPECT EXAMPLE — LOGGING + TIMING + AUDIT
@Aspect
@Component                         // must be a Spring bean!
public class LoggingAspect {

    // ── Reusable pointcut ─────────────────────────
    @Pointcut("execution(* com.app.service.*.*(..))")
    public void serviceLayer() {}    // empty method, just a handle

    // ── @Before ───────────────────────────────────
    @Before("serviceLayer()")
    public void logBefore(JoinPoint jp) {
        String method = jp.getSignature().getName();
        Object[] args = jp.getArgs();
        log.info("Calling: {} with args: {}", method, args);
    }

    // ── @AfterReturning — access return value ─────
    @AfterReturning(pointcut = "serviceLayer()", returning = "result")
    public void logReturn(JoinPoint jp, Object result) {
        log.info("Returned: {}", result);   // can inspect, NOT modify
    }

    // ── @AfterThrowing — access exception ─────────
    @AfterThrowing(pointcut = "serviceLayer()", throwing = "ex")
    public void logError(JoinPoint jp, Exception ex) {
        log.error("Exception in {}: {}", jp.getSignature(), ex.getMessage());
    }

    // ── @Around — timing + modify return + retry ──
    @Around("serviceLayer()")
    public Object timeExecution(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed();       // ← MUST call this
            long elapsed = System.currentTimeMillis() - start;
            log.info("{} took {}ms", pjp.getSignature(), elapsed);
            return result;                        // can modify return value
        } catch (Exception e) {
            log.error("Failed after {}ms", System.currentTimeMillis() - start);
            throw e;                              // re-throw or suppress
        }
    }

    // ── Custom annotation-based pointcut ──────────
    @Around("@annotation(com.app.annotation.Audited)")
    public Object auditMethod(ProceedingJoinPoint pjp) throws Throwable {
        saveAuditLog(pjp.getSignature(), pjp.getArgs());
        return pjp.proceed();
    }
}
◆ Interview Questions
Must Know
What happens if you forget to call proceed() in an @Around advice?
▾
The actual method is never executed. The @Around advice completely replaces the method. If you return null without calling proceed(), the caller gets null (or default value for primitives). This is intentional in some cases — like a caching aspect that returns a cached value without calling the real method. But forgetting it accidentally is a common bug that can silently break functionality with no exception thrown.
Compare
When should you use @Around vs @Before + @AfterReturning?
▾
Use @Around when you need to: modify the return value, modify method arguments, conditionally skip method execution, implement retry logic, catch and handle exceptions, measure timing (need both before and after in same method). Use @Before/@After when you only need to observe (logging, auditing) without modifying behavior — simpler and less error-prone since you can't accidentally prevent method execution.
Tricky
If multiple aspects apply to the same method, what is the execution order?
▾
If multiple aspects apply, order is undefined by default. Control it with @Order(n) on the aspect class — lower value = higher priority. High priority aspects run their @Before first, their @After last (like nested try-finally blocks). Example: @Order(1) SecurityAspect wraps @Order(2) LoggingAspect. Security runs @Before first, then Logging @Before, then the method, then Logging @After, then Security @After.
⚡ Think of it like Russian nesting dolls — outer aspect wraps inner aspects.
◆ Memory Tip
💡
Advice types = different moments in a restaurant visit:
@Before = waiter checks reservation before seating you.
@After = staff clears the table (always, whether you liked the food or not).
@AfterReturning = chef asks for feedback only if the meal was good.
@AfterThrowing = manager apologizes only if something went wrong.
@Around = the entire maître d' experience — controls everything from arrival to departure.

◆ Core Concepts
🧹
What JdbcTemplate solves — Raw JDBC requires: open connection → create PreparedStatement → execute → iterate ResultSet → handle exceptions → close ResultSet → close Statement → close Connection. JdbcTemplate handles ALL resource management and exception handling automatically via the Template Method pattern.
🗺️
RowMapper<T> — Maps each ResultSet row to an object. Implement mapRow(ResultSet rs, int rowNum). Use BeanPropertyRowMapper.newInstance(User.class) for automatic mapping when column names match field names.
📊
Key Methods — queryForObject(sql, type, args) — expects exactly 1 row. query(sql, rowMapper, args) — returns List. queryForList(sql, type) — returns List of scalar values. update(sql, args) — INSERT/UPDATE/DELETE, returns rows affected. batchUpdate(sql, batchArgs) — bulk operations.
⚠️
Exception Translation — JdbcTemplate converts checked SQLExceptions into unchecked DataAccessExceptions. No more try-catch-SQLException boilerplate. The exception hierarchy is database-agnostic (e.g., DuplicateKeyException, DataIntegrityViolationException).
🔄
NamedParameterJdbcTemplate — Extension that uses named params (:name) instead of ? positional params. More readable for complex queries. Pass params via MapSqlParameterSource or SqlParameterSource.
◆ Memorizable Code
JDBC TEMPLATE — CRUD PATTERNS
@Repository
public class UserRepository {
    private final JdbcTemplate jdbc;

    public UserRepository(JdbcTemplate jdbc) { this.jdbc = jdbc; }

    // ── SELECT single object ──────────────────────
    public User findById(long id) {
        return jdbc.queryForObject(
            "SELECT * FROM users WHERE id = ?",
            new BeanPropertyRowMapper<>(User.class),
            id
        );  // throws EmptyResultDataAccessException if not found
    }

    // ── SELECT list ───────────────────────────────
    public List<User> findAll() {
        return jdbc.query(
            "SELECT * FROM users",
            (rs, rowNum) -> new User(   // inline RowMapper lambda
                rs.getLong("id"),
                rs.getString("name"),
                rs.getString("email")
            )
        );
    }

    // ── SELECT scalar value ───────────────────────
    public int countUsers() {
        return jdbc.queryForObject("SELECT COUNT(*) FROM users", Integer.class);
    }

    // ── INSERT / UPDATE / DELETE ──────────────────
    public int save(User user) {
        return jdbc.update(
            "INSERT INTO users (name, email) VALUES (?, ?)",
            user.getName(), user.getEmail()
        );  // returns rows affected
    }

    // ── Named params (NamedParameterJdbcTemplate) ─
    @Autowired
    private NamedParameterJdbcTemplate namedJdbc;

    public User findByEmail(String email) {
        MapSqlParameterSource params = new MapSqlParameterSource()
            .addValue("email", email);
        return namedJdbc.queryForObject(
            "SELECT * FROM users WHERE email = :email",
            params, new BeanPropertyRowMapper<>(User.class)
        );
    }

    // ── Batch insert ──────────────────────────────
    public int[] batchInsert(List<User> users) {
        return jdbc.batchUpdate(
            "INSERT INTO users (name, email) VALUES (?, ?)",
            users, users.size(),
            (ps, user) -> {
                ps.setString(1, user.getName());
                ps.setString(2, user.getEmail());
            }
        );
    }
}
◆ Interview Questions
Must Know
What does JdbcTemplate handle automatically that raw JDBC doesn't?
▾
JdbcTemplate eliminates: 1. Opening/closing connections (returns to pool). 2. Creating and closing Statements/PreparedStatements. 3. Iterating ResultSets. 4. Exception handling — converts checked SQLException to unchecked DataAccessException. 5. Transaction management integration. You only provide the SQL and the RowMapper — JdbcTemplate handles the rest via the Template Method pattern.
Tricky
What exception does queryForObject throw when no row is found? How do you handle it?
▾
queryForObject throws EmptyResultDataAccessException (subclass of DataAccessException) when no rows are returned, and IncorrectResultSizeDataAccessException when more than one row is returned. Best practice: wrap in a try-catch or use a query() + stream().findFirst() approach to return Optional: jdbc.query(sql, rowMapper, id).stream().findFirst() — returns Optional<User> cleanly.
⚡ Senior tip: Never use queryForObject when 0 results is a valid outcome — it will throw, not return null.
Compare
RowMapper vs ResultSetExtractor — when to use which?
▾
RowMapper<T>: Maps one ResultSet row to one object. Called per row. Spring handles the iteration. Use for standard queries returning a list of objects.
ResultSetExtractor<T>: Gives you the entire ResultSet — you handle iteration yourself. Use when: one result set should produce a complex object (e.g., parent + children), you need to process rows in groups, or you need special traversal logic. More flexible but more code.

◆ Core Concepts
📐
JPA vs Hibernate — JPA (Jakarta Persistence API) is the specification — a set of interfaces and annotations (@Entity, @Table, EntityManager). Hibernate is the most popular implementation of JPA. Spring Data JPA wraps JPA/Hibernate with repository abstractions. You program to JPA interfaces; Hibernate does the work.
🔑
EntityManager — JPA's core API. Manages the persistence context (first-level cache). Operations: persist() (insert), find() (select by PK), merge() (update detached entity), remove() (delete), createQuery() (JPQL). Spring manages the EntityManager lifecycle — inject with @PersistenceContext.
💾
Entity States — 4 states: Transient (new, not managed), Managed/Persistent (in persistence context, changes auto-synced to DB), Detached (was managed, context closed — changes NOT tracked), Removed (marked for deletion). Understanding state transitions is critical.
⚡
N+1 Problem — When fetching a list of N entities, and for each entity a separate query fires for a related entity. Results in N+1 total queries. Solve with: JOIN FETCH in JPQL, @EntityGraph, or FetchType.EAGER on the relationship (last resort).
🔄
FetchType — LAZY (default for collections): related data loaded only when accessed. EAGER: related data loaded immediately with the parent. Prefer LAZY — it avoids loading unnecessary data. But: accessing lazy data outside a transaction throws LazyInitializationException.

JPA ENTITY + ENTITY MANAGER + N+1 FIX
// ── JPA Entity ────────────────────────────────────
@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true)
    private String email;

    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY,
               cascade = CascadeType.ALL)
    private List<Order> orders;   // LAZY — not loaded until accessed
}

// ── EntityManager usage ───────────────────────────
@Repository
public class UserDao {
    @PersistenceContext                  // Spring injects scoped EM proxy
    private EntityManager em;

    public void save(User user) {
        em.persist(user);               // Transient → Managed
    }

    public User update(User detachedUser) {
        return em.merge(detachedUser);  // Detached → Managed (returns new)
    }

    public void delete(long id) {
        User u = em.find(User.class, id);
        em.remove(u);                    // Managed → Removed
    }
}

// ── N+1 Problem & Fix ─────────────────────────────
// ❌ BAD: N+1 — 1 query for users + N queries for orders
List<User> users = em.createQuery("SELECT u FROM User u", User.class).getResultList();
users.forEach(u -> u.getOrders().size()); // each triggers a query!

// ✅ FIX 1: JOIN FETCH in JPQL
List<User> users = em.createQuery(
    "SELECT DISTINCT u FROM User u LEFT JOIN FETCH u.orders", User.class
).getResultList();                       // 1 query, all data fetched

// ✅ FIX 2: @EntityGraph (Spring Data JPA)
@EntityGraph(attributePaths = {"orders"})
List<User> findAll();                     // eager-loads orders for this query
◆ Interview Questions
Must Know
What is the difference between persist() and merge() in JPA?
▾
persist(): Takes a transient (new, no ID) entity and makes it managed. The entity must not already have an ID. Modifying the object after persist() still updates it (within the same transaction).
merge(): Takes a detached entity (has ID, not tracked by EM) and returns a new managed copy. The original detached object is NOT managed after merge(). Always use the returned object: User managed = em.merge(detached).
⚡ Common mistake: Using the original detached object after merge() instead of the returned managed copy — changes to the original are lost.
Tricky
What is LazyInitializationException and how do you fix it?
▾
LazyInitializationException occurs when you access a LAZY-loaded association outside of an active persistence context (after the transaction/session closes). Common scenario: loading a User in the service layer, closing the transaction, then accessing user.getOrders() in the controller. Fixes:
1. JOIN FETCH or @EntityGraph — load eagerly for that specific query
2. DTO projection — fetch only what you need, no lazy associations
3. Open Session in View — keep session open per request (anti-pattern in most cases)
4. @Transactional on the calling method — extends transaction scope
⚡ Best practice: Use DTOs at service layer boundaries — never expose entities to presentation layer. Eliminates LazyInitializationException entirely.
Must Know
Explain the N+1 problem and its solutions.
▾
When you load N entities and each entity's lazy collection triggers a separate query. Example: 100 users × 1 query each for their orders = 101 total queries. Solutions in order of preference:
1. JOIN FETCH in JPQL — single query with a join
2. @EntityGraph — define fetch plan on the repository method
3. Batch fetching — @BatchSize(size=20) on the collection — fetches in groups
4. DTO projection — select only needed columns, no association loading
5. FetchType.EAGER — last resort, loads everywhere even when not needed
Tricky
What is the first-level cache in Hibernate and when can it cause bugs?
▾
The first-level cache is the persistence context (EntityManager/Session). Within one transaction, if you call em.find(User.class, 1L) twice, Hibernate returns the cached object — no second DB query. This is usually beneficial but causes bugs when: 1. Bulk updates via JPQL (bypasses the cache — entities in cache are stale). 2. Expecting the DB to have changed from another session. Fix: call em.refresh(entity) or em.clear() to invalidate the cache.
⚡ The second-level cache (L2) is optional and shared across sessions — configured with providers like Ehcache, Redis. Unlike L1, L2 is not enabled by default.
◆ Memory Tip
💡
JPA = contract. Hibernate = implementation. Spring Data JPA = convenience layer on top.

Entity states mnemonic — TMDR: Transient (born) → Managed (working) → Detached (retired) → Removed (fired).

N+1 is your production enemy. Always think: "Am I loading a collection in a loop?" If yes, use JOIN FETCH or @EntityGraph.

Spring TX
Transactions: Programmatic vs Declarative
ACID guarantees in Spring — @Transactional is the most important annotation to understand deeply
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory Tip
◆ Core Concepts
📝
Programmatic Transactions — Manually manage transactions using TransactionTemplate or PlatformTransactionManager. Fine-grained control (begin, commit, rollback inline in code). More verbose. Use when transaction boundaries are dynamic or conditional.
✨
Declarative Transactions (@Transactional) — Preferred. Annotate a method or class with @Transactional — Spring wraps it in a proxy that begins the transaction before the method and commits/rolls back after. No transaction code in your business logic. Works via AOP.
🔄
Propagation — Defines behavior when a transactional method calls another. Key ones: REQUIRED (default — join existing or create new), REQUIRES_NEW (always new transaction, suspends existing), NESTED (savepoint within existing), SUPPORTS (join if exists, non-transactional if not), MANDATORY (must have existing, else exception), NEVER (must not have existing).
🔐
Isolation Levels — Controls visibility of concurrent transactions. READ_UNCOMMITTED (dirty reads possible), READ_COMMITTED (default in most DBs), REPEATABLE_READ, SERIALIZABLE (strictest — no concurrency issues but lowest throughput). Each level prevents certain concurrency problems: dirty read, non-repeatable read, phantom read.
⚠️
Rollback Rules — By default, Spring rolls back only on unchecked exceptions (RuntimeException and Error). Checked exceptions do NOT trigger rollback by default. Override with rollbackFor = Exception.class or noRollbackFor = SpecificException.class.
🎭
Proxy Limitation — Same self-invocation problem as AOP — calling a @Transactional method from within the same class bypasses the proxy. The called method runs in the same transaction as the caller, ignoring its own @Transactional settings.

DECLARATIVE + PROGRAMMATIC TRANSACTIONS
// ── Declarative: @Transactional (preferred) ───────
@Service
public class OrderService {

    @Transactional                       // REQUIRED by default
    public void placeOrder(Order order) {
        orderRepo.save(order);
        paymentService.charge(order);   // if this throws → rollback
        inventoryService.deduct(order);
    }

    // Rollback for checked exception too
    @Transactional(rollbackFor = Exception.class)
    public void importData(File file) throws IOException { }

    // Read-only optimization (tells DB + JPA to skip dirty-checking)
    @Transactional(readOnly = true)
    public List<Order> getAllOrders() {
        return orderRepo.findAll();
    }

    // REQUIRES_NEW: separate transaction, commits independently
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logAudit(String action) {
        auditRepo.save(new AuditLog(action));
        // commits even if parent TX rolls back!
    }

    // Custom isolation
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void criticalOperation() { }
}

// ── Programmatic: TransactionTemplate ────────────
@Service
public class BatchService {
    private final TransactionTemplate txTemplate;

    public BatchService(PlatformTransactionManager txMgr) {
        this.txTemplate = new TransactionTemplate(txMgr);
        txTemplate.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
    }

    public String runBatch() {
        return txTemplate.execute(status -> {
            try {
                doWork();
                return "success";
            } catch (Exception e) {
                status.setRollbackOnly(); // programmatic rollback
                return "failed";
            }
        });
    }
}
◆ Interview Questions
Must Know
By default, which exceptions trigger @Transactional rollback?
▾
By default, Spring rolls back on unchecked exceptions (subclasses of RuntimeException) and Error. Checked exceptions do NOT trigger rollback — the transaction commits even if a checked exception propagates out. Override with @Transactional(rollbackFor = Exception.class) to include checked exceptions, or noRollbackFor = SpecificRuntimeException.class to exclude specific unchecked ones.
⚡ Classic interview question: "If a method throws IOException, does @Transactional roll back?" Answer: No, unless you specify rollbackFor = IOException.class.
Tricky
What is the difference between REQUIRES_NEW and NESTED propagation?
▾
REQUIRES_NEW: Creates a completely independent transaction. The outer transaction is suspended. If the inner commits and outer rolls back, the inner's work is preserved. Use for audit logging that must succeed regardless of business logic.

NESTED: Creates a savepoint within the existing transaction. If inner rolls back, it only reverts to the savepoint — the outer transaction can still commit. If outer rolls back, everything including the inner is rolled back. More efficient (no separate DB connection), but not all DBs/drivers support savepoints.
Tricky
@Transactional on a private method — does it work?
▾
No. @Transactional works via AOP proxy. Spring's proxy can only intercept public methods (for CGLIB proxies) or methods on the interface (for JDK proxies). Putting @Transactional on a private method is silently ignored — no error, no transaction. Also, calling a @Transactional method from within the same class (self-invocation) bypasses the proxy. Always put @Transactional on public service methods and call them from different beans.
⚡ This is one of the most common production bugs with Spring transactions. "Why isn't my transaction working?" — check for private method or self-invocation.
Concept
What does readOnly=true do in @Transactional?
▾
It's a hint to the underlying infrastructure that no modifications will be made. Effects: 1. JPA — skips dirty checking at flush time (performance gain for large object graphs). 2. Database — some DBs optimize read-only transactions (no locking, read from replica). 3. Spring Data — automatically adds readOnly=true on query methods. It does NOT prevent you from writing — it's just a hint, not enforcement. Use on all read-only service methods.

Must Know
Where should @Transactional be placed — on the interface or implementation?
▾
Always on the implementation class or method, not the interface. Spring recommends this because: if you use CGLIB proxying (the default), annotations on the interface are not inherited by the proxy. Spring's documentation explicitly says: "do not annotate interfaces — only concrete classes." Putting @Transactional on an interface method technically works with JDK dynamic proxies, but CGLIB ignores interface annotations. Since Spring Boot defaults to CGLIB proxies, always annotate the implementation.
◆ Memory Tip
💡
@Transactional = "All or nothing" enforced by Spring's proxy bouncer.

The proxy starts a transaction, lets your method run, then either commits (method returned normally) or rolls back (RuntimeException thrown). Remember the 3 traps:
1. Self-invocation = bypasses proxy = no transaction
2. Private method = proxy can't intercept = no transaction
3. Checked exception = no rollback by default
