◆ Must-Know Patterns
🪟
Sliding Window — For subarray/substring problems with a constraint. Move right pointer to expand, left pointer to shrink when constraint violated. Time: O(n). Use when: "max subarray sum with constraint", "smallest window containing all chars", "longest substring without repeat".
👆👆
Two Pointers — Left + right pointers on sorted array, move toward each other. Use when: pair sum = target, remove duplicates in-place, container with most water. Time: O(n). Requires sorted or specific structure.
🔍
Binary Search on Answer — When answer is a range and you can verify it. "Find minimum X such that condition holds." while(lo < hi) { mid = lo+(hi-lo)/2; if valid(mid) hi=mid; else lo=mid+1; } Pattern applies to: capacity allocation, kth smallest, search in rotated array.
➕
Prefix Sum — Precompute cumulative sums. Query any subarray sum in O(1). prefix[i] = prefix[i-1] + arr[i]. Subarray [l,r] = prefix[r] - prefix[l-1]. Extended: 2D prefix sums, prefix XOR for XOR range queries.
🗺️
HashMap Frequency — Count occurrences, find missing/duplicate, track first occurrence. Most "find the pair/triplet" problems reduce to: iterate + look up complement in map. Time: O(n), Space: O(n).
🔄
Kadane's Algorithm — Maximum subarray sum. maxEndingHere = max(num, maxEndingHere + num). Track global max. O(n), O(1) space. Extend for: max product subarray (track min too), circular subarray.

// ── Sliding Window: longest substr no repeat ──────
int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> map = new HashMap<>();
    int max = 0, left = 0;
    for (int r = 0; r < s.length(); r++) {
        char c = s.charAt(r);
        if (map.containsKey(c)) left = Math.max(left, map.get(c) + 1);
        map.put(c, r);
        max = Math.max(max, r - left + 1);
    }
    return max;
}

// ── Two Pointers: pair sum in sorted array ────────
int[] twoSum(int[] arr, int target) {
    int l = 0, r = arr.length - 1;
    while (l < r) {
        int sum = arr[l] + arr[r];
        if      (sum == target) return new int[]{l, r};
        else if (sum < target)  l++;
        else                    r--;
    }
    return new int[]{-1, -1};
}

// ── Prefix Sum ────────────────────────────────────
int[] buildPrefix(int[] arr) {
    int[] pre = new int[arr.length + 1];
    for (int i = 0; i < arr.length; i++) pre[i+1] = pre[i] + arr[i];
    return pre;
}
int rangeSum(int[] pre, int l, int r) { return pre[r+1] - pre[l]; }

// ── Kadane's Algorithm: max subarray ─────────────
int maxSubarray(int[] nums) {
    int cur = nums[0], best = nums[0];
    for (int i = 1; i < nums.length; i++) {
        cur  = Math.max(nums[i], cur + nums[i]);
        best = Math.max(best, cur);
    }
    return best;
}

// ── Binary Search template ────────────────────────
int binarySearch(int[] arr, int target) {
    int lo = 0, hi = arr.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2; // avoids overflow vs (lo+hi)/2
        if      (arr[mid] == target) return mid;
        else if (arr[mid] <  target) lo = mid + 1;
        else                          hi = mid - 1;
    }
    return -1;
}
◆ Interview Questions
Must Know
Why use mid = lo + (hi - lo) / 2 instead of (lo + hi) / 2?
▾
(lo + hi) can overflow a 32-bit integer when both are large (e.g., both near Integer.MAX_VALUE). lo + (hi - lo) / 2 avoids this because hi - lo is always smaller than hi. This is a subtle but critical bug that caused a famous Java binary search bug that existed for nearly a decade in the JDK itself.
⚡ Always use the safe form in interviews — it signals you understand overflow.
Tricky
How do you find the duplicate in an array of n+1 integers where values are 1 to n, without modifying the array and O(1) space?
▾
Use Floyd's Cycle Detection (Tortoise & Hare) — treat the array as a linked list where each value points to its index. A duplicate creates a cycle. Two-phase: 1) find cycle intersection with slow/fast pointers. 2) Reset slow to start, move both at speed 1 — they meet at the duplicate. Time O(n), Space O(1).
⚡ Classic hard problem — but knowing Floyd's = instant senior signal.
Must Know
What is the sliding window pattern and when do you use it?
▾
Sliding window maintains a contiguous subarray/substring that satisfies a constraint. Expand window by moving right pointer; shrink by moving left pointer when constraint is violated. Two variants: Fixed window (size k — both pointers move together) and Variable window (size varies — shrink until valid). Use for: max/min subarray with constraint, longest/shortest substring patterns. Time O(n) — each element enters and exits window at most once.
Tricky
How to rotate an array by k positions in O(n) time and O(1) space?
▾
Three-reversal trick: 1) Reverse entire array. 2) Reverse first k elements. 3) Reverse remaining n-k elements. Example: [1,2,3,4,5], k=2 → reverse all: [5,4,3,2,1] → reverse first 2: [4,5,3,2,1] → reverse last 3: [4,5,1,2,3]. This works because rotation = rearranging two halves.
⚡ Remember: k = k % n to handle k > n edge case.
Concept
Explain Kadane's algorithm and its extension to maximum product subarray.
▾
Kadane's max sum: at each index, either start fresh (nums[i]) or extend previous (prev + nums[i]). For max product, negatives flip sign — a minimum can become maximum. Track both maxProd and minProd at each step: newMax = max(nums[i], nums[i]*maxProd, nums[i]*minProd). The triple max accounts for: start fresh, extend positive, flip negative minimum.

💡

• "Subarray with constraint" → Sliding Window
• "Pair sum / in-place modify" → Two Pointers (needs sorted or structure)
• "Range sum query" → Prefix Sum
• "Max/Min subarray sum" → Kadane's
• "Find in sorted / monotonic" → Binary Search

// ── Merge Sort ────────────────────────────────────
void mergeSort(int[] arr, int l, int r) {
    if (l >= r) return;
    int mid = l + (r - l) / 2;
    mergeSort(arr, l, mid);
    mergeSort(arr, mid + 1, r);
    merge(arr, l, mid, r);
}
void merge(int[] arr, int l, int mid, int r) {
    int[] left  = Arrays.copyOfRange(arr, l, mid+1);
    int[] right = Arrays.copyOfRange(arr, mid+1, r+1);
    int i = 0, j = 0, k = l;
    while (i < left.length && j < right.length)
        arr[k++] = (left[i] <= right[j]) ? left[i++] : right[j++];
    while (i < left.length)  arr[k++] = left[i++];
    while (j < right.length) arr[k++] = right[j++];
}

// ── Quick Sort ────────────────────────────────────
void quickSort(int[] arr, int lo, int hi) {
    if (lo < hi) {
        int pi = partition(arr, lo, hi);
        quickSort(arr, lo, pi - 1);
        quickSort(arr, pi + 1, hi);
    }
}
int partition(int[] arr, int lo, int hi) {
    int pivot = arr[hi], i = lo - 1;
    for (int j = lo; j < hi; j++)
        if (arr[j] <= pivot) { i++; int tmp=arr[i]; arr[i]=arr[j]; arr[j]=tmp; }
    int tmp=arr[i+1]; arr[i+1]=arr[hi]; arr[hi]=tmp;
    return i + 1;
}

// ── Counting Sort (for small integer range) ───────
int[] countingSort(int[] arr, int max) {
    int[] count = new int[max + 1];
    for (int n : arr) count[n]++;
    int idx = 0;
    for (int i = 0; i <= max; i++)
        while (count[i]-- > 0) arr[idx++] = i;
    return arr;
}
◆ Interview Questions
Must Know
What sorting algorithm does Java use internally and why?
▾
Arrays.sort() uses two different algorithms: Dual-Pivot Quicksort for primitive arrays (int[], long[], etc.) — faster in practice due to better cache behavior, not stable. TimSort for Object arrays — a hybrid of merge sort and insertion sort, stable, guaranteed O(n log n). TimSort exploits natural "runs" in real-world data for superior performance on partially-sorted data.
⚡ Why two? Primitives don't need stability (no identity distinction). Objects may need stable sort to preserve equal elements' relative order.
Tricky
Why is QuickSort generally faster than MergeSort despite the same O(n log n) average?
▾
QuickSort is typically faster in practice for three reasons: 1. Cache locality — partitioning operates on contiguous memory; merge sort's auxiliary array causes cache misses. 2. No extra space — in-place vs O(n) auxiliary for merge. 3. Constant factor — partition is simpler per step. However, QuickSort has O(n²) worst case on bad pivots, while MergeSort guarantees O(n log n). For practical sorted/nearly-sorted data, MergeSort or TimSort wins.
Tricky
How would you find the kth largest element efficiently?
▾
Option 1 — QuickSelect: O(n) average, O(n²) worst. Like QuickSort partition, but recurse only on the side containing kth element. Option 2 — Min-Heap of size k: O(n log k). Maintain a heap of k elements; pop when size > k. Final top = kth largest. Option 3 — Sort + index: O(n log n) — simple but slowest. For streaming data, the heap approach is best since you don't need all elements at once.
⚡ QuickSelect: pivot index == (n-k) means found. If higher, recurse left. If lower, recurse right.
Concept
What makes a sorting algorithm "stable" and when does it matter?
▾
A stable sort preserves the relative order of equal elements. Stable: Merge Sort, TimSort, Insertion Sort, Counting Sort, Bubble Sort. Unstable: QuickSort, HeapSort, Selection Sort. Matters when: sorting objects by one field that have equal values in another field (e.g., sort employees by department, then by name — a stable sort preserves name order within each department). Also matters in multi-pass sorting (sort by secondary key first, then primary key with stable sort).
◆ Memory Tip
💡
Sorting choice cheat sheet:
• Default → Arrays.sort() / Collections.sort() (TimSort)
• Need guaranteed O(n log n) + O(1) space → Heap Sort
• Need stable + worst-case guarantee → Merge Sort
• Small integer range → Counting Sort
• Nearly sorted data → Insertion Sort (O(n) best case)

Stable = "Marriage preserving order" (MICS): Merge, Insertion, Counting, (Bubble)

// ── BFS — Level order / shortest path ────────────
void bfs(Node root) {
    Queue<Node> q = new LinkedList<>();
    q.offer(root);
    while (!q.isEmpty()) {
        Node n = q.poll();
        process(n);
        if (n.left  != null) q.offer(n.left);
        if (n.right != null) q.offer(n.right);
    }
}

// ── DFS — inorder iterative ───────────────────────
List<Integer> inorder(Node root) {
    List<Integer> res = new ArrayList<>();
    Deque<Node> stack = new ArrayDeque<>();
    Node cur = root;
    while (cur != null || !stack.isEmpty()) {
        while (cur != null) { stack.push(cur); cur = cur.left; }
        cur = stack.pop();
        res.add(cur.val);
        cur = cur.right;
    }
    return res;
}

// ── Monotonic Stack: next greater element ─────────
int[] nextGreater(int[] nums) {
    int[] res = new int[nums.length];
    Arrays.fill(res, -1);
    Deque<Integer> stack = new ArrayDeque<>(); // stores indices
    for (int i = 0; i < nums.length; i++) {
        while (!stack.isEmpty() && nums[i] > nums[stack.peek()])
            res[stack.pop()] = nums[i]; // i is next greater for popped
        stack.push(i);
    }
    return res;
}

// Kth largest with min-heap of size k
int kthLargest(int[] nums, int k) {
    PriorityQueue<Integer> heap = new PriorityQueue<>();
    for (int n : nums) {
        heap.offer(n);
        if (heap.size() > k) heap.poll(); // remove smallest
    }
    return heap.peek();                   // top of min-heap = kth largest
}

// ── LRU Cache (LinkedHashMap trick) ──────────────
class LRUCache extends LinkedHashMap<Integer, Integer> {
    private final int cap;
    public LRUCache(int cap) {
        super(cap, 0.75f, true); // accessOrder=true
        this.cap = cap;
    }
    @Override
    protected boolean removeEldestEntry(Map.Entry<Integer,Integer> e) {
        return size() > cap; // evict LRU automatically
    }
}
◆ Interview Questions
Must Know
How does a HashMap work internally in Java? What happens on hash collision?
▾
HashMap uses an array of buckets. Hash of key → bucket index (hash % capacity). Collision handling: Java uses Separate Chaining — each bucket holds a linked list. In Java 8+, when a bucket's chain length exceeds 8 AND table size ≥ 64, the chain is converted to a Red-Black Tree (O(log n) instead of O(n) per bucket). When tree shrinks below 6, it reverts to a list. Default load factor 0.75 — rehashes (doubles capacity) when 75% full.
⚡ Key: HashMap is O(1) average, O(n) worst case (all keys same bucket). Java 8's tree fix makes worst case O(log n) per bucket.
Tricky
Design an LRU Cache in O(1) get and put. What data structures are needed?
▾
HashMap + Doubly Linked List. HashMap for O(1) key lookup. Doubly linked list for O(1) insertion at head (most recent) and removal from tail (least recent). HashMap stores key → list node reference. On get: move node to head. On put: if exists, update + move to head; if new, add at head; if over capacity, remove tail node and its HashMap entry. In Java: use LinkedHashMap(capacity, 0.75f, true) with removeEldestEntry override — accessOrder=true maintains access order automatically.
Tricky
When should you use a TreeMap over a HashMap?
▾
Use TreeMap when you need: 1. Keys in sorted order (NavigableMap). 2. Range queries — subMap(from, to), headMap(key), tailMap(key). 3. Floor/ceiling keys — floorKey(k) (largest key ≤ k), ceilingKey(k) (smallest key ≥ k). 4. First/last key. Cost: O(log n) vs HashMap's O(1). Keys must implement Comparable or you must provide a Comparator.
Concept
Explain the difference between BFS and DFS with their application scenarios.
▾
BFS (Breadth-First): Queue-based, explores level by level. Guarantees shortest path in unweighted graphs. Uses more memory (stores entire frontier). Best for: shortest path, level-order traversal, minimum steps, "nearest" problems.
DFS (Depth-First): Stack/recursion, explores as far as possible before backtracking. Less memory (just the current path). Best for: connected components, cycle detection, topological sort, maze solving, all paths, tree serialization.
⚡ "Shortest" = BFS. "All paths / connected" = DFS. Always.

◆ Core Concepts
🔐
synchronized — Mutual exclusion on a monitor lock. Method-level (lock = this) or block-level (lock = specified object). Prevents concurrent access. Causes: thread contention, deadlock if not careful. synchronized is re-entrant — same thread can acquire the same lock twice.
👁️
volatile — Guarantees visibility (writes flushed to main memory, reads from main memory — no CPU cache). Does NOT guarantee atomicity (i++ is not atomic even on volatile). Use for: flags, stop-signals, double-checked locking DCL. Think of volatile as "always read from RAM, not CPU cache".
⚛️
Atomic Classes — AtomicInteger, AtomicLong, AtomicReference in java.util.concurrent.atomic. Use CAS (Compare-And-Swap) CPU instruction — lock-free thread-safe operations. incrementAndGet(), compareAndSet(expected, update). Faster than synchronized for single variables.
🏊
ExecutorService — Thread pool management. Executors.newFixedThreadPool(n), newCachedThreadPool(), newSingleThreadExecutor(), newScheduledThreadPool(n). Submit tasks via submit() (returns Future) or execute() (fire-and-forget). Always shutdown properly: shutdown() then awaitTermination().
🔮
CompletableFuture — Async programming without blocking. Chain operations: thenApply() (transform), thenAccept() (consume), thenCompose() (flat-map), thenCombine() (combine two futures). Handle errors: exceptionally(), handle().
💀
Deadlock Conditions — All 4 must hold: Mutual Exclusion (only one thread holds resource), Hold and Wait (holding one, waiting for another), No Preemption (can't force-release), Circular Wait (A waits for B, B waits for A). Prevention: always acquire locks in same order; use tryLock with timeout.

// ── Double-Checked Locking (Singleton) ───────────
public class Singleton {
    private static volatile Singleton instance;  // volatile required!
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null)           // double check!
                    instance = new Singleton();
            }
        }
        return instance;
    }
}
◆ Interview Questions
Must Know
What is the difference between volatile and synchronized?
▾
volatile: Guarantees visibility — changes are immediately visible to all threads (no CPU cache). Does NOT guarantee atomicity — compound operations like i++ (read-modify-write) are still unsafe. No locking, no mutual exclusion. Fast.
synchronized: Guarantees both visibility AND atomicity — only one thread in the block at a time. Creates happens-before relationship. Slower due to lock acquisition. Use volatile for simple flags; synchronized for compound operations on shared state.
⚡ volatile is NOT a lightweight synchronized. They solve different problems.
Tricky
Why must volatile be used in double-checked locking for Singleton?
▾
Without volatile, the JVM/CPU can reorder instructions (write to instance before the constructor finishes, due to Java Memory Model reordering). Another thread can see a non-null but partially initialized object. volatile prevents reordering — establishes a happens-before guarantee that all writes before the volatile write are visible after the volatile read. Alternative: use the Initialization-on-Demand Holder idiom which is simpler and correct without volatile.
Must Know
What is the difference between Thread, Runnable, and Callable?
▾
Runnable: functional interface with run() — no return value, no checked exception. Callable<T>: functional interface with call() — returns a value of type T, can throw checked exceptions. Thread: actual thread of execution; can be constructed with Runnable. In modern Java: prefer Callable + ExecutorService.submit() which returns Future<T> for async result retrieval. Never extend Thread directly — prefer composition over inheritance.
Tricky
What is a ThreadLocal and when would you use it in a Spring application?
▾
ThreadLocal<T> provides a per-thread variable — each thread gets its own isolated instance. Spring uses ThreadLocal heavily internally: SecurityContextHolder stores authentication per-request thread; TransactionSynchronizationManager tracks current transaction; RequestContextHolder holds HttpServletRequest. Use in Spring for: user context propagation, request-scoped data in non-Spring-managed components, database connection binding. Critical: always call remove() in a finally block — thread pool threads are reused and can carry stale values.
⚡ Memory leak risk: ThreadLocal in thread pools — always clean up in finally.
Concept
What thread pool sizes should you choose for CPU-bound vs I/O-bound tasks?
▾
CPU-bound tasks: Pool size = number of CPU cores (or cores + 1 for occasional blocking). More threads = context-switching overhead. Runtime.getRuntime().availableProcessors().
I/O-bound tasks: Pool size can be much larger — threads spend most time waiting (network, DB, disk). Formula: cores × (1 + wait_time/compute_time). If wait ratio is 9:1, pool size = 10× core count. Modern alternative: Virtual threads (Java 21) — create one per task regardless of count, JVM handles scheduling efficiently.

💡
Concurrency primitives by guarantee:
• volatile = Visibility only (no atomicity)
• synchronized = Visibility + Atomicity + Mutual Exclusion
• AtomicXxx = Visibility + Atomicity (no mutex, CAS-based, lock-free)
• ReentrantLock = All of synchronized + tryLock + fairness + interruptible

Deadlock prevention: "Always lock in alphabetical order" — consistent lock ordering breaks circular wait.

◆ Interview Questions
Must Know
What is the difference between WHERE and HAVING?
▾
WHERE filters individual rows before grouping — cannot use aggregate functions. HAVING filters groups after GROUP BY — can use aggregate functions. Order of SQL execution: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. Example: WHERE salary > 50000 (filter rows first), HAVING AVG(salary) > 60000 (filter groups by avg).
Tricky
What is the difference between RANK(), DENSE_RANK(), and ROW_NUMBER()?
▾
For salaries [100, 90, 90, 80]:
ROW_NUMBER(): 1, 2, 3, 4 — always unique, arbitrary order for ties
RANK(): 1, 2, 2, 4 — ties get same rank, next rank skips (gap)
DENSE_RANK(): 1, 2, 2, 3 — ties get same rank, next rank is consecutive (no gap)
⚡ Use ROW_NUMBER() to get exactly one row per group (e.g., latest order per user). DENSE_RANK() for "top 3 distinct salary levels".
Tricky
Why can applying a function to an indexed column break index usage?
▾
The index stores raw column values. When you apply a function like UPPER(email) or YEAR(created_at) in the WHERE clause, the database can't use the index because the index values don't match the function output — it must compute the function for every row (full table scan). Fix: store data in the normalized form you'll query (lowercase emails), or use a functional index (CREATE INDEX ON users (UPPER(email))), or rewrite the query with a range: created_at BETWEEN '2024-01-01' AND '2024-12-31'.
Concept
What is a covering index and why is it powerful?
▾
A covering index includes all columns needed by a query — the database can answer the query entirely from the index without touching the main table. Example: SELECT name, email FROM users WHERE dept = 'Engineering' — if the index is on (dept, name, email), it's a covering index. No heap fetch needed. Often 10-100× faster for read-heavy queries. Trade-off: larger index size, slower writes. In PostgreSQL, this is an "index-only scan".
Must Know
What is the N+1 query problem in SQL/ORM and how do you fix it at the SQL level?
▾
N+1 means 1 query to fetch N parents, then N separate queries for each parent's children — total N+1 queries. At SQL level: use a single JOIN to fetch all data together, then reconstruct in application. Or use IN clause: fetch all parent IDs, then one query WHERE parent_id IN (...) to fetch all children at once. At ORM level: JOIN FETCH in JPQL or @EntityGraph. At DB design level: denormalization or materialized views for frequently joined data.
◆ Memory Tip
💡
SQL execution order (not writing order):
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT

Index killer checklist (things that break index usage):
• Function on column in WHERE: UPPER(col), YEAR(date)
• Leading wildcard: LIKE '%suffix' (trailing OK: LIKE 'prefix%')
• Type mismatch: comparing varchar to integer
• OR conditions (sometimes) — use UNION instead
• NOT IN with NULLs

◆ Interview Questions
Tricky
What happens if you use a mutable object as a HashMap key and then mutate it?
▾
The key's hashCode() changes after mutation. The HashMap still stores the entry in the original bucket (based on old hash). When you try to get the value, the new hash maps to a different bucket — the entry is effectively lost. You won't find it with get() or containsKey(). This is why keys must be immutable (or at least their hash-relevant fields must not change). String and Integer are immutable — ideal keys.
⚡ Classic interview trap: "You have a key in the map but get() returns null."
Must Know
What is the contract between equals() and hashCode()?
▾
The contract: 1. If a.equals(b) is true, then a.hashCode() == b.hashCode() MUST be true. 2. If a.hashCode() == b.hashCode(), a.equals(b) MAY be false (collision). Breaking rule 1 breaks HashMap/HashSet — equal objects map to different buckets and are treated as different entries. Always override both together. The IntelliJ/Lombok-generated implementations follow this contract. A common bug: override equals() without hashCode() — results in duplicate entries in HashSet for "equal" objects.
Concept
What is the difference between fail-fast and fail-safe iterators?
▾
Fail-fast (ArrayList, HashMap, HashSet): throws ConcurrentModificationException if the collection is structurally modified during iteration (uses a modCount check). Detects bugs early. Do not modify the collection while iterating — use Iterator.remove() or a copy.
Fail-safe (ConcurrentHashMap, CopyOnWriteArrayList): operates on a copy of the collection, so modifications during iteration don't cause exceptions. May not see the latest state. Trade-off: memory and freshness vs safety.
Tricky
Why should you avoid the Stack class and use ArrayDeque instead?
▾
Stack extends Vector, which is a legacy synchronized class. This means every Stack operation acquires a lock — performance overhead even in single-threaded code. Also, it exposes Vector methods (like get(int)) that violate the stack abstraction. ArrayDeque is unsynchronized, faster, and implements the Deque interface — use push()/pop()/peek() for stack semantics. If you need a thread-safe stack, use ConcurrentLinkedDeque or synchronize explicitly.
◆ Memory Tip
💡
Collection cheat: "What matters most?"
• Order + fast access → ArrayList
• Fast ends (queue/stack) → ArrayDeque
• Key-value, fast lookup → HashMap
• Key-value, insertion order → LinkedHashMap
• Key-value, sorted → TreeMap
• Unique values → HashSet
• Unique + sorted → TreeSet
• Thread-safe map → ConcurrentHashMap (not Hashtable!)

equals + hashCode rule: "Equal objects MUST have equal hashes. Equal hashes MAY have unequal objects."

Exception Handling
Not just try-catch — exception design is an API decision that affects callers forever
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
◆ Exception Hierarchy
🌳
Throwable — Root. Two branches: Error (JVM-level, unrecoverable: OutOfMemoryError, StackOverflowError — never catch) and Exception (application-level). Exception splits into: Checked (must handle or declare: IOException, SQLException) and Unchecked/RuntimeException (optional handling: NullPointerException, IllegalArgumentException).
✅
Checked vs Unchecked Design Rule — Use checked when caller CAN reasonably recover (file not found → retry). Use unchecked for programming errors (null arg, illegal state — programmer should have prevented this). Spring/modern Java prefers unchecked — checked exceptions in APIs are "viral" and force callers to handle them.
🔗
Exception Chaining — Wrap low-level exceptions in meaningful higher-level ones. Pass original as cause: throw new ServiceException("Failed to load user", e). Preserves the full stack trace. Never swallow exceptions silently — at minimum log them.
🛡️
try-with-resources — Auto-closes resources implementing AutoCloseable. Resources closed in reverse declaration order. Suppressed exceptions: if both try body AND close() throw, the body's exception is primary and close's is "suppressed" (accessible via getSuppressed()).
⚠️
Anti-patterns — Pokemon exception handling (catch Exception and do nothing), swallowing (empty catch block), returning null on exception (use Optional or throw), using exceptions for flow control (extremely slow — exceptions involve stack trace capture).

} finally {
    if (conn != null) conn.close(); // always runs (prefer try-with-resources)
}
◆ Interview Questions
Tricky
Can a finally block prevent an exception from propagating? Can it override a return value?
▾
Yes to both — this is a dangerous anti-pattern. If the finally block contains a return statement, it swallows any exception thrown in the try block and returns the finally's value. Similarly, if finally throws its own exception, the original exception is lost. Example: try throws RuntimeException, finally returns 42 — caller sees 42, original exception is gone. This is why you should never put return or throw in finally.
⚡ Never put return or throw in a finally block — it silently swallows exceptions.
Must Know
What is the difference between throw and throws?
▾
throw is a statement — actually throws an exception instance: throw new IllegalArgumentException("bad input"). throws is a method declaration keyword — declares that a method might throw checked exceptions, forcing callers to handle or declare them: void readFile() throws IOException. Required for checked exceptions; optional (but good documentation) for unchecked ones. Spring/modern Java prefers unchecked exceptions to avoid the checked exception "tax" on API callers.
Concept
Why are exceptions slow and when should you avoid using them for control flow?
▾
Exception creation involves capturing the full stack trace — walking the call stack and building a StackTraceElement array. This is expensive (microseconds to milliseconds). Using exceptions for normal flow (e.g., using NumberFormatException to detect non-numeric strings instead of isNumeric() check) can be 100-1000× slower than a regular conditional. Always validate before attempting, not catch-to-detect. If you must create exceptions without the overhead, override fillInStackTrace() to return this.
◆ Memory Tip
💡
Exception design decision tree:
→ Can caller reasonably recover? → Checked exception
→ Programming bug (should not happen)? → Unchecked (RuntimeException)
→ JVM-level catastrophe? → Error (don't catch)

Exception best practices:
• Always pass the original cause in the constructor (chain!)
• Never swallow silently — at least log
• Don't use exceptions for flow control (performance)
• prefer try-with-resources over try-finally for resources

Must Know
Which design patterns does Spring Framework heavily use?
▾
Singleton — default bean scope. Factory Method — BeanFactory creates beans. Proxy — AOP, @Transactional, @Cacheable wrap beans in proxies. Tricky
What is the difference between Strategy and Template Method patterns?
▾
Both define an algorithm with interchangeable parts. Template Method: uses inheritance — base class defines skeleton algorithm with abstract/hook methods; subclasses override specific steps. The algorithm structure is fixed. Hard to change at runtime. Strategy: uses composition — algorithm is encapsulated in an interface; implementations are plugged in at runtime. More flexible, easier to test, follows "favor composition over inheritance". Spring Boot auto-configuration uses Strategy — you swap implementations by providing different beans.
Tricky
What is the difference between Decorator and Proxy patterns?
▾
Both wrap another object and delegate to it, but their intent differs: Proxy controls access — same interface, adds access control, caching, logging, lazy initialization (client doesn't know it's a proxy). Decorator adds behavior/features — same interface, wraps to enhance functionality dynamically (you know you're decorating). Proxy: Spring AOP, JDK Proxy. Decorator: BufferedReader wrapping FileReader, HttpServletRequestWrapper. Key: Proxy hides its existence; Decorator is explicitly composed by the client.
Concept
When should you use the Builder pattern vs a constructor with many parameters?
▾
Use Builder when: 4+ constructor parameters (readability drops), optional parameters (avoid telescoping constructors), immutable object needed with complex construction. Constructor pitfall: new User("Alice", null, null, true, false, 30) — what do the booleans mean? Builder self-documents: new User.Builder().name("Alice").active(true).age(30).build(). Lombok's @Builder generates this for free. Also use when the object needs to be validated before construction — validate in build().

CS · Principles
SOLID Principles
The 5 commandments of object-oriented design — know them, recognize violations, propose fixes
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
◆ The 5 Principles
1️⃣
S — Single Responsibility Principle — A class should have only one reason to change. One class = one job. Violation: a User class that handles validation, DB persistence, AND email sending. Fix: split into UserValidator, UserRepository, UserNotificationService. "Reason to change" means different stakeholders — business logic changes shouldn't force DB code to change.
2️⃣
O — Open/Closed Principle — Open for extension, closed for modification. Add new behavior by adding new code, not changing existing. Violation: adding a new payment type requires modifying PaymentService with a new if/else branch. Fix: PaymentStrategy interface — add new types by creating new classes, not editing old ones.
3️⃣
L — Liskov Substitution Principle — Subtypes must be substitutable for their base types without altering correctness. Violation: Square extends Rectangle where setting width also changes height — breaks the rectangle invariant. Fix: don't force IS-A when behavior differs; prefer composition or a common interface.
4️⃣
I — Interface Segregation Principle — Clients should not be forced to depend on interfaces they don't use. Violation: a Worker interface with work(), eat(), sleep() — robots implement it but eat() makes no sense. Fix: split into Workable, Eatable, Sleepable — implement only what applies.
5️⃣
D — Dependency Inversion Principle — High-level modules should not depend on low-level modules. Both should depend on abstractions. Violation: OrderService directly creates new MySQLOrderRepository(). Fix: OrderService depends on OrderRepository interface; the implementation is injected (Spring DI). This is what Spring IoC/DI implements.
◆ Memorizable Code
SOLID — Violations vs Fixes
// ── S: Single Responsibility ──────────────────────
// ❌ BAD: One class does too much
class UserService {
    public void createUser(User u) { validate(u); saveToDb(u); sendEmail(u); }
}
// ✅ GOOD: Split responsibilities
class UserService {
    public void createUser(User u) {

}

interface PaymentProcessor { void charge(Order o); }
class CardProcessor   implements PaymentProcessor { public void charge(Order o){} }
class PaypalProcessor implements PaymentProcessor { public void charge(Order o){} }

class Rectangle { int w, h; void setW(int w){this.w=w;} void setH(int h){this.h=h;} }
class Square extends Rectangle {
    void setW(int w){ this.w=this.h=w; } // breaks rectangle invariant

// ❌ BAD: depends on concrete class
class OrderService {

}
// ✅ GOOD: depends on abstraction, injected
interface OrderRepository { void save(Order o); }

@Service
class OrderService {

◆ Interview Questions
Must Know
How does Spring's DI implement the Dependency Inversion Principle?
▾
Perfectly — Spring is DIP in action. Instead of OrderService creating new MySQLOrderRepository() (depending on concrete class), it depends on OrderRepository interface. Spring's IoC container provides the concrete implementation at runtime (injection). You can swap MySQLOrderRepository for PostgreSQLOrderRepository or a mock in tests without changing OrderService. High-level module (OrderService) doesn't know or care about the low-level module (MySQLOrderRepository).
Tricky
Can you give a real-world example of an LSP violation?
▾
Classic: Square extending Rectangle. A Rectangle allows independent width/height setting. A Square must keep them equal. Code expecting a Rectangle — setWidth(5); setHeight(10); assert area() == 50 — fails with Square (area = 100 because setHeight also set width to 10). The subtype changes behavior the parent promised. Another: ReadOnlyList extends ArrayList but throws UnsupportedOperationException on add() — violates ArrayList's contract. Fix: don't extend if behavior changes; use composition or a different interface hierarchy.
Concept
How do SOLID principles relate to each other in practice?
▾
They reinforce each other: SRP keeps classes small and focused. OCP encourages interfaces — which enables DIP (depend on abstractions). LSP enforces that those abstractions are honest. ISP keeps interfaces lean so LSP violations are less likely. Violating one often violates others — a God class (SRP violation) usually also violates OCP (every new feature requires modifying it) and DIP (hardcoded dependencies). SOLID principles collectively push toward: small classes, abstraction-based design, and high testability.

💡

S — One class, one job ("one reason to change")
O — Add features by adding code, not editing old code
L — Subclass must behave like parent (no surprise behavior)

🧠 Memory
◆ JVM Memory Areas
🏗️
Heap — Where all objects live. Shared across all threads. Managed by GC. Divided into: Young Generation (Eden + Survivor S0/S1 — most objects die young), Old Generation/Tenured (long-lived objects — promoted from Young after multiple GC cycles). Where OutOfMemoryError occurs when full.
📦
Stack — Each thread has its own stack. Stores stack frames (local variables, operand stack, method return address). Created per method call, destroyed on return. StackOverflowError when too deep (infinite recursion). Primitive variables and object references (not objects themselves) live here.
📚
Metaspace (Java 8+, replaced PermGen) — Stores class metadata (class structure, method bytecode, static variables). Grows dynamically in native memory (no fixed size — less likely to cause OOM than PermGen). OutOfMemoryError: Metaspace if too many classes loaded (e.g., dynamic class generation).
⚡
JIT Compiler — Hotspot JVM identifies hot methods (frequently called) and compiles bytecode to native machine code at runtime. Two tiers: C1 (client, quick compile, less optimized) and C2 (server, slower compile, heavily optimized). JIT enables Java to approach C++ performance for long-running apps. JVM warm-up period = JIT doing its work.
🗑️
Garbage Collectors — Serial GC (single-threaded, small apps), Parallel GC (throughput, multiple threads), G1 GC (default in Java 9+, balanced — divides heap into regions, aims for predictable pause times), ZGC/Shenandoah (low latency — <10ms pauses, concurrent collection). Choose GC based on throughput vs latency needs.
🔍
GC Roots & Reachability — GC collects objects NOT reachable from GC roots. Roots: active thread stacks, static variables, JNI references. Objects reachable = alive; unreachable = eligible for GC. Memory leaks in Java: objects are kept alive unintentionally (in static collections, listeners not deregistered, ThreadLocal not cleared).

◆ Interview Questions
Must Know
What is the difference between Stack and Heap memory in Java?
▾
Heap: All objects (instances of classes). Shared across threads. Managed by GC. Larger. Where OutOfMemoryError happens.
Stack: Per-thread. Stores method call frames — local primitive variables AND object references (but NOT the objects themselves, those are on heap). Automatically freed when method returns. Where StackOverflowError happens. Each thread has its own stack (thread-safe by default). Accessing heap objects across threads requires synchronization; stack variables are inherently thread-safe (private to each thread).
Tricky
Why does new String("hello") == "hello" return false?
▾
String literals ("hello") are interned in the String constant pool (part of Metaspace/heap). new String("hello") creates a new object on the heap — a different reference. == compares references, not values. Even though both contain "hello", they're different objects. Solution: always use .equals() for string comparison. s.intern() adds a string to the pool and returns the pool reference — then == works, but this is rarely needed in modern Java.
⚡ Senior insight: String pool is why you should never use == for Strings — it works for literals by accident (same pool reference), but breaks for runtime strings.
Must Know
What causes memory leaks in Java and how do you detect them?
▾
Java GC collects objects not reachable from GC roots. Memory "leaks" when objects remain reachable but are logically unused. Common causes: 1. Static collections (static Map cache) that grow forever. 2. Event listeners not deregistered. 3. ThreadLocal not removed (especially in thread pools). 4. Inner class holding reference to outer class. 5. Interned strings or class metadata in Metaspace (dynamic class generation). Detection: -XX:+HeapDumpOnOutOfMemoryError + analyze with Eclipse MAT or VisualVM. Look for large retained sets.
Tricky
What is the parent delegation model in class loading and why does it exist?
▾
When a ClassLoader needs to load a class, it first delegates to its parent. Only if the parent fails does it try itself. Order: Bootstrap → Extension → Application → Custom. Why: Security (ensures core Java classes like java.lang.String are always loaded from trusted bootstrap loader — can't be replaced with malicious versions), consistency (same class loaded by different custom loaders = different Class objects — causes ClassCastException across ClassLoader boundaries), and avoid duplicate loading. Spring Boot's executable JAR uses a custom ClassLoader (LaunchedURLClassLoader) on top of this hierarchy.
Concept
What is JIT compilation and what is JVM warmup?
▾
The JVM starts executing bytecode in interpreted mode (slow but immediate). The JIT (Just-In-Time) compiler monitors method invocations — when a method becomes "hot" (called frequently, typically 10,000+ times), it compiles it to optimized native machine code. Two-tier: C1 compiles quickly with basic optimization; C2 compiles slowly with aggressive inlining, loop unrolling, and escape analysis (can eliminate heap allocations). Warmup period: the initial phase where JIT is profiling and compiling — performance improves over time. This is why Java microservices may respond slowly initially (cold start), and why GraalVM native-image (ahead-of-time compilation) is appealing for short-lived processes — no warmup needed.
