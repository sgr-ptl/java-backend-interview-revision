Spring Security
Authentication & Authorization
Who are you? What are you allowed to do? — the two pillars of security
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Authentication = proving who you are (login with username+password). Authorization = checking what you're allowed to do (can this user delete orders?). Spring Security handles both through a filter chain that intercepts every HTTP request before it hits your controller.
◆ Core Concepts
🔐
Authentication — Verifying identity. Spring Security's flow: AuthenticationFilter extracts credentials → AuthenticationManager delegates to AuthenticationProvider → provider validates against UserDetailsService (loads user from DB) → returns Authentication object stored in SecurityContext. UserDetailsService must implement loadUserByUsername().
🎫
Authorization — Controlling access after authentication. Two approaches: Method-level (@PreAuthorize, @Secured on service methods) and URL-level (rules in SecurityFilterChain). Spring checks the GrantedAuthority list from the authenticated user's principal.
⚙️
SecurityFilterChain — The heart of Spring Security config. A chain of filters applied to HTTP requests. You define rules: which URLs need authentication, which roles are required, which authentication mechanism to use (form login, HTTP Basic, JWT). Multiple chains can exist (e.g., API vs admin UI).
👤
SecurityContext & SecurityContextHolder — After successful authentication, a SecurityContext holding the Authentication object is stored in SecurityContextHolder (thread-local by default). Access anywhere: SecurityContextHolder.getContext().getAuthentication(). Cleared after request completes.
🔑
Password Encoding — Never store plain-text passwords. Use BCryptPasswordEncoder (recommended — adaptive cost factor, built-in salting). Spring Security's PasswordEncoder interface: encode(raw) to store, matches(raw, encoded) to verify. Default rounds: 10 (configurable 4–31).
🏷️
Roles vs Authorities — Role: a high-level grouping (ROLE_ADMIN, ROLE_USER). Spring automatically prepends "ROLE_" prefix. hasRole("ADMIN") checks for "ROLE_ADMIN". Authority/Permission: fine-grained privilege (orders:read, users:delete). hasAuthority("orders:read"). A user can have both roles and granular authorities.

Spring Security — SecurityFilterChain + UserDetailsService + @PreAuthorize
// ── SecurityFilterChain (Spring Boot 3 / Security 6) ─
@Configuration
@EnableWebSecurity
@EnableMethodSecurity          // enables @PreAuthorize, @PostAuthorize
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(AbstractHttpConfigurer::disable) // disable for REST APIs
            .sessionManagement(s -> s
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**", "/public/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .requestMatchers(HttpMethod.DELETE).hasAuthority("orders:delete")
                .anyRequest().authenticated()  // all others need login
            )
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder(12); // cost factor 12
    }

    @Bean
    public AuthenticationManager authManager(AuthenticationConfiguration config)
            throws Exception {
        return config.getAuthenticationManager();
    }
}

// ── UserDetailsService — loads user from DB ───────
@Service
public class UserDetailsServiceImpl implements UserDetailsService {

    private final UserRepository userRepo;

    @Override
    public UserDetails loadUserByUsername(String username) {
        User user = userRepo.findByEmail(username)
            .orElseThrow(() -> new UsernameNotFoundException(username));
        return org.springframework.security.core.userdetails.User
            .withUsername(user.getEmail())
            .password(user.getPasswordHash())
            .roles(user.getRole().name())   // auto-prefixes ROLE_
            .build();
    }
}

// ── Method-level security ─────────────────────────
@Service
public class OrderService {

    @PreAuthorize("hasRole('ADMIN') or #userId == authentication.principal.id")
    public List<Order> getUserOrders(Long userId) { ... }

    @PreAuthorize("hasAuthority('orders:delete')")
    public void deleteOrder(Long id) { ... }

    @PostAuthorize("returnObject.ownerId == authentication.principal.id")
    public Order getOrder(Long id) { ... } // check AFTER loading
}

// ── Access current user anywhere ──────────────────
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();

// Or in controller method:
@GetMapping("/me")
public UserDto me(@AuthenticationPrincipal UserDetails user) {
    return userService.findByEmail(user.getUsername());
}
Must Know
What is the difference between Authentication and Authorization in Spring Security?
▾
Authentication = verifying who you are. You prove identity (username/password, token, certificate). Result: a Authentication object stored in SecurityContextHolder.
Authorization = determining what you can do. After authentication, Spring checks the user's roles/authorities against the required ones. Happens in AccessDecisionManager or via @PreAuthorize SpEL expressions. Real-world analogy: Authentication = showing your ID at the door. Authorization = checking if your membership level allows access to the VIP lounge.
⚡ Spring Security runs Authentication first. Authorization only happens after a valid Authentication exists.
Must Know
How does Spring Security's filter chain work? What happens to every HTTP request?
▾
Spring Security inserts a DelegatingFilterProxy into the Servlet container's filter chain. This delegates to a FilterChainProxy which holds your SecurityFilterChain. Every request passes through ~15+ security filters in order: SecurityContextPersistenceFilter (load context), CsrfFilter, LogoutFilter, UsernamePasswordAuthenticationFilter, JwtAuthFilter (custom), ExceptionTranslationFilter (converts security exceptions to 401/403), FilterSecurityInterceptor (checks authorization). Each filter can: abort the chain (deny request), modify the request, or pass through.
💡 Your custom JWT filter goes BEFORE UsernamePasswordAuthenticationFilter to set the SecurityContext before Spring checks authorization.
Tricky
What is the difference between @PreAuthorize and @Secured? When do you use each?
▾
@Secured: Simple role checking. @Secured("ROLE_ADMIN") — checks if user has ROLE_ADMIN. No SpEL — just string role names. Available since Spring 2.
@PreAuthorize: Full Spring Expression Language support. @PreAuthorize("hasRole('ADMIN') or #userId == authentication.principal.id") — complex conditions, access to method arguments (via #paramName), access to return value (in @PostAuthorize). Requires @EnableMethodSecurity. Always prefer @PreAuthorize — it's more expressive and flexible. Use @PostAuthorize when the decision depends on the return value (e.g., only owner can see their data).
Tricky
Why is BCrypt preferred over MD5 or SHA-256 for password hashing?
▾
MD5 and SHA-256 are fast hash functions — designed for speed. This is terrible for passwords: an attacker with a GPU can compute billions of hashes per second (brute force/rainbow tables). BCrypt is intentionally slow — the cost factor (rounds) makes it computationally expensive. BCrypt also includes a built-in random salt per hash, so identical passwords produce different hashes (defeats rainbow tables). Argon2 and scrypt are even better. Modern OWASP recommendation: Argon2 > bcrypt > PBKDF2 for password hashing.
⚡ "Slow is good for password hashing." BCrypt's whole value is that it takes 100ms+ per attempt — fast enough for users, too slow for brute-force attackers.
Concept
What is CSRF and when should you disable it?
▾
Cross-Site Request Forgery (CSRF): an attacker tricks a logged-in user's browser into making an unwanted request to your server (the browser sends session cookies automatically). Spring Security enables CSRF protection by default by requiring a CSRF token in state-changing requests (POST, PUT, DELETE). Disable CSRF when: your API is stateless (no cookies/sessions — using JWT in headers instead). JWT in Authorization header is not sent automatically by browsers from other origins, so CSRF is not a risk. Keep CSRF on for web apps that use cookie-based sessions (traditional Spring MVC apps).
⚠️ Never disable CSRF for session-based web apps. Only disable for stateless JWT REST APIs.
Senior
How do you implement row-level security in Spring Security? (e.g., a user can only see their own orders)
▾
Three approaches: 1. @PostAuthorize SpEL: @PostAuthorize("returnObject.ownerId == authentication.principal.id") — loads the record then checks. Simple but wastes DB call if unauthorized. 2. @PreAuthorize with param: @PreAuthorize("#userId == authentication.principal.id") — checks before DB call using method param. 3. Service-layer filtering: Add the current user's ID to all DB queries programmatically: orderRepo.findByUserIdAndId(currentUserId, orderId) — most efficient, database enforces the filter. Option 3 is the most secure and performant for production systems.
⚡ Spring Data's @Query combined with the current user's ID from SecurityContextHolder is the most robust pattern for row-level security.
💡
Authentication = WHO. Authorization = WHAT.
Think of a hotel: Authentication = checking in with your ID (they know who you are). Authorization = your keycard only opens your room, not the penthouse.

Spring Security flow: Request → Filter Chain → Authentication (who?) → SecurityContext stores it → Authorization (what can they do?) → Your Controller

Password rule: Always BCrypt. Never MD5/SHA. Slow = Secure for passwords.

Security Filters & Interceptors
The pipeline that protects every request — understand the order, understand the power
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Every HTTP request flows through a chain of "security checkpoints" before reaching your controller. Each checkpoint (filter) can inspect, modify, or block the request. Your JWT validation, CORS handling, and session management all happen here as separate, independent filters.
◆ Key Security Filters (in execution order)
FilterOrderWhat It Does
DisableEncodeUrlFilter1stPrevents session ID in URL
SecurityContextPersistenceFilter2ndLoads SecurityContext from session/repo
CorsFilter~3rdHandles CORS preflight and headers
CsrfFilter~4thValidates CSRF token on state-changing requests
LogoutFilter~5thHandles logout endpoint
Your Custom JWT FilterBefore AuthFilterValidates JWT, sets SecurityContext
UsernamePasswordAuthenticationFilter~8thHandles form login, sets authentication
AnonymousAuthenticationFilter~11thSets anonymous authentication if no auth found
ExceptionTranslationFilter~13thConverts AccessDeniedException→403, AuthException→401
FilterSecurityInterceptorLastChecks URL-level authorization rules
🔄
OncePerRequestFilter — Base class for custom security filters. Guarantees your filter runs exactly once per request (important for forwarded/internal requests). Override doFilterInternal(). Used for JWT validation, API key checks, request ID injection, rate limiting.
🌐
CORS Configuration — Cross-Origin Resource Sharing. When your frontend (localhost:3000) calls your backend (localhost:8080), the browser sends a preflight OPTIONS request. Spring Security's CorsFilter responds with allowed origins/methods/headers. Configure via CorsConfigurationSource bean. Do this in Spring Security, not just @CrossOrigin — security filter runs before @CrossOrigin is processed.
⚡
ExceptionTranslationFilter — Catches two types of security exceptions: AuthenticationException (not authenticated → 401, sends to login page or WWW-Authenticate challenge) and AccessDeniedException (authenticated but no permission → 403 or redirect to access-denied page). This is why you get 401 vs 403 without writing any error handling code.
📋
GenericFilterBean vs OncePerRequestFilter — GenericFilterBean: Servlet filter integrated with Spring. Can run multiple times per request (for forwards/includes). OncePerRequestFilter: extends GenericFilterBean, adds guarantee of single execution per request. Always use OncePerRequestFilter for security filters — prevents JWT being validated twice if the request is forwarded internally.
Custom Security Filter + CORS Config + Rate Limiter Filter
// ── Custom OncePerRequestFilter ───────────────────
@Component
public class ApiKeyFilter extends OncePerRequestFilter {

    @Value("${api.key}")
    private String validApiKey;

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                     HttpServletResponse res,
                                     FilterChain chain)
            throws ServletException, IOException {

        String apiKey = req.getHeader("X-API-Key");

        if (apiKey != null && apiKey.equals(validApiKey)) {
            // Set authentication in SecurityContext
            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(
                    "api-client", null,
                    List.of(new SimpleGrantedAuthority("ROLE_API"))
                );
            SecurityContextHolder.getContext().setAuthentication(auth);
        }

        chain.doFilter(req, res); // always continue chain
    }

    @Override
    protected boolean shouldNotFilter(HttpServletRequest req) {
        return req.getServletPath().startsWith("/public/"); // skip public
    }
}

// ── CORS Configuration ────────────────────────────
@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("https://myapp.com", "http://localhost:3000"));
    config.setAllowedMethods(List.of("GET","POST","PUT","DELETE","OPTIONS"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true);
    config.setMaxAge(3600L); // cache preflight 1hr

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
}

http.addFilterAfter(loggingFilter, SecurityContextPersistenceFilter.class);
http.addFilterAt(myFilter, UsernamePasswordAuthenticationFilter.class);
Must Know
Where should you place a custom JWT filter relative to UsernamePasswordAuthenticationFilter? Why?
▾
Place it BEFORE UsernamePasswordAuthenticationFilter using http.addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class). Why: Your JWT filter reads the token, validates it, and sets the Authentication in SecurityContextHolder. This must happen BEFORE Spring's authorization checks (which happen in FilterSecurityInterceptor at the end of the chain). If you place it after, the request hits authorization without an authenticated user and gets rejected with 401 before your filter even runs.
⚡ Rule: anything that establishes authentication goes BEFORE UsernamePasswordAuthenticationFilter. Anything that needs authentication to be established goes AFTER.
Tricky
What is the difference between CORS and CSRF? Are they related?
▾
They're completely different problems: CORS (Cross-Origin Resource Sharing) controls which origins (domains) can make browser-based requests to your API. A mechanism that browsers enforce. You configure it to allow your frontend's domain. CSRF (Cross-Site Request Forgery) is an attack where a malicious site tricks your logged-in browser into sending a request to your API (which automatically includes cookies). Protection: CSRF tokens that the malicious site can't access. They're unrelated but both deal with "cross-site" concerns. You can have: CORS enabled + CSRF enabled (traditional web apps), CORS enabled + CSRF disabled (stateless JWT REST APIs).
⚡ CORS = "who can call my API from a browser?" CSRF = "prevent unauthorized actions by tricking the browser".
Tricky
What does ExceptionTranslationFilter do? How does it decide between 401 and 403?
▾
ExceptionTranslationFilter catches two security exception types from downstream filters: AuthenticationException → the user is NOT authenticated → 401 response (or redirect to login). Delegates to AuthenticationEntryPoint (for REST: sends WWW-Authenticate header, returns JSON error). AccessDeniedException → the user IS authenticated but lacks permission → 403 response. Delegates to AccessDeniedHandler (for REST: returns JSON error). Customize both: implement AuthenticationEntryPoint and AccessDeniedHandler and register them in SecurityFilterChain. Without customization, Spring sends HTML error pages — not ideal for REST APIs.
Senior
How would you implement request rate limiting in Spring Security?
▾
Implement a custom OncePerRequestFilter that uses a rate limiter. Options: 1. Bucket4j + Redis: Store token bucket per IP/user in Redis. On each request: consume a token from the bucket; if empty, return 429 Too Many Requests. 2. Resilience4j RateLimiter: Use as a filter or in gateway. 3. Spring Cloud Gateway: Has built-in RequestRateLimiter filter (Redis token bucket). Placement: put the rate limiter filter very early in the chain — before authentication, to protect even the login endpoint from brute force. Use IP-based rate limiting for unauthenticated endpoints, user-based for authenticated ones.
💡
Filter chain = assembly line for requests. Each station (filter) does one job: check CSRF token, validate JWT, check rate limits, enforce CORS. They run in order. If any filter says NO, the request stops there.

Always use OncePerRequestFilter for security filters — prevents double execution on forwards.

Position rule: JWT filter → BEFORE UsernamePasswordAuthenticationFilter. Logging/audit → AFTER SecurityContextPersistenceFilter (so you have context).

Security · Tokens
JWT — JSON Web Tokens
Stateless authentication tokens — self-contained, signed, and verifiable without a database
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: JWT is like a signed hotel keycard — the hotel issues it (auth server signs it), you carry it (stored in browser), and every door (API endpoint) reads it to know who you are WITHOUT calling back to the front desk. The signature proves it hasn't been tampered with.
◆ JWT Structure
Header
{"alg":"RS256","typ":"JWT"}

.

User data
.
Signature
HMAC_SHA256(base64(hdr)+"."+base64(payload),secret)
Tamper proof
Each part is Base64Url encoded → separated by dots → eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJhbGljZSJ9.<sig>
🏷️
Standard Claims (Registered) — iss (issuer), sub (subject — user ID), aud (audience), exp (expiration — Unix timestamp), iat (issued at), nbf (not before), jti (JWT ID — unique token identifier, used for blacklisting). Always include exp — tokens must expire.
🔏
Signing Algorithms — HS256/HS512 (HMAC): symmetric — same secret for sign and verify. Simple but secret must be shared with all verifiers. RS256/RS512 (RSA): asymmetric — private key signs, public key verifies. Auth server keeps private key; any service can verify with the public key (from JWKS endpoint). Prefer RS256 for microservices.
🔄
Access + Refresh Token Pattern — Access token: short-lived (15 min), sent in every request, stateless. Refresh token: long-lived (7–30 days), stored in DB, used only to get new access tokens. When access token expires: client sends refresh token to /auth/refresh → server validates refresh token in DB → issues new access token. Revoke by deleting refresh token from DB.
🚫
JWT Vulnerabilities — "alg: none" attack: attacker modifies header to {"alg":"none"}, removes signature — library accepts if misconfigured. Fix: always specify allowed algorithms explicitly. Secret brute-force: weak HMAC secrets can be cracked offline. Fix: 256+ bit random secret. JWT stored in localStorage: vulnerable to XSS. Fix: httpOnly cookies for web apps, or short expiry + refresh rotation.
⚠️
JWT is NOT encrypted by default — Anyone can decode the payload (Base64 is trivially reversible). JWT only guarantees that the payload hasn't been tampered with (via signature). Never put sensitive data in JWT payload (passwords, credit cards). Use JWE (JSON Web Encryption) if you need encryption.
JWT — Utility + Filter + Login Endpoint
// ── JWT Utility (using io.jsonwebtoken:jjwt) ──────
@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;

    private final long ACCESS_TTL  = 15 * 60 * 1000;  // 15 min
    private final long REFRESH_TTL = 7 * 24 * 60 * 60 * 1000; // 7 days

    private SecretKey key() {
        return Keys.hmacShaKeyFor(Decoders.BASE64.decode(secret));
    }

    public String generateAccess(UserDetails user) {
        return Jwts.builder()
            .subject(user.getUsername())
            .claim("roles", user.getAuthorities())
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + ACCESS_TTL))
            .signWith(key())
            .compact();
    }

    public Claims parse(String token) {
        return Jwts.parser().verifyWith(key()).build()
            .parseSignedClaims(token).getPayload();
    }

    public boolean isValid(String token) {
        try { parse(token); return true; }
        catch (JwtException e) { return false; }
    }
}

// ── JWT Authentication Filter ─────────────────────
@Component
public class JwtFilter extends OncePerRequestFilter {

    private final JwtUtil jwt;
    private final UserDetailsService uds;

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                     HttpServletResponse res,
                                     FilterChain chain)
            throws ServletException, IOException {

        String header = req.getHeader("Authorization");
        if (header == null || !header.startsWith("Bearer ")) {
            chain.doFilter(req, res); return; // no token — continue
        }

        String token = header.substring(7);
        if (jwt.isValid(token)) {
            Claims claims = jwt.parse(token);
            UserDetails user = uds.loadUserByUsername(claims.getSubject());
            var auth = new UsernamePasswordAuthenticationToken(
                user, null, user.getAuthorities()
            );
            auth.setDetails(new WebAuthenticationDetailsSource().buildDetails(req));
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        chain.doFilter(req, res);
    }
}

// ── Login endpoint ────────────────────────────────
@PostMapping("/auth/login")
public ResponseEntity<TokenResponse> login(@RequestBody LoginRequest req) {
    Authentication auth = authManager.authenticate(
        new UsernamePasswordAuthenticationToken(req.email(), req.password())
    );
    UserDetails user = (UserDetails) auth.getPrincipal();
    return ResponseEntity.ok(new TokenResponse(
        jwtUtil.generateAccess(user),
        jwtUtil.generateRefresh(user)
    ));
}
Must Know
Explain the three parts of a JWT token. What does each contain?
▾
Header (Base64Url encoded JSON): algorithm used (alg: HS256 or RS256) and token type (typ: JWT).
Payload (Base64Url encoded JSON): claims — registered (sub, exp, iat, iss, jti) and custom (roles, userId, email). NOT encrypted — readable by anyone with the token.
Signature: HMACSHA256(base64(header) + "." + base64(payload), secret) — verifies the token hasn't been modified. If anyone changes even one character of header or payload, the signature becomes invalid.
⚡ JWT payload is visible to anyone who has the token. NEVER put passwords, credit cards, or PII in JWT claims.
Must Know
How do you invalidate a JWT token before it expires? (e.g., on logout)
▾
JWT is stateless — the server doesn't store it, so you can't "delete" it. Options: 1. Token Blacklist: store revoked JWT IDs (jti claim) in Redis with TTL = token's remaining lifetime. On each request, check if jti is in the blacklist. Simple but requires a DB lookup per request. 2. Short Access Token TTL: 5–15 minute access tokens + invalidate the refresh token in DB on logout. The access token technically still works until expiry but the damage window is small. 3. Rotate Secrets: change the signing secret (invalidates ALL tokens — nuclear option). Best practice: option 2 — short-lived access + DB-tracked refresh tokens. This balances statelessness (access) with control (refresh).
⚡ "Can JWT be revoked?" is a classic interview question. The answer: not natively — you need a blacklist or short TTL + refresh token pattern.
Tricky
What is the "alg:none" JWT vulnerability? How do you prevent it?
▾
An attacker modifies the JWT header to {"alg":"none"} and removes the signature entirely. A poorly configured JWT library might accept this as valid (treating "none" as a valid algorithm). This allows the attacker to craft arbitrary claims without a valid secret. Prevention: always explicitly specify which algorithms your JWT parser accepts. With JJWT: Jwts.parser().verifyWith(key).build() — if no key is set, it rejects "none". Never use any JWT library that accepts "none" algorithm. Audit your library configuration.
Tricky
HS256 vs RS256 — when would you use each?
▾
HS256 (HMAC-SHA256): Symmetric — same secret for signing AND verification. Simple, fast. Problem: every service that needs to verify tokens must share the secret. Secret management becomes complex in microservices. Best for: single service or tightly controlled environments.
RS256 (RSA-SHA256): Asymmetric — auth server has private key (signs), all services have public key (verify only). Public key can be freely distributed (JWKS endpoint). Services can verify tokens without ever knowing the private key. Best for: microservices architecture where multiple services need to validate tokens independently. OAuth2/OIDC typically uses RS256.
💡 In microservices: use RS256 + expose public key via /oauth2/jwks endpoint. Services self-validate without calling auth server.
Senior
Should JWT be stored in localStorage or HttpOnly cookies? What are the tradeoffs?
▾
localStorage: Simple to use, accessible to JavaScript. Vulnerable to XSS — if attacker injects script, they can steal the token. The attacker then has full access until token expires.
HttpOnly Cookie: Not accessible to JavaScript — immune to XSS. But requires CSRF protection (since cookies are sent automatically). Set with SameSite=Strict or SameSite=Lax to mitigate CSRF. Requires cookie handling infrastructure.
Recommendation: HttpOnly cookies with SameSite=Strict for web apps. For mobile/native apps: secure local storage (OS-protected) is acceptable since XSS is less of a concern. The XSS risk with localStorage is severe in modern SPAs — prefer cookies for production web apps.
💡
JWT = Signed Hotel Keycard. Hotel creates it (auth server signs it), you carry it (browser stores it), every door reads it independently (no DB call) but can't be faked (signature).

JWT structure mantra: Header.Payload.Signature
Header = "how was it signed?"
Payload = "who is this and what can they do?" (NOT encrypted!)
Signature = "I promise nobody tampered with this"

Invalidation rule: Short-lived access (15 min) + revocable refresh (in DB) = best balance of statelessness and control.

Security · Standards
OAuth2 & OpenID Connect
The industry-standard authorization framework — "Login with Google" is OAuth2 + OIDC
🟡 Medium
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: OAuth2 solves "letting App A access your data on Service B without giving App A your password." You tell Service B "yes, App A can read my emails" — B gives App A an access token. OpenID Connect (OIDC) adds "and also tell App A who I am" — adds identity on top of OAuth2.
◆ OAuth2 Roles & Grant Types
👥
OAuth2 Roles — Resource Owner: the user who owns the data. Client: the application requesting access (your app). Authorization Server: issues tokens (Keycloak, Okta, Auth0, Google). Resource Server: the API that holds the protected data (your Spring Boot app with Bearer token validation).
🔄
Grant Types (Flows) — Authorization Code (most secure, with PKCE for SPAs): user logs in at auth server → code → exchange for token. Client Credentials: machine-to-machine (no user), service-to-service calls. Implicit: deprecated — token in URL (insecure). Resource Owner Password: deprecated — user gives password to client (anti-pattern). Device Flow: TVs/devices without browser.
🆔
OpenID Connect (OIDC) — Thin identity layer on top of OAuth2. Adds: ID Token (JWT with user info: sub, name, email, picture), UserInfo endpoint (fetch more user details), discovery document (/.well-known/openid-configuration), and standardized scopes (openid, profile, email). OAuth2 = authorization. OIDC = authentication + authorization.
🎟️
Token Types — Access Token: short-lived, sent to Resource Server to access APIs. Refresh Token: long-lived, used to get new access tokens without re-login. ID Token (OIDC only): JWT about the user's identity (for the client, not for APIs).
🛡️
PKCE (Proof Key for Code Exchange) — Extension for Authorization Code flow. Prevents authorization code interception attacks. Client generates a code_verifier (random secret), sends code_challenge (hash of verifier) with auth request. Token exchange includes code_verifier — server verifies they match. Mandatory for public clients (SPAs, mobile apps) that can't securely store secrets.
Spring Boot — OAuth2 Resource Server + Client Configuration
// ── Resource Server (validates JWT from external auth server)
@Configuration
@EnableWebSecurity
public class ResourceServerConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated())

Must Know
What is the difference between OAuth2 and OpenID Connect?
▾
OAuth2 is an authorization framework — it delegates access. It answers: "Can App A access my data on Service B?" It issues access tokens that prove permission, but doesn't define who the user is. OAuth2 itself doesn't have a concept of "identity."
OpenID Connect (OIDC) is an authentication layer built on top of OAuth2. It adds the ID Token (a JWT containing user identity: sub, name, email) and a UserInfo endpoint. OIDC answers: "Who is this user AND what can they access?" "Login with Google" uses OIDC — Google authenticates you AND authorizes your app to read your profile.
⚡ OAuth2 = "what can this app do?" OIDC = "who is this user AND what can this app do?" OIDC extends OAuth2, not the other way around.
Must Know
Explain the Authorization Code flow. Why is it the most secure grant type?
▾
Steps: 1. User clicks "Login with Google". 2. App redirects to Google auth server with client_id, redirect_uri, scope, state. 3. User authenticates with Google. 4. Google redirects back with a short-lived code in URL. 5. App's backend exchanges code for tokens using client_secret (server-to-server, never exposed to browser). 6. App receives access_token + id_token. Why secure: tokens are never in the URL (only code is, briefly). The code is useless without client_secret. Add PKCE for public clients (SPAs) that can't store secrets. The code is single-use and short-lived.
Tricky
When would you use Client Credentials grant type vs Authorization Code?
▾
Authorization Code: when a human user is involved. The app acts on behalf of a user. Example: user clicks "Connect to Spotify" in your app.
Client Credentials: machine-to-machine, no user involved. Your Order Service calls your Inventory Service. The client (Order Service) authenticates with its own credentials (client_id + client_secret) to get an access token. No redirect, no user consent. Think: service accounts / API keys but with OAuth2 infrastructure.
💡 In microservices: use Client Credentials for service-to-service calls. Use Authorization Code for user-facing login flows.
Tricky
What is PKCE and why is it needed for SPAs and mobile apps?
▾
Single-page and mobile apps are "public clients" — they can't securely store a client_secret (the code runs on the user's device). Without a client_secret, the Authorization Code flow is vulnerable: if an attacker intercepts the authorization code, they can exchange it for tokens. PKCE fixes this: the app generates a random code_verifier locally, sends its hash (code_challenge) with the auth request. When exchanging code for token, sends the original code_verifier. Auth server verifies the hash matches — only the legitimate app that started the flow can complete it. No shared secret needed. All public clients (SPAs, native apps) MUST use PKCE.
Senior
How does Spring Security Resource Server validate JWT tokens from an external auth server without calling the auth server on every request?
▾
Spring Security's oauth2ResourceServer().jwt() uses the JWKS (JSON Web Key Set) endpoint of the auth server. On first request: Spring fetches the public keys from /.well-known/jwks.json and caches them in memory. On every subsequent request: Spring validates the JWT signature locally using the cached public key — no network call. If a new key ID (kid) appears in a token header that's not in cache, Spring re-fetches JWKS. Key rotation: auth server adds a new key to JWKS, gradually tokens signed with old key stop being issued, Spring eventually fetches the new keys. The issuer-uri in application.yml points Spring to the discovery document which contains the JWKS URI automatically.
⚡ Local validation = no network call per request = massive performance benefit at scale.
💡
OAuth2 = Valet parking key. You give the valet a special key that only works for parking (not your house). If the valet loses it, they can't rob your house. That's OAuth2 — limited, delegated access without giving your password.

Grant type selector:
• User is involved → Authorization Code (+ PKCE for SPAs)
• Machine-to-machine → Client Credentials
• Implicit/Password → Deprecated. Don't use.

OAuth2 vs OIDC: OAuth2 = what can you do? OIDC = who are you AND what can you do?

Must Know
What does Spring Data JPA's save() method actually do? When does it INSERT vs UPDATE?
▾
save() uses a SimpleEntityInspector to determine if the entity is new: if the entity's ID is null (or 0 for primitives), it calls entityManager.persist() (INSERT). If the entity has an ID, it calls entityManager.merge() (tries UPDATE — if not found in DB, may INSERT depending on JPA provider). Critical: merge() returns a new managed instance — always use the return value of save(), not the object you passed in. The original object passed in is NOT managed after merge(). Customize "is new" logic by implementing Persistable<ID>.
⚡ Always: User saved = userRepo.save(user); — use saved, not user after this.
Tricky
What is the difference between CrudRepository, JpaRepository, and when would you use each?
▾
CrudRepository: basic CRUD — save, findById, findAll, delete, count. Framework-agnostic (works with MongoDB, Redis, etc.). PagingAndSortingRepository: adds findAll(Pageable) and findAll(Sort). JpaRepository: extends PagingAndSorting + adds JPA-specific: flush(), saveAllAndFlush(), deleteInBatch(), getReferenceById(). Use JpaRepository when: you need batch operations, flush control, JPA-specific features. Use CrudRepository if you might switch to another data store (reactive MongoDB, Redis) — less coupling to JPA.
Tricky
When should you use projections instead of returning full entities?
▾
Always prefer projections for API responses. Full entities: load all columns (including large CLOBs, sensitive fields), load all lazy associations when serialized (N+1 risk), expose internal DB structure to clients. Projections: SELECT only needed columns (better SQL, less memory), no accidental lazy loading, naturally maps to your API contract, prevents over-posting/sensitive field exposure. Rule of thumb: never serialize JPA entities directly to JSON — always map to a DTO or use interface/record projections.
⚡ Returning @Entity directly from @RestController is an anti-pattern — it leaks DB structure and causes lazy-loading surprises.
Senior
What is the difference between @OneToMany with cascade=ALL and orphanRemoval=true?
▾
CascadeType.ALL: operations (persist, merge, remove, refresh, detach) on the parent cascade to children. If you save a User with a new Address in the list, the Address is also saved automatically. If you delete the User, all their Addresses are deleted too.
orphanRemoval=true: if a child entity is removed from the parent's collection, it is automatically deleted from the DB. Example: user.getOrders().remove(order) → the order is DELETE'd. Without orphanRemoval, removing from the collection just breaks the association; the child remains in the DB with a null foreign key.
💡 Use orphanRemoval=true when children have no meaning without their parent (Address → User). Use without it when children can exist independently.
💡
Spring Data JPA = "implement the interface, get the code for free."
You define: what queries you want. Spring generates: how to execute them.

save() rule: null ID → INSERT. Has ID → merge (UPDATE). Always use the returned object.

Projection rule: Entity = DB shape. DTO/Projection = API shape. Never expose entities directly. Use projections for reads, entities for writes.

Repository hierarchy: CrudRepository ⊂ PagingAndSorting ⊂ JpaRepository. Use JpaRepository unless you need store-agnostic code.

// ── Derived Query Methods ─────────────────────────
public interface UserRepository extends JpaRepository<User,Long>,
                                          JpaSpecificationExecutor<User> {

    Optional<User> findByEmailIgnoreCase(String email);
    List<User> findByRoleAndActiveTrue(Role role);
    List<User> findByNameContainingIgnoreCase(String name);
    List<User> findByCreatedAtBetween(Instant from, Instant to);
    List<User> findByRoleIn(List<Role> roles);
    boolean existsByEmail(String email);
    long countByRoleAndActiveTrue(Role role);

    // ── @Query — JPQL ──────────────────────────────
    @Query("SELECT u FROM User u WHERE u.role = :role AND u.active = true ORDER BY u.name")
    List<User> findActiveByRole(@Param("role") Role role);

    // JPQL join fetch (avoids N+1)
    @Query("SELECT DISTINCT u FROM User u LEFT JOIN FETCH u.orders WHERE u.id = :id")
    Optional<User> findByIdWithOrders(@Param("id") Long id);

    // ── @Query — Native SQL ────────────────────────
    @Query(value = """
        SELECT u.*, COUNT(o.id) as order_count
        FROM users u
        LEFT JOIN orders o ON o.user_id = u.id
        WHERE u.active = true
        GROUP BY u.id
        HAVING COUNT(o.id) > :minOrders
        """, nativeQuery = true)
    List<User> findActiveUsersWithMinOrders(@Param("minOrders") int min);

    // ── @Modifying — UPDATE query ──────────────────
    @Modifying(clearAutomatically = true)
    @Transactional
    @Query("UPDATE User u SET u.active = false WHERE u.lastLogin < :cutoff")
    int deactivateInactiveUsers(@Param("cutoff") Instant cutoff);
}

// ── Specification: dynamic filter ─────────────────
public class UserSpec {
    public static Specification<User> hasRole(Role role) {
        return (root, query, cb) ->
            role == null ? null : cb.equal(root.get("role"), role);
    }
    public static Specification<User> nameLike(String name) {
        return (root, query, cb) ->
            name == null ? null
                : cb.like(cb.lower(root.get("name")), "%" + name.toLowerCase() + "%");
    }
    public static Specification<User> isActive() {
        return (root, query, cb) -> cb.isTrue(root.get("active"));
    }
}

// Usage: combine specs dynamically
Specification<User> spec = UserSpec.hasRole(role)
    .and(UserSpec.nameLike(searchName))
    .and(UserSpec.isActive());
List<User> users = userRepo.findAll(spec);
Must Know
How does Spring Data JPA parse derived query method names? What's the limit?
▾
Spring Data parses the method name into a query by splitting it at keywords. Parts: find...By (prefix), then property names separated by And/Or, optionally modified by keywords (IgnoreCase, Containing, etc.). Example: findByEmailIgnoreCaseAndRoleIn → WHERE LOWER(email) = LOWER(?) AND role IN (?). This happens at application startup — if Spring can't parse the method name, it fails fast (great for catching typos early). Limits: method names get unwieldy beyond 3-4 conditions. Long names are hard to read and maintain. Switch to @Query for complex queries.
Tricky
What does @Modifying do and why is clearAutomatically=true important?
▾
@Modifying tells Spring Data that this @Query is an update/delete operation (not a SELECT). Without it, Spring would try to execute it as a query and map results. clearAutomatically=true: after an UPDATE/DELETE JPQL query, the JPA persistence context (first-level cache) still holds stale versions of updated entities. If you load an entity after running a bulk update on it, you might get the OLD values from cache. clearAutomatically=true clears the persistence context after the bulk operation — subsequent loads hit the database and return fresh data.
⚡ Always use clearAutomatically=true with @Modifying bulk operations — otherwise you'll get stale data from the entity cache after the update.
Tricky
What is the difference between JPQL and native SQL in @Query? When do you use native?
▾
JPQL (Java Persistence Query Language): uses entity class names and field names — database-agnostic. SELECT u FROM User u WHERE u.email = :email. Spring translates to DB-specific SQL. Supports Spring Data paging naturally.
Native SQL: real SQL for your specific database — table and column names. SELECT * FROM users WHERE email = ?1. Use native when: JPQL can't express the query (window functions, DB-specific functions, CTEs, complex aggregations), or for maximum performance control. Limitation: paging with native queries requires a countQuery: @Query(value="...", countQuery="SELECT COUNT(*) FROM ...", nativeQuery=true).

How do JPA Specifications work? When are they better than derived query methods?
▾
A Specification<T> wraps a single predicate using the JPA Criteria API. Each Specification can return null for "no filter" — Spring Data ignores null predicates. Combine with and(), or(), not(). Use when: search forms with multiple optional filters (user picks any combination of: role, name, date range, active status). With derived methods, you'd need 2^n methods for n optional filters. With Specifications, you compose them dynamically. Repository must extend JpaSpecificationExecutor<T> to get findAll(Specification) method.

🟢 Essential
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: Returning ALL records is always wrong for production — it kills DB performance and network bandwidth. Spring Data makes paging trivial: pass a Pageable object to any repository method, get back a Page<T> with the records AND metadata (total count, pages, current page, has-next). Your REST controller exposes ?page=0&size=20&sort=name,asc.
◆ Core Concepts
📄
Pageable — Encapsulates pagination request: page number (0-indexed), page size, and Sort instructions. Create with: PageRequest.of(page, size), PageRequest.of(page, size, Sort.by("name")), PageRequest.of(page, size, Sort.by(Direction.DESC, "createdAt")). In controllers: Spring MVC auto-binds Pageable from query params (?page=0&size=20&sort=name,asc) with @PageableDefault for defaults.
📊
Page<T> vs Slice<T> — Page<T>: runs TWO queries — data query + COUNT query. Returns total count, total pages, current page info. Needed when you want to show "Page 3 of 47." Slice<T>: runs ONE query + peek at next page. No total count. More efficient for "load more" / infinite scroll UIs where total doesn't matter. List<T>: just the data, no pagination metadata. Use when count is irrelevant.
🔀
Sort — Sort.by("name") ascending (default). Sort.by(Direction.DESC, "createdAt"). Sort.by("lastName", "firstName") — multi-column sort. Sort.unsorted(). Combine with Pageable: PageRequest.of(0, 20, Sort.by(DESC, "createdAt").and(Sort.by("name"))).
⚠️
Paging with Joins — Using findAll(Pageable) on entities with EAGER fetch joins causes HHH90003004 warning: Hibernate fetches ALL records in memory then paginates — defeating the purpose! Fix: use @Query with countQuery separately, or use Specification, or fetch associations in a second query after pagination. Never use JOIN FETCH with pagination on the main query.
🛡️
Sort Injection Prevention — When accepting sort fields from user input (?sort=email,asc), validate allowed fields. An attacker could try to sort by a column that reveals sensitive data via timing, or cause a DB error. Maintain a whitelist of sortable fields. Spring Data's Pageable binding helps but always validate sort field names.
Paging & Sorting — Repository + Controller + Response
// ── Repository ────────────────────────────────────
public interface UserRepository extends JpaRepository<User, Long> {

    // Page = data + total count (2 queries)
    Page<User> findByRole(Role role, Pageable pageable);

    // Slice = data + has-next (1 query — for "load more" UI)
    Slice<User> findByActive(boolean active, Pageable pageable);

    // Paged JPQL query
    @Query("SELECT u FROM User u WHERE u.name LIKE %:name%")
    Page<User> searchByName(@Param("name") String name, Pageable p);

    // Native SQL paging requires separate countQuery
    @Query(
        value = "SELECT * FROM users WHERE active = true ORDER BY created_at DESC",
        countQuery = "SELECT COUNT(*) FROM users WHERE active = true",
        nativeQuery = true
    )
    Page<User> findAllActiveNative(Pageable pageable);
}

Must Know
What is the difference between Page and Slice in Spring Data? When would you use each?
▾
Page<T>: runs the data query + a COUNT query. Returns: data, totalElements, totalPages, currentPage. Needed when: showing "Page 3 of 47" to the user, or when the UI needs total count for pagination controls.
Slice<T>: runs only the data query + fetches one extra record to check hasNext. No COUNT. More efficient. Needed when: "Load More" / infinite scroll UIs where total count doesn't matter — just know if there's more data. Significantly faster on large tables where COUNT can be expensive.
⚡ If your UI shows total pages/records → use Page. If it shows "Load More" button → use Slice. Never use Page when you don't show the total.
Tricky
Why does paging fail with JOIN FETCH in JPQL? What is the warning and how do you fix it?
▾
When you use JOIN FETCH in a paged JPQL query, Hibernate can't translate LIMIT/OFFSET to SQL correctly because the JOIN multiplies rows. Hibernate logs: HHH90003004: firstResult/maxResults specified with collection fetch; applying in memory! — meaning it fetches ALL records first (ignoring LIMIT), loads them into memory, THEN paginates in Java. This breaks pagination performance completely. Fix options: 1. Two-query approach: first paginate entity IDs, then fetch full data with JOIN FETCH for those IDs. 2. Use @EntityGraph on the repository method (Hibernate optimizes this). 3. Use DTO projection with a regular JOIN (no FETCH). 4. Accept separate queries (use @BatchSize).
Tricky
How does Spring MVC auto-bind Pageable from request parameters? Can it be abused?
▾
Spring's PageableHandlerMethodArgumentResolver auto-binds: ?page=0&size=20&sort=name,asc. @PageableDefault sets defaults if params absent. Abuse risk: without limits, a user can request ?size=1000000 — returning all records, killing the DB. Fix: @PageableDefault(max=100) sets maximum page size. Also, ?sort=internalColumn can expose DB columns via sort error messages or timing attacks. Fix: validate allowed sort fields with a whitelist in the controller or use @SortDefault. Always set a maximum page size in production.
⚡ Always set max page size. The default Spring max is 2000 — too large for most use cases. Set it to 100 or lower.

How do you implement cursor-based (keyset) pagination and why is it better than offset for large datasets?
▾
Offset pagination (LIMIT 20 OFFSET 10000): DB must scan and discard 10,000 rows to get to page 500. Gets slower as pages increase. Inconsistent if data is inserted/deleted between pages (items skipped or duplicated).
Cursor/keyset pagination: uses a pointer (the last seen ID or timestamp) instead of offset: WHERE id > lastSeenId ORDER BY id LIMIT 20. Always uses the index — O(log n) not O(n). Consistent — new insertions don't shift pages. Limitation: can't jump to arbitrary pages (only next/previous). Spring Data doesn't support it natively — implement with a custom @Query using @Param("lastId"). Ideal for social feeds, infinite scroll, large datasets.
💡 For APIs with >100K records, cursor-based pagination is significantly more efficient than offset-based. Consider it for high-traffic endpoints.
💡
Paging = reading a book, not the whole library. You don't read all 50,000 pages at once — you open at a specific page. Pageable = "give me page 5 with 20 items per page, sorted by date." Page<T> = the page content + "this is page 5 of 250 total pages."

Page vs Slice: Page = "Page 5 of 47" (needs count). Slice = "Load More" (just needs hasNext).

Paging + JOIN FETCH = danger. Hibernate loads ALL in memory. Always separate paging from eager loading.
