Unit & Integration Testing with @SpringBootTest
The testing pyramid — right annotation per layer means fast, focused, trustworthy tests
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: The testing pyramid has 3 layers: Unit tests (bottom, many, fast) test one class in isolation with mocked dependencies. Integration tests (middle) test components working together with a real Spring context. End-to-end tests (top, few, slow) test the whole stack. Spring Boot provides specific annotations for each layer — pick the right one and your tests stay fast and focused.
◆ Test Slice Annotations Cheatsheet
AnnotationLoadsUse ForSpeed
No annotationNothing (pure JUnit)Unit tests — plain Java logic, no Spring⚡ Fastest
@WebMvcTestMVC layer onlyController logic, request mapping, validation🚀 Fast
@WebFluxTestWebFlux layer onlyReactive controllers🚀 Fast
@DataJpaTestJPA + in-memory DBRepository queries, entity mapping⚡ Fast
@JsonTestJackson onlyJSON serialization/deserialization⚡ Fastest
@SpringBootTestFull contextIntegration tests, full-stack tests🐢 Slow
🔬
@SpringBootTest — Loads the complete ApplicationContext (all beans). webEnvironment options: MOCK (no server, mocked servlet — default for MockMvc), RANDOM_PORT (real server on random port — use with TestRestTemplate), DEFINED_PORT (uses configured port), NONE (no web context, for batch/CLI tests). Slow to start but tests real wiring.
🎯
@WebMvcTest(Controller.class) — Loads ONLY the web layer for specified controller(s): controllers, filters, interceptors, @ControllerAdvice, WebMvcConfigurer. Services and repositories are NOT loaded — must be @MockBean'd. Fast. Uses MockMvc for sending requests. Best for: testing request mapping, @Valid validation, response format, exception handling.
🗄️
@DataJpaTest — Loads JPA repositories, entity classes, data source. By default uses an in-memory H2 database and wraps each test in a transaction that auto-rolls back. Services are NOT loaded. Use @AutoConfigureTestDatabase(replace=NONE) to test against your real DB. Use TestEntityManager to set up test data.
📦
@MockBean vs @Mock — @Mock (Mockito): creates a mock outside Spring context. Fast, for unit tests with @ExtendWith(MockitoExtension.class). @MockBean (Spring): creates a mock AND registers it in the ApplicationContext, replacing any existing bean of that type. Use @MockBean in @WebMvcTest/@SpringBootTest. Adding @MockBean breaks context caching — use sparingly.
⚡
Context Caching — Spring caches the ApplicationContext between test classes with the same configuration. Loading a context takes 2–20 seconds — without caching, large suites would be impossibly slow. Tests with identical configuration share one context. Avoid @DirtiesContext (forces context reload per test class — very slow). Design stateless tests with @BeforeEach resets instead.
🕵️
@SpyBean — Wraps the real Spring bean as a Mockito spy — calls the real implementation by default, allows stubbing specific methods. Use when you want to verify that a real bean method was called, or override just one method while keeping the rest real. Contrast with @MockBean (replaces entirely) and @Spy (Mockito only, not in Spring context).

@WebMvcTest + @DataJpaTest + @SpringBootTest
// ── @WebMvcTest — controller layer only ──────────
@WebMvcTest(UserController.class)
class UserControllerTest {

    @Autowired private MockMvc mockMvc;
    @MockBean  private UserService userService;   // mock the service
    @Autowired private ObjectMapper mapper;

    @Test
    void getUser_found_returns200() throws Exception {
        UserDto dto = new UserDto(1L, "Alice", "alice@x.com");
        Mockito.when(userService.findById(1L)).thenReturn(Optional.of(dto));

        mockMvc.perform(get("/api/users/1")
                .accept(MediaType.APPLICATION_JSON))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.name").value("Alice"))
            .andExpect(jsonPath("$.email").value("alice@x.com"));
    }

    @Test
    void getUser_notFound_returns404() throws Exception {
        Mockito.when(userService.findById(99L)).thenReturn(Optional.empty());
        mockMvc.perform(get("/api/users/99"))
            .andExpect(status().isNotFound());
    }
}

◆ Interview Questions
Must Know
What is the difference between @SpringBootTest and @WebMvcTest?
▾
@SpringBootTest: loads the FULL application context — all beans (controllers, services, repos, security config, etc.). Slow startup (2–20s) but tests real integration. Use for: integration tests where multiple layers must work together, end-to-end HTTP tests with TestRestTemplate, testing cross-cutting concerns like security and transactions.

@WebMvcTest: loads ONLY the web layer for a specific controller. No services, no repos. Faster (< 2s). All dependencies must be @MockBean'd. Use for: testing controller request mapping, @Valid validation, @ControllerAdvice error handling, response format — in isolation.
⚡ Prefer @WebMvcTest for controller tests — 10–50× faster than @SpringBootTest for the same coverage.
Must Know
What is Spring's test context caching and why does it matter for test speed?
▾
Spring caches the ApplicationContext keyed by its configuration (which @Configuration classes, profiles, properties, @MockBean definitions). Tests with identical config share one cached context — loading it once, reusing for all matching test classes.

Without caching: 200 test classes × 5-second startup = 1000 seconds just in startup. With caching: often 1–5 contexts total regardless of test count.

What breaks caching: @DirtiesContext (marks as dirty, forces reload), different @MockBean combinations per test class, different profiles, different properties. Solution: consolidate @MockBean usage, use shared base test classes, avoid @DirtiesContext.
💡 Profile your test suite: if startup times dominate, you likely have too many unique ApplicationContext instances being created.
Tricky
What is the difference between @Mock, @MockBean, @Spy, and @SpyBean?
▾
@Mock (Mockito): creates a Mockito mock, no Spring context. Use in unit tests with @ExtendWith(MockitoExtension.class). All methods return null/empty by default.

@MockBean (Spring): creates a Mockito mock AND replaces the bean of that type in the Spring ApplicationContext. Use in @WebMvcTest/@SpringBootTest when Spring context is active.

@Spy (Mockito): wraps a real object. Calls real methods by default. Override specific methods. No Spring context.

@SpyBean (Spring): wraps the real Spring bean as a spy. Real implementation runs unless stubbed. Use to verify real bean method calls without replacing the bean entirely.
Tricky
How do you test with a real PostgreSQL database instead of H2?
▾
Use Testcontainers: @DataJpaTest + @AutoConfigureTestDatabase(replace=NONE) + @Testcontainers. Declare a static @Container PostgreSQLContainer<?> — Testcontainers starts a real PostgreSQL Docker container. Use @DynamicPropertySource to inject the container's JDBC URL, username, and password into Spring's datasource config at runtime.

Benefits: tests against the real DB engine (catches H2 vs PostgreSQL SQL differences), no manual DB setup, container spins up/down automatically, shareable across test classes (static container).
⚡ H2 silently accepts some invalid SQL that PostgreSQL rejects. Use Testcontainers for anything with complex queries or DB-specific features.
Senior
How do you test a Spring Security-secured endpoint?
▾
Three approaches:

1. @WithMockUser (Spring Security Test): inject a fake authenticated user into the SecurityContext for the test. @WithMockUser(roles="ADMIN"). Fast, no real auth. Use to test that security rules are correctly configured (right role = 200, wrong role = 403).

2. @WithUserDetails(username): loads a real UserDetails from your UserDetailsService mock. Tests that your role mapping from UserDetails to granted authorities works correctly.

3. Full auth flow with TestRestTemplate: in @SpringBootTest, call the real login endpoint to get a JWT token, then include it in subsequent request headers. Tests the complete authentication pipeline end-to-end.
◆ Memory Tip
💡
Testing pyramid = right annotation per layer:
• Service logic → no annotation + @ExtendWith(MockitoExtension.class)
• Controller behavior → @WebMvcTest + @MockBean services
• Repository queries → @DataJpaTest (H2 or Testcontainers)
• Full integration → @SpringBootTest(RANDOM_PORT) + TestRestTemplate

Context caching mantra: "Same config = shared context." @MockBean differences create different contexts — consolidate them to keep your test suite fast.

@MockBean = replaces bean. @SpyBean = wraps real bean.

Mocking with Mockito
when() · verify() · ArgumentCaptor · Spy — isolating the class under test
🔴 Must Know
📌 Concepts
💻 Code
❓ Interview Q&A
🧠 Memory
🧠
In plain English: A mock is a fake replacement for a dependency. When testing OrderService you don't want to actually hit a real database or charge a real payment — you tell the mock: "when called with X, return Y." This lets you test just your class's logic in isolation, fast and without side effects.
🎭
Mock vs Stub vs Spy — Mock: fake object, all methods return empty/null by default, records all calls for verification. Stub: pre-programmed with when().thenReturn() to return specific values. Mockito mocks are both mock + stub. Spy: wraps a real object — real methods run by default, but you can stub specific ones and verify calls.
📋
Stubbing with when() — when(mock.method(arg)).thenReturn(val): return a value. thenThrow(ex): throw exception. thenAnswer(inv → ...): compute dynamically. thenReturn(first).thenReturn(second): different values on consecutive calls. For void methods or spies: doReturn(val).when(spy).method() or doThrow(ex).when(mock).voidMethod().
✅
Verification with verify() — verify(mock).method(arg): called exactly once. verify(mock, times(3)).method(): exactly 3 times. verify(mock, never()).method(): never called. verify(mock, atLeast(2)).method(). Use InOrder for ordered verification. Don't over-verify — only check meaningful interactions, not every call.
🎣
ArgumentCaptor — Captures the actual argument passed to a mock method for assertion afterward. Use when: the argument is constructed internally (you can't pass it in yourself). Pattern: ArgumentCaptor<Order> captor = ArgumentCaptor.forClass(Order.class) → verify(repo).save(captor.capture()) → assertThat(captor.getValue().getStatus()).isEqualTo(PENDING).
🎯
Argument Matchers — any(), any(User.class), anyString(), anyLong(), eq("exact"), argThat(pred). Rule: if you use any matcher, ALL arguments in the same method call must use matchers — can't mix matchers and exact values. Use eq() to specify exact values alongside matchers.
⚠️
Don't Mock What You Don't Own — Avoid mocking third-party classes (JPA, HttpClient) — you likely misunderstand their contract. Wrap third-party code in your own interface and mock THAT. Also avoid mocking: value objects (use real instances), simple utilities. Over-mocking = brittle tests that break on refactoring, not behavior changes.
◆ Memorizable Code
Mockito — when() + verify() + ArgumentCaptor + Spy
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock  private OrderRepository orderRepo;
    @Mock  private PaymentService  paymentSvc;
    @InjectMocks private OrderService orderService;

    @Test
    void placeOrder_success() {
        // Arrange
        Order saved = new Order(1L, BigDecimal.TEN);
        Mockito.when(orderRepo.save(any(Order.class))).thenReturn(saved);
        Mockito.when(paymentSvc.charge(eq(1L), any())).thenReturn(true);

        // Act
        Order result = orderService.placeOrder(new OrderRequest(1L, BigDecimal.TEN));

        // Assert + Verify
        assertThat(result.getId()).isEqualTo(1L);
        Mockito.verify(orderRepo, times(1)).save(any());
        Mockito.verify(paymentSvc).charge(eq(1L), any());
    }

    @Test
    void placeOrder_paymentFails_throws() {
        Mockito.when(paymentSvc.charge(anyLong(), any()))
               .thenThrow(new PaymentException("Declined"));

        assertThatThrownBy(() -> orderService.placeOrder(req))
            .isInstanceOf(PaymentException.class)
            .hasMessageContaining("Declined");

        Mockito.verify(orderRepo, never()).save(any()); // must NOT save
    }

    @Test
    void placeOrder_capturesSavedEntity() {
        // ArgumentCaptor: capture what was passed to mock
        ArgumentCaptor<Order> captor = ArgumentCaptor.forClass(Order.class);

        orderService.placeOrder(new OrderRequest(1L, BigDecimal.TEN));

        Mockito.verify(orderRepo).save(captor.capture());
        Order captured = captor.getValue();
        assertThat(captured.getStatus()).isEqualTo(OrderStatus.PENDING);
        assertThat(captured.getAmount()).isEqualByComparingTo(BigDecimal.TEN);
    }

    // ── Spy: real object, override one method ────────
    @Spy
    private PriceCalculator calc = new PriceCalculator();

    @Test
    void spy_overridesOneMethod() {
        // Override only formatPrice — real calcTotal still runs
        Mockito.doReturn("£99").when(calc).formatPrice(any());

        BigDecimal total = calc.calcTotal(items); // real method!
        String fmt   = calc.formatPrice(total);  // stubbed
        assertThat(fmt).isEqualTo("£99");
    }

    // ── Consecutive return values ─────────────────────
    Mockito.when(service.findStatus(anyLong()))
           .thenReturn("PROCESSING")  // 1st call
           .thenReturn("PROCESSING")  // 2nd call
           .thenReturn("COMPLETED");  // 3rd+ calls
}
◆ Interview Questions
Must Know
What is the difference between @InjectMocks and @Autowired?
▾
@InjectMocks (Mockito): creates an instance of the class under test and injects all @Mock/@Spy fields from the test class using constructor, setter, or field injection. NO Spring context needed — pure Mockito. Fast. The class under test is a real instance, not a mock.

@Autowired (Spring): injects the real Spring bean from the ApplicationContext. Requires Spring context (i.e., inside @SpringBootTest, @WebMvcTest etc.). Slow (context startup).

Rule: unit tests → @InjectMocks + @Mock. Integration tests with Spring → @Autowired.
⚡ @InjectMocks does NOT understand @Qualifier — multiple deps of the same type may inject wrong bean silently. Prefer constructor injection in production code to avoid this.
Tricky
When should you use ArgumentCaptor vs argThat() matchers?
▾
ArgumentCaptor: capture the argument AFTER the fact, then assert with AssertJ/JUnit assertions. Better for: complex assertions, readable error messages when assertion fails, when you need to inspect multiple properties of the captured object. verify(repo).save(captor.capture()); assertThat(captor.getValue().getStatus()).isEqualTo(PENDING).

argThat(predicate): inline assertion during verify. Better for: simple single-property checks. verify(repo).save(argThat(o → o.getStatus() == PENDING)). Error messages are less readable on failure. Avoid for complex assertions — ArgumentCaptor gives clearer failure output.
Tricky
Why should you avoid over-mocking? What is the "don't mock what you don't own" rule?
▾
Over-mocking makes tests brittle — they break when implementation changes, not when behavior changes. If you mock internal collaborators, you're testing implementation details, not contracts.

"Don't mock what you don't own" (Freeman & Pryce): avoid mocking third-party classes (JPA EntityManager, HttpClient, etc.) because you likely misunderstand their behaviour and could mock them incorrectly. Instead, wrap third-party code in your own interface (Repository, HttpGateway) and mock THAT interface — which you own and understand.

Mock only the boundary of your system — repository interfaces, external HTTP clients, email senders — not internal helpers.
Senior
What are the best practices for structuring unit tests with Mockito?
▾
Follow the AAA pattern (Arrange–Act–Assert): setup stubs in Arrange, call the real method in Act, assert results AND verify important interactions in Assert.

Name tests descriptively: methodName_condition_expectedBehavior.

One test, one concept — don't test multiple scenarios in one test method.

Verify only meaningful interactions — don't verify every mock call. If a class uses 5 mocks but the test only cares about 2, only verify those 2.

Stub only what's needed — don't set up stubs that aren't called. Unused stubs are noise.

Use @ExtendWith(MockitoExtension.class) over MockitoAnnotations.openMocks(this) — cleaner and avoids manual teardown.
💡
Mockito workflow: Arrange → Act → Assert + Verify

when(mock.method(arg)).thenReturn(val) — stub (Arrange)
Call the real method (Act)
assertThat(result)... — assert output (Assert)
verify(mock).method(arg) — verify key interactions (Assert)

When to use what:
• Control return value → thenReturn()
• Test exception handling → thenThrow()
• Capture what was passed → ArgumentCaptor
• Partial real behavior → @Spy + doReturn()

🧠
In plain English: MockMvc = pretend HTTP through the DispatcherServlet — fast, no real network socket. TestRestTemplate = real HTTP to a real running server — slower but more realistic. MockMvc is for most controller tests; TestRestTemplate for the most critical full-stack paths.

🎭
MockMvc — Calls your controller through the DispatcherServlet without a real TCP connection. Tests: request mapping, @Valid validation, @ControllerAdvice error handling, content negotiation, response body and headers. Auto-configured by @WebMvcTest. Rich assertion API: status(), jsonPath(), header(), content().
🌐
TestRestTemplate — Auto-configured by @SpringBootTest(RANDOM_PORT). Makes real HTTP calls. Handles redirects automatically. Returns ResponseEntity with error status instead of throwing exceptions (unlike RestTemplate). Use with @LocalServerPort to get the actual port. Good for: full auth flows, cookie testing, security integration testing.
📍
jsonPath() assertions — Powered by Jayway JsonPath. jsonPath("$.name").value("Alice"): simple field. jsonPath("$.items").isArray(): is array. jsonPath("$.items.length()").value(3): array length. jsonPath("$.address.city").value("NYC"): nested. jsonPath("$.errors.email").exists(): field exists.
🔐
Testing Secured Endpoints — @WithMockUser(roles="ADMIN"): inject fake auth for the test. @WithMockUser(authorities="orders:read"): fine-grained authorities. No actual authentication happens — just sets up SecurityContext. For full auth flow testing: use @SpringBootTest + TestRestTemplate and call the real login endpoint.

// ── MockMvc comprehensive test ────────────────────
@WebMvcTest(OrderController.class)
class OrderControllerTest {
    @Autowired private MockMvc mockMvc;
    @MockBean  private OrderService orderSvc;
    @Autowired private ObjectMapper mapper;

    @Test
    void createOrder_valid_returns201() throws Exception {
        OrderDto created = new OrderDto(1L, BigDecimal.TEN);
        when(orderSvc.create(any())).thenReturn(created);

        mockMvc.perform(
                post("/api/orders")
                .contentType(MediaType.APPLICATION_JSON)
                .content(mapper.writeValueAsString(
                    new CreateOrderReq(1L, BigDecimal.TEN)))
            )
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.id").value(1))
            .andExpect(jsonPath("$.amount").value(10))
            .andDo(print());           // print request+response for debug
    }

    // Validation failure test
    @Test
    void createOrder_negativeAmount_returns400() throws Exception {
        mockMvc.perform(post("/api/orders")
                .contentType(MediaType.APPLICATION_JSON)
                .content("""{"customerId":1,"amount":-5}"""))
            .andExpect(status().isBadRequest())
            .andExpect(jsonPath("$.errors.amount").exists());
    }

    // ── Security testing with @WithMockUser ───────────
    @Test
    @WithMockUser(roles = "ADMIN")
    void delete_asAdmin_returns204() throws Exception {
        doNothing().when(orderSvc).delete(1L);
        mockMvc.perform(delete("/api/orders/1"))
            .andExpect(status().isNoContent());
    }

    @Test
    void delete_noAuth_returns403() throws Exception {
        mockMvc.perform(delete("/api/orders/1"))  // no @WithMockUser
            .andExpect(status().isForbidden());
    }
}

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderIntegrationTest {
    @Autowired private TestRestTemplate restTemplate;

    @Test
    void createAndRetrieveOrder() {
        ResponseEntity<OrderDto> created = restTemplate
            .withBasicAuth("user", "password")
            .postForEntity("/api/orders",

            .getForEntity("/api/orders/{id}", OrderDto.class,

◆ Interview Questions
Must Know
When would you use MockMvc vs TestRestTemplate?
▾
MockMvc: testing controller behavior in isolation. Request mapping, @Valid validation, response format, @ControllerAdvice error handling, content negotiation. No real HTTP server — fast. Rich fluent assertion API. Use for the majority of controller tests.

TestRestTemplate: full integration tests where the complete stack matters — real security filter chain, real database, real HTTP request/response cycle, cookie/redirect handling. Use with @SpringBootTest(RANDOM_PORT). Slower but representative of production. Use for critical end-to-end paths only — not every endpoint.
⚡ Strategy: 80% @WebMvcTest with MockMvc (fast, focused). 20% @SpringBootTest with TestRestTemplate (slow, comprehensive) for critical paths.
Tricky
What are the limitations of @WithMockUser?
▾
@WithMockUser injects a pre-built UsernamePasswordAuthenticationToken into the SecurityContext — no actual authentication happens. This means it bypasses: JWT parsing, UserDetailsService loading, password verification, and any custom AuthenticationProvider logic.

It only tests that your authorization rules are correctly configured (does ROLE_ADMIN grant access to /admin/**?). It does NOT test authentication.

To test authentication itself: use @SpringBootTest + TestRestTemplate and actually call the login endpoint, get a real JWT, and use it in subsequent requests. Or use @WithUserDetails to test that your UserDetailsService correctly loads roles for a given username.
Tricky
How do you test a controller endpoint that accepts Pageable?
▾
Pass pagination parameters as request params in MockMvc: mockMvc.perform(get("/api/users").param("page","0").param("size","10").param("sort","name,asc")). Spring's PageableHandlerMethodArgumentResolver auto-binds these to a Pageable object.

For the service stub, use any(Pageable.class) matcher. Return a real PageImpl<> object: when(service.findAll(any(Pageable.class))).thenReturn(new PageImpl<>(users, pageable, 100)).

Assert on the JSON response: jsonPath("$.content.length()").value(users.size()), jsonPath("$.totalElements").value(100), jsonPath("$.totalPages").value(5).
💡 Use ArgumentCaptor<Pageable> to verify the controller passed the correct page/size/sort to the service.
Senior
How do you test @ControllerAdvice exception handling with MockMvc?
▾
@WebMvcTest automatically picks up @ControllerAdvice beans in the same package. Trigger the exception by stubbing the service to throw it: when(service.findById(99L)).thenThrow(new EntityNotFoundException("User 99 not found")). Then send the request and assert on the error response format.

Example: mockMvc.perform(get("/api/users/99")).andExpect(status().isNotFound()).andExpect(jsonPath("$.status").value(404)).andExpect(jsonPath("$.message").value("User 99 not found")).

Test all error paths: 404 not found, 400 validation error, 403 forbidden, 500 unexpected error. This ensures your error response format is consistent across all endpoints.
💡
MockMvc = fast, isolated. TestRestTemplate = slow, realistic.
Build most tests with MockMvc. Use TestRestTemplate only for the most critical integration paths.

MockMvc cheat sheet:
perform(get("/url").param("k","v")) → make request
.andExpect(status().isOk()) → check status
.andExpect(jsonPath("$.name").value("x")) → check body field
.andExpect(header().string("Location","...")) → check header
.andDo(print()) → print for debugging

@WithMockUser tests authorization rules, NOT authentication. Use full auth flow with TestRestTemplate to test auth itself.

Event Handling — ApplicationEvent & @EventListener
Decouple services with the Observer pattern — publish once, react anywhere, no direct dependency
🟡 Medium

🧠 Memory
🧠
In plain English: Instead of OrderService calling EmailService, InventoryService, and AuditService directly (tight coupling), OrderService just shouts "Order was placed!" — and any service that cares can listen and react. Adding a new reaction never touches OrderService. This is the Observer pattern built into Spring.

📡
Spring Event System — Any Spring bean can publish events via ApplicationEventPublisher.publishEvent(event). Any @Component method annotated with @EventListener can listen. Events are synchronous and same-thread by default — listeners run in the publisher's thread, within the same transaction.
📦
Event Classes — Extend ApplicationEvent (traditional) or use any POJO (Spring 4.2+). POJO events are preferred — no Spring coupling in your domain model. Use record for immutable events: public record OrderPlacedEvent(Long orderId, String email, BigDecimal amount) {}. Events should be immutable — they describe something that already happened.
💾
@TransactionalEventListener — Runs listener at a specific point in the publisher's transaction. AFTER_COMMIT (default): runs only after the transaction successfully commits. Critical for irreversible actions: sending an email after order is committed to DB guarantees the order exists when the email is sent. Other phases: BEFORE_COMMIT, AFTER_ROLLBACK, AFTER_COMPLETION.
🔄
Async Listeners with @Async — Combine @TransactionalEventListener + @Async to run listeners asynchronously in a thread pool after the transaction commits. Requires @EnableAsync. Prevents slow listeners (email sending, external API calls) from blocking the publisher's response. With @Async, the listener runs OUTSIDE the original transaction.
🏗️
Built-in Spring Events — ContextRefreshedEvent (context fully ready — all beans initialized), ApplicationReadyEvent (Spring Boot — app ready to serve, use for post-startup initialization), ContextClosedEvent (before shutdown), ApplicationFailedEvent (startup failed). Listen with @EventListener(ApplicationReadyEvent.class).
🎯
Event Filtering with Conditions — @EventListener(condition = "#event.amount.compareTo(@bigOrderThreshold) > 0"): SpEL condition, listener only runs when true. Order multiple listeners with @Order(1): lower value = higher priority. Listen for multiple event types: @EventListener({OrderPlacedEvent.class, OrderUpdatedEvent.class}).
◆ Memorizable Code
ApplicationEvent + @EventListener + @TransactionalEventListener + @Async
// ── POJO Event (no Spring coupling) ──────────────
public record OrderPlacedEvent(
    Long       orderId,
    String     customerEmail,
    BigDecimal amount,
    Instant    placedAt
) {}

// ── Publisher ─────────────────────────────────────
@Service
public class OrderService {
    private final ApplicationEventPublisher publisher;
    private final OrderRepository repo;

    @Transactional
    public Order placeOrder(OrderRequest req) {
        Order order = repo.save(Order.from(req));

        publisher.publishEvent(new OrderPlacedEvent(
            order.getId(), req.getEmail(),
            req.getAmount(), Instant.now()
        ));
        return order;
        // OrderService has ZERO knowledge of email/inventory/audit!
    }
}

// ── Synchronous listener (in publisher's TX) ──────
@Component
public class InventoryListener {
    @EventListener
    public void onOrderPlaced(OrderPlacedEvent event) {
        inventoryService.deduct(event.orderId()); // same TX as publisher
    }
}

// ── @TransactionalEventListener — AFTER_COMMIT ────
@Component
public class EmailNotificationListener {

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Async      // async: don't block publisher thread
    public void sendConfirmation(OrderPlacedEvent event) {
        // Runs ONLY after order is committed to DB
        // Guaranteed: order exists in DB when this runs
        emailService.sendOrderConfirmation(
            event.customerEmail(), event.orderId()
        );
    }
}

// ── Conditional listener ───────────────────────────
@Component
public class BigOrderAlertListener {
    @EventListener(condition = "#event.amount.compareTo(@threshold) > 0")
    public void alertOnBigOrder(OrderPlacedEvent event) {
        alertService.notifyManager(event); // only for large orders
    }
}

// ── Listen to Spring Boot lifecycle ───────────────
@Component
public class StartupInitializer {
    @EventListener(ApplicationReadyEvent.class)
    public void onReady() {
        log.info("App ready! Warming up cache...");
        cacheService.warmup();  // all beans ready, safe to call
    }
}

// ── Enable @Async ─────────────────────────────────
@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean
    public TaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor ex = new ThreadPoolTaskExecutor();
        ex.setCorePoolSize(5); ex.setMaxPoolSize(20);
        ex.setQueueCapacity(100);
        return ex;
    }
}
◆ Interview Questions
Must Know
What problem does Spring's event system solve? Why not call services directly?
▾
Direct service calls create tight coupling: OrderService must depend on EmailService, InventoryService, AuditService. Adding a new action on order placement requires modifying OrderService — violating the Open/Closed Principle.

With events: OrderService publishes an OrderPlacedEvent and knows nothing about who listens. New reactions are added by creating new listeners — OrderService unchanged. This is the Observer pattern built into Spring.

Benefits: loose coupling, easier unit testing of OrderService (no need to mock EmailService etc.), separate deployment of concerns, easy to add/remove side effects without risk to core business logic.
⚡ Events are best for: cross-cutting concerns (email, audit, analytics) where the publisher shouldn't care about the reaction.
Must Know
What is @TransactionalEventListener and why is it critical for correctness?
▾
Regular @EventListener runs synchronously within the publisher's transaction. If the listener sends an email, and the publisher's transaction later rolls back — you've sent an email for data that was rolled back.

@TransactionalEventListener(phase = AFTER_COMMIT) delays execution until AFTER the transaction successfully commits. The listener ONLY runs if the transaction committed successfully — no email for rolled-back orders.

Other phases: BEFORE_COMMIT (last chance to affect the transaction), AFTER_ROLLBACK (cleanup after failure), AFTER_COMPLETION (always, like finally).
⚠️ Never send emails, charge payments, or call external APIs in a plain @EventListener. Always use @TransactionalEventListener(AFTER_COMMIT) for irreversible side effects.
Tricky
How do you test that an event was published?
▾
Three approaches:

1. Mock ApplicationEventPublisher: @MockBean ApplicationEventPublisher publisher in @WebMvcTest/@SpringBootTest. Then verify: ArgumentCaptor<OrderPlacedEvent> captor = ArgumentCaptor.forClass(OrderPlacedEvent.class); verify(publisher).publishEvent(captor.capture()); assertThat(captor.getValue().orderId()).isEqualTo(1L). Simplest for unit/slice tests.

2. @RecordApplicationEvents (Spring Test 5.3+): @SpringBootTest + @RecordApplicationEvents, inject ApplicationEvents events. After action: assertThat(events.stream(OrderPlacedEvent.class).count()).isEqualTo(1).

3. @SpyBean on listener: verify the listener's method was called after the event was published.
Tricky
What are the limitations of Spring's in-process event system?
▾
Spring ApplicationEvents are in-process only. Key limitations:

1. No durability: if the app crashes before an async listener executes, the event is lost — no retry, no dead-letter queue.
2. Single JVM: events don't cross service boundaries.
3. No ordering guarantee for async listeners.
4. No built-in replay capability.

For distributed systems: use Kafka or RabbitMQ which provide: persistence (events survive crashes), delivery guarantees (at-least-once/exactly-once), replay, cross-service communication.

Use Spring Events for: simple in-process decoupling, lightweight synchronous notifications within one service. Use Kafka/RabbitMQ for: critical events, cross-service, must-not-lose semantics.
Senior
What is the Transactional Outbox Pattern and how does it relate to Spring Events?
▾
The Outbox Pattern solves the dual-write problem: you want to save data to DB AND publish a message to Kafka in the same "unit of work" — but they're different systems, so you can't do both in one transaction.

Solution: write the event to an outbox table in the same DB transaction as the business data. A separate process (like Debezium CDC) reads from the outbox table and publishes to Kafka/RabbitMQ. The outbox row is only created if the main transaction commits — guaranteed consistency.

How it relates: @TransactionalEventListener(AFTER_COMMIT) is good for in-process side effects. But for cross-service event publishing to Kafka, the Outbox Pattern is the robust solution — especially when network partitions or Kafka downtime could cause you to miss publishing an event.
💡
Events = "I did something" announcement. Listeners = "I care about that" reaction.
Publisher never knows the listeners. Adding a new reaction never touches the publisher. Open/Closed Principle in action.

Critical rules:
• Irreversible actions (email, payment) → @TransactionalEventListener(AFTER_COMMIT)
• Slow listeners → add @Async so they don't block the publisher
• POJO records > ApplicationEvent subclasses (no Spring coupling)
• Spring Events = single JVM only → use Kafka for cross-service

AFTER_COMMIT mnemonic: "Don't email about the pizza until it's actually baked."
